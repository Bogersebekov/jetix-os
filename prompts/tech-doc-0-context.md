## 4. Ключевые принципы — должны быть отражены в тех.документе

### 4.1 Beta-first, не перфекционизм
- Цель — **работающий тех.документ v1-beta**, не идеальный
- Первичная цель Ruslan: $50K до 30.06.2026. Документ — инструмент, не самоцель.
- Что НЕ делать: описывать все возможные edge cases, формальную математику, полный ISO 25010 mapping

### 4.2 Полуручной режим (Часть 5 SYSTEM-DESIGN-HUMAN)
- Система **ничего не делает сама** на v1-beta
- Всё по команде Ruslan'а
- Автоматизация — после обкатки
- Тех.документ должен это поддерживать (не закладывать обязательные cron'ы / event-driven)

### 4.3 Kay принцип — инфраструктура важнее инструментов
- Методология (папки / wiki / роли) должна работать **и без AI**
- Если AI-провайдер отключится — можно посадить человека-оператора и система продолжит работать
- Vendor lock-in — **невозможен архитектурно**

### 4.4 GitHub-style work (Часть 6 SYSTEM-DESIGN-HUMAN)
- Проекты = main (чистая ключевая информация)
- Daily drafts = песочница (бунтовать, изобретать)
- Не выходим из песочницы для решения задач

### 4.5 Чистота — главный критерий (Часть 4 SYSTEM-DESIGN-HUMAN §4.0)
- Код системы жизни чистый как на GitHub
- Каждая информация имеет заранее заготовленное место
- Никаких разрастаний, никакого грязного кода

### 4.6 Границы — что система НЕ делает (Часть 2.4 + 5.7)
- Не принимает стратегические решения без Ruslan'а
- Не публикует контент без одобрения
- Не совершает финансовые операции
- Не удаляет данные
- Не заменяет человеческие отношения
- Не выходит в WebFetch/WebSearch автономно

## 7. Методология — параллельная мульти-перспективная ревизия

Чаты 1, 2, 3, 4 работают **параллельно и независимо**:
- Каждый видит общий контекст (этот файл) + v1-beta SYSTEM-DESIGN-HUMAN
- Каждый выдаёт свой markdown-отчёт в `reviews/tech-doc-{N}-*.md`
- **Не общаются друг с другом** — чистота независимых взглядов

Чат 5 (Синтезатор) работает **после**:
- Читает 4 отчёта
- Собирает финальный тех.документ (SYSTEM-DESIGN-TECH + AGENT-PROTOCOLS + DATA-FLOWS + ARCHITECTURE-TARGET)
- Ruslan ревьюит и утверждает

**Почему 5 чатов, а не 1:** один Claude может попасть в свой угол зрения и пропустить уязвимости. 5 чатов дают кросс-верификацию.
