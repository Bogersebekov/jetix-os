## Контекст и принципы работы над документом

### Зачем этот документ
Фундамент второй фазы строительства Jetix OS. Без техники, без кода, без схем
на архитектурном языке. **Только на человеческом:** что система должна делать,
для кого, что выдавать на выходе, какие потоки информации, какие действия, какие
состояния, какие границы. Из этого документа вырастают AGENT-PROTOCOLS.md,
SYSTEM-DESIGN-TECH.md, DATA-FLOWS.md, ARCHITECTURE-TARGET.md (Шаг 4).

### Принципы (из манифеста 2026-04-16)
1. **Не упрощаем.** 12 агентов остаются 12 агентами. Если часть — декорация, оживляем.
2. **Фундамент сначала.** До клиентов, до видео-pipeline'ов, до бизнеса.
3. **Два языка.** Сначала человеческий (этот документ). Потом архитектурный (Шаг 4).
4. **Уровень детализации — инженеры ракет.** Прочитал — понятно что и почему, без догадок.
5. **Утверждаем на каждом уровне.** Этот документ полностью завершается и фиксируется до Шага 4.
6. **Система жизни = код.** Чистота, модульность, версионирование, тесты, документация.
7. **Заметки и стратегия — после.** Сначала каркас, потом заливка смыслом, потом стратегия.

### Что в этот документ НЕ пишем
- Код, команды, пути файлов, API.
- Схемы на архитектурном языке (компоненты, шины, state machines формально).
- Технические решения ("использовать JSONL + inotifywait"). Только "система делает X при Y".
- Инструкции агентам на техническом уровне (это Шаг 4: AGENT-PROTOCOLS.md).

### Зависимости
- **Вход:** `ARCHITECTURE-CURRENT.md` (Шаг 1, 337 строк), inventory-отчёт,
  Miro фреймы L1/L2/L3, Notion-страница архитектуры памяти+KB (Karpathy+),
  Манифест, CLAUDE.md, все идеи wiki/ + Notion-дампы от Фазы α.
- **Выход:** `SYSTEM-DESIGN-HUMAN.v1.md` (Шаг 3) + Notion-чистовик +
  N визуализаций в Miro.

## Точка Б (результат к концу Шага 2)

Полноценно описанная целевая система Jetix OS со всех сторон:

### 1. Текстом на человеческом языке — документ с 7 частями + мета-секцией

| # | Часть | Что внутри |
|---|-------|------------|
| 0 | Мета | Как пользоваться этим документом (выше) |
| 1 | Видение / Стратегия | WHY — зачем система, горизонт |
| 2 | Цели и результаты | WHAT — что выдаёт, метрики, границы (**NEW**) |
| 3 | Пользователи / Роли | WHO — кто работает в системе + human-in-loop |
| 4 | Потоки информации | HOW (data) — где течёт, где оседает |
| 5 | Действия / Триггеры | HOW (behavior) — что делает + failure modes |
| 6 | Состояния / Жизненный цикл | HOW (evolve) — как элементы живут + self-improvement |
| 7 | Открытые вопросы | TBD — нерешённое, сквозное |

### 2. Все релевантные идеи / видения / заметки подтянуты
Каждый нетривиальный тезис опирается на конкретный источник: ссылка на
`wiki/ideas/*`, на Notion-идею, на voice-memo, на knowledge-base/ legacy.
Источники привязаны через `[[...]]` wikilinks или прямую цитату.

### 3. В Miro — несколько визуализаций
Которые позволяют систему описать, пояснить её устройство и в чём её преимущество
перед обычной системой задач/проектов. Минимум:
- **Общая схема системы** (видение единым взглядом: пользователь → агенты →
  знания → действия → результат)
- **Диаграмма результатов** (что на входе / что на выходе / какие метрики) — **NEW**
- **Диаграмма потоков информации** (откуда приходит, где оседает, как возвращается)
- **Диаграмма жизненного цикла сущностей** (идея → claim → experiment → summary)
- **Диаграмма преимущества** — ценностное предложение системы
- (опц.) **Карта ролей** — кто из 12 агентов за что отвечает в человеческих понятиях

### 4. Документ — фундамент для всего дальнейшего
Из него растут:
- Шаг 3: фиксация как `SYSTEM-DESIGN-HUMAN.v1.md` + Notion-чистовик
- Шаг 4: перевод на архитектурный язык (`AGENT-PROTOCOLS.md`,
  `SYSTEM-DESIGN-TECH.md`, `DATA-FLOWS.md`, `ARCHITECTURE-TARGET.md`,
  обновлённые `agents/{id}/system.md`)
- Шаги 5-7: внедрение, тестирование, обработка оставшихся заметок
- Шаг 8-10: декомиссия Notion, legacy миграция, `STRATEGY-2026-04-16-to-2026-06-30.md`

## 7 частей документа — пустые seed'ы

> Заполняются по порядку выше. Каждая часть может содержать:
> заготовки тезисов, цитаты Ruslan, ссылки на идеи из `wiki/`, открытые вопросы,
> варианты формулировок (отмечать `[вариант А] / [вариант Б]` если не решили).

### Часть 1. Видение / Стратегия

#### 1.1 Что такое Jetix OS (ядро в одной фразе)

**Jetix OS — это инструмент для трекинга и управления своими ресурсами,
который позволяет управленцу отделять важное от неважного, направлять
ресурсы на важные проекты и вопросы, а неважное делегировать агентам
и автоматизации.**

За счёт этого система позволяет **эффективно и удобно передвигаться из Точки А
в Точку Б** — достигать целей, будь то стратегических или тактических.

Такой же по сути инструмент как **велосипед, машина, мотоцикл** — но для
информации, времени, связей, инструментов и внешней среды. Сам по себе мёртвый.
Работает только через пользователя с целями.

> "Система сама по себе жить не может. Это один из инструментов для
> пользователей этой системы. Точно так же как велосипед создан для человека
> который на нём едет — для медведя будет мало толку от велосипеда."
> — Ruslan, диктовка 2026-04-17

> "Jetix OS — это инструмент для трекинга и управления ресурсами, который
> позволяет управленцу отделять важное от неважного, направлять ресурсы на
> важное, а неважное — делегировать. За счёт этого она эффективно и удобно
> двигает из Точки А в Точку Б."
> — Ruslan, диктовка 2026-04-17

#### 1.2 Шесть типов ресурсов которые трекает и помогает использовать

1. **Информация** — что потребляем, что производим (знания, заметки, инсайты) — [[ideas/information-discipline-as-core-skill]]
2. **Время** — личное + команды (если есть) — [[ideas/day-influence-scale-0-100]]
3. **Финансы** — свои + управляемые средства
4. **Связи / нетворкинг** — кто есть в окружении, какие соединения создают
   возможности. Пример: 5 друзей, которые сами бы не связались, соединяются
   через систему → новый проект → revenue — [[ideas/truth-clan-acceleration]]
5. **Внешние обстоятельства** — окружение превращённое в ресурс через анализ.
   Не просто «деревья вокруг», а «из этих деревьев можно построить 3 дома,
   из этих 10 машин — подземный бункер». Сырые данные внешней среды
   превращаются в готовый к использованию ресурс.
6. **Инструменты** — какими пользуемся в работе (Claude Code, Antigravity,
   Obsidian, Notion, AI-сервисы, любые скрипты/плагины/фреймворки). Система
   помогает:
   - **трекать** какие инструменты есть в арсенале
   - **быстрее понимать** как они работают, что можно с ними сделать
   - **настраивать** их под задачу
   - **использовать на максимум** — выжимать из каждого инструмента больше
     пользы, чем без системы

##### Мета-функция по ВСЕМ 6 типам ресурсов — приоритизация

По каждому из 6 ресурсов выше система помогает **отделять важное от неважного**.
Это не ещё один ресурс, а **сквозная функция** поверх всех:

- **Информация** → что читать сейчас vs что в backlog
- **Время** → на что тратить час vs на что 5 минут
- **Финансы** → во что инвестировать vs чего избегать
- **Связи** → с кем углубляться vs с кем поддерживать контакт
- **Внешние обстоятельства** → какие сигналы ловить vs какие игнорировать
- **Инструменты** → какой освоить на 100% vs какой просто знать

**Результат:** ресурсы пользователя (внимание, энергия, деньги) идут на
**важное**, а **неважное** — делегируется агентам или автоматизации.

Это мост к Части 2 «Цели и результаты» — именно через эту мета-функцию
система выдаёт время и энергию пользователю.

#### 1.3 Что Jetix OS делает сверх простого трекинга

Система — не пассивный блокнот. Она активно:

- **Соединяет** то что без неё не соединилось бы
- **Превращает сырые данные** в ресурс через анализ (точка 5 выше)
- **Ускоряет обучение** — помогает быстрее находить нужную информацию,

### Часть 2. Цели и результаты

#### 2.1 Что система выдаёт на выходе

Девять конкретных outputs. Перечислены по порядку значимости для Ruslan:

##### 2.1.1 Время
**1-2 часа жизни ежедневно** — за счёт автоматизации рутины.

Не нужно каждый раз:
- Писать шаблоны плана на день
- Выдумывать новую систему анализа недели
- Делать ту же рутинную работу, что и вчера

Система забирает это на себя, время возвращается пользователю.

##### 2.1.2 Энергия и состояние фокуса / потока
Забрав рутину, система **освобождает энергию на стратегические задачи**.

Пример: вместо того чтобы **проводить звонки продаж вручную** каждый день, система делает звонки полуавтоматом — а пользователь работает над **стратегией продаж, видением компании, управлением жизнью на стратегическом уровне**.

> "Она непосредственно даёт энергию на более стратегические уровни — на управление своей жизнью и бизнесом более стратегически."
> — Ruslan, диктовка 2026-04-17

##### 2.1.3 Контент и интеллектуальные артефакты
Не просто контент. **Производство полезной структурированной информации** на основе твоих мыслей и наработок — которую в будущем можно использовать как:
- Статьи
- Научные работы
- Обучение персонала
- Философские работы

Связь: [[ideas/writing-is-telepathy]].

##### 2.1.4 Артефакты wiki — накопленное знание
Сохранение знаний компаундирующим способом: каждая новая идея / claim / experiment встраивается в граф.

Эффект: **быстрее решать следующие задачи**, потому что уже накопленные знания доступны мгновенно. См. [[concepts/collaborative-project-versioning]].

##### 2.1.5 Лиды и контакты (sales-контур)
Система делает **shallow work за тебя** — ходит по форумам, сообществам, интернету, ищет подходящих людей.

Пользователь **остаётся с людьми общаться** на более глубоком уровне. Эффекты:
- Ускорение поиска клиента / партнёра
- Высвобождение времени Ruslan на работу над **продуктом**
- Более качественный коннект с людьми (система готовит контекст)

##### 2.1.6 Работа с информацией / обучение
Система делает 3 вещи с информацией:
1. **Планирование** — что именно изучать в ближайшее время
2. **Обработка + сохранение** изученного материала
3. **Быстрое структурированное описание любого топика** — глубоко и качественно, **95%-го покрытия** достаточно чтобы идти решать задачи без дополнительного ресёрча

Это «меняет правила игры» для обучения. См. [[ideas/information-discipline-as-core-skill]].

##### 2.1.7 Рука на пульсе рынка (real-time)
Бизнес-output: картина рынка в реальном времени. Чего раньше не было — теперь есть:
- Входящие и исходящие комментарии
- События рынка
- Изменения сигналов

→ **Быстрое управление бизнесом чуть ли не в реальном времени**.

##### 2.1.8 Преобразование информации в гипотезы + возможности
Это самый важный output — система превращает «информационный шум» в **использованные возможности на 300%**.

Механика:
- Видео / статьи / разговоры → система всасывает
- На выходе: **сводка идей / гипотез**, новые инструменты в арсенал, потенциальные партнёры в CRM
- Сохраняет каждого интересного человека, каждый инструмент, каждую гипотезу
- **Соединяет** гипотезы ⇄ инструменты ⇄ партнёров — быстро тестирует

> "Система будет как сохранятор и использователь возможностей на все 300%. Будет сохранять каждого человека, каждый инструмент, гипотезу — и потом за счёт этого соединять их вместе."
> — Ruslan, диктовка 2026-04-17

##### 2.1.9 Обработка созвонов / встреч / длинных коммуникаций
Быстрое извлечение **ключевых инсайтов + задач** из любых разговоров:
- Созвонов (1-на-1, групповых)
- Встреч
- Длинных переписок / чатов
- Подкастов / интервью

**Раньше:** чтобы понять что важного было в часовом созвоне — нужно переслушать час записи или прочитать полный transcript.

**Сейчас (с Jetix OS):** система даёт:
- Структурированный digest по разговору
- **Список задач**, которые надо выполнить из этого разговора
- Ключевые insights — сохраняются в wiki
- Контакты / упоминания людей — идут в CRM

**Эффект:**
- Ни один важный инсайт из коммуникации не теряется
- Экономия часов на re-listening
- **Управление проектами и задачами** — работа с задачами идёт из первоисточника
  (сказано → зафиксировано → в план), а не «потом вспомню»

Это output стыкуется с ресурсом «Время» (2.1.1) и с активным поиском гипотез (2.1.8), но достоин отдельного упоминания как **важный практический шаблон использования системы**.

- **Комментарии** — ок, может публиковать автоматически
- **Статьи** — проходят **проверку Ruslan'ом**
- **Сообщения потенциальным партнёрам** — на начальных этапах **проходят проверку** (чтобы было качественно, глубоко, как надо)

##### 2.4.3 Не совершает финансовые операции
Вообще. Никаких платежей, переводов, подписок. Сейчас жёсткая граница без исключений.

##### 2.4.4 Не удаляет данные (почти)
**Сейчас** — вообще ничего не удаляет.
**Планируется (позже):** рутинные штуки (логи, временные файлы) — раз в неделю чистка, с разрешения Ruslan.

Принцип: append-only по умолчанию, удаление — через человека.

##### 2.4.5 Не заменяет человеческие отношения
Очевидно. Система не заменяет:
- Ментора (Антон)
- Экспертов (Владислав, Родион)
- Друзей, семью
- Клиентов как живых людей

Она **помогает** в этих отношениях (контекст, напоминания, идеи), но не становится их заменой.

##### 3.1.1 Как использую — четыре режима
Постоянное взаимодействие в течение дня. Четыре главных режима:

**1. Стратегическое управление (источник правды)**

Система — это единственный источник правды для стратегических данных и целей.

- Заношу в систему стратегические цели (годовые, 5-летние, 10-летние — см. 1.6)

- Аналогично планирую день: цель дня + план работы → весь день работаю в рамках этого плана
- Решения всех масштабов (сегодня / неделя / месяц / год) опираются на то что зафиксировано в системе

**2. Ускорение рутинных задач**
- Напоминания о встречах → через интеграцию с календарём
- Трекинг времени (Toggl-интеграция) — система пишет мои часы, потом даёт сводки
- Шаблоны документов, планов, анализов — не переделываю каждый раз

**3. Место для рефлексии**
Куда я могу закинуть свои мысли и быстро проанализировать их с разных сторон.

**4. «Jarvis на максималках» — digital operator**

Это ключевая роль:
- Центральный Claude Code знает мою ситуацию досконально
- Я быстро взаимодействую / коммуницирую с ним
- Он помогает с другими инструментами: работа с другими AI, выдача контекста, настройка инструментов
- Контекст НЕ забывается, только улучшается
- Система учится, становится умнее → совместная работа ускоряется

> "У нас будет этот Claude, который знает всё (ну, важное), не забывает контекст, наоборот только улучшает его. В такой совместной работе быстрее решаем задачи."

#### 3.2 Агенты — команда внутри системы

##### 3.2.1 Центральная фигура — «мега-Claude»

##### 3.2.2 Роли агентов — что важно на v1-beta

- Задача: вытягивать из всех материалов (видео, статьи, созвоны) **всё релевантное для продаж**
- Содержит свою базу: контакты, инсайты, паттерны
- Параллельно работает над проектами: например «книга про продажи», «3 открытых вопроса про ICP»
- Всё что касается этих тем — заносит в wiki под свою ветку

Пример — **inbox-processor**:
- Секретарь по входящим
- Приоритизирует, роутит, извлекает задачи

Пример — **sales-outreach**:
- Подроль внутри sales-lead
- Ищет клиентов, пишет первые касания, трекает responses

Когда появится команда (1-5 лет, см. 1.6), у каждого члена команды будут **свои агенты** аналогичного уровня. 12 агентов — это первая итерация архитектуры, не окончательный состав.

```
(агенты-«пираньи» периодически разгрызают, см. 4.5)
  ↓
Если важное → wiki/sources/{date}-{slug}.md
              + извлечённые: wiki/ideas/{slug}.md / claims / concepts
              + edges в wiki/graph/edges.jsonl
Если неважное → остаётся в raw/ как архив
```
**Батч-обработка:** заметки идут пакетами (раз в неделю analytics-сессия) —
агенты решают что куда.

##### 4.2.2 Звонок / встреча
```
Запись звонка → транскрипт (raw/transcripts/)
  ↓
Агент анализирует → структурированный digest
  ↓ ↓ ↓
  ├─ Insights → wiki/sources/ + ideas / concepts
  ├─ Задачи → tasks/master.md (или tasks per project)
  ├─ Контакты → crm/{clients,partners,personal}.md
  └─ Решения принятые → decisions/
```
(Это output 2.1.9.)

##### 4.2.3 Идея
Идея может прийти из любого источника. Маршрут:
```
Идея в inbox / raw/
  ↓
Недельный аналитический проход
  ↓
Агент думает: «куда эту идею?»
  ↓ ↓ ↓
  ├─ Связана с проектом → projects/{x}/ideas.md + ссылка из wiki/ideas/
  ├─ Системная → ideas-pool/inbox.md
  └─ Сразу гипотеза → hypotheses/active.md
```

##### 4.2.4 Решение (NEW важная сущность)
**Зачем:** не принимать одно и то же решение по 20 раз.
```
Решение принято (life-level или project-level)
  ↓
Куда фиксируется?
  ├─ Life-level (что-то про мою жизнь, привычки, принципы)
  │   → decisions/life-decisions-log.md
  │   → сводка по месяцу: decisions/2026-MM-decisions.md
  │
  └─ Project-level (что-то про конкретный проект)
      → projects/{slug}/decisions.md
      → ссылка из соответствующего страт.документа
```
**Связь с проектами:** все решения на 80% подвязаны к проектам, life-level —
оставшиеся 20% (общие жизненные принципы, привычки, ритуалы).

**Здоровье/привычки** — это решения **другого уровня**, живут в `health/`,
но логически — это разновидность решений (что регулярно делать, что не делать).

> "Должно быть место для решений отдельно для всей жизни, отдельно для каждого
> проекта. Лог должен вестись чтобы реально пользоваться решениями и не тратить
> время на обдумывание одних и тех же по 50 раз."
> — Ruslan, диктовка 2026-04-17

##### 4.2.5 Гипотеза
Особый цикл — **тестирование**.
```
Гипотеза формулируется
  ↓
hypotheses/active.md (тестируется сейчас)
или hypotheses/backlog.md (на ожидании)
  ↓
Тестирование
  ↓ ↓ ↓
  ├─ Подтвердилась → hypotheses/validated-archive.md → дальше:
  │   ├─ Новый проект (если scope большой)
  │   ├─ В wiki/claims/ (как утверждение с evidence)
  │   └─ В задачи (если конкретные действия)
  │
  └─ Опроверглась → archive с пометкой "не работает"
```

#### 4.3 Слои хранения информации (моя рекомендация — best-practice)

**4 слоя информации**, каждый со своим назначением:

| Layer | Назначение | Где живёт | Кто пишет |
|-------|------------|-----------|-----------|
| **L1 Raw** | Immutable первоисточники | `raw/`, `inbox/`, `daily-log/drafts/` | Только человек / автотул |
| **L2 Processed** | Sources + atomic ideas | `wiki/sources/`, `wiki/ideas/` | Агенты после ingest |
| **L3 Synthesized** | Concepts, claims, foundations | `wiki/concepts/`, `wiki/claims/`, `wiki/foundations/` | Агенты + ручная корректировка |
| **L4 Strategic** | Решения, страт.документы | `strategy/`, `decisions/`, `projects/{x}/strategy.md` | Только Ruslan + одобрение |

**Правила перехода между слоями:**
- L1 → L2: через skill `/ingest` (автоматически, агенты)
- L2 → L3: через `/consolidate` + `/build-graph` (периодически)
- L3 → L4: только **через явное решение Ruslan** (граница 2.4.1)
- Обратные потоки (L4 → L3): через writeback (см. 4.4)

**Зачем 4 слоя:** L1 не теряется, L2-L3 могут переписываться без страха
(они — производные L1), L4 — компас (редко меняется, влияет на всё ниже).

#### 4.4 Writeback — обратные потоки (как знание возвращается)

Это **best-practice 2026** для Karpathy-style wiki систем — что я закладываю
в Jetix OS:

##### 4.4.1 Что такое writeback
Когда система **отвечает на вопрос** или **синтезирует** — результат должен
**возвращаться обратно в wiki**, чтобы знание компаундилось, а не пере-открывалось
каждый раз.

##### 4.4.2 Конкретные writeback-маршруты

| Триггер | Что записывается | Куда |
|---------|------------------|------|
| `/ask` отвечает на вопрос | Ответ с цитатами | `wiki/comparisons/{date}-{slug}.md` |
| Ответ создаёт **новую связь** | Edge type | `wiki/graph/edges.jsonl` |
| Ответ создаёт **новый claim с evidence** | Page | `wiki/claims/{slug}.md` |
| Ответ выявляет **противоречие** | Edge type `contradicts` + alert | `wiki/graph/edges.jsonl` + `/lint` отчёт |
| Ответ **использован в решении** | Backlink | `decisions/...` ↔ `wiki/comparisons/...` |
| Решение Ruslan приводит к **новой стратегии** | Update | `strategy/...` |

##### 4.4.3 Почему это критично
- **Без writeback** wiki замораживается (тонна материала, но не растёт)
- **С writeback** каждый ответ обогащает базу (compounding знание)
- Это уже частично работает в текущей wiki/ (60+199 идей импортированы, 572 edges)

##### 4.4.4 Cycle время writeback
- **Real-time** writeback после каждого `/ask` (immediate)
- **Daily** — `/lint` проверяет здоровье wiki (orphans, contradictions)
- **Weekly** — `/consolidate` + `/build-graph` (rebuild communities)
- **Monthly** — review writeback metrics (что добавилось, что не использовалось)

#### 4.5 Связи между компонентами — полная связность

Ключевая особенность: **всё в тексте + всё в одной плоскости** → **полная
связность**.

> Все они полностью связаны. Например, можем раз в месяц делать так: берём
> все гипотезы (с 20 разных проектов) — натягиваем на ОДИН только проект и
> смотрим как это может быть связано. Или натягиваем стратегический документ
> с одного проекта на другой и тоже смотрим как работает.
> — Ruslan, диктовка 2026-04-17

##### 4.5.1 Граф связей (концептуальный)

```
    Стратегические документы (life + projects)
              ↑              ↓
        ┌─────┼──────────────┼─────┐
        ↓     ↓              ↓     ↓
   Проекты ←→ Решения ←→ Гипотезы ←→ Идеи
        ↑     ↑              ↑     ↑
        └─────┼──────────────┼─────┘
              ↑              ↓
            Задачи ←→ Wiki (sources, claims, concepts)
              ↑              ↓
              └─────────┬────┘
                        ↓
                CRM (clients, partners, personal)
                        ↓
                Daily Log + черновики (песочница)
```

##### 4.5.2 «Натягивания» — еженедельная и ежемесячная аналитика

| Период | Что «натягиваем» | Зачем |
|--------|------------------|-------|
| **Еженедельно** | Все задачи × все проекты | «Кто кому может помочь?» |
| **Еженедельно** | Активные гипотезы × проекты | «Какую гипотезу применить?» |
| **Ежемесячно** | Все гипотезы × один проект | Глубокий анализ потенциала |
| **Ежемесячно** | Страт.документ проекта A × проект B | Cross-fertilization |
| **Ежемесячно** | Решения × проекты | «Не нарушаем ли свои принципы?» |
| **Ежеквартально** | Wiki overall × жизненная стратегия | Re-alignment |

Эта аналитика — **главный механизм** компаундирования знания через систему.

##### 4.5.3 Кто делает «натягивания»
- **Агент** запускает аналитику по расписанию (еженедельно/ежемесячно)
- Возвращает **report**: «обнаружены связи X, Y, Z, рекомендую обсудить»
- **Ruslan** ревьюит → решает что делать с insights → результат идёт в
  decisions/ или новые задачи

#### 4.6 Миграция Notion → wiki

(См. полный план в `design/NOTION-MIGRATION-PLAN.md` §5 Фазы α/β/γ/δ.)

**Краткое резюме потока:**

### Часть 5. Действия / Триггеры

#### 5.0 Главный принцип v1-beta — полуручной режим

> На данном этапе (v1-beta, следующий месяц) **система ничего не делает сама**.
> Она делает **только то что я ей скажу**. Если ничего не скажу — ничего не
> происходит.
>
> Автоматизация и активный режим — **после того как базовая архитектура
> подтвердит свою работоспособность**. Не сейчас.
> — Ruslan, диктовка 2026-04-18

Это осознанное решение. Причины:
- **Риск-менеджмент** — сначала убедиться что архитектура работает, потом автоматизировать
- **Обучение методологии** — пока Ruslan сам запускает каждое действие, он видит что работает / не работает, накапливает паттерны
- **Отсутствие случайностей** — автономное поведение на неотлаженной базе может создать мусор / ошибки / плохие артефакты
- **Beta-first** — автоматизация усложнит v1-beta без гарантий пользы

**Что из этого следует:**
- Нет cron'ов / триггеров по расписанию
- Нет event-driven действий
- Нет активного поиска в интернете
- **Всё через прямую команду Ruslan'а** или запуск шаблона

#### 5.1 Автоматические действия — нет (v1-beta)

В этой версии: **нет автоматических действий**.

Всё что в CLAUDE.md / существующих skills помечено как "cron" / "scheduled" —
**деактивировано** до выхода v1 (после обкатки beta).

Maintenance-действия (`/lint`, `/build-graph`, `/consolidate`) запускаются
**только по команде Ruslan'а**.

(После обкатки в v1-final — настройка автоматизации станет отдельным проектом
"jetix-os-automation", см. backlog в Приложении.)

#### 5.2 Действия по команде Ruslan — основной режим работы

**Каждое действие системы = прямая команда Ruslan'а.** Два способа:

**(а) Через Claude Code** — разговор с центральным Claude:
- Просьбы на естественном языке ("проанализируй week", "создай проект X")
- Явные skills: `/plan-day`, `/close-day`, `/ingest`, `/ask`, `/lint`, `/consolidate`, `/build-graph`
- Запуск отдельных агентов через Task tool ("запусти sales-researcher для X")

**(б) Через шаблоны** — запускаешь готовый workflow:
- Открываешь страницу дня → автоматически подтягивается шаблон
- Создаёшь новый проект через `projects/_template/` → шаблон раскатывается

**Внутри каждого workflow** — потом, по мере опыта — Ruslan будет давать
**более глубокие указания**: что внутри этого workflow делать автоматом,
что вручную. **Но это уже следующая итерация**, не v1-beta.

#### 5.3 Действия по событию (event-driven) — нет (v1-beta)

В этой версии **нет event-driven реакций**.

Система не реагирует автоматически на:
- Новый файл в `inbox/`
- Новое сообщение в mailbox агента
- Новую запись в CRM
- Дедлайн
- Закрытие задачи

Ruslan сам запускает обработку когда считает нужным.

#### 5.4 Эскалация — один уровень

Упрощённая модель v1-beta: **любой агент с проблемой → сразу Ruslan**.

Не многоуровневая (агент → lead → manager → Ruslan). В v1-beta пропускаем
слой lead/manager — всё идёт прямо.

**Как эскалация выглядит:**
1. Агент встречает проблему (неясность / невозможность / противоречие / failure)
2. Записывает в свой `scratchpad.md` + возвращает короткий отчёт в чат
3. Ruslan принимает решение
4. Даёт следующую команду

#### 5.5 Ритуалы — по команде, не по расписанию

**Утренний ритуал** (по команде Ruslan):
- Ruslan открывает Daily Log страницу дня
- Командой запускает планирование (через Claude Code)
- Claude подтягивает: календарь (если интеграция есть), вчерашний close-day, фокус из priorities.json, страт.документ недели
- Обсуждаем фокус дня + ключевые действия
- Ruslan утверждает → план дня зафиксирован

**Вечерний ритуал** (по команде `/close-day`):
- Ruslan запускает
- Claude анализирует черновики дня (песочница, см. 6.2.4)
- Раскидывает результаты: wiki / задачи / энергия / план на завтра
- Таблица с ключевыми данными Daily Log обновляется
- Ruslan ревьюит

**Еженедельный ритуал** (по команде, в конце недели):
- Ruslan запускает
- Claude анализирует данные за неделю (логи дней, задачи done/blocked, insights)
- Создаёт 2 документа: отчёт за прошлую неделю + план на следующую
- Запускаются "натягивания" из §4.5.2 (задачи × проекты, гипотезы × проекты)
- Ruslan ревьюит отчёт + утверждает план

**Ежемесячный ритуал** (по команде):
- То же, но масштаб месяца + глубже
- Отчёт + черновик плана следующего месяца + обновление страт.документа

**Все ритуалы запускает Ruslan.** Никаких cron'ов / автозапусков на v1-beta.

#### 5.6 Failure modes и recovery — базовые инструкции для агентов

Минимальный набор правил которые агенты **обязаны соблюдать** при любом сбое:

##### 5.6.1 SAFE-SAVE паттерн
При любой ошибке / прерывании / неясности:
1. `git add -A && git commit -m "[agent-id] SAFE-SAVE: stopped at <что>"`
2. `git push origin main`
3. Запись в `scratchpad.md`: где остановился, что сделано, что нужно
4. Короткий отчёт в чат

**Критерий:** не терять прогресс. Лучше лишний коммит, чем потерянная работа.

##### 5.6.2 Когда Notion MCP недоступен
- Если агент зависит от `mcp__notion-*` и MCP упал → SAFE-SAVE + репорт
- Fallback: если есть локальные dumps в `raw/notion-*` — использовать их
- Не пытаться обойти через другие источники без команды Ruslan'а

##### 5.6.3 Когда Claude API ограничен (529 / rate limit)
- 3 retry с backoff (5s → 15s → 45s)
- Если не помогло → SAFE-SAVE + репорт
- Fallback на Haiku — **только по команде Ruslan'а**, не автоматически

##### 5.6.4 Когда агент "потерял контекст" / путается
- Сразу стоп, не продолжать "наугад"
- Репорт в чат: "я не понимаю контекст, последнее что помню — X, что делать?"
- Ruslan корректирует или перезапускает сессию

##### 5.6.5 Когда неясно что делать
- **Не гадать.** Ставить в bucket UNCLEAR.
- Запрос к Ruslan'у: "варианты A / B / C, что выбрать?"
- Ждать ответ.

##### 5.6.6 Когда Ruslan offline
- Система **не продолжает** агрессивно работать. Ждёт.
- Только безопасные задачи (ingest уже начатого, `/lint`, push коммитов)
- Никаких решений (они требуют Ruslan)
- Никакой внешней коммуникации (ни postов, ни emails, ни messages)

##### 5.6.7 Git conflict / дубли
- Дубли: SAFE-SAVE, репорт Ruslan'у с описанием
- Git conflict: НЕ резолвить автоматически, SAFE-SAVE, ждать Ruslan'а
- Никогда `git push --force`, `git reset --hard` — без Ruslan'а

#### 5.7 Запреты — что система НЕ делает никогда

Параллель с 2.4 (границы) в разрезе действий. Агенты **обязаны** соблюдать:

| Запрет | Причина |
|--------|---------|
| Публиковать контент без одобрения Ruslan | 2.4.2 |
| Совершать финансовые операции | 2.4.3 |
| Удалять данные | 2.4.4 |
| Принимать стратегические решения без Ruslan | 2.4.1 |
| Коммуницировать с реальными людьми (email, сообщения) без одобрения | 3.1.5 |
| `git push --force` / `git reset --hard` / rebase main | безопасность репо |
| Трогать `.env`, `private/`, `~/.ssh/`, `config/*.secret.*` | безопасность credentials |
| Писать в `dashboard/` без явной команды | отдельный трек |
| Выходить в интернет (WebFetch / WebSearch) при автономной работе | защита от мусора |

Любое нарушение → немедленный стоп + репорт.

#### 5.8 Активный режим (автономный поиск) — отложен

**Отложено для v1-final** (следующая итерация после обкатки).

Когда базовая архитектура v1-beta подтвердит свою работоспособность (после 1-2 месяцев реального использования) — включим:
- Активный поиск лидов / клиентов / партнёров
- Активный поиск видео / статей по темам
- Автоматические "натягивания" еженедельные
- Event-driven реакции

В v1-beta — **всё по команде Ruslan'а**.

### Часть 6. Состояния / Жизненный цикл

#### 6.0 Глобальное видение — зачем вообще "система"

Глобально видение такое:

> У человека есть **жизнь**. Он куда-то идёт — в какую-то Точку Б. Знает он
> о ней или нет. Основные ресурсы которые у него есть для этого пути —
> **время** и **внимание**.
>
> У всех 24 часа в день. **Парадокс:** если очень хочется в Точку Б, скорее
> всего нужно больше чем 24 часа.
>
> Если человек использует Jetix OS — значит он **знает свою Точку Б** и хочет
> **адекватно распоряжаться вниманием и временем**, чтобы:
> - двигаться в нужном направлении
> - не остановиться посреди дороги
> - не забрести не туда
> — Ruslan, диктовка 2026-04-17

**Jetix OS = инструмент движения из Точки А в Точку Б.** Это глобальная
рамка для всего что описано ниже.

#### 6.1 Компоненты системы — что вообще есть

Прежде чем описывать жизненные циклы — фиксируем **из чего состоит** Jetix OS.
13 групп компонентов, разделённых на сущности (что), документы, базы и роли.

##### 6.1.1 Проекты — главная сущность управления (мини-жизни)
**Проект = жизнь в миниатюре.** Имеет:
- Видение / Зачем
- Точка А (где сейчас)
- Точка Б (куда идём)
- Шаги реализации
- **Ресурсы** выделенные с уровня выше (с уровня жизни)

**Что является проектом:** всё что занимает **больше недели**.

**Направления проектов любые:**
- Research (понимание мира вокруг)
- Self-improvement (улучшение себя, навыков)
- Улучшение системы (навыки/инструменты Jetix OS)
- Получение влияния
- Сбор философской базы
- Любая комбинация

**Взаимодействие проектов:** могут сотрудничать (общие данные, leverage), но
чаще **конкурируют** за ресурсы (внимание, время, деньги). **Задача
управленца (Ruslan) — решать кому больше внимания, кому меньше.**

> "Эти проекты могут быть вообще любых уровней направлений. Все что больше
> недели занимает — это по сути проект. Они между собой могут взаимодействовать
> или вообще борются за ресурсы. Задача управленца — между этими проектами
> решать кому больше внимания, кому меньше."
> — Ruslan, диктовка 2026-04-17

##### 6.1.2 Задачи
Атомарные единицы работы. Пул:
- **Общий пул** задач (все)
- **Пул на каждый проект** (фильтр)

Хранится в виде таблиц.

##### 6.1.3 Идеи
Сырые мысли / гипотезы / варианты. Таблица + связь с проектами.

##### 6.1.4 Гипотезы
Отдельная таблица, два уровня:
- **Гипотезы по жизни в целом** (системные)
- **Гипотезы по каждому проекту** (как этот проект улучшить, ускорить)

##### 6.1.5 Ресурсы (как сущность учёта)
6 типов из Части 1.2 — каждый трекается. Конкретно:
- Время (через Toggl)
- Внимание (через Daily Log + рефлексию)
- Финансы (отдельный учёт)
- Связи (через CRM — см. 6.1.9)
- Внешние обстоятельства (через wiki)
- Инструменты (см. 6.1.6)

##### 6.1.6 Инструменты — таблица + инструкции
Каждый инструмент имеет:
- Карточку (что это, для чего)
- **Инструкции** как использовать (детально и адекватно)
- Уровень освоения / приоритет

##### 6.1.7 Wiki (общая + локальные)
**Несколько wiki, не одна:**
- **Общая wiki** знаний (текущая `wiki/` в репо)
- **Локальные wiki** под темы / ключевые проекты (срезы общей через teги/symlinks)
- **Wiki рефлексии** — отдельная (см. 6.1.11)
- **Wiki здоровья / привычек** — отдельная (см. 6.1.12)

##### 6.1.8 Стратегические документы — два уровня
**Уровень жизни:**
- Стратегический документ всей жизни (5 / 10 лет)
- Месячный (отчёт + план)
- Недельный (отчёт + план)

**Уровень проекта:**
- Стратегический документ каждого основного проекта
- Видение, точка А/Б, ресурсы, шаги

**Назначение:** направляют **меня** и **агентов работающих над проектом**.

##### 6.1.9 CRM — три отдельные базы
1. **CRM клиентов** — отдельная база, отдельный агент
2. **CRM партнёров / отношений с людьми** — другая логика, другие поля
3. **CRM для девушек** (личное) — отдельная, чтобы трекать контакты и этапы общения

Разделение принципиальное — у каждой свои стадии, поля, формат взаимодействия.

##### 6.1.10 Daily Log + черновики (песочницы)
- **Шаблон Daily Log** — структура страницы дня
- **Под-страницы черновиков** на каждой странице дня — **песочницы**, где идёт реальная работа

Принципиальная архитектура работы — см. 6.2.4 «GitHub-style».

##### 6.1.11 Рефлексия — отдельный отсек
Отсек / направление для рефлексии:
- Свой агент
- Своя wiki
- Своя структура входа (когда + как фиксируется)

##### 6.1.12 Здоровье, привычки, спорт, ощущения
Один отдел / одна wiki — но всё вместе:
- Трекинг привычек
- Спорт
- Здоровье
- Ощущения

Разделение этого от рабочих компонентов — намеренное (другая логика, другой формат).

##### 6.1.14 Решения — отдельная сущность (NEW)
**Зачем:** не принимать одно и то же решение по 20 раз. Решение, однажды
принятое, должно быть зафиксировано и **переиспользуемо**.

**Два уровня:**
- **Life-level решения** (общие принципы, привычки, ритуалы) — `decisions/life-decisions-log.md`
- **Project-level решения** (что-то про конкретный проект) — `projects/{slug}/decisions.md`

**Пропорция:** ~80% решений подвязаны к проектам, ~20% — life-level.

**Здоровье/привычки** — это решения **другого уровня**, живут в `health/`,
но логически — разновидность решений (что регулярно делать).

**Лог:** append-only. Каждое решение — дата, контекст, само решение, обоснование,
последствия. См. 4.2.4.

##### 6.1.13 Агенты — центральный + специализированные
- **Центральный Claude Code** — главный собеседник Ruslan'а (см. 3.2.1).
  **Задача архитектуры:** контекст сохраняется, агент **умнеет с каждой сессией**,
  использует все самые последние наработки.
- **Специализированные агенты** — роли (sales-lead, inbox-processor и т.д. — см. 3.2.2)

> "Насчёт агентов улучшения вообще без понятия — это к тебя касается. Просто запиши
> самый наилучший самый качественный вариант который только есть на данный момент,
> чтобы они там умнели, контекст сохранялся."
> — Ruslan, диктовка 2026-04-17

**Best-practice 2026 для self-improvement агентов** (что я закладываю как
работающий вариант — детализируется в Шаге 4):
1. **Persistent memory** — каждый агент имеет `strategies.md` где фиксирует
   "когда X — делай Y", обновляется после каждой большой задачи (Karpathy
   System Prompt Learning)
2. **Context preservation** — `scratchpad.md` для текущей сессии, `mailbox.jsonl`
   для длинной recall
3. **Versioning** — каждое изменение system prompt создаёт новую версию,
   старые архивируются (rollback всегда возможен)
4. **A/B testing** — новые версии тестируются на 10% задач, метрики
   сравниваются, через 7 дней либо принимается, либо откатывается
5. **Цикл улучшения:** работа → feedback (от Ruslan или результата) →
   meta-agent анализирует → предлагает изменения strategies.md → Ruslan
   одобряет → внедрение

Это уже частично заложено в архитектуре (`agents/{id}/strategies.md` существуют
у всех 12, см. ARCHITECTURE-CURRENT.md), но **пустые** — наполняются по мере
работы.

#### 6.2 Жизненные циклы — как компоненты живут

##### 6.2.1 Жизненный цикл проекта
**Рождение проекта:**
Проект может появиться из:
- **Идеи** (что-то «прочитал и захотелось сделать»)
- **Прямого решения Ruslan** ("нужно делать X")
- **Задач, которые выросли в проект** (если задач по теме > N — это проект)
- **Действий** (начали что-то делать → сложилось в проект)

**Старт проекта:**
1. Описывается **стратегический документ проекта** (см. 6.1.8):
   - Точка А (где сейчас)
   - Точка Б (куда идём)
2. **Фиксируются ресурсы** выделенные на проект (время, деньги, агенты,
   приоритет внимания)
3. Создаётся структура: задачи / гипотезы / открытые вопросы / wiki срез

**Работа над проектом:**
- Появляются **задачи** (см. 6.2.2)
- Появляются **гипотезы** (как ускорить, как улучшить)
- В каждом проекте есть **место под открытые вопросы** — на их основании
  агенты **сами вытягивают ключевую информацию и задачи** из входящих данных

**Финал проекта:**
- Проект достиг Точки Б → закрывается, результаты переходят в wiki / CRM /
  case study
- Либо проект приостанавливается → ресурсы перераспределяются

##### 6.2.2 Жизненный цикл задачи
**Задача появляется из:**
- Целей проекта (декомпозиция страт документа)
- Голосовых заметок Ruslan
- Метода работы (что-то сделал → "о, надо это изучить")
- **Любого источника** — задача может прийти откуда угодно

**Стадии:** базово — backlog → today → in-progress → done / blocked.
(детализация — отдельная задача под v1-final, beta-минимум здесь достаточно).

**Связь:** задача ∈ проект (но могут быть «orphan» задачи без проекта —
быстрая мысль).

##### 6.2.3 Жизненный цикл стратегического документа

**5 лет / 10 лет — стратегический документ жизни:**
- Сейчас закладываем
- Один документ, держится перед глазами

**Месяц:**
- В конце месяца — **анализ** (отчёт за прошлый месяц)
- Создание **страт документа на следующий месяц** (видение + ключевые задачи)
- Получается **2 документа**: отчёт + план
- Шаблоны (templates) для обеих

**Неделя:**
- Анализ данных за неделю
- Планирование + ключевые задачи на следующую неделю
- Тоже 2 документа: отчёт + план

**Все эти документы влияют на действия пользователя и агентов** — они
**направляют работу под собой**.

##### 6.2.4 Жизненный цикл дня — главный практический цикл

**Утро (планирование):**
1. Открываю **шаблон Daily Log**
2. Вместе с агентом обсуждаем что делаем
3. Определяем:
   - **Фокус дня**
   - **Ключевые действия**
4. **Закрываем предыдущий день** (если не закрыт)
5. **Toggl-интеграция** (планируется): API подтягивает время прошлого дня
   автоматически в систему

**В течение дня — GitHub-style работа** ⭐ (ключевая методология):

> Работа в течение дня происходит **как ветки на GitHub**.
>
> - **Проекты = main** — там чистая ключевая информация, всё аккуратно,
>   адекватно прочитал и понял.
> - **Черновики дня = ветки/песочница** — там бунтуем, изобретаем, творим
>   что хотим. Не выходим из песочницы для решения задач.
>
> На странице дня создаются **под-страницы черновиков**, в них идёт реальная
> работа — берутся чистовики из других проектов, делаются новые версии,
> отшлифовываются.
> — Ruslan, диктовка 2026-04-17

Это прямая аналогия с git workflow: ничего «грязного» не попадает в main
проектов. Песочница свободна от перфекционизма — main документов сохраняет
качество.

**Вечер (закрытие дня):**
1. Заношу Toggl данные (если не автомат)
2. **Даю агенту задачу** проанализировать черновики дня:
   - Что добавить в **wiki**
   - Что в **задачи** (для проектов)
   - Что зафиксировать (энергия, состояние)
   - Возможно — план на завтра
3. Всё сохраняется в таблицу с ключевыми данными Daily Log
4. **Daily Log один** — для работы и трекинга состояния. Агенты
   по-разному его обрабатывают и раскидывают:
   - Рабочие данные → проекты / wiki / задачи
   - Состояние → wiki здоровья (6.1.12)
   - Insights → wiki общая
   - Контакты → CRM

##### 6.2.5 Связь между днями
**В целом — слабая.** Каждый день как самостоятельная сессия.

Связь идёт через:
- **Анализ недели** (см. 6.2.3) — там сводятся данные нескольких дней
- **Цель дня может быть любой** (необязательно по проектам — даже «отдых»
  как полноценная цель)

> "Каждый день просто как какая-то новая жизнь. Можем просто взять цель
> "отдых" и идти отдыхать. Цель дня может быть любая."
> — Ruslan, диктовка 2026-04-17

##### 6.2.6 Информационные потоки в систему — три канала

(Это пересекается с Частью 4, но важно зафиксировать здесь как часть life-cycle.)

**Откуда информация попадает в систему:**

1. **Входящий поток моих заметок** — основной источник:
   - Голосовые заметки Ruslan
   - Текстовые просьбы / запросы
   - Разговоры с Claude Code (вот этот, центральный)
2. **Записи звонков и встреч** — отсюда задачи / идеи / инструменты / контакты
3. **Видео / статьи / внешний контент** — те которые Ruslan **посчитает важными**.
   Полуавтоматический режим — **система не зарастает мусором**.

**Принцип фильтрации:** будут **инструкции** что является:
- Проектом (требует страт документ + ресурсы)
- Задачей (требует action)
- Идеей (требует обработки → может стать проектом / claim)
- Ресурсом (трекается)
- Прочее (отбрасывается)

6 ресурсов, 9 outputs, 13+ компонентов, и т.д.) — в v1/v2 вынести в
Приложение "Glossary".

##### 7.4.2 Разбиение документа при >5000 строк
Сейчас ~1900 строк. Если в v1/v2 превысит 5000 — разбивать на
`SYSTEM-DESIGN-HUMAN-part-N.md`? Или оставлять монолитом с collapsible?
Решать когда достигнем.

##### 7.4.3 Miro-визуализации
В DoD плановали 5+. На v1-beta сделаны только inventory-артефакты Шага 1
(11 фреймов). Новые под 7 частей — можно сделать **после** release v1-beta,
отдельной сессией. Это не блокирует release.

##### 7.4.4 Перевод в Notion-чистовик
На v1-beta — не делаем. Это Шаг 3 (фиксация) отдельно. Когда — после
release и 1-2 недель использования текущей версии.

##### 7.4.5 User-facing версия (для широкой аудитории)
Из FOUNDATION-DOCS-RESEARCH §14: отдельный документ для "человека с улицы"
по Divio 4-docs theory. Отдельный проект на будущее.

#### 7.5 Критерии выхода v1-beta → v1-final

Что должно случиться чтобы перейти к v1-final:

- [ ] Документ реально используется — 2+ недели ежедневной работы с ним
- [ ] Хотя бы 5 из 7 частей получили реальные правки от использования
- [ ] Технический документ (Шаг 4) создан методологией 5 чатов и утверждён
- [ ] Первая партия задач выполнена по новой методологии
- [ ] Явно обозначено минимум 3 пункта из §7.1 которые готовы к закрытию
- [ ] Ruslan говорит: "ок, пора фиксировать v1-final"

Только при всех 6 ✓ — переход. До этого — v1-beta продолжает жить как
рабочий документ.
