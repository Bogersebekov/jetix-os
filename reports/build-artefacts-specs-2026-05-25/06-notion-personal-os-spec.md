# 📋 Phase 5 — Спека Notion Personal OS (Layer 1+2, 5 баз core)

> **Что это.** Чертёж implementation-scope личного Notion-шаблона для Build: какие именно базы
> входят в demo-ready версию, как они устроены *функционально*, как тестер их форкает и
> использует. НЕ полный дизайн (он уже есть в Personal OS Plan) и **НЕ конкретные поля/формулы**
> — это пишется при самой сборке. Здесь — что demo должно покрыть, чтобы Дмитрий реально начал
> пользоваться.
>
> **Важно про границу.** Personal OS Plan описывает 8/11/14 баз (LITE/STANDARD/FULL). Для Build
> мы НЕ строим всё — мы строим **demo-ready 5 баз core**, достаточные, чтобы тестер форкнул и
> прожил с ними ≥1 неделю. [src: Personal OS Plan §10 Week 1 — F3, R-med]

---

## 1. Назначение (one-liner)

Рабочий (не дизайн-макет) личный Notion-шаблон из 5 баз + хаб, который smart T3 может
**форкнуть в свой workspace за ≤30 минут**, настроить под себя и реально использовать неделю —
чтобы дать живой feedback, где он ломается.

## 2. Цель (поведенческая)

Не «тестер посмотрел и сказал красиво», а **тестер форкнул → прожил неделю → вернул
structured feedback (5-7 точек)**: где затык, чего не хватает, где Jetix-логика мешает. Это
первый реальный proof point «философия работает на personal level». [src: Personal OS Plan §1
proof point — F3, R-med]

## 3. Primary audience (умный партнёр)

**smart T3** — Notion-comfortable, НЕ developer.
- **Bloom-ступень:** 3 (Применяет) → 5 (Оценивает) — это уже trial, не просмотр.
- **Знает:** как пользоваться Notion; что такое база/связь/вид; ценит порядок в жизни.
- **НЕ знает:** код, формулы Notion на уровне инженера, наш Jetix-жаргон в полях.
- **Хочет:** инструмент, который реально помогает жить, а не ещё одну систему для поддержки.
- **Отпугнёт:** 50 баз сразу, формулы, требующие технаря, Jetix-термины в названиях полей,
  отсутствие fork-инструкций.
- *Примеры роли (IP-1):* Дмитрий (первый, гуманитарий) → Сева → другие из ближнего круга.

## 4. Secondary audience (mention only)

T1-методолог — посмотрит как на «доказательство, что философия операционализируется», но для
него demo вторично (его primary — Видео A + Team OS). Инженер — захочет FULL 14 баз, но это не
Build-scope.

## 5. Ключевые сообщения (что demo доносит самим фактом работы)

1. **Философия Jetix ужимается в лёгкий личный инструмент** — не тяжёлая машина на 17 агентов,
   а человек + Claude Code как один помощник. [src: Personal OS Plan §1 — F3, R-med]
2. **Файлы = правда, Notion = витрина** — данные твои навсегда, vendor lock-in исключён.
   [src: Personal OS Plan §8 — F3, R-med]
3. **Голос → только черновик** — Claude Code предлагает, ты решаешь; никогда не перезаписывает.
4. **Гипотезы в центре** — «что хочу узнать» как первоклассная база (отличие от обычных
   to-do систем). [src: Personal OS Plan §5 Hypotheses — F3, R-med]
5. **Форкается за час, настраивается под нишу** — твой POINT A/B, твои проекты, не «дневник Васи».

## 6. Структура (5 баз core + хаб — функции, не поля)

R1-вопрос «какие именно 5» (§15). Default-набор из Personal OS Plan §10 Week 1:

| # | База | Функция (что делает) | Почему в core |
|---|---|---|---|
| 1 | 📅 Daily Log | дневник, append-only, главный голосовой ввод | сердце ежедневного использования |
| 2 | 🚀 Projects | проекты: тип + чекпоинт + «застрял?» | без неё нет «что я делаю» |
| 3 | ❓ Hypotheses | «что хочу узнать» + как проверю | отличительная черта подхода |
| 4 | 🤝 CRM (People) | люди + история касаний + что предлагаю/прошу | связь с миром |
| 5 | 🧭 Strategic | POINT A (где я) → POINT B (куда хочу) | направление; прозу пишет человек |
| — | 🎯 Command Center | одна страница-хаб, всё видно | точка входа |

**Что НЕ входит в Build demo (но есть в полном плане):** Knowledge (Sources/Claims), Reviews,
Life Pulse, Ideas Bank, Reference — это Week 2+ или для более глубоких ниш. Для первого trial
5 баз достаточно, чтобы прожить неделю. [src: Personal OS Plan §10 «Неделя 1 ядро» — F3, R-med]

**Дисциплина зеркала (illustrative, не финал):** каждая строка базы = один `.md` на диске,
свойства Notion = поля YAML-шапки. Конкретные поля и формулы — пишутся при сборке, НЕ в этой
спеке. Здесь только: «у Projects есть поле-индикатор “застрял” = активный проект без касания
>14 дней». Это пример функции, не готовая формула. [src: Personal OS Plan §5 — F3, R-med]

## 7. Hook начала (первое впечатление при открытии demo)

Тестер открывает Command Center и за 30 секунд видит **не пугающую сложность, а порядок**: «вот
сегодня, вот мои проекты, вот что я хочу узнать». Hook = «о, это я могу сразу понять и
использовать», а не «боже, тут 14 баз и формулы». Низкий порог входа = главная задача первого
экрана. [src: Personal OS Plan §0 «форк за 30-60 мин» — F3, R-med]

## 8. CTA конца (что тестер делает после demo)

Не «купи», а **«форкни и попробуй неделю → дай feedback»**. Это CTA-08-подобное (trial), но
бесплатное и без обязательств. Явная fork-инструкция (1 клик дублирования + что менять в Layer
2). После недели — сессия feedback (по шаблону «звонок» из Personal OS §7). [src: Personal OS
Plan §9 fork за 30-60 мин — F3, R-med]

## 9. Формат / длина

- **Формат:** Notion workspace (публичная ссылка-дубликат или приватный инвайт — R1 §15) + 1
  страница fork-инструкции + (опц.) Claude Code helper-prompts.
- **Объём:** 5 баз + хаб; LITE-уровень формул (не инженерный).
- **Время сборки Ruslan:** Personal OS Plan §10 Week 1 ≈ 4 дня на 5 баз + хаб + засев.

## 10. Зависимости

- **Pre-reqs:** Видео B желательно (тестер приходит к Personal OS, поняв парадигму); решение
  «какие 5 баз» (R1).
- **Блокирует:** trial Дмитрия → feedback #1 → правка шаблона → онбординг Севы; Team OS demo
  (Phase 6 строится поверх Personal OS).
- **Параллельно:** видео A/B (Personal OS не зависит от видео технически).
- [src: Personal OS Plan §15 + Execution Plan §6 — F3/F2-F3, R-med]

## 12. R12 paired-frame check (8 вопросов)

| # | Вопрос | Применение к Personal OS demo | Вердикт |
|---|---|---|---|
| 1 | Цена ≤ польза? | бесплатный инструмент, экономит время на порядок | ✅ |
| 2 | Конкретно? | да — 5 конкретных баз с понятными функциями | ✅ |
| 3 | Соразмерно? | просим только feedback за бесплатный инструмент | ✅ |
| 4 | По ступени? | trial-ступень; нет partnership pitch внутри demo | ✅ |
| 5 | Канал уместен? | Notion fork — нормальный канал | ✅ |
| 6 | Не доим / не запираем? | **fork-friendly ключевое** — отдельный workspace, Jetix НЕ получает данные форка, уйдёшь — данные у тебя | ✅ |
| 7 | Нет манипуляции? | нет FOMO/«закрытого клуба»; апдейты добровольны | ✅ |
| 8 | Стоп-сигнал готов? | если в полях всплывёт lock-in механика — убрать | ✅ |

**Главная R12-зона (вопрос 6):** lock-in через данные. Защита: форк = **отдельный workspace**,
Jetix не получает данные из форкнутых копий, экспорт в файлы в любой момент, нет механик
удержания/FOMO. [src: Personal OS Plan §9 «этика встроена R12» — F3, R-med]

## 13. Acceptance criteria

- ✅ Дмитрий открывает demo → видит 5 баз → **форкает в свой workspace за ≤30 минут**.
- ✅ Использует **≥1 неделю** реально (не просто посмотрел).
- ✅ Возвращает **structured feedback из 5-7 точек** (где ломается / чего не хватает).
- ✅ Нет Jetix-жаргона в названиях баз/полей; формулы LITE-уровня (без технаря).
- ✅ Есть явная fork-инструкция; форк = отдельный workspace (данные изолированы).
- ✅ Голосовой ввод (если включён) пишет только черновики, не перезаписывает.

## 14. Анти-паттерны

- ❌ **Over-engineering** — 50 баз / FULL 14 сразу. Build = 5 core.
- ❌ **Jetix-специфичная терминология в полях** — «cohort target ontology» вместо «цель».
- ❌ **Формулы, требующие технаря** — отпугивают гуманитария.
- ❌ **Нет fork-инструкции** — тестер не знает, как взять к себе.
- ❌ **Авто-перезапись голосом** — главное табу (потеря данных).
- ❌ **Хранение только в Notion** без файлового зеркала — vendor lock-in.
- ❌ **Имена-личности в Layer 1** — ломает fork-friendly («Дневник Васи»).

## 15. Варианты / R1 decisions (surface)

- **R1-N1 Какие 5 баз?** Default: Daily Log + Projects + Hypotheses + CRM People + Strategic.
  *Альтернатива:* заменить Strategic на Life Pulse для гуманитария (Дмитрий-ниша центрирует Life
  Pulse + Daily Log). [src: Personal OS Plan §9 ниша гуманитарий — F3, R-med]
- **R1-N2 Уровень формул:** lite (минимум) vs medium (несколько индикаторов). *Факт:* lite =
  проще форкать гуманитарию; medium = больше пользы инженеру.
- **R1-N3 AI integration helpers** — включить готовые Claude Code prompts в demo или нет? *Факт:*
  с ними мощнее, но добавляет порог настройки.
- **R1-N4 Канал раздачи форка** — публичная ссылка Notion / приватные инвайты / git scaffold.
- **R1-N5 Степень автоматизации Claude Code** — A ручной / B скрипты / C гибрид (рекомендуемый
  старт по плану). [src: Personal OS Plan §13 — F3, R-med]
