## §3 FPF-Scope (explicit — per prompt §0.10 + memory `feedback_fpf_lens_first`)

> Без этого Phase 4-6 = «каша непонятная». Фиксирую FPF-линзу ДО re-audit/breakdown/sessions.

1. **ЧТО research'им (re-plan):** strategic plan-as-document — план как **объект-планирование** (FPF P-система планирования, не набор шагов) для **Jetix-as-Workshop-Mastery-Network** = body of Vision. Артефакт уровня «4D-экстент плана»: план существует во времени (предшественники → текущий → следующий) и в пространстве направлений (16).
2. **На каком уровне сравниваем:** план-документ × план-документ — predecessor (handoff 27.05 + Sprint substrate) vs new re-plan 28.05. За этим уровнем — Foundation/system level (FPF Part E constitution), который НЕ трогаем (R2 STRICT).
3. **С чем сравниваем:** Sprint 25-27.05 selected canonical docs (substrate, §4 working list) + voice insights v16.
4. **Зачем (acceptance predicate):** Ruslan-instruction explicit — «ещё раз посмотреть всё ли подойдёт + что нужно по каждому направлению + что можем сейчас + strategic sessions queue + какие данные нужны». ЭТО = acceptance predicate этого run'а. Re-plan refuted, если хоть одно не выполнено.
5. **FPF primitives применимы:**
   - **P-плана (4D extent)** — план = объект во времени+направлениях, не список.
   - **role-method-work separation per direction** — для каждого direction: какая роль (U.Episteme abstract) выполняет какой метод над каким work-product. IP-1 STRICT: #5 Партнёры роли абстрактны, исполнители (Левенчук / Tseren / Kaiser) = RUSLAN-LAYER примеры.
   - **F-G-R per claim** — каждый per-direction verdict + re-audit verdict несёт Formality (F2 voice / F3 synthesis) / Group-scope / Reliability (R-low surface … R-high grounded).
   - **U.Episteme abstract roles** — founder, strategist, PM, sales-lead, methodolog, advisor = role-types; конкретные люди = bindings.
   - **gradient достаточности (Simon satisficing)** — re-audit verdict = «достаточно ли doc хорош для своей роли», не «идеален ли».
6. **Claims F-G-R triple** требуется per per-direction verdict (Phase 5) и per-doc verdict (Phase 4): F2/F3 + G (этот prompt) + R (low/med/high по grounding).

**FPF-следствие для re-audit:** doc оценивается не «нравится/не нравится», а **«выполняет ли свою роль в 4D-экстенте плана»** — KEEP (роль активна, актуален), REFINE (роль активна, нужна правка под v16/V4), MERGE (роль дублирует соседа), DEFER (роль валидна но не Wave 1), ARCHIVE (роль закрыта — superseded).
