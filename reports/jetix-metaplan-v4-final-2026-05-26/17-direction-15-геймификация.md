# Direction #15 — Геймификация жизни / Engagement: portfolio spec (V4)

> **Foundation grounding.** Jetix = мега-мастерская мирового уровня: место, где люди становятся
> мастерами в эпоху AI, вместе двигают фронтир и не дают системе доить или запирать себя.
> Три грани: Мастерская (ГДЕ) · Мастерство (ЧТО прокачивают) · **Сеть (КАК) + Мастерство (engagement-обёртка)**.
>
> **Геймификация как грань.** Направление #15 — не просто «фича». Это слой, который превращает
> абстрактную «прокачку» (Мастерство) в видимое переживание каждый день. Без этого слоя мастерство
> остаётся идеей. С этим слоем — человек чувствует, что растёт. Грань Сеть: soft focal points,
> social coordination, cohort solidarity — без давления и FOMO. Грань Мастерство: skill trees,
> achievement markers, quest framing — без vanity metrics и dopamine loops.
>
> **R12 CENTRAL DISCIPLINE.** Геймификация = **PRIMARY R12 surface V4** (максимальный риск
> манипуляции во всей системе). Любой механик геймификации может легко соскользнуть в dark pattern.
> Этот документ — spec для создания портфолио материалов, которые называют эту границу явно,
> объясняют разницу между «геймификацией для смысла» и «геймификацией для удержания», и дают
> конкретные инструменты для первого. R12 paired-frame: influence-ethics-expert + propaganda-expert +
> nlp-expert + recruitment-dynamics-expert — все AUTO-FIRED на это направление.
>
> **Что здесь.** Spec для 12 artefacts (§A..§L). Каждый раздел = инструкция для создателя контента,
> НЕ публичный текст. R11 STRICT: никаких готовых публичных фрагментов. R1 STRICT: формулировки —
> опции и описания, финальный prose за Русланом. IP-1 STRICT: нет имён executor-агентов.

## Введение: тезис (Ruslan voice 26.05)

**Тезис направления (для создателей — R1 surface, формулировки за Русланом):**

Жизнь интересно жить, когда есть **вызов / прогрессия / измеримый рост мастерства / достижения**.
Jetix даёт инструменты геймифицировать свою жизнь — привычки, навыки, проекты, здоровье, вклад в
сообщество — чтобы было интересно жить, следить за своей статистикой, прокачиваться.
Цель: **наполнить жизнь смыслом, видеть его каждый день, чувствовать тягу к жизни** — триада
O-138 «жить чтобы жить», ощутимая в ежедневной статистике.

**Фундаментальная разница (ось всего направления):**

| Геймификация для смысла (Jetix путь) | Геймификация для удержания (тёмный путь) |
|---|---|
| Метрика = насколько ты вырос | Метрика = время в приложении |
| Intrinsic motivation (SDT) | Variable-reward / dopamine loop |
| Opt-out всегда, без штрафа | Sunk-cost / streak-pressure |
| Прогрессия = mastery marker | Прогрессия = вовлечённость-прокси |
| Flow state (challenge ≈ skill) | Pseudo-urgency (FOMO) |
| Смысл-центрично | Retention-центрично |
| Виден реальный рост | Виден рост в системе |

**R12 paired-frame note.** Приведённая таблица — не дихотомия «хорошее vs плохое». Это диагностический
инструмент для создателя. Каждый механик нужно проверять по обеим колонкам перед внедрением.
Граница подвижна: один и тот же механик (например, streak) может быть в обоих колонках в зависимости
от того, как сконструирован. Именно поэтому §J — обязательный шаг, а не опциональный.

## Sub-areas обзор

Семь sub-areas, которые охватывает направление:

**1. Personal stats dashboard** (Life Pulse Layer 1 + Habits + deep work minutes tracking).
Notion templates существуют как прообразы. Sub-area дает человеку «зеркало роста»: не мотивирует
через стыд или сравнение, а через видимость своего прогресса.

**2. Achievement system** (milestones/badges — НЕ vanity metrics).
Ачивки как meaningful progression markers: ачивка «первые 100 часов глубокой работы» = маркер
реального порога. Ачивка «опубликовал первый метод» = факт в мире, а не click count.

**3. Skill trees** (визуализация тем vs уровней, нелинейная развитость).
Связан с Direction #13 Мастерство. Skill tree — не геймплей, а карта: показывает, где человек,
откуда пришёл, куда может идти. Нелинейность важна: нет «правильного порядка» — есть уникальный
путь каждого мастера.

**4. Quests** (project-as-quest framing для life management).
Проект = квест: есть начало, вызов, прогрессия, завершение, артефакт. Не «задача в to-do», а
осмысленное приключение. Framing меняет отношение, не меняя механику.

**5. Schelling coordination** (мягкие focal points в сообществе — НЕ manipulation).
Soft coordination: публичные обязательства, синхронные спринты, видимость прогресса соседа —
без давления, сравнения рейтингов или штрафов за отставание. Coordination через вдохновение.

**6. Virtual economy hooks** (cohort tokens / contribution credits — R12 STRICT).
Самый опасный sub-area. No pay-to-win. No extraction. R12 anti-extraction: вклад в сообщество
получает признание, но признание ≠ привилегии, которые нельзя получить без токенов. Fork-and-leave
без потери накопленного мастерства (только токены остаются в кооперативе — это справедливо).

**7. Meaning embedding** (смысл каждый день).
Самый важный sub-area: без него вся геймификация — пустая обёртка. Инструменты делают смысл
видимым: «сегодня ты потратил 3 часа на это — это 1.2% от 250 часов, которые нужны для N-уровня».
Смысл не создаётся инструментом — он уже есть в работе. Инструмент его отражает.

## §A — Главный документ «Геймификация жизни без тёмных паттернов»

**Тип:** 1-pager (summary) + long-form spec (полный документ).

**Назначение 1-pager:** вход для T1 методологов и потенциальных участников кланов.
Должен за 1 страницу дать: тезис (зачем геймифицировать жизнь), разницу (смысл vs удержание),
3-5 конкретных механик, явный список того, чего Jetix NOT делает (anti-patterns), call-to-action.

**Spec для 1-pager:**
- Структура: hook (1 абзац) → тезис (3 предложения) → механики (3-4 bullet с иконкой) →
  anti-pattern box («что мы НЕ делаем: ...») → CTA.
- Тон: прямой, без пафоса. Не «мы делаем жизнь игрой», а «мы делаем рост видимым».
- Anti-dark-pattern framing: явный, в отдельном выделенном блоке, не в сноске. Не извинение —
  позиция.
- R12 check: каждая перечисленная механика должна пройти тест §J перед включением.

**Spec для long-form:**
- Разделы: (1) Почему геймификация жизни? (тезис + контекст эпохи AI) →
  (2) SDT-фундамент (autonomy/competence/relatedness как основа — кратко) →
  (3) 7 механик (каждая с описанием + anti-pattern pitfall) →
  (4) Чёрный список dark patterns (явно, с примерами из индустрии) →
  (5) Как настроить (ref на §E SOP) →
  (6) AI роль (ref на §K) →
  (7) Партнёры (ref на §L).
- Объём long-form: 3000-5000 слов.
- Sourcing: SDT (Deci & Ryan), Csikszentmihalyi flow, Deterding et al. gamification critique,
  Jane McGonigal (позитивная сторона), но также критика gamification в HR/edtech.
- R12 permeation: каждая секция содержит explicit note о том, где риск манипуляции.

**R12 check (per artefact):** Документ сам по себе не может быть dark pattern. Риск: если список
«преимуществ» написан как продающий текст без балансирующей критики → propaganda risk.
Mitigation: R6 честность — явно называть риски и ограничения механик.

## §B — Видео: hook + scenes + closing + anti-manipulation framing

**Тип:** видео-spec (сценарий структуры, НЕ скрипт).

**Формат:** 5-8 минут (средней длины, не short-form — тема требует контекста).
Альтернативно: 90-секундный hook + полная версия (два формата из одного материала).

**Hook spec (0:00-0:30):**
- Визуальный вход: статистика дня — deep work hours, навык-прогресс, вклад в сообщество.
- Нарратив hook: «Что если бы ты каждый день видел, насколько вырос? Не лайки. Не стрики.
  Реальный рост.» (пример формулировки — R1, финальный текст за Русланом).
- Anti-TikTok framing уже в hook: то, что мы показываем — противоположность алгоритма.

**Scenes spec (0:30-6:30):**
- Scene 1 (1:00): Проблема — жизнь без прогрессии. Что чувствует человек без видимого роста.
  R6: честно, без преувеличения.
- Scene 2 (1:30): SDT micro-объяснение (без академизма). Autonomy / competence / relatedness —
  три фактора, при которых мотивация устойчива. Пример: skill tree vs leaderboard.
- Scene 3 (1:30): 3 механики в действии (пример из жизни участника — R1 surface, Ruslan
  может использовать себя как Founder-as-Exhibit). Статистика → ощущение → решение следующего дня.
- Scene 4 (0:45): ЯВНОЕ называние тёмных паттернов. «Мы специально не делаем X, Y, Z — вот почему».
  Это не сноска. Это полноценная сцена. Причина: аудитория задаётся этим вопросом, и ответ в середине
  видео — лучший способ не потерять доверие.
- Scene 5 (0:45): Schelling coordination — community без давления. Пример: синхронный спринт
  как focal point, а не соревнование.

**Closing spec (6:30-8:00):**
- Не продажный. Не «зарегистрируйся». Скорее: «попробуй настроить dashboard на этой неделе».
  Action = simple, бесплатный первый шаг.
- Повтор anti-dark-pattern позиции в 1 предложении.

**R12 check (per artefact):** Видео — высокий риск propaganda mechanics (emotional hook, social proof).
Mitigation: Scene 4 явная, не спрятана. Emotion в hook работает через relatable situation,
не через manufactured urgency. Нет countdown, нет «ограниченных мест».

## §C — Курс: «Дизайн собственной геймификации жизни»

**Тип:** курс для T1 методологов как architects (люди, которые хотят спроектировать свою систему,
а не просто использовать готовую).

**Аудитория:** T1 методологи, участники кланов, потенциальные партнёры (Мастерская-партнёры),
которые хотят понять принципы и адаптировать под себя или под свой клан.

**Формат:** 4-6 модулей по 45-90 минут. Возможен self-paced + cohort sprint вариант (Hackathon
#16 как activation venue).

**Модули spec:**

- Модуль 1: SDT-фундамент. Autonomy / competence / relatedness. Почему внешняя мотивация
  подрывает внутреннюю (crowding out effect). Применение: как спроектировать систему, которая
  усиливает, а не замещает внутреннюю мотивацию.

- Модуль 2: Диагностика жизни без геймификации. Где сейчас нет видимости прогресса?
  Что хочется видеть, но не видишь? Участник выходит с картой «белых пятен» в своей статистике.

- Модуль 3: Механики — обзор и anti-pattern тест. Каждая механика (dashboard / achievement /
  skill tree / quest / coordination / economy) с SDT-mapping + dark-pattern risk score +
  opt-out check. Участник выбирает механики для своей системы, осознанно исключая риски.

- Модуль 4: Дизайн своей системы. Практический workshop: участник строит свой dashboard
  (Notion или другой инструмент), skill tree, первый набор achievements. Выходной артефакт —
  рабочая v0.1 личной системы.

- Модуль 5: Social coordination без давления. Как использовать Schelling focal points в
  своём клане или малой группе. Что делать, если кто-то выпал (не санкции — поддержка).

- Модуль 6: AI-ассистент в геймификации. Как AI помогает отслеживать статистику и генерировать
  идеи — и где граница, за которой AI начинает nudge-manipulate. Участник выходит с policy
  для использования AI в своей системе.

**R12 check (per artefact):** Курс — риск превращения в «продукт», который создаёт зависимость
от платформы. Mitigation: выходной артефакт (личная система) полностью принадлежит участнику.
Курс не встраивает lock-in механики. Возможность fork-and-leave с артефактами.

## §D — Книга (если применимо)

**Применимость:** YES. Тема «геймификация жизни» достаточно широка для книги. Но только если
позиционирование чёткое: не «gamification how-to» (таких много), а «anti-manipulative life design
through gamification» — ниша, которую почти никто не занял с этой позиции.

**Рабочее название (опция — R1):** «Прокачка без ловушек: как геймифицировать жизнь и не стать
заложником системы». Или более прямо: «Жизнь как мастерство: дизайн прогрессии без тёмных паттернов».

**Структура книги (spec):**

- Часть I: Почему жизнь без вызова — скучная. Феноменология отсутствия прогрессии.
  Кризис смысла как кризис видимости роста. R6: честно, без клише.

- Часть II: SDT и flow — научный фундамент. Краткое, без академизма. Три потребности.
  Flowstate Csikszentmihalyi. Почему это важно для дизайна системы.

- Часть III: Тёмная сторона геймификации. Честный разбор: Duolingo streaks, Fitbit shaming,
  leaderboards в корпорациях. Механизм dark pattern: как retention-логика проникает в жизнь.

- Часть IV: 7 механик Jetix-подхода. Каждая механика — глава: описание, SDT-alignment,
  пример внедрения, anti-pattern check, opt-out design. Skill tree глава связана с Мастерством.

- Часть V: Дизайн своей системы. Пошаговое руководство (≈ курс Модуль 4 в письменном виде).
  Founder-as-Exhibit: Ruslan показывает свою реальную систему (per §I).

- Часть VI: Сообщество и координация. Schelling coordination без давления. R12 anti-extraction
  в virtual economy. Как community остаётся средством, а не целью.

- Заключение: «Живи так, чтобы было интересно жить» — триада O-138 в конкретных инструментах.

**Объём:** 50-80К слов. Целевая аудитория: T1 методологи + образованные люди, которые
хотят структурированный подход к жизни без gamification-bullshit.

**R12 check (per artefact):** Книга — риск стать «самопомощью», которая создаёт зависимость от
фрейма. Mitigation: явная цель книги — дать инструменты для независимой системы, не встроить
читателя в платформу. Последняя страница должна быть anti-lock-in по тону.

## §E — Инструкции/SOP: настройка Life Pulse dashboard / skill tree / quests

**Тип:** практические операционные инструкции с decision trees + anti-patterns (что НЕ делать).

**Компоненты SOP:**

**SOP-1: Life Pulse Dashboard Setup**
- Pre-requisite check: зачем ты это делаешь? (anti-pattern: «потому что так надо»).
  Участник формулирует 1-3 вопроса, на которые dashboard должен отвечать каждую неделю.
- Decision tree: какие метрики включать? Критерий включения: метрика отвечает на твой вопрос.
  Критерий исключения: метрика красивая, но не отвечает на твой вопрос.
- Notion template setup (step-by-step): 5 шагов. Habit tracker интеграция. Deep work minutes.
  Energy/mood как leading indicator (опционально — не принудительно).
- Anti-patterns section (явно в SOP, не в сноске):
  - НЕ добавляй метрики «потому что интересно»: dashboard-inflation → signal-to-noise → abandon.
  - НЕ проверяй dashboard чаще 1 раза в день: real-time tracking = anxiety loop.
  - НЕ ставь цели-числа без контекста: «1000 минут deep work» без привязки к проекту = empty.
  - НЕ сравнивай dashboard с чужим: контекст разный, сравнение бессмысленно.
- Weekly review ritual (15 минут): 3 вопроса, 1 action.

**SOP-2: Skill Tree Design**
- Принцип: skill tree = карта, не рельса. Нет правильного порядка.
- Step 1: список тем, которые хочешь прокачать (из Direction #13 Мастерство substrate).
- Step 2: разбивка по уровням — не «новичок/средний/эксперт», а собственные маркеры
  (что конкретно умеешь на каждом уровне).
- Step 3: визуализация (Notion, Miro, draw.io — инструмент не принципиален).
- Step 4: связи между темами (нелинейность: тема A помогает теме B).
- Anti-patterns:
  - НЕ копируй чужое дерево: твой путь мастерства уникален.
  - НЕ добавляй темы «для полноты»: каждая ветка должна быть осмысленной для тебя.
  - НЕ оценивай «% завершения»: skill tree не завершается.
- Update cadence: раз в месяц обновить (не чаще — tree не меняется быстро).

**SOP-3: Quest Design для проектов**
- Что такое quest framing: проект = начало + вызов + прогрессия + завершение + артефакт.
  Это не переименование задачи в «квест» — это изменение способа думать о проекте.
- Step 1: сформулировать вызов (не «сделать X», а «решить Y через X»).
- Step 2: определить прогрессионные ворота (3-5 milestone с видимым артефактом).
- Step 3: описать «что у тебя будет, когда закончишь» (outcome + artifact).
- Decision tree: когда quest framing подходит, а когда нет
  (routine tasks не нуждаются в quest framing — anti-pattern: всё превращать в квест).
- Anti-patterns:
  - НЕ создавай квесты ради квестов: если проект неосмысленный, framing не помогает.
  - НЕ делай квест без реального вызова: «квест» из routine task = empty gamification.
  - НЕ ставь fake milestones: прогрессионные ворота должны быть реальными.

**R12 check (per artefact):** SOP — низкий risk, но anti-patterns sections критичны.
Без них SOP становится инструкцией для создания системы, которая может drift в dark patterns.

## §F — Шаблоны/Templates

**Тип:** набор конкретных шаблонов для немедленного использования.

**Template 1: Life Pulse Notion DB**
- Spec: database с полями: Date / Deep Work Minutes / Habits Completed (checkbox list) /
  Energy (1-5 — опционально) / Weekly Focus (текст) / Week Rating (1-5) / Notes.
- View: Weekly Calendar (primary) + Rolling 4-week table + Rolling 3-month bar chart.
- Принцип дизайна: минимальный ввод (2-3 минуты в день). Если ввод занимает >5 минут — shorten.
- Anti-pattern protection встроена в дизайн: нет поля «сравнение с прошлой неделей»
  (это субъективное суждение — за человеком). Нет публичного view по умолчанию.

**Template 2: Habits Tracker**
- Spec: минималистичный tracker. Не Notion — может быть простая таблица или бумага.
  Принцип: отслеживать 3-5 привычек max (anti-inflation). Checkmark или крест, без оценок.
- Anti-pattern: tracker не должен вызывать guilt за пропущенный день. Дизайн: нет
  «streak counter» как primary view. Есть «% за месяц» как более мягкий индикатор.
- Weekly reset: в конце недели смотришь на паттерн, не на конкретные дни.

**Template 3: Skill Tree Visualization**
- Spec: Miro template с pre-built structure. 3 уровня глубины. 5-7 основных веток.
  Цветовая кодировка: explored / learning / mastered (не %). Связи между ветками.
- Customization guide: как адаптировать под свою область мастерства.
- Anti-pattern: нет «глобального % завершения» — только per-branch depth visibility.

**Template 4: Achievement Schema (meaningful markers)**
- Spec: YAML/Notion schema для создания собственных achievements.
  Поля: title / description / criteria (что именно произошло) / evidence (ссылка на артефакт) /
  date / domain / significance (почему это важный порог — свободный текст).
- Принцип: каждое achievement = реальный событие в мире (сделал X, опубликовал Y, прошёл Z).
  НЕ «потратил 100 часов» (vanity) — а «впервые решил задачу такого типа самостоятельно».
- Стартовый набор meaningful milestones (примеры, не prescriptive):
  «первая публикация», «первый платный клиент», «первый mentor», «первая статья на иностранном языке».
- Anti-pattern: нет автоматических badges за активность (time spent / logins / streaks).
  Badges только за meaningful progression markers.

**Template 5: Quest Card**
- Spec: single-page Notion template для одного квеста.
  Поля: Quest Name / Why This Quest? / Challenge Statement / Milestones (3-5) /
  Final Artifact / Stakeholders (кто заинтересован) / Status / Log (append-only).
- Quest framing reminder встроен в template: «квест» начинается с вопроса, не с задачи.

**R12 check (per artefact):** Templates — самый «безопасный» artefact, но именно поэтому
важно встроить anti-pattern protection в дизайн. Шаблон, который легко дрейфует в dark pattern
при неаккуратном использовании, должен иметь явные guardrails.

## §H — FAQ (10-30 вопросов)

**Тип:** честные ответы на реальные вопросы. Не marketing FAQ — диалог с критиком.

**Spec для ответов:** прямые, без defensiveness, без spin. Если вопрос указывает на реальный
риск — ответ признаёт риск и объясняет mitigation, а не отрицает.

**Список вопросов (spec — финальные формулировки и ответы за Русланом):**

1. «Это не то же самое, что залипалово в играх?»
   — Ответ должен: признать сходство механик → объяснить разницу в цели → описать конкретные
   design decisions, которые предотвращают залипание.

2. «Не манипуляция ли это?»
   — Ответ должен: дать честное определение манипуляции (влияние без согласия) → показать,
   где геймификация может быть манипуляцией → объяснить, почему opt-out и intrinsic focus
   это меняют.

3. «Зачем статистика? Я и так знаю, как работаю.»
   — Ответ должен: признать, что не всем нужна статистика → объяснить, что dashboard решает
   конкретную проблему (visibility роста), а не «нужен всем» → предложить попробовать 2 недели.

4. «Я ненавижу стрики. Почему у вас их нет?»
   — Ответ должен: объяснить, почему streak как primary mechanic = dark pattern → что вместо
   этого (rolling % за период).

5. «Badges и achievements — это детство.»
   — Ответ должен: согласиться, что плохо реализованные badges = детство → объяснить
   meaningful markers vs vanity badges → дать пример реального achievement.

6. «Skill tree — это очень сложно для начала.»
   — Ответ должен: согласиться → предложить минимальную версию (3 темы, 2 уровня).

7. «Как это связано с AI?»
   — Ответ: AI = учёт и напоминание, человек = смысл и цель. Ref §K.

8. «Что если я забуду заполнять dashboard?»
   — Ответ: это нормально. Gap = информация (что мешает). Система не должна вызывать guilt.

9. «Нет ли здесь риска превратить жизнь в работу?»
   — Ответ должен: честно признать риск → объяснить design principle «геймификация следует
   за смыслом, не создаёт его» → предупредить о признаках over-gamification.

10. «Почему нет leaderboard?»
    — Ответ: leaderboard = social comparison → reduces intrinsic motivation (SDT research).
    Coordination без ранжирования → Schelling focal points.

11. «Что с токенами / виртуальной экономикой?»
    — Ответ должен: объяснить R12 принцип → конкретно: что токены могут делать, что — нет →
    fork-and-leave без штрафа.

12. «Как не превратить quest framing в самообман?»
    — Ответ должен: согласиться с риском → объяснить критерии реального квеста (реальный вызов,
    реальный артефакт) → предупредить о fake quest pattern.

**+ 10-15 дополнительных вопросов** охватывающих: сравнение с Duolingo/Habitica/Bereal,
B2B применение (геймификация для команды), отношения skill tree и образования #7,
virtual economy и R12 anti-extraction, AI и манипулятивный nudge.

**R12 check (per artefact):** FAQ — низкий риск, высокая ценность. Честный FAQ снижает
manipulation risk самого направления: аудитория видит, что авторы думали о рисках.

## §J — R12 Paired-Frame Check (CORE — PRIMARY SURFACE)

**Тип:** ОБЯЗАТЕЛЬНЫЙ audit doc. Все 5 influence experts AUTO-FIRED. Это не дополнение к §A-§L,
это фундамент всего направления.

**Структура §J:**

### J.1 Полный список anti-patterns (явный запрет)

Каждый из следующих паттернов **запрещён** в любом Jetix-gamification artefact:

1. **Addictive engagement loops** — цикл action→reward→action без паузы и рефлексии.
   Распознавание: «не могу остановиться, потому что потеряю прогресс».

2. **Variable-reward exploitation (slot machine mechanics)** — случайное вознаграждение
   с непредсказуемым расписанием. Самый сильный behavioural hook.
   Распознавание: «не знаю, когда получу следующую награду — но жду».

3. **Sunk-cost mechanics** — дизайн, при котором уход «дорого стоит» (потеря накопленного).
   Распознавание: «остаюсь, потому что уже столько вложил».

4. **Manufactured FOMO/urgency** — искусственное ограничение («осталось 3 места», «акция
   заканчивается») без реального ограничения.
   Распознавание: urgency не объяснима реальными причинами.

5. **Social-pressure streaks** — streak как primary mechanic, с guilt/shame за разрыв.
   Распознавание: пропуск одного дня чувствуется как провал.

6. **Pay-to-win virtual economy** — доступ к важным функциям/статусу через платёж.
   Распознавание: деньги → преимущество в мастерстве (а не просто удобство).

7. **Vanity metrics as goals** — оптимизация показателей, которые выглядят как прогресс,
   но не коррелируют с реальным ростом.
   Распознавание: «мои цифры растут, но я не чувствую, что становлюсь лучше».

8. **Dopamine-not-flow design** — система оптимизирована для быстрых пиков удовольствия,
   а не для состояния потока (challenge ≈ skill).
   Распознавание: много маленьких наград вместо редких значимых.

### J.2 Intrinsic motivation test per mechanic (SDT framework)

Для каждого механика перед внедрением — 3 вопроса:

**Autonomy check:** Может ли человек отключить/изменить этот механик без потерь?
Если нет → не включаем или перепроектируем.

**Competence check:** Этот механик отражает реальный рост компетенции, или только активность?
Если только активность → это vanity metric, не meaningful marker.

**Relatedness check:** Если механик социальный — он соединяет людей или создаёт сравнение?
Соединение (coordination, solidarity) → OK. Сравнение (ranking, shame) → не включаем.

### J.3 8 вопросов для audit любого нового механика

Перед добавлением любого нового механика в систему — пройти все 8:

1. Какой тип мотивации этот механик усиливает — intrinsic или extrinsic?
2. Что происходит, если человек его отключает? Есть ли потери, штрафы, давление?
3. Отражает ли этот механик реальный рост или только активность в системе?
4. Создаёт ли этот механик зависимость от платформы или независимость от неё?
5. Если механик социальный — соединяет или сравнивает?
6. Есть ли в механике variable reward schedule? Если да — зачем, и можно ли его заменить?
7. Что чувствует человек, когда выходит из системы через месяц? Стыд / потеря / тревога — стоп.
8. R12 anti-extraction: не ли этот механик извлекает value из участника сверх agreed share
   (time, data, attention, money)?

### J.4 Opt-out verification checklist

Для каждого публичного продукта (курс, dashboard, система) — проверить:

- [ ] Все данные участника доступны для экспорта в любой момент без потерь качества.
- [ ] Отписка/выход возможен без ритуалов «вы уверены?» × 3.
- [ ] После выхода: нет писем «вы нас покидаете», нет streak-shame.
- [ ] Накопленные достижения/artifacts принадлежат участнику, не платформе.
- [ ] Нет «бонусов за возвращение» — они создают psychological pull против свободного выбора.

### J.5 Influence-ethics expert verdict

**Propaganda-expert (framing risk):** Видео §B и 1-pager §A используют emotional hooks.
Это само по себе не манипуляция — но hook должен работать через recognition (узнавание),
а не через manufactured fear или social proof overload. Проверить: нет ли в материалах
«все успешные люди делают X», «не хочешь отставать?», «последний шанс».

**NLP-expert (verbal mechanics):** Язык материалов должен избегать presuppositions,
которые встраивают принятие тезиса до его обоснования. Пример красной флажки:
«когда ты начнёшь геймифицировать свою жизнь...» (встраивает «ты начнёшь» как данность).
Лучше: «если ты решишь попробовать...».

**Recruitment-dynamics-expert (cohort mechanics):** Schelling coordination §5 sub-area —
это recruitment mechanic. Soft focal points работают через social visibility и peer effect.
Граница: focal point = voluntary opt-in, видимый другим только с согласия участника.
Нет «позора перед группой» за выход из спринта.

**Gamification-engagement-expert (механики):** PRIMARY DRIVER проверяет каждый механик
по J.2 и J.3 перед включением в spec. Ни один механик не попадает в §F templates
без прохождения J.3 audit.

**Summary verdict (per influence-ethics-expert):** Направление #15 является R12-compliant
при условии: (1) §J является частью любого operational product, не только этого spec;
(2) anti-pattern list §J.1 явно включён в каждый шаблон §F; (3) Founder-as-Exhibit §I
включает failure stories; (4) все материалы проходят J.3 audit перед публикацией.

## §K — AI tooling integration

**Тип:** spec для AI-роли в геймификации жизни. R1 surface для финальных формулировок.

**Принцип (O-182 alignment):** AI tracks, human decides. AI = операционная рутина (учёт,
статистика, напоминание). Человек = смысл (что отслеживать, зачем, что значит прогресс).

**Что AI может делать в этой системе:**

1. **Автоматический учёт** — собирать данные из calendar, time tracker (Toggl), git commits,
   Notion updates → агрегировать в Life Pulse без ручного ввода.

2. **Weekly review summary** — генерировать еженедельный отчёт: что сделано, где паттерны,
   без оценок и суждений (только факты + вопросы для рефлексии).

3. **Skill tree updates** — предлагать обновление skill tree на основе завершённых проектов/
   курсов (не автоматически — предложение для принятия человеком).

4. **Quest design assistant** — помогать формулировать challenge statement и milestones
   для нового квеста (brainstorm, не предписание).

5. **Anti-pattern alert** — если система замечает признаки dark pattern drift (dashboard
   проверяется >10 раз в день, участник не открывал систему 21 день после streak) — один
   тихий alert. Не повторяющийся. Не guilt-inducing.

**Что AI НЕ должен делать (explicit запрет):**

- НЕ ставить цели за человека (даже «рекомендованные» — это nudge).
- НЕ отправлять reminder без explicit opt-in для каждого типа reminder.
- НЕ оценивать прогресс как «хороший/плохой» — только факты.
- НЕ использовать social proof в nudge («другие члены уже завершили X»).
- НЕ создавать urgency («заполни отчёт сегодня, пока не поздно»).
- НЕ агрегировать данные за пределами явно согласованного scope (R12 anti-extraction).

**AI policy template (встроить в §F шаблоны):**
Каждый участник при setup формулирует AI policy для своей системы:
«AI может делать: [X, Y, Z]. AI не может делать: [A, B, C]. AI alerts мне только если: [условие]».
Это не административная бюрократия — это инструмент осознанного opt-in.

**R12 check (per artefact):** AI — самый высокий скрытый risk в этом направлении.
Именно через AI nudge тёмные паттерны входят незаметно (personalized variable reward,
social comparison через «другие участники», urgency через timing). Mitigation: AI policy
обязателен, не опционален. Scope AI явно ограничен в каждом шаблоне.

## §L — Partner-extension hook

**Тип:** spec для партнёров/кланов, которые добавляют свою геймификацию под R12 floor.

**Принцип:** Jetix определяет R12 floor (минимальные требования anti-dark-pattern compliance).
Партнёры могут строить поверх, но не ниже. Anti-dark-pattern compliance — MANDATORY, не optional.

**R12 floor для партнёров (binding):**

1. **Anti-pattern запрет абсолютный:** ни один партнёр не может внедрять паттерны из §J.1
   в своей геймификации под брендом Jetix или в клане Jetix. Нарушение = выход из кооператива.

2. **Opt-out без штрафа:** любой участник любого клана может выйти из любой partner-gamification
   системы без потери своего мастерства, достижений и накопленных артефактов.

3. **Data sovereignty:** данные участника, собранные партнёром, не могут использоваться
   за пределами явно согласованного scope без нового согласия участника.

4. **Transparency requirement:** партнёр публикует список всех механик своей системы +
   anti-pattern audit результат (по J.3) — доступный для любого участника до вступления.

**Процесс добавления partner-gamification:**

- Step 1: партнёр проводит J.3 audit для каждого механика (самостоятельно или с помощью
  Jetix influence-ethics review).
- Step 2: результаты audit → shared doc с Jetix + участниками клана.
- Step 3: участники клана голосуют (opt-in, не default-on). Порог: 2/3 согласны → запуск.
- Step 4: 30-дней trial с weekly check-in (есть ли drift в dark patterns?).
- Step 5: post-trial review. Сохранить / изменить / отключить.

**Fork-and-leave protection:** партнёр не может создавать механики, которые делают уход дороже
стандартного. R12 principle: «members can fork-and-leave without penalty».

**Template для partner compliance declaration:**
Партнёр заполняет декларацию: список механик → тип мотивации (intrinsic/extrinsic) →
J.3 audit results → opt-out design → data scope → update schedule (пересмотр раз в квартал).

**R12 check (per artefact):** Partner hook — системный риск. Один партнёр с dark patterns
может подорвать доверие ко всей системе. Mitigation: R12 floor binding + transparency requirement +
participant opt-in (не default-on). Нарушение = exit. Без компромисса.

## §V4-delta: что нового по сравнению с V3

**V3 не содержало Direction #15** — это полностью новое направление V4.

**Что изменилось в системе с появлением #15:**

1. **Мастерство получило engagement-обёртку.** Direction #13 Мастерство описывает «что».
   Direction #15 Геймификация описывает «как это ощущается каждый день». Без #15 Мастерство
   — абстракция. С #15 — живая практика.

2. **Сеть получила social coordination механику.** Schelling coordination (#15 sub-area 5)
   даёт Сети (#16 Хакатоны, #12 Workshop) конкретный инструмент мягкой координации без
   давления и без leaderboards.

3. **V4 стал «движком».** V3 = структура. V4 = структура + движок (#16 events) + переживание
   (#15 gamification). Направление #15 закрывает вопрос «зачем это всё каждый день» —
   человек видит свой рост, чувствует прогрессию, понимает куда идёт.

4. **R12 surface изменился.** В V3 R12 был главным образом про экономику (#8 R12 Обещание,
   #4 Заработок). В V4 #15 добавляет психологический R12 (manipulation через mechanics).
   Это расширяет R12 discipline с экономической плоскости на поведенческую.

## Итоговая позиция (для читателя spec)

Direction #15 — Геймификация жизни — это не «сделать жизнь игрой». Это инструменты сделать
рост видимым, прогрессию ощутимой, и смысл — измеримым каждый день.

Главная дисциплина: геймификация следует за смыслом, не создаёт его. Если в жизни нет смысла,
никакая система статистики его не добавит. Если смысл есть — инструменты делают его видимым.

R12 central: каждый механик геймификации — это точка, где забота о человеке может легко
соскользнуть в эксплуатацию внимания. Разница тонкая, но принципиальная.
Jetix держит эту линию явно: через §J audit, через anti-pattern design в шаблонах,
через AI policy, через R12 floor для партнёров.

Метрика успеха direction #15: участник, который использует инструменты полгода, чувствует,
что вырос — и чувствует, что это его рост, а не рост в системе.
