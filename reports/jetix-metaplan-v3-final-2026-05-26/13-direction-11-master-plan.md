> **Lens:** ROY systems-expert + investor-expert + philosophy-expert. F2-F3 derivative,
> интегрирует только существующий substrate. R1 поверхностно (формулировки = Ruslan-authored).
> R11 STRICT: спецификации артефактов, НЕ публичный текст. R12 paired-frame STRICT.

## Вводный контекст: зачем Master Plan существует как отдельное направление

Jetix — мега-мастерская. Три грани: 🏛️ Мастерская · 🎯 Мастерство · 🌍 Сеть.

Core statement (R1, формулировка = Ruslan-authored): *«Jetix — мега-мастерская мирового уровня:
место, где люди становятся мастерами в эпоху AI, вместе двигают фронтир и не дают системе
доить или запирать себя.»*

**Зачем нужен отдельный Master Plan?** Direction #6 «Видение» — это витрина (≤2K, короткое тело
Vision'а: зачем / для кого / won't-compromise). Direction #11 — другой жанр. Это long-form
манифест с дугой, у которого есть части, публичная граница, gated-зона и временны́е горизонты.
Разница принципиальная: #6 объясняет что такое Jetix, #11 объясняет куда идёт и почему именно
так, а не иначе.

**Жанровые референсы** (из метаплана-v2 §9):
- Tesla Master Plan (Elon Musk, 2006/2016) — структура 4 частей с горизонтами, честное
  признание что «мы ещё не там», анонс будущего как проект а не обещание.
- OpenAI Charter — safety-в-топе, миссия первична, коммерция вторична; прецедент честного
  разговора о рисках.
- Berkshire письма-манифесты (Buffett) — тон без маркетинга, цифры как факты, «мы долго
  думаем и делаем то что говорим».

**GAP (из таблицы 14 направлений):** ❌ create — жанра с нуля нет. Весь substrate есть
(Foundation, PLATFORM-LIFECYCLE, R12, member-journey), но собственного long-form документа
«Master Plan» пока не существует. Это самый высокорычажный create-gap: публичный документ
такого жанра создаёт доверие, разгружает все остальные артефакты от объяснений «почему gated»
и «что будет в Scale», и становится ссылочным документом для всего сообщения.

**Почему public/gated разбивка здесь критична.** Tesla/Mondragón-урок: публичные обещания
больших планов без готового продукта = hype-trap и потеря доверия при каждой задержке. Решение:
Parts 1-2 публичные (Build/Run — уже работает или будет работать в ближайший горизонт),
Parts 3-4 gated (Scale/Mature — только для партнёров и участников, когда реально достижимы).
$1T и Network State = gated hard (trigger: $100K + 20+ мастерских + published artifact — per
memory `project_balaji_outreach_target`).

## §A Главный документ «Master Plan»

**Формат:** 1-pager (one-liner + краткое тело) + long-form (~4K основной текст, 4 части).

### Структурная спецификация 1-pager

Один экран. Функция: якорь для всех остальных коммуникаций. Что должно быть:
1. One-liner (R1 — формулировка Ruslan-authored, финал за ним)
2. Три грани одной строкой каждая (🏛️ / 🎯 / 🌍)
3. 4 части с горизонтами — без цифр участников (сценарные), с маркерами public/🔒gated
4. Won't-compromise одним абзацем

Чего НЕ должно быть: числа доходов / Network State / $1T / обещания сроков как гарантий.

### Структурная спецификация long-form

Длина: ~4K слов основного текста. Тон: прямой, без воды (Berkshire-референс), с признанием
неопределённости там где она есть (Tesla-референс: «возможно это звучит безумно, но вот
почему мы считаем это достижимым»).

**Архитектура документа:**

**Преамбула / Why this document:** объяснение жанра — это не промо и не roadmap с датами.
Это описание дуги: откуда мы идём, через какие этапы, с каким принципом на каждом. Читатель
должен понять: Jetix — не очередная платформа, а structural bet на то как должно выглядеть
мастерство в AI-эру.

**Part 1 Build — Виртуальная мастерская** (PUBLIC, ~6 мес горизонт)
- Что строим: первая версия пространства онлайн — не курсы, а среда для работы руками
- Кто крутит маховик на этом этапе: основатель, вручную
- Что должно произойти до выхода из Build: Build exit ≥80% выходов (маховик работает без
  постоянного ручного толчка)
- Чего НЕ делаем: не масштабируем до подтверждённой петли, не арендуем физ. место, не
  обещаем когорты
- Триггеры перехода Build→Run (публично названные, без конкретных цифр как обязательств):
  маховик работает, петля подтверждена, минимум несколько платящих участников

**Part 2 Run — Hybrid + первое место** (PUBLIC, ~1-2 года горизонт)
- Что меняется: появляется физическое арендованное место (Берлин, anti-WeWork — аренда,
  не покупка), гибридный формат, партнёры со-создают курсы, несколько клетов-кланов
- Кто крутит маховик: основатель + партнёры + петля обратной связи
- Финансово (сценарно, не обещание): ступень 5 ≈ €1500/мес; revenue-share 75/25 с партнёрами
- Принцип: Run начинается только при подтверждённой петле. Если петли нет — остаёмся в Build.
- Триггеры Run→Scale: кооперативная структура подтверждена, несколько автономных клетов,
  внутренние стандарты переносимы без основателя

**Part 3 Scale — 3-10 мастерских + mesh** (🔒 GATED — партнёры и участники уровня Practitioner+)
*Зачем gated:* на этом этапе появляются смарт-контракты, кооперативный legal, mesh-топология.
Публичное раскрытие до достижения = hype. Gated = честность, не секретность.
- Mesh, не star: равноправные cells, координация через Charter + R12 + общая wiki, не приказ
- Кооперативный legal: worker-owner, 5:1 ratio, fork-and-leave механика
- R12-escalation: 🔴 смарт-контракты 4 класса

**Part 4 Mature — 30-100+ network, member-owned** (🔒 GATED/internal — только участники;
Network State как возможный сценарий gated-hard, trigger: $100K + 20+ мастерских + artifact)
- Сеть сама крутит маховик: основатель = один из хранителей, не центр
- Member-owned: кооператив, каждый участник = co-owner с правами и обязанностями
- R12-escalation: 🔴+ механический контроль + внешний аудит
- Что такое Network State здесь (gated-only): не государство, не утопия — сеть людей с
  общей идентичностью и инфраструктурой, способная к самоуправлению (Balaji-референс,
  gated trigger не достигнут)

**Closing:** почему этот план настоящий. Не «мы обещаем X к году Y», а «мы зафиксировали
принципы, которые не нарушим ни ради скорости роста, ни ради инвесторских требований».
R12 как стержень: нет этапа где мы начнём доить или запирать.

## §B Видео (manifesto long-form)

**Формат:** видео 15-25 минут, канал YouTube / основная площадка.

**Структурная спецификация:**

**Hook (0-90 сек):** Провокационный вопрос или наблюдение, которое ломает ожидание. Не
«мы создали платформу», а что-то про образование, AI-эру или мастерство, что заставляет
остановиться. Формулировка = Ruslan-authored.

**Scene 1 — Зачем (почему мир нуждается в Мастерской):** gap в 3 пунктах — AI ломает
образование, tacit knowledge негде передавать, платформы доят. Тон: точный диагноз без
излишнего драматизма.

**Scene 2 — Part 1 Build:** что строим прямо сейчас, что уже есть (честно), чего нет ещё.
Показываем маховик и принцип его переключения. Демонстрирует: мы в процессе, не в мечтах.

**Scene 3 — Part 2 Run:** куда идём в ближайший горизонт. Физическое место как следствие
работающей петли, не как само-цель. Берлин — первый кандидат.

**Scene 4 — Почему 3/4 gated:** объяснение без апологетики. «Мы не прячем план — мы не
продаём мечту. Scale и Mature откроем когда будем там.» Это честность, а не секретность.

**Scene 5 — Member journey:** Visitor → Apprentice → Practitioner → Master → Master of
Masters. fork-and-leave звучит первым (anti-cult сигнал).

**Scene 6 — Won't-compromise:** R12 / 5:1 / fork-and-leave / workshop ≠ bootcamp /
mastery ≠ диплом / mesh ≠ star. Без пафоса.

**Closing:** призыв к действию — Practitioner-уровень или тестер, не подписка и не мечта.
Формулировка = Ruslan-authored.

**Технические требования по жанру:** монтаж без инфографики-ради-инфографики, прямая речь
без телесуфлёра (или с, но без ощущения), b-roll умеренно. Длина = не менее 12 минут (жанр
manifesto требует глубины).

**Структурная спецификация (chapters):**

4. **Chapter 3 — Что такое Сеть** (🌍) — mesh-топология, кооператив, fork-and-leave, R12
5. **Chapter 4 — Part 1 Build** — развёрнутая история + принципы + чему научил Build

этапа. Это структурное решение, не маркетинговый ход.

**Тон:** Berkshire-манифест (факты, признание неопределённости, долгий горизонт без сроков
как гарантий). Объём: ~50-80K слов для full-arc.

## §E Инструкции/SOP — Public/Gated Reveal Protocol

**Применимость:** для этого направления не SOP в смысле «как сделать X», а протокол раскрытия
информации. Это операционно важный артефакт.

**Gated Reveal Protocol — структура:**

**Принцип:** информация о Parts 3-4 раскрывается участникам и партнёрам, но не публикуется
открыто, пока соответствующий этап не достигнут. Это не засекречивание — это честность.
Публично мы говорим: «Parts 3-4 существуют, они gated, вот почему».

**Уровни доступа:**
- Public: Parts 1-2 (всё описанное в §A выше), one-liner, 3 грани, won't-compromise
- Practitioner+: Part 3 в деталях (после вступления и первого результата)
- Master+: Part 4 в деталях (после устойчивого участия ≥3 мес)
- Internal (основатель + Stewards): Network State как сценарий (gated-hard trigger = $100K
  + 20+ мастерских + published artifact)

**Что говорить при вопросе «а что в Part 3/4?»:** прямой ответ: «Там масштаб и кооперативная
структура. Мы не раскрываем детали пока не там, потому что обещать то что ещё не создано —
это hype. Когда будем там — откроем». Это не уклонение, это стандарт.

**Что НЕЛЬЗЯ делать (anti-patterns):**
- Намекать на Network State / $1T в публичных материалах
- Создавать ощущение «если подпишешься, узнаешь секрет»
- Использовать gated как инструмент продаж («exclusive access»)
- Называть сценарные цифры как прогнозы или обещания

**Триггеры для раскрытия следующего уровня:** решение = Ruslan + Stewards коллегиально.
AI мониторит прогресс по индикаторам (см. §K), Ruslan принимает gate-переход.

## §G Презентация (10-20 slides)

**Формат:** 15 слайдов, Google Slides / Notion-embed. Функция: объяснить дугу Jetix за 10-15
минут (для партнёрских переговоров, онбординга тестеров, пресс-материалов).

**Структура слайдов:**

1. **Cover:** One-liner (R1) + 3 грани иконками
2. **Диагноз (почему сейчас):** 3 проблемы — AI/образование / tacit gap / extraction
3. **Что такое Мастерская:** 3 грани развёрнуто, 1 слайд на каждую (слайды 3-5)
4. **Маховик:** кто крутит на каждом этапе (Build/Run/Scale/Mature) — визуально
5. **Timeline (4 части):** горизонтальная шкала, public/🔒gated разбивка, без конкретных
   дат как обязательств
6. **Member journey:** Visitor → ... → Master of Masters, fork-and-leave звучит явно
7. **Won't-compromise:** R12 / 5:1 / fork-and-leave / won't-do список
8. **Где мы сейчас (Точка А):** честно — что есть, чего нет
9. **Следующий шаг (Точка B):** Build exit триггеры
10. **Призыв к действию:** не подписка — конкретная роль (тестер / методолог-партнёр / ...)

**Дизайн-принцип:** минимум инфографики, максимум содержания. Если слайд можно убрать без
потери смысла — убрать. Цветовая схема = 4 этапа: 🟢 Build / 🔵 Run / 🟡 Scale / 🔴 Mature
(соответствует R12-escalation сигналам).

**Anti-pattern слайдов:** нет слайда «рынок на $X трлн», нет «наши конкуренты», нет «join
now». Жанр — манифест, не питч-дек.

## §H FAQ (10-30 вопросов)

Честные ответы на неудобные вопросы. Принцип составления: если вопрос задавали хоть раз —
он в FAQ. Если вопрос каверзный — тем более.

**Структура FAQ:**

**Блок 1 — Про план и жанр:**

*Q: «$1T — это серьёзно или маркетинг?»*
Это gated-сценарий для Parts 3-4, не публичное обещание. Мы не раскрываем его пока не там.
Если вы читаете это в публичных материалах — значит что-то пошло не так.

*Q: «Когда будет оффлайн-место?»*
Part 2 Run. Первое место = арендованное, Берлин, только при подтверждённой петле. Дата не
фиксирована — это trigger-based, не calendar-based.

*Q: «Зачем Part 3/4 gated?»*
Чтобы не обещать то чего нет. Tesla-принцип: пиши о том что делаешь, не о том что мечтаешь.
Когда будем там — откроем.

*Q: «Это культ?»*
Нет. Признак: выход звучит первым. Fork-and-leave anytime, без штрафов, забираешь долю.
Культ удерживает выход. Мы строим обратное.

**Блок 2 — Про участие:**

*Q: «Сколько стоит?»*
Apprentice (1-4 ступени) — free. Practitioner ступень 5 — сценарно ~€1500/мес. Цифра
сценарная, уточняется до начала Run.

*Q: «Что значит fork-and-leave?»*
В любой момент можно уйти, забрав то что заработал (знания, проекты, долю при кооперативе).
Без потери лица, без штрафов, без осуждения.

*Q: «Как стать партнёром?»*
Партнёр Build сейчас = методолог или тестер. Не YouTuber, не «просто верю в идею». Конкретная
роль на конкретном этапе. Форма для связи = [placeholder].

**Блок 3 — Про Network State / $1T:**

*Q: «Что такое Network State?»*
Balaji Srinivasan-концепция: сеть людей с общей идентичностью, инфраструктурой, самоуправлением.
Для Jetix это gated-сценарий Part 4, trigger не достигнут. Мы не говорим об этом публично
пока не там.

*Q: «Почему вы вообще думаете о $1T?»*
Потому что Mondragón начинал с маленькой мастерской в Стране Басков и вырос в 81 кооператив.
Масштаб возможен, если стандарты не нарушаются на каждом шаге. $1T = gated longshot, не обещание.

**Блок 4 — Про устройство:**

*Q: «Что такое R12?»*
Правило: никакой механизм не может выкачивать больше согласованной доли из участников и не
может удерживать их от выхода. Юридически реализуется через кооперативную структуру, 5:1 ratio
и fork-and-leave механику.

*Q: «Кто принимает решения?»*
Build: основатель. Run: основатель + Stewards. Scale+: Stewards + участники через Charter.
Mature: сеть сама. Переход = trigger-based, не произвольный.

*Q: «Как Jetix зарабатывает?»*
Ступени участия (Practitioner+), партнёрские revenue-share (75/25), будущие кооперативные
механики. Подробно в #3 Бизнес и #4 Заработок.

## §I Worked Examples (3-5)

Примеры показывают как абстрактная дуга выглядит в реальных переходах. IP-1 STRICT — имена
ролей, не конкретных людей.

**Example 1 — Точка А → Точка В (Build exit)**

*Ситуация:* Substrate готов (4 LOCKED документа, Foundation, 180 контактов, 17 агентов),
но видео нет, Wave 1 не отправлен, петля не подтверждена. Маховик = основатель вручную.

*Что происходит в Master Plan:* это середина Part 1 Build. Правильное место. Build exit
требует ≥80% выходов без постоянного ручного толчка. Пока петли нет — остаёмся здесь.

*Переход:* когда видео A/B/C вышли, Notion 5 баз внедрены, Charter проверен, Wave 1 отправлен,
1-2 T1 партнёра + 3-5 тестеров активны, юр + счёт оформлены — маховик начинает крутиться
сам. Это Build exit. Тогда можно думать о Run.

**Example 2 — Точка В → Точка С (Build→Run)**

*Ситуация:* Build exit пройден, первые тестеры прошли Apprentice-ступени, один из T1-партнёров
ведёт когорту. Выручка начинается. Возникает вопрос: арендовать место?

*Master Plan читает:* Part 2 Run начинается только при подтверждённой петле. Петля = минимум
одна когорта платит, партнёр со-создаёт контент, выручка покрывает операционные расходы.
Место = арендованное, не купленное (anti-WeWork). Берлин — первый кандидат.

*Anti-триггер:* если петли нет — место не арендуется, даже если есть инвестор готовый дать
деньги на аренду. Это won't-compromise.

**Example 3 — Точка С → Точка D (Run→Scale)**

*Ситуация (проекция):* 3+ когорты прошли, €10K/мес, 1-й клан формируется, T4-партнёры
появляются. Основатель думает: пора переходить в Scale?

*Master Plan читает:* переход Run→Scale требует клетовой автономии — cells крутят свой
маховик без основателя. Кооперативная структура подтверждена legально. Внутренние стандарты
переносимы без основателя. Пока хоть одного из этого нет — Run продолжается.

*R12-escalation:* переход в Scale = переход в 🔴 зону. Смарт-контракты 4 класса должны быть
проверены до начала Scale, не после. Это gated-артефакт (Part 3 раскрывается Practitioner+).

**Example 4 — Member journey через Master Plan**

*Visitor:* смотрит видео (§B), читает 1-pager (§A). Понимает what и why. Принимает решение
пробовать — или нет.

*Apprentice (ступени 1-4, free):* первый результат руками. Знакомится с методом, мастерской,
форматом. В конце ступени 4: продолжать как Practitioner или fork-and-leave. Обе опции
равноценны и звучат так.

*Practitioner:* собственные проекты, роли вбок (Investor / Consultant), доступ к Part 3
gated-контенту Master Plan.

*Master:* ведёт когорты, mastery-момент публичен. Может стать Steward. Доступ к Part 4
gated-контенту (Mature).

**Example 5 — Партнёр строит Clan Master Plan**

*Ситуация:* T1-методолог хочет запустить свой клан внутри Jetix-mesh. Берёт шаблон §F.

*Что происходит:* заполняет own Part 1-2 (публично), own Part 3-4 (gated в своём клане).
Jetix проверяет только R12-compliance. Содержание = автономия клана. fork-and-leave работает
в обе стороны: клан может уйти из Jetix-mesh с атрибуцией, но без штрафов.

## §J R12 Paired-Frame Check

**8 вопросов R12 per Master Plan (anti-extraction / anti-manipulation check):**

**1. Не доим ли мы ожиданием?**
Проверка: нет ли в документе формулировок типа «подпишись и узнай секрет Parts 3-4».
Anti-pattern: gated как инструмент продаж. Стандарт: gated = честность, прямо объясняемая.

**2. Не запираем ли мы людей?**
Проверка: fork-and-leave звучит явно в каждом ключевом документе (§A, §B, §G, §H).
Anti-pattern: умолчание о выходе, пока человек не в системе. Стандарт: выход звучит первым.

**3. Не манипулируем ли сценарными цифрами?**
Проверка: все цифры (€/мес, участники, сроки) маркированы как «сценарно», «проекция»,
«не обещание». Anti-pattern: «вы будете зарабатывать X». Стандарт: «сценарно — X».

**4. Не создаём ли hype через gated?**
Проверка: Parts 3-4 не описаны намёками в публичных материалах. Anti-pattern: «если хочешь
знать что дальше...» как крючок. Стандарт: «Parts 3-4 gated, вот почему — честно».

**5. Network State / $1T — нет ли в публичном тексте?**
Проверка: grep по всем публичным артефактам (§A публичная часть, §B, §G, §H). Должно быть
0 совпадений. Anti-pattern: упоминание даже намёком в public-контексте. Trigger не достигнут.

**6. Не претендуем ли AI на стратегию?**
Проверка: все формулировки One-liner / Part-descriptions / won't-compromise = Ruslan-authored
или явно помечены «формулировка = Ruslan-authored». AI = spec-builder, не автор стратегии.

**7. Не используем ли массовое рекрутирование раньше времени?**
Проверка: призыв к действию в §B/§G — конкретная роль на Build (методолог, тестер),
не «присоединяйся к движению». Anti-pattern: Scale-риторика в Build-материалах.

**8. R12-escalation растёт с этапом — зафиксировано ли?**
Проверка: Build 🟢 / Run 🟡 / Scale 🔴 / Mature 🔴+ явно в §A Part descriptions и §I
Examples. Anti-pattern: говорим о смарт-контрактах как о «уже работает» до Scale.

**Anti-patterns Master Plan жанра:**

- **Hype:** «мы изменим мир» без деталей как → заменить на конкретные механики (5:1, mesh,
  fork-and-leave)
- **Обещания как факты:** «к 2027 году у нас будет X» → заменить на «сценарно» или убрать
- **Network State как vapourware:** упоминание без достигнутого trigger → не упоминать публично
- **Cult-framing:** «нас могут понять только избранные» → запрещено, выход звучит первым
- **Exclusivity-trap:** gated = привилегия, а не честность → переформулировать

## §K AI Tooling Integration

**Принцип (per O-182):** AI отслеживает прогресс по этапам (рутина), человек принимает
gate-переход (решение). AI ≠ триггер перехода, AI = мониторинг и алерт.

**Что AI делает в контексте Master Plan:**

*Build progress monitoring:*
- Отслеживает checklist Build exit (видео вышли / Notion внедрён / Wave 1 отправлен /
  T1-партнёры / юр-счёт) — статус в daily/weekly отчёт
- Алертит при появлении anti-триггеров (выгорание-сигналы / methodology drift / R12-нарушение)
- Проверяет что новые артефакты не нарушают public/gated разбивку (grep по Network State /
  $1T в публичных файлах)

*R12-compliance monitoring:*
- Автоматически проверяет новые draft-материалы по 8 вопросам §J
- Если находит anti-pattern — Halt-Log-Alert, не auto-fix (per Part 6b §I.2)
- Escalation: F4 grade ≤5s alert, не silent swallowing

*Clan Master Plan tracking:*
- При появлении нового clan-document проверяет R12-compliance section (§F шаблон)
- Если R12-compliance section отсутствует — алерт Ruslan, не auto-approve

**Что AI НЕ делает:**
- Не принимает решение о gate-переходе (Build→Run / Run→Scale / etc)
- Не формулирует One-liner / won't-compromise (R1 surface)
- Не раскрывает gated-контент автоматически при достижении численных показателей (триггер
  = Ruslan + Stewards коллегиально, AI только сигнализирует что показатели достигнуты)
- Не публикует Master Plan или его части автоматически

**Инструментарий (существующий substrate):**
- `swarm/lib/hooks/` — pre-commit проверка gated-контента в публичных файлах
- `comms/mailboxes/` — алерт-поток от мониторинга
- `swarm/approvals/log.jsonl` — Halt-Log-Alert записи при R12-нарушениях
- Daily plan-of-day интеграция: Build exit checklist в утреннем контексте

## §L Partner-Extension Hook

**Как партнёр строит свой Master Plan под Clan**

**Принцип:** форк Jetix-методологии в автономный клан — это feature системы, не баг. Mesh
работает только если каждый узел реально автономен. Партнёрский Master Plan — документ
этой автономии.

**Скелет Clan Master Plan (4 части, public/gated своё):**

Шаблон (§F) — базовая структура. Ключевые точки расширения:

*Part 1 Build клана (PUBLIC):* специализация клана — чем отличается от других клетов. Это
не «мы лучше», это «мы для другого» (ниша: тип мастерства, регион, язык, инструменты).
Маховик Part 1 клана = кто крутит (обычно основатель клана).

*Part 2 Run клана (PUBLIC):* как клан входит в Jetix-mesh на Run-этапе. Общая wiki,
Charter-compliance, совместные Demo Days / экспедиции — через стандарты, не через приказ
центра.

*Part 3 Scale клана (🔒 GATED — своё решение клана):* клан сам решает что внутри, когда
раскрывать. Jetix не контролирует содержание gated-зоны клана, только R12-compliance.

*Part 4 Mature клана (🔒 GATED/internal):* долгосрочное видение клана в сети. Клан = co-owner
своего пространства. Связь с Jetix-сетью = добровольная, через стандарты.

**R12 Compliance в партнёрском Master Plan:**

Обязательный раздел. Четыре элемента:
1. fork-and-leave клана из Jetix-mesh: как реализовано, без штрафов, с атрибуцией
2. fork-and-leave участников из клана: механика, явно звучит первой
3. 5:1 ratio: подтверждение что revenue split не нарушает максимальный коэффициент
4. extraction policy: что клан не делает (не доит аудиторию, не запирает знания)

**Что Jetix проверяет при принятии Clan Master Plan:**
- R12-compliance section присутствует и заполнена
- fork-and-leave упомянут явно
- gated-контент не продаётся как «секрет» (hype anti-pattern)
- части 3-4 не раскрываются публично до достижения клана соответствующего этапа

**Что Jetix НЕ проверяет:**
- Содержание специализации клана (автономия)
- Конкретные цифры и сроки клана (сценарные — ответственность клана)
- Методологические выборы внутри стандартов

**Выход клана из Jetix-mesh:**
Клан может выйти из сети в любой момент. Берёт свои материалы, участников (их выбор),
методологию — с атрибуцией «основано на Jetix-методологии», без штрафов. Это R12-принцип
applied на уровне кланов.
