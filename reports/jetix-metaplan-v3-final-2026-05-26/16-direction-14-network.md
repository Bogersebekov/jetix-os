# Direction #14 — 🌍 Сеть: портфельный спек

## Введение: грань Foundation — КАК распределено

Jetix — мега-мастерская мирового уровня. Три грани Foundation:
**Мастерская** (🏛️ ГДЕ — пространство, 8 зон, физическая/виртуальная инфраструктура) ·
**Мастерство** (🎯 ЧТО прокачивают — методы, выбор, вечная тренировка, фронтир) ·
**Сеть** (🌍 КАК распределено — **это направление**).

**One-liner R1 (Ruslan-ready):** «Jetix — мега-мастерская мирового уровня: место, где люди становятся мастерами в эпоху AI, вместе двигают фронтир и не дают системе доить или запирать себя.»

Direction #14 «Сеть» — архитектурный ответ на вопрос «как это работает на масштабе»: не одна мастерская, а **mesh из автономных cells** в разных городах и странах, связанных общими стандартами (Charter + R12 + метод-канон + общая wiki), а не центральной властью. Это не просто масштабирование — это **структурный иммунитет** от тех болезней, которые убили большинство сетевых проектов: founder cult, extraction, lock-in, engagement-trap.

**Почему Сеть = primary R12 surface.** Из всех 14 направлений именно здесь концентрируется максимальный соблазн нарушений: масса людей + движение ресурсов + рекрутинг + cross-border power dynamics. Каждый механизм Сети требует explicit R12-проверки. Paired-frame influence-ethics-expert AUTO-FIRED (recruitment-dynamics-expert в паре).

**GAP:** ❌ create. Документы этого направления ещё не существуют — это их spec.

**R11 STRICT:** всё написанное ниже — описания ЧЕГО должен содержать каждый документ, не сами публичные тексты. Смарт-контракты = Scale-phase (gated), не текущий момент. Цифры — сценарные.

## §A — Главный документ «Сеть Мастерских»

### A.1 One-pager (≤500 слов, публичный уровень)

**Аудитория.** Мастер (или будущий) в любом городе, который слышит «Jetix — это сеть» и хочет понять: что это значит практически. Уровень: знает что такое co-working и bootcamp, подозревает что они его доят.

**Формат.** Markdown-источник → лендинг-секция (HTML/React) + PDF-версия (A4). Три варианта заголовка:
- (a) «Сеть мастерских: mesh, не франшиза»
- (b) «Берлин. Белград. NYC. Как это работает без центра»
- (c) «Как 30 мастерских остаются одной сетью, не превращаясь в корпорацию»

**Разделы внутри one-pager:**
1. Проблема — один абзац: глобальные сети либо превращаются в звёзды (центр диктует), либо распадаются (никто не координирует). Оба варианта проигрышные для членов.
2. Что такое mesh — три строки: равноправные cells + общие стандарты + каждая автономна + может форкнуться.
3. Что это значит для тебя — три bullet: приехал в другой город → есть где jam'нуть; ресурс предлагаешь добровольно → никто не изымает; уходишь → забираешь долю.
4. Кем ты можешь быть в сети — 8 ролей одной строкой (без вербовочного давления).
5. Что держит сеть вместе — Charter + R12 + метод-канон + общая wiki. Не личность founder'а.
6. CTA — конкретный следующий шаг.

**Ключевые слова/метафоры:** Mondragón, гильдия, mesh, cell, Charter — не «платформа», не «экосистема», не «движение» (последнее — тонкая зона рекрутинга, R12 сигнал).

### A.2 Long-form spec (≤4K слов, полный артефакт)

**Аудитория.** Потенциальный Steward/Connector, партнёр рассматривающий открытие cell, инвестор оценивающий масштабируемость, пресса.

**Разделы long-form:**

1. **Топология: mesh vs star** — почему выбор топологии это не эстетика, а R12-требование. Star → центр контролирует → extraction inevitable. Mesh → каждая cell автономна → extraction structurally blocked. Mondragón-паттерн (81 кооператив + внутренний банк, каждый автономен — reference, не цитата).

2. **Строение cell** — что есть у каждой cell минимально: локальное пространство (виртуальное или физическое), локальный Charter (производный от Network Charter + локальные overrides), локальное управление (Совет cell), локальная культура (язык, ритм, ниши), связь с mesh (общая wiki, cross-cell protocol).

3. **Coordination layer** — что держит mesh вместе без центральной власти: (а) Charter (общий, публичный, с R12-clause обязательным), (б) R12 enforcement (смарт-контракты — Scale phase; сейчас: явные процедуры в Charter), (в) метод-канон (общая педагогика — Direction #1 Метод), (г) общая wiki (shared knowledge layer), (д) cross-cell protocol (как запустить проект с людьми из другой cell).

4. **Мульти-язык** — английский primary для cross-cell коммуникации, локальный язык per cell без ограничений. Явная оговорка: английский = инструмент координации, не культурное доминирование. Cell вправе работать полностью на локальном языке внутри.

5. **Передвижение** — travel program (приехал → получаешь временный доступ), cross-cell projects (проект с людьми из других cells, mesh-координация), sabbaticals (временный переход в другую cell), Demo Days (месячные, рождают cross-cell проекты), expeditions (5-7 человек из разных cells, 1-2 недели в нишу). **R12 проверка каждого:** добровольно, не отработка лояльности, выход без штрафа.

6. **8 ролей в сети** — Local Master · Network Steward · Connector · Investor · Consultant · Researcher · Master Mentor · Visiting Practitioner. Для каждой: что делает в mesh, как взаимодействует с другими cells, R12-ограничения роли. Особо: нет роли «вербовщик» — Connector соединяет согласных, не убеждает несогласных. Steward сам подотчётен Charter + R12.

7. **Ресурсы pooling (9 типов)** — знакомства · склады/помещения · хаты/гаражи · машины · бизнес-идеи · деньги (internal pool) · информация · инструменты · skills. Для каждого типа: opt-in механизм, consent процедура, cap правила (5:1 где применимо), fork-and-leave протокол. Явный принцип: ресурс остаётся под контролем владельца, пул = gift + согласие, не сбор дани. Не «всё общее по умолчанию».

8. **9 активностей** — походы в ниши · поездки в города · учёба · соревнования · школа · отдых · путешествия · создание сообществ · open-source. Для каждой: как развивает мастера, R12-проверка (развивает не удерживает, anti-TikTok).

9. **Направление (не enforcement)** — триада O-138 + «голодные/натренированные/дисциплинированные» = вектор культуры, поддерживаемый средой, не enforced сверху. Нельзя исключить члена за «недостаточную дисциплину». Нет давления «настоящий мастер всегда голоден». Уважение к ритму. Выход без стыда.

10. **Lifecycle защиты** — явный закон: защита должна расти быстрее системы. Seed (1 cell виртуальная): Charter + R12 процедуры явные. Build (1 физическая): pooling opt-in формы + cross-cell protocol. Scale (3-10 cells): смарт-контракты Ethereum (gated, Phase 2+). Mature (30-100+): полная on-chain R12 enforcement.

**Сцены (15 сек – 3:30):**

- Сцена 2 «Что такое mesh» (45 сек): каждая cell автономна — показываем Берлин-cell, NYC-cell, Belgrade-cell. Между ними линии (не иерархия). Coordination layer = Charter + R12 + wiki = стандарты, не приказы.
- Сцена 3 «Ты в сети» (45 сек): конкретный маршрут — приехал в другой город, есть где работать и встречать людей. Участвуешь в expedition. Запускаешь cross-cell проект. Нет главного — есть партнёры.
- Сцена 4 «Что держит вместе» (30 сек): не founder, не бренд — Charter публичен, R12 встроен структурно, fork-and-leave существует. Мастерская не держит — удерживает только ценностью.
- Сцена 5 «8 ролей» (30 сек): быстрый перебор ролей с иконками. Ключевой момент: можно менять роль, совмещать, выходить.

**Закрытие (3:30 – 4:00):** «Сеть — не организация которая тебя держит. Это инфраструктура которую ты используешь пока хочешь.» CTA: Charter публичен, вход — первые 4 сессии бесплатно.

**R12 проверка видео:** нет призыва «стань частью движения» (манипулятивный рекрутинг) — только описание устройства. Нет сцен «вместе мы сила» без оговорки о выходе. Mesh показывается через автономию, не через «семью».

## §C — Курс: Network Steward / Connector Training

**Применимость:** ДА — роли Steward и Connector требуют формальной подготовки.

**Аудитория:** человек который берёт на себя роль Network Steward (отвечает за здоровье cell в mesh) или Connector (соединяет людей из разных cells).

**Формат:** асинхронный курс, 4-6 модулей, практические задания, peer-review внутри cell.

**Модули:**

**Модуль 1 — Топология и зачем она важна.** Что такое mesh vs star на практике. Как определить что твоя cell начинает дрейфовать в сторону star (признаки). Как вернуть в mesh.

**Модуль 2 — Charter: читать и применять.** Структура Network Charter. Как работает локальный override. Когда нужно созывать Совет cell. Как принять нового члена (Welcome-frame + free 1-4). Как обработать конфликт без эскалации в центр (которого нет).

**Модуль 3 — R12 для Steward.** 4 action-класса R12 (extraction_beyond_share / wage_ratio_violation / non_consensual_distribution / fork_prevention_attempt). Как распознать их в реальных ситуациях. Как остановить до того как произошло. Steward = первая линия R12 enforcement.

**Модуль 4 — Pooling: opt-in процедуры.** 9 типов ресурсов. Форма согласия. Как объяснить члену разницу между «предлагаю ресурс в пул» и «ресурс изымают». Как обработать запрос на ресурс. Как вернуть ресурс из пула.

**Модуль 5 — Connector: соединять согласных.** Разница между Connector и вербовщиком. 8 диагностических вопросов перед приглашением. Как объяснить выход не как провал. Promoter-бонус = за качество fit, не за объём приглашений.

**Модуль 6 — Cross-cell projects.** Как запустить проект с другими cells. Mesh-координация (никто не начальник). Ресурсы из пула (opt-in+consent на уровне cross-cell). Документирование и sharing back (знание обогащает сеть).

**R12 проверка курса:** нет модуля «как убедить людей остаться» — есть модуль «как уважительно обработать выход». Нет геймификации удержания (time-based rewards). Метрика курса = качество решений, не часы в системе.

## §E — Инструкции / SOP

### E.1 — Как открыть новую cell

**Шаг 1 — Намерение.** Человек или группа заявляет намерение открыть cell в городе X. Заполняет pre-application форму (5 вопросов: кто, где, какие ниши, какие ресурсы, зачем сейчас).

**Шаг 2 — Charter adoption.** Читают Network Charter. Принимают R12-clause (обязательно, не опционально). Формируют локальный Charter-override (культура / язык / ритм) — в пределах Network Charter, не против него.

**Шаг 3 — Seed period (3 мес).** Виртуально или малая физическая группа. Первые 4 сессии открыты бесплатно (Welcome-frame). Steward проходит Steward Training (§C, Модули 1-4 обязательно).

**Шаг 4 — Cell activation.** Совет cell (минимум 3 члена) подтверждает. Cell появляется в Network Map. Получает доступ к shared wiki и cross-cell protocol.

**Шаг 5 — Ongoing.** Ежеквартальный Charter review. Ежегодный R12-аудит (Steward отчитывается перед Советом cell, не перед центром). Любой член может запросить fork-and-leave в любой момент.

### E.2 — Pooling opt-in процедура

**Принцип:** ресурс не входит в пул автоматически при вступлении в cell. Каждый тип ресурса — отдельный opt-in.

**Процедура:**
1. Владелец ресурса заполняет Pooling Consent Form (шаблон — §F.2): тип ресурса, условия использования, cap (если применимо), срок consent (renewabe), условия отзыва.
2. Cell Steward регистрирует в Pool Registry (простой markdown список или shared doc).
3. Запрос на ресурс: заявитель → Steward → согласует с владельцем → подтверждение. Никакого «берём без спроса».
4. После использования: feedback в Pool Registry. Опционально: взаимный обмен (не обязательный).
5. Отзыв consent: владелец в любой момент, без объяснений, без штрафа.

**Anti-patterns:** нет автоматического pooling при вступлении. Нет «вы обязаны шарить как член». Нет негативных последствий за отказ дать ресурс в пул.

### E.3 — Cross-cell project protocol

**Инициация:** любой член любой cell. Публикует brief в cross-cell channel (стандартная форма: задача, нужные роли/skills, ресурсы, timeline). Connector из другой cell реагирует.

**Координация:** никто не назначается «главным». Роли определяются компетенцией и договорённостью. Решения — consent (любой может заблокировать с аргументом, не большинством). Ресурсы — opt-in per individual.

**Документация:** результат и learnings идут в shared wiki. Обогащает всю сеть. Это не «работа на центр» — это contribution которая возвращается ко всем.

**R12 проверка:** нет механизма «выгодно центру больше чем участникам». Ресурсный вклад учитывается в пользу участников (не изымается). Fork-and-leave работает даже внутри активного cross-cell проекта (с graceful handoff).

### E.4 — Decision trees (ключевые развилки)

**Дерево 1: «Что делать если cell начинает дрейфовать в star»**
- Симптом: один человек принимает большинство решений → Steward поднимает на Совет cell → Charter review → если не исправлено: любой член может инициировать fork.
- Симптом: founder настаивает на особых правах → Charter-override не может противоречить Network Charter → эскалация: Charter публичен, R12-clause binding.
- Симптом: выход сопровождается социальным давлением → Steward обязан сказать «ВЫХОД ЗВУЧИТ ПЕРВЫМ» буквально.

**Дерево 2: «Что делать если подозрение на extraction»**
- Шаг 1: Steward диагностирует по 4 action-классам (§J.3).
- Шаг 2: если extraction_beyond_share → стоп, R12-процедура, возврат ресурсов.
- Шаг 3: если fork_prevention_attempt → немедленный публичный ответ: «fork-and-leave доступен, вот процедура».

### E.5 — Anti-patterns и как их распознать

| Anti-pattern | Сигнал | Защита |
|---|---|---|
| Star topology | Один человек = центр решений | Charter + mesh + fork |
| Founder cult | «Без него сеть развалится» | Founder = 1 из многих; уход не ломает |
| Non-consensual pooling | «Как член, ты должен шарить» | Opt-in + consent form + отзыв в любой момент |
| Fork prevention | «Уйдя ты предашь нас» | Fork-and-leave: забираешь долю, без стыда |
| Engagement trap | Метрика = время в системе | Метрика = рост мастерства |
| Манипулятивный рекрутинг | Promoter-бонус за объём | Бонус за качество fit, 8 вопросов before invite |
| Uniformity | «Все должны говорить по-английски» | Локальная автономия культуры |
| Extraction beyond share | Treasury растёт, члены нет | 5:1 cap + 75/25 + fork-and-leave забирает долю |

## §F — Шаблоны / Templates

### F.1 — Cell Charter (шаблон)

**Структура шаблона (разделы):**
1. Идентификация cell (название, город, дата основания, Совет cell — минимум 3 члена)
2. Network Charter adoption (явная ссылка + дата принятия)
3. R12-clause (обязательный раздел — не опциональный override): 4 action-класса запрещены, fork-and-leave доступен, extraction механизм описан
4. Локальные overrides (язык, ритм встреч, ниши, культурные традиции) — в пределах Network Charter
5. Welcome-frame: первые 4 сессии открыты бесплатно, вход не требует commitment
6. Pooling rules (ссылка на §E.2 + локальные дополнения)
7. Exit procedure: как уйти из cell (без штрафа), как уйти из проекта, как форкнуться (со своей долей)
8. Steward обязанности (Charter review, R12-аудит, ведение Pool Registry)
9. Revision cadence (ежеквартально + по необходимости)

### F.2 — Pooling Consent Form

**Поля:**
- Владелец ресурса (role-type, не имя — IP-1)
- Тип ресурса (из 9 типов)
- Описание (что конкретно)
- Условия использования (кто может, для чего, на каких условиях)
- Cap (если есть — например, «не более 2 запросов в месяц»)
- Срок consent (3 мес / 6 мес / бессрочно до отзыва)
- Условия отзыва (в любой момент, письменно Steward'у)

## §H — FAQ (10-30 вопросов)

**Раздел строится вокруг самых острых вопросов — тех, где честный ответ важнее красивого.**

**Q: Кто главный в сети? Кто принимает решения?**
A: Никто один. Mesh — это децентрализованная структура. Каждая cell управляется своим Советом. Network-уровневые решения (изменение Charter, новый R12-enforcement) — через consent-процесс всех Stewards. Founder — первый член сети, не её глава навечно. Если network работает только пока founder активен — это баг, не фича.

**Q: Что я обязан класть в общий пул ресурсов?**
A: Ничего обязательного. Pooling — полностью opt-in. Вступление в cell не означает автоматического взноса ресурсами. Ты предлагаешь то, что хочешь, на условиях которые ты устанавливаешь, и можешь отозвать в любой момент. Пул = подарок + согласие, не членский взнос.

**Q: Могу ли я уйти из cell? Что я теряю?**
A: Можно уйти в любой момент без объяснений. Ты не теряешь долю накопленного (если участвовал в pooling — по правилам fork-and-leave забираешь свою долю). Нет механизма «исключения» за «недостаточную дисциплину» или «несоответствие культуре».

**Q: Что если я не согласен с решением Совета cell?**
A: Несколько опций: (а) поднять на Совет с контраргументами (consent-процесс), (б) инициировать Charter review, (в) форкнуться — создать новую cell со своими правилами (в пределах Network Charter), забрав свою долю ресурсов.

**Q: Почему английский primary? Это не культурное доминирование?**
A: Английский = инструмент координации между cells в разных странах, не культурный стандарт. Внутри cell — полная автономия языка. Cell может работать полностью на русском, немецком, сербском. Cross-cell коммуникация на английском — прагматика, не иерархия.

**Q: Кто контролирует Charter? Founder может его изменить в одностороннем порядке?**
A: Charter публичен. Изменения через consent-процесс Stewards. Founder не имеет особых прав на изменение Charter. R12-clause — non-negotiable (не может быть убран даже consensus'ом, если это нарушает R12).

**Q: Как рассчитываются финансовые доли? Кто получает что?**
A: Базовая структура: 75/25 (75% участнику, 25% в treasury cell + network) + 5:1 wage cap (максимальный разрыв в вознаграждении 5:1 внутри cell). При форке: участник забирает пропорциональную долю treasury. Конкретные цифры — в Charter каждой cell (в пределах сетевых caps). Смарт-контракты (on-chain enforcement) — Scale phase.

**Q: Как вы предотвращаете превращение в секту?**
A: Структурно, не через намерения. Anti-секта чек-лист из 9 пунктов встроен в Charter как обязательный. ВЫХОД ЗВУЧИТ ПЕРВЫМ — буквально: в Welcome-frame для нового члена первым делом объясняют как уйти. Метрика = рост мастерства, не время в сети. Founder cult structurally blocked: сеть не зависит от одного человека.

**Q: Что если одна cell начинает вести себя как центр власти?**
A: Charter + Network Charter — публичные. Любой Steward другой cell может поднять вопрос. Если нарушение R12 — стоп-процедура. Если системный drift — другие cells не обязаны взаимодействовать с нарушителем. Открытость fork означает что здоровые члены могут выйти.

**Q: Как вы гарантируете что AI-координация не создаёт скрытый центр?**
A: AI = рутинная координация (matching, wiki sync, scheduling). Decisions = люди. Нет алгоритма который «продвигает» людей или «наказывает» низкую активность. AI-layer подотчётен R12 так же как и human-layer (per §K).

**Q: Могу ли я одновременно быть в двух cells?**
A: Да. Роль Visiting Practitioner предполагает временное участие в другой cell. Постоянное членство в двух cells — возможно по согласованию с обоими Советами. Нет ограничений «лояльности одной cell».

**Q: Что значит «promoter-бонус за качество, не объём»?**
A: Если ты приглашаешь кого-то в cell (Connector-роль) и они хорошо fit — через 6 мес ты получаешь recognition/reward. Не за факт приглашения и не за количество приглашений. Никаких реферальных лестниц в стиле Pinduoduo. Цель: качество соответствия, не рост числа.

## §I — Worked Examples (5 сценариев)

### I.1 — Cross-cell проект: методолог + исследователь + тестер

Методолог из Берлин-cell публикует brief: нужен исследователь и тестер для 3-месячного проекта (новый инструмент диагностики методов). Исследователь из NYC-cell и тестер из Belgrade-cell откликаются. Запускают cross-cell protocol. Никто не назначается PM — решения через consent. Ресурсы: методолог вносит 40ч/мес (opt-in в cross-cell pool), исследователь — 20ч/мес + доступ к базе данных (consent form), тестер — 15ч/мес. Через 3 мес: инструмент задокументирован и загружен в shared wiki. Все три cell обогащены. Финансово (сценарно): outcome = consulting gig €8K → 75/25 split per cell Charters → каждый участник получает соразмерно вкладу.

### I.2 — Expedition: 6 человек из 4 cells в нишу

Connector из London-cell организует expedition «AI-pedagogy sprint»: 6 человек из Берлин/NYC/Belgrade/London cells, 10 дней в онлайн-интенсиве + 3 дня физически в Берлин. Полностью добровольно. Каждый едет за свой счёт (нет «обязательных командировок»). Результат: 3 новых метода задокументированы в shared wiki, 1 cross-cell проект запущен, 2 случайных conexiones → новые Connector-связи. R12 проверка: организатор не «начальник экспедиции» — координатор. Никто не обязан ехать физически. Нет штрафа за отказ.

### I.3 — Pooling сценарий: хата как временный ресурс

Член Берлин-cell (Visiting Practitioner-тип) на 2 месяца уезжает в экспедицию. Заполняет Pooling Consent Form: предлагает квартиру как temporary workspace для другого члена. Условия: 1 человек, некурящий, max 6 недель, с предупреждением за 1 неделю. Steward регистрирует в Pool Registry. Исследователь из той же cell берёт на 4 недели (подтверждение через Steward → согласование с владельцем → ок). После: feedback в Registry (всё нормально). Ресурс остаётся под контролем владельца весь процесс.

### I.4 — Fork сценарий: здоровый выход

6 членов Belgrade-cell хотят сфокусироваться на niche которая сильно отличается от текущего направления cell. Инициируют fork: создают Belgrade-AI-cell. Процедура: публичная (Charter требует). Их доля treasury: пропорционально вкладу за время в cell (рассчитано по формуле в Charter). Принимают Network Charter + R12-clause. Старая Belgrade-cell продолжает работать. Обе остаются в mesh. Нет конфликта, нет «предательства» — это feature, не bug.

### I.5 — Welcome-frame: новый член слышит про выход первым

Потенциальный Apprentice приходит на первую бесплатную сессию NYC-cell. Steward проводит Welcome (15 мин структурированный разговор). Первое что звучит: «Первые 4 сессии бесплатно и без commitment. Уйти можно в любой момент. Если решишь — скажи мне, я объясню как. Никакого давления.» Потом: что такое cell, роли, как всё устроено. CTA в конце — не «вступи», а «приди на следующую сессию если понравилось». Это anti-манипуляция на уровне первого контакта.

### J.5 — Тонкие зоны (где культы ломаются не сразу)

**Зона 1: «Культура направления» vs enforcement.**
Триада O-138 + «голодные/натренированные/дисциплинированные» — это вектор культуры, не требование. Тонкая зона: если культура создаёт неявное давление («настоящий член всегда голоден» / «ты что, не дисциплинирован?»), это дрейф к секте без явных нарушений R12. Защита: явные оговорки в Charter («уважение к ритму» / «выход без стыда» / «нет единственного правильного темпа»). Steward обязан диагностировать неявное давление так же как явное.

**Зона 2: «Мы — семья» фрейм.**
Семейный фрейм распространён в community-проектах и может быть аутентичным. Проблема: «семья не предаёт» → выход воспринимается как предательство. В контексте Сети: заменить на «профессиональная гильдия» / «мастерская» — сильный connection без семейного lock-in.

**Зона 3: Steward-power.**
Steward получает значительную операционную власть (Charter enforcement, Pool Registry, Welcome-frame). Риск: Steward становится де-факто мини-центром. Защита: ежеквартальная ротация Steward (опция в Charter), Совет cell как коллективная власть, Steward подотчётен Совету (не наоборот).

**Зона 4: Рост ≠ здоровье.**
Проблема масштабирующихся сетей: growth становится метрикой успеха → появляется давление привлекать всё больше → рекрутинг → manipulation creep. Защита: метрика сети = quality of mastery outputs + экономическое здоровье cells, не number of members. Anti-hype stance (Tesla/Mondragón pattern): рост как следствие, не цель.

## §K — AI Tooling Integration

**Принцип (per O-182):** AI = координация рутина (matching, wiki sync, scheduling, pool tracking). Человек = cross-cell trust, strategic decisions, consent процессы.

**Текущий этап (Build):**

- **Wiki sync:** AI автоматически тегирует контрибьюции в shared wiki, формирует cross-cell relevant digest (кому что полезно из других cells). Человек = review + публикация.
- **Matching:** AI предлагает потенциальные cross-cell connections (по skills, niches, проектам). Connector = финальное решение + 8 вопросов.
- **Pool Registry:** AI ведёт учёт opt-in ресурсов, напоминает о renewal сrokov consent, флагирует expired consent. Steward = финальные действия.
- **Charter health-check:** AI периодически анализирует паттерны активности на сигналы дрейфа (star-pattern, retention-metric signs). Совет cell = финальный суд.

**Scale phase (gated — Programmable Ethereum, per RUSLAN-ACK commit `8a3d800`):**

- Смарт-контракты автоматизируют 5:1 wage-cap enforcement, fork-and-leave долю (on-chain расчёт), R12-clause compliance.
- Mondragón-ratio cap + QF revenue distribution + fork-and-leave exit tokens.
- Per-Clan opt-in через Charter (не все cells обязаны переходить одновременно).

**R12 проверка AI-layer:**

- Нет алгоритма который «продвигает» активных и «штрафует» неактивных.
- AI-matching критерии = открытые и публичные.
- Нет retention-optimization в AI (никаких «напомнить что человек не заходил 3 дня»).

## §L — Partner-Extension Hook: как партнёр открывает новый cell

**Сценарий:** T1-тип партнёр (методолог, исследователь или любой Connector-роль) хочет открыть cell в своём городе или для своей ниши.

**Mondragón spawn pattern (Layer 4):**
Mondragón начинался с одного кооператива. Каждый новый кооператив в сети создавался самостоятельно, принимал общие принципы, получал поддержку внутреннего банка, сохранял полную автономию. Не франшиза, не филиал — новый кооперативный субъект в сети.

**Процедура открытия cell партнёром:**

1. **Pre-application.** Партнёр заполняет форму (5 вопросов из §E.1 Шаг 1). Steward любой существующей cell review'ит (не «одобряет» — диагностирует R12-readiness).

2. **Charter onboarding.** Партнёр читает Network Charter. R12-clause = обязателен, non-negotiable. Steward Training — Модули 1-4 минимум.

3. **R12 compliance check.** Перед активацией — явная проверка: 4 action-класса понятны? Fork-procedure в локальном Charter? Pooling opt-in процедуры описаны? Welcome-frame с выходом первым?

4. **Seed period.** 3 мес виртуально. Поддержка от Network (shared wiki, cross-cell protocol доступен). Никакой финансовой «платы за франшизу» (это extraction_beyond_share если принудительно).

5. **Activation.** Cell появляется в Network Map. Становится полноправным узлом mesh.

**Что партнёр получает от сети:**
- Доступ к shared wiki и метод-канону.
- Cross-cell protocol (проекты с другими cells).
- Network Map visibility.
- Steward Training и курсы.

**Что сеть НЕ диктует партнёру:**
- Ценообразование внутри cell.
- Язык и культуру.
- Ниши (в пределах R12).
- Операционный ритм.

**R12 compliance при spawn:**
- Нет royalty или franchise fee (extraction_beyond_share риск).
- Нет exclusive territory (это был бы fork-prevention — другой cell не может открыться рядом).
- Нет «стандартного бренда который нельзя менять» без локального Charter override права.
- Cell может выйти из сети (super-fork) забирая свои материалы. Только R12-clause binding на всё время участия.
