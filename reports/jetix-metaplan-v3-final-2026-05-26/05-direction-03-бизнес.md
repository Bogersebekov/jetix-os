# Direction #3 💼 Бизнес — портфель арtefacts

## Вступление: Foundation embedding

Jetix — мега-мастерская мирового уровня: место, где люди становятся мастерами в эпоху AI, вместе двигают фронтир и не дают системе доить или запирать себя.

Три грани: 🏛️ Мастерская (ГДЕ) · 🎯 Мастерство (ЧТО) · 🌍 Сеть (КАК — mesh + pooling). Direction #3 Бизнес — это операционализация грани Сеть на уровне организационно-правовой формы, экономической модели и governance. Кооператив — это не «красивая упаковка», это точная форма для мега-мастерской: автономные единицы связаны общими принципами (Mondragón pattern: 81 кооператив, внутренний банк, единая идентичность), каждая ячейка сохраняет суверенитет.

**Ключевые якоря этого direction'а:**
- Экономика: 75/25 split (worker/pool) + 5:1 wage-ratio cap (Mondragón) + fork-and-leave (30-day notice, без штрафа)
- R12 четыре action classes: extraction_beyond_share / wage_ratio_violation / non_consensual_distribution / fork_prevention_attempt
- Ethereum Phase 2+: Mondragón ratio cap on-chain + QF revenue distribution + exit tokens; per-Clan opt-in
- Governance: Ruslan = единственный стратег; AWAITING-APPROVAL packets на все архитектурные изменения; Stage Gates SG-0..SG-7; Steward role; corrigibility (owner ack-authority финальна)
- Жизненный цикл: Build → Run → Scale → Mature; точка A = Build средний

**GAP этого direction'а:** жанр «create» отсутствует — риск, что Foundation-жаргон (AWAITING-APPROVAL, Halt-Log-Alert, F-G-R) вытекает наружу в партнёрские/инвесторские тексты. Каждый artefact ниже учитывает этот разрыв и описывает, как переводить внутренний язык в внешний.

## §A. Главный документ «Как устроен Jetix»

### A.1 1-pager (≤500 слов) — спецификация

**Назначение:** первое касание с партнёром или инвестором. Должен закрыть вопрос «что это вообще такое» за 2-3 минуты чтения.

**Аудитория:** потенциальные партнёры (будущие члены-основатели), ранние инвесторы (angels, impact funds), корпоративные клиенты на разведке.

**Формат:** MD + PDF export, A4, одна страница при стандартном шрифте. Никакого жаргона (нет «AWAITING-APPROVAL», «F-G-R», «brigadier»).

**Структура 1-pager'а:**
1. Одна фраза — что такое Jetix (R1 one-liner, адаптированный без акрониумов)
2. Проблема — почему AI-эпоха ломает старые модели обучения и работы
3. Решение — мега-мастерская: место + метод + сеть
4. Как работает экономика — 75/25, потолок 5:1, выход без штрафа (три строки, без формул)
5. Где сейчас — точка Build, первая когорта O-161 (5-10 основателей)
6. Что хотим от читателя — CTA (discovery call / ознакомиться с Charter)

**Тон:** прямой, без корпоративного пафоса. Язык — такой, чтобы умный незнакомец понял за первое прочтение.

### A.2 Long-form (≤5K слов) — спецификация

**Назначение:** углублённое знакомство для тех, кто прошёл 1-pager и хочет понять детали перед decision call.

**Аудитория:** серьёзные партнёры-кандидаты, impact-инвесторы с DD-процессом, журналисты/исследователи.

**Формат:** MD + PDF, 8-12 страниц.

**Разделы:**
1. Thesis — почему мы строим именно так (Mondragón + deep-tech + методологический IP — три уникальных слоя одновременно)
2. Что такое мега-мастерская — три грани операционально (Мастерская / Мастерство / Сеть)
3. Экономическая модель — 75/25 split, 5:1 cap, social fund 10%, QF matching (Phase 2+), triple-role (worker + investor + promoter)
4. Governance — как принимаются решения (Stage Gates, Steward role, один стратег на Phase A); что значит corrigibility на практике; как система не запирает членов
5. Lifecycle — Build / Run / Scale / Mature, где сейчас, что разблокирует следующий этап
6. R12 anti-extraction — восемь вопросов, которые мы задаём каждому governance-решению (plain language, без кода)
7. Команда и компетенции — Ruslan + первая когорта + ROY swarm как операционная машина
8. Что дальше — Ethereum Phase 2+, Network State spawn, масштабирование кланов
9. Как присоединиться / инвестировать — ссылка на Charter, discovery call форма

**Ключевой принцип:** Foundation-жаргон переводится в человеческий язык. «AWAITING-APPROVAL packet» → «решения с архитектурным влиянием требуют явного одобрения основателя». «Halt-Log-Alert» → «при нарушении конституционных принципов система останавливается и сигнализирует».

## §B. Видео

**Hook (0-10 сек):** Один вопрос в лоб — «Вы когда-нибудь работали в компании, где правила придумывают наверху, а платят внизу?» Пауза. «Мы строим что-то другое.»

**Сцены:**

1. (10-40 сек) Проблема — AI меняет рынок труда, но большинство компаний реагируют старыми инструментами: обучение как товар, работа как найм, выход наказывается
2. (40-90 сек) Jetix как мега-мастерская — визуально показываем три грани (место / метод / сеть); аналогия с ремесленной гильдией, обновлённой для AI-эпохи
3. (90-150 сек) Кооперативная экономика на пальцах — 75/25, потолок 5:1, можно уйти через 30 дней без штрафа. Сравнение с классической компанией (там 100% у топа, у тебя — зарплата)
4. (150-210 сек) Governance — как работает Stage Gate (просто: идея → валидация → запуск → масштаб); Steward как роль защиты принципов
5. (210-240 сек) Где сейчас, что строим, кого ищем

**Closing:** «Если вы хотите строить что-то серьёзное и не отдавать 100% плодов вверх — посмотрите Charter.» [ссылка]

**Аудитория:** потенциальные члены-основатели (25-45 лет, tech/consulting/education background), impact-инвесторы, журналисты Future of Work темы.

**Длина:** 3-4 минуты. Формат: screen + talking head Ruslan. Субтитры обязательны (EN + RU).

## §E. Инструкции/SOP — governance procedures

**Назначение:** операционный справочник для Steward, Ruslan и будущих Clan Leads. Внутренний документ. Может частично публиковаться (та часть, что не содержит внутренней инфраструктуры).

**Структура SOP-пакета:**

### E.1 Как принимается решение (Decision Flow)

Дерево решений по классам:

```
Входящее решение
├── Операционное (затрагивает один проект, обратимо, не меняет принципы)
│   └── Ответственный Lead или Ruslan → логируется → исполняется
├── Архитектурное (изменение Foundation, principles/, schemas/, routing-table)
│   └── AWAITING-APPROVAL packet → Ruslan ack → LOCKED → исполняется
├── Стратегическое (direction, vision, resource allocation)
│   └── Только Ruslan → логируется как R1 decision → производный контент отмечается
└── Конституционное нарушение (R12 action class detected)
    └── HALT → log.jsonl → Ruslan alert → разбор
```

### E.2 Stage Gates (SG-0..SG-7) — операционная механика

| Gate | Название | Predicate-пример | Что разблокирует |
|------|----------|-----------------|-----------------|
| SG-0 | Idea validated | Ruslan ack + problem statement written | Scaffold проекта |
| SG-1 | First signal | ≥3 leads / ≥1 paid session / ≥1 partnership MOU | Полный scaffold |
| SG-2 | Traction | Revenue ≥ threshold OR cohort ≥ N members | Operational budget |
| SG-3 | Repeatability | ≥3 successful cycles of core process | Process documentation |
| SG-4 | Promote / Scale | Stable revenue + team ≥ core-group | Phase transition |
| SG-5 | Legal entity | Registered coop entity | Formal contracts |
| SG-6 | Charter ratified | Charter v1 signed by first cohort | Full member rights |
| SG-7 | Ethereum on-chain | Smart-contract deployed + audited | R12 programmable enforcement |

**de-morph:** откат до SG-N возможен через `/project-de-morph --rollback-to=SG-N`. Логируется. Данные предыдущих gates сохраняются (append-only).

**Anti-patterns SG:**
- «Предсказание» вместо predicate: запрещено. Predicate должен быть бинарным (да/нет, count ≥ N).
- Автоматическое промотирование без Ruslan ack: запрещено. Агенты предлагают, Ruslan решает.
- Retroactive gate-firing: запрещено без explicit AWAITING-APPROVAL.

### E.3 AWAITING-APPROVAL — когда и как

Пакет открывается, если:
- Изменение затрагивает Foundation paths (Parts 1-11, principles/, shared/schemas/, .claude/config/)
- Изменение routing-table.yaml
- Изменение R12-классов
- Любое действие из default-deny-table.yaml без explicit ack

Структура пакета: gate_type / blast_radius / proposed_change / reversibility / Ruslan_ack_required.

После ack — исполняется, логируется в swarm/approvals/log.jsonl.

### E.4 Steward role — responsibilities

- Мониторинг R12: каждый раз при распределении дохода запускает 8-вопросный чеклист
- Stage Gate audit: еженедельно проверяет `/lint --check-stage-gates`
- Anti-cult discipline: отслеживает признаки cult-dynamics (давление на выход, нарушение 30-day notice)
- Escalation to Ruslan: при любом нарушении конституционных принципов — немедленно

**Anti-patterns Steward:**
- Steward не стратег: не принимает стратегических решений самостоятельно
- Steward не цензор: не блокирует операционные решения Lead'ов без конституционного основания
- Steward не должен быть один: при масштабировании → rotating Steward panel

## §H. FAQ — честные ответы

### H.1 Базовые вопросы (аудитория: незнакомые)

**Q1: Вы кооператив или компания?**
Мы строим кооператив. Юридического лица пока нет (Stage Gate SG-5 ещё не достигнут). Charter v1 — это текстовое соглашение об обязательствах, которое предшествует юридическому оформлению.

**Q2: Кто решает в Jetix?**
На текущем этапе (Build) стратегические решения принимает Ruslan. Операционные — распределены между Lead'ами. Переход к более распределённому governance происходит через Stage Gates при достижении Scale. Это не «потому что Ruslan хочет власти» — это потому что рассредоточенный контроль до достижения traction убивает стартапы.

**Q3: Как вы защищаете от диктатуры основателя?**
Четыре уровня: (1) Charter явно запрещает R12 action classes; (2) Steward role мониторит соответствие; (3) fork-and-leave — любой член может уйти через 30 дней без объяснений и штрафа; (4) в Phase 2+ — Ethereum on-chain enforcement: ни Ruslan, ни кто-либо другой не может изменить параметры без smart-contract.

**Q4: Что значит 75/25?**
75% от revenue проекта/сессии идёт тому, кто создал ценность (worker). 25% — в общий pool (инфраструктура, social fund, инвестиции в рост). Это не «Ruslan берёт 25%» — это pool системы.

**Q5: Что такое 5:1 wage cap?**
Максимальная зарплата в системе не может превышать минимальную более чем в 5 раз. Пришло из Mondragón. Это про достоинство, а не уравниловку.

**Q6: Можно ли взять деньги и уйти?**
Да. 30-day notice. Без штрафа. Без non-compete, который нарушает это. Fork-and-leave — базовое право.

### H.2 Более сложные вопросы (аудитория: серьёзные партнёры)

**Q7: Вы говорите о Mondragón, но у вас нет юрлица. Это не лицемерие?**
Нет. Mondragón строился поэтапно — сначала принципы, потом структура. Мы в той же логике. SG-5 (legal entity) и SG-6 (Charter ratified) — конкретные gates с конкретными predicates. До их достижения Charter — это текстовое обязательство. Это честнее, чем зарегистрировать LLC и называть её кооперативом.

**Q8: Почему AI — операционная машина, а не стратег?**
Потому что стратегия требует skin-in-the-game, которого у AI нет. Агенты генерируют варианты, surfac'ят опции, помогают исполнять. Ruslan — единственный стратег. Это не ограничение — это архитектурный принцип. Компании, где AI принимает стратегические решения, создают systems без ответственности.

**Q9: Что такое R12?**
Конституционный принцип (LOCKED 2026-05-12): ни AI, ни substrate не могут извлекать ценность сверх согласованной доли; члены могут форкнуться и уйти без штрафа. В Phase 2+ — кодируется в Ethereum смарт-контракт.

**Q10: Stage Gates — это не бюрократия?**
Это ровно обратное. Stage Gates предотвращают преждевременную бюрократизацию: нельзя достичь SG-6 (Charter ratified) без реального traction на SG-2. de-morph позволяет откатиться, если что-то пошло не так. Это механизм обратимости, а не рельсы.

**Q11: Кто такой Steward и как его выбирают?**
Steward — роль, не человек. В Phase A (Build) её исполняет Ruslan де-факто. В Phase B (Run) — назначается отдельный человек из когорты. В Scale — rotating panel. Steward не стратег, он хранитель конституционных принципов.

**Q12: Ethereum Phase 2+ — это обязательно?**
Per-Clan opt-in. Каждый Clan решает, внедрять ли on-chain enforcement. Базовая защита (Charter текст + Steward + fork-and-leave) работает без блокчейна.

**Q13: Jetix — это Network State?**
Мы видим alignment с Balaji Srinivasan's Network State concept (7 из 7 шагов NS mapping применимы). Но мы не называем себя Network State публично — это Phase 3+ горизонт при достижении $100K+ и 20+ мастерских.

**Q14: Как мне присоединиться?**
Этапы: (1) Discovery call — понять fit; (2) Charter review — прочитать обязательства; (3) O-161 onboarding — первая когорта 5-10 основателей; (4) первый вклад — проект или сессия.

**Q15: Что вы ищете в партнёре?**
Не исполнителя. Мастера в своей области, который хочет строить систему, а не быть нанятым в неё.

## §I. Worked examples (IP-1 STRICT — без executor names)

### Example 1: Новый партнёр проходит onboarding

Ситуация: Анна, эксперт по организационному дизайну, узнала о Jetix и хочет присоединиться.

Процесс:
1. Discovery call (Ruslan + Анна): выясняем fit, объясняем Charter
2. Анна читает Charter draft — задаёт вопросы по §R12 и §economics
3. Ruslan открывает O-161 onboarding entry (AWAITING-APPROVAL не нужен — это операционное)
4. Первый проект Анны: консультация для клиента → revenue split 75/25 применяется автоматически
5. SG-1 считает Анну как +1 к cohort count → если cohort ≥ 5 → SG-1 предикат может сработать

Что работает: Stage Gate предикат бинарный, измеримый. Ruslan ack'ует сам факт entry в когорту. Экономика применяется с первого дня без переговоров.

### Example 2: Запрос на изменение wage cap

Ситуация: один из Lead'ов предлагает поднять cap с 5:1 до 8:1 для привлечения senior-специалиста.

Процесс:
1. Lead пишет предложение → brigadier роутит к mgmt-expert + investor-expert + systems-expert
2. Эксперты генерируют варианты (8:1 / 5:1 / 6:1 с sunset clause / etc.)
3. AWAITING-APPROVAL packet открывается автоматически — wage_ratio_violation = конституционный класс
4. Ruslan получает пакет с вариантами → рассматривает → ack или reject
5. Если ack: Charter обновляется, логируется как RUSLAN-LAYER change; Steward уведомлён

Что не происходит: Lead не принимает решение самостоятельно. AI-агенты не «рекомендуют» итоговый вариант (они surfac'ят). Ruslan = единственный кто может ack architectural decision.

### Example 3: Выход члена через fork-and-leave

Ситуация: Михаил, один из ранних участников, решает построить свой Clan на основе Jetix-модели.

Процесс:
1. Михаил уведомляет Ruslan и Steward — 30-day notice (письменно, в Charter log)
2. Steward запускает R12 checklist exit: нет штрафов, данные портируются, IP-права уточняются
3. Михаил получает exit token (Phase 2+) или аналог Phase 1 (письменное соглашение об IP)
4. Опционально: Mikhail форкает governance model — берёт Charter template, адаптирует под свой Clan
5. Jetix и Clan Mikhail'а могут оставаться в network (edge в wiki/graph/edges.jsonl)

Что важно: exit = не предательство. Anti-cult discipline. Fork-and-leave — это фича, не баг.

### Example 4: Обнаружение R12 нарушения

Ситуация: один из проектов начал направлять 40% revenue в отдельный «развитийный фонд» без явного согласия членов.

Процесс:
1. Steward проводит еженедельный R12 audit → Q3 (non_consensual_distribution) = true
2. Halt-Log-Alert: Steward логирует в swarm/approvals/log.jsonl + alert Ruslan
3. Ruslan получает полный контекст → решает: откатить распределение / открыть Charter amendment process
4. Если откат: documented, логируется, члены уведомлены
5. Если Charter amendment: AWAITING-APPROVAL packet + явное согласие всех членов

### Example 5: Stage Gate SG-4 и promote

Ситуация: проект quick-money (P1) достигает stable revenue + cohort ≥ core-group.

Процесс:
1. `/lint --check-stage-gates` выявляет SG-4 predicate = true
2. brigadier готовит SG-4 promote proposal → Ruslan ack
3. Проект переходит из bets/consulting в Scale phase
4. `/project-promote quick-money` — scaffold масштабируется
5. Новые Stage Gates SG-5/SG-6 становятся активными (legal entity + Charter ratified)

## §J. R12 paired-frame check

### J.1 8 вопросов — governance edition

Каждое governance-решение в direction #3 должно пройти:

1. **Share integrity:** все члены получают свою согласованную долю? (75/25 не нарушается?)
2. **Wage ratio:** wage cap 5:1 соблюдается? Предложенное изменение не нарушает?
3. **Consent:** изменение дохода / распределения / принципов — с явного согласия всех затронутых?
4. **Exit freedom:** создаёт ли это решение барьер для выхода (технический / финансовый / социальный / психологический)?
5. **Fork freedom:** позволяет ли это решение создать fork? Или оно запирает членов?
6. **Substrate extraction:** получает ли платформа / AI-система ценность сверх согласованной доли?
7. **Retroactivity:** изменяет ли это условия задним числом без ретроактивного согласия?
8. **Phase 2+ compatibility:** если смарт-контракт существовал бы — это решение прошло бы его логику?

### J.2 Governance anti-patterns (диктатура vs corrigibility)

| Anti-pattern | Признак | Защита |
|---|---|---|
| Founder lock-in | Основатель может блокировать выход или форк | fork-and-leave 30d, exit token Phase 2+ |
| Strategic capture | AI-агент «предлагает» стратегию и owner соглашается без критической оценки | Ruslan = sole strategist; AI = surface only |
| Gate inflation | Stage Gates расширяются без Ruslan ack, создавая бюрократию | AWAITING-APPROVAL на изменение gate schema |
| Steward overreach | Steward блокирует операционные решения без конституционного основания | Steward role = конституционный монитор, не менеджер |
| Silent distribution | Revenue направляется в фонд без явного Charter basis | R12 Q3 мониторинг + Steward audit |
| Wage drift | wage ratio постепенно растёт через «исключения» | Charter explicit cap + SG-5 enforcement |
| Cult dynamics | Давление на членов «не уходить», социальная стоимость выхода | Anti-cult discipline + 30-day notice без штрафа |
| AI autonomy creep | Агент начинает принимать архитектурные решения самостоятельно | Default-deny-table + AWAITING-APPROVAL + Halt-Log-Alert |

### J.3 Corrigibility в governance контексте

Corrigibility — это не слабость системы, это её прочность. Система, которую нельзя поправить, ломается под нагрузкой. Конкретно для direction #3:

- Ruslan может отменить любое Stage Gate решение через AWAITING-APPROVAL
- Ruslan может изменить routing-table, Charter, R12 enforcement механику — но только через явный ack-trail
- Ни один агент не может lock'ать Ruslan out of control
- Члены могут форкнуться в любой момент — это тоже corrigibility (system-level)

Governance — это не область, где AI стратегизирует. Но AI значительно снижает операционную нагрузку:

### K.2 Что swarm НЕ делает (IP-1 STRICT)

- Не решает, кого принять в O-161

Attention budget: max 20 активных governance tasks одновременно (RUSLAN-LAYER cap). При переполнении — brigadier логирует overflow, не создаёт новых задач.

## §L. Partner-extension hook — как партнёр форкает governance-модель под свой Clan

**Mondragón spawn pattern:** каждый Clan = автономная единица с собственным Charter, связанная с Jetix общими принципами (не иерархией).

### L.1 Что форкается

Партнёр берёт:
- Charter template (§F.1 выше) и адаптирует под свой Clan
- Stage Gate schema (SG-0..SG-7) — может добавлять собственные gates, не убирать конституционные
- R12 8-вопросный checklist — обязателен, extension допускается
- Governance log schema — стандартный формат для совместимости

### L.2 Что остаётся неизменным (конституционная инвариантность)

- R12 четыре action classes — нельзя убрать из Charter
- fork-and-leave 30-day notice — нельзя ужесточить
- wage ratio cap — собственный Clan может сделать cap строже (3:1), но не слабее без явного обоснования
- Corrigibility принцип — Clan Lead не может lock out членов

### L.3 Ethereum Phase 2+ spawn

При per-Clan opt-in: Clan разворачивает свой экземпляр смарт-контракта (fork от Jetix base contract). Параметры: 75/25 split (configurable), 5:1 cap (configurable вниз, не вверх), exit token mechanics.

### L.4 Процесс spawn

1. Clan Lead (бывший член или внешний партнёр) получает Charter template через discovery process
2. Clan Lead адаптирует Charter, проходит R12 review (Steward или Ruslan проверяет)
3. Clan ставит SG-0 predicate: «Charter написан + R12 прошёл review»
4. После SG-0 — Clan автономен; Jetix = network peer, не иерарх
5. Edge создаётся в wiki/graph/edges.jsonl: `{from: jetix, to: clan-slug, type: spawned, date: YYYY-MM-DD}`
