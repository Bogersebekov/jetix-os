# Phase 8 — Fork-friendly mechanics + R12 (3 layers)

> **Что в этой фазе.** Как работает форк на каждом из трёх слоёв архитектуры v2.
> Почему R12 enforcement применяется ТОЛЬКО к Jetix overlay (Layer 2 Part B), а не
> к generic baseline. Мета-R12 инсайт: навязывать кооперативную модель всем = само
> по себе анти-R12. Fork-and-leave mechanic: как уйти в любой момент с данными и
> без штрафов — на каждом уровне.

---

## §1 Базовый паттерн: два слоя внутри каждого уровня

Прежде чем разбирать форк по слоям — один универсальный принцип, который работает на всей архитектуре.

Любой шаблон состоит из двух частей:

- **«Грамматика» (часть A)** — структура, типы связей, базовые поля, топология баз. Это не трогается при форке. Если переделать грамматику, теряется совместимость и способность получать обновления из апстрима.
- **«Словарь» (часть B)** — всё, что специфично для конкретного человека, бизнеса, сообщества: названия ролей, ниши, overlay под конкретную модель управления.

Форкать = взять грамматику как есть + адаптировать словарь под себя.

Анти-паттерн: вписать личные имена или брендовые термины в Layer A. «Daily Log» — окей, fork-friendly. «Дневник Васи» в заголовке шаблона — ломает форк для всех, кто не Вася.

## §2 Layer 1 — полная свобода (Personal Core)

### Что это

Layer 1 — личная операционная система. Дневник, задачи, проекты, CRM для личных контактов, привычки, знания, голосовые заметки, энергия. Всё, что нужно конкретному человеку для работы с собой.

### Как форкнуть

Путь один: **скопировать → адаптировать → использовать без какой-либо связи с апстримом**.

- Отдельный воркспейс — данные не пересекаются ни с чьими другими.
- Никаких ссылок на Jetix, никаких обязательств.
- Пять базовых ниш адаптации (инженер, исследователь, предприниматель, методолог, гуманитарий) меняют только «что в центре» — какие базы активны и как расставлены приоритеты. Грамматика не меняется.

### Данные и уход

Данные в Layer 1 — полностью твои. Никакой синхронизации с внешними системами без явного шага пользователя. Перестал пользоваться шаблоном — данные остаются в твоём воркспейсе. Никаких «subscription lock-in», никаких хранилищ на чужой стороне.

Форк Layer 1 не требует никакого согласования. Это инструмент — взял, адаптировал, используешь.

### R12 на Layer 1

R12 (анти-экстракция) на уровне Personal Core не требует специальной инфраструктуры: здесь один человек, нет распределения доходов, нет ролей с разной властью. Базовая «порядочность» Layer 1 — данные не покидают воркспейс без явного действия пользователя. Этого достаточно.

## §3 Layer 2 — командное сотрудничество: три пути форка

Layer 2 — самый гибкий уровень с точки зрения форка. Именно здесь живёт ключевое различие между generic baseline и Jetix overlay.

### Три пути для форкера

**Путь 1: Swap overlay — заменить Jetix overlay на свой**

Берёшь generic baseline Layer 2 + создаёшь собственный overlay вместо Jetix-специфичного. Например:
- IT-команда в корпоративной среде: overlay с корпоративными approval-процессами и SLA.
- Небольшой стартап: overlay с cap table, board-логикой, investor reporting.
- Творческое агентство: overlay с утилизацией, project margin, portfolio.

Generic baseline (Part A) — нейтральный, чистый, без чьих-либо ценностных установок. Overlay (Part B) — целиком твой, включается / выключается как блок.

**Путь 2: Use generic baseline clean — вообще без overlay**

Если не нужен никакой специфичный overlay — используешь только Part A (generic). Хватает для большинства малых команд: задачи, встречи, knowledge base, базовый CRM для команды, shared Goals.

Никакого R12, никакого Jetix, никакого Mondragón. Просто инструмент.

**Путь 3: Use Brand pattern — готовый шаблон для специфичного типа организации**

Brand pattern — это специализированный вариант Part A для трёх типовых контекстов:
- Blogger / creator: контент-план, аудитория, метрики охвата, партнёрства.
- Corporate team: структура проектов с approvals, compliance, отчётность.
- Startup: product roadmap, метрики роста, fundraising pipeline.

Это не overlay в смысле ценностей — это адаптация грамматики под специфику типа организации. Никакого R12-enforcement.

### Jetix overlay (Part B) — что это и для кого

Jetix overlay существует как ОДИН конкретный пример того, как может выглядеть Layer 2 Part B для кооперативной организации с R12-дисциплиной. Это не «правильный» путь — это RUSLAN-LAYER инстанс. Другие организации могут создать совершенно другой overlay под свои ценности и модель.

Что включает Jetix overlay:
- Роли в формате кооператива (10 ролей, горизонтальная координация)
- Contribution Ledger — история вклада каждого участника
- R12 enforcement mechanics (подробно в §5)
- Charter structure — явный документ с условиями, которые можно обновлять через re-ack
- Fork-and-leave protection полного цикла (не только «ты можешь уйти», а конкретные механики)

Если кто-то хочет построить кооператив с другими ценностями — он не берёт Jetix overlay, он создаёт свой.

### Fork-and-leave из команды (Layer 2)

При уходе из командного пространства:

- **Личные данные** (Layer 1 воркспейс) не трогаются — они в отдельном воркспейсе.
- **Командные данные, к которым есть доступ** — экспорт всегда доступен (базовая порядочность, не зависит от overlay).
- **При Jetix overlay**: дополнительно сохраняется строка Contribution Ledger + накопленная доля + 30-day notice при изменении Charter.

Ключевой момент: **уход из команды (Layer 2) не влияет на Personal OS (Layer 1)**. Это архитектурно — раздельные воркспейсы, не «по обещанию».

## §4 Layer 3 — универсальный бизнес-фундамент: форк для любого основателя

### Что это

Layer 3 — операционная система для бизнеса. 12 групп баз: стратегия, люди, финансы, продукт, продажи, операции, знания, партнёрства, риски, коммуникации, рост, governance. Нейтральный, без встроенных ценностей бизнес-инструмент.

### Как форкнуть

**ANY founder** берёт generic Layer 3 base → адаптирует под свой бизнес → опционально добавляет overlay.

Шаги форка:
1. Дублировать template-воркспейс.
2. Переименовать под свой бизнес.
3. Пройтись по 12 группам баз — включить нужные, отключить лишние.
4. Активировать extension points (каждая база имеет явный блок «если X-бизнес → добавь поля Y»).
5. Опционально — наслоить overlay (Jetix / VC / agency / SaaS / свой / никакого).
6. Первая неделя адаптации: внести реальные данные, собрать Command Center.

Весь процесс — самостоятельно, без обращения к автору шаблона, ≤3 часов.

### Примеры overlay для Layer 3

| Overlay | Для кого | Что добавляет |
|---|---|---|
| **Jetix cooperative** | Кооператив с R12 | Роли + Mondragón + Charter + fork-and-leave |
| **VC-startup** | Стартап с инвесторами | Cap table, board votes, investor reporting, cliff/vest |
| **Agency** | Сервисное агентство | Utilization, project margin, client health |
| **SaaS** | Продуктовая компания | MRR, churn, cohort retention, activation |
| **Solo consultant** | Фриланс / консалтинг | Simple pipeline, project tracking, rate management |
| **Без overlay** | Любой бизнес | Generic Layer 3 достаточно |

Jetix = ОДИН пример. Не лучший, не единственный — конкретный инстанс для конкретной организации.

### Fork-and-leave из бизнеса Layer 3

При уходе основателя или ключевого участника:

- **Данные бизнеса** — экспорт доступен (Notion позволяет экспортировать весь воркспейс).
- **Continuity**: при правильно выстроенном Layer 3 бизнес не зависит от конкретного основателя — это и есть смысл «Scale: founder = 1 из многих» (Layer 3 фаза зрелости).
- **При Jetix overlay**: Contribution Ledger сохраняет историю + proportional share treasury + 30-day notice при Charter change.

## §5 R12 enforcement — ТОЛЬКО для Jetix overlay (Layer 2 Part B + Layer 3 Jetix overlay)

### Критически важное разграничение

R12 **не enforced** на generic baseline ни Layer 2, ни Layer 3.

Generic baseline = нейтральный бизнес-инструмент. Бизнес может быть капиталистическим стартапом, корпоративной командой, кооперативом, агентством — generic база работает для всех. Встраивать R12 enforcement в generic = навязывать конкретную ценностную позицию.

R12 включается **только** если организация выбирает Jetix cooperative overlay. Это явный, осознанный выбор.

### Что такое R12

R12 — правило анти-экстракции: никто в системе не может извлекать ценность сверх согласованного. Члены могут уйти в любой момент без штрафов. Ratio cap ограничивает неравенство.

Конкретно при Jetix overlay:

**8 paired-frame вопросов** (каждый денежный шаблон / каждое ценностное предложение проходит через них):
1. Цена ≤ польза для конкретного человека?
2. Конкретно сформулировано?
3. Соразмерно отношениям?
4. Соответствует стадии отношений?
5. Канал подачи уместен?
6. Нет «доения» / «запирания»?
7. Нет манипуляции?
8. Стоп-сигнал готов и понятен?

Хоть один FAIL → не применяется / HALT.

**4 action classes (детекторы)**:
- `extraction_beyond_share` — забрать сверх договорённого → HALT
- `wage_ratio_violation` — разрыв в выплатах >5× (Mondragón cap) → HALT
- `non_consensual_distribution` — опубликовать чужое без согласия → HALT
- `fork_prevention_attempt` — язык «теперь не уйдёшь» / штрафы за уход → HALT

**Mondragón 5:1 ratio cap**: максимальная выплата не может превышать минимальную более чем в 5 раз. В Jetix Founding Partner разрыв ещё меньше. Enforced на уровне Charter + (Phase 2+) smart contract.

**Charter + re-ack**: любое изменение условий требует явного согласия каждого + 30-day opt-out window без штрафов.

### Мета-R12 инсайт

Навязывать R12 всем организациям, которые используют шаблон, = само по себе анти-R12.

Если generic Layer 3 «по умолчанию» встраивает Mondragón и кооперативные роли — это идеологический lock-in. Основатель стартапа в капиталистической модели не может «уйти» от кооперативной грамматики. Это именно то, с чем борется R12 — невозможность уйти без потерь.

Поэтому **generic = нейтральный, R12 = opt-in через overlay**. Это и есть R12 на мета-уровне: не запираем даже в «правильную» модель.

## §6 Universal R12 best practices (полезно для кооперативных организаций)

Независимо от overlay, есть практики которые полезны любой организации с несколькими участниками — не как enforcement, а как здравый смысл:

**Anti-extraction mindset (любой бизнес)**:
- Условия партнёрства / сотрудничества — записаны явно, доступны всем участникам.
- Изменение условий — с уведомлением заранее, не постфактум.
- Нет скрытых «штрафов за уход» в договорах.

**Fork-and-leave как базовая порядочность**:
- Данные участника — его. Экспорт всегда доступен.
- Уход не сопровождается «обнулением» вклада в прошлом.
- Нет пунктов «если уйдёшь, теряешь доступ к своим материалам».

**Mondragón-aware ratio cap (кооперативным организациям)**:
- Разрыв в вознаграждении >10× начинает создавать проблемы мотивации и доверия.
- 5:1 (Mondragón модель) — практически проверенный ориентир для cooperative-adjacent организаций.
- Это не закон, это best practice для тех, кому важно сохранить горизонтальную культуру.

Эти практики не навязываются. Они просто есть в разделе «полезно для cooperative organizations» Layer 3 generic — как один из разделов «considerations», не как enforcement.

## §7 Fork-and-leave mechanics: сквозная картина по трём слоям

| Слой | Что сохраняется при уходе | Механика |
|---|---|---|
| **Layer 1** | Все личные данные (отдельный воркспейс, не трогается) | Архитектурная изоляция |
| **Layer 2 generic** | Доступ к командным данным + экспорт | Базовая порядочность |
| **Layer 2 Jetix overlay** | + Contribution Ledger строка + накопленная доля + 30-day notice | Charter + overlay enforcement |
| **Layer 3 generic** | Полный экспорт воркспейса | Notion native export |
| **Layer 3 Jetix overlay** | + Proportional treasury share + Contribution Ledger + 30-day opt-out | Charter + R12 mechanics |

**Ключевые свойства fork-and-leave на всех слоях:**

**Exit anytime.** Нет периода «обязательного пребывания» без согласия участника. Если Charter содержит такой период — он должен быть явно согласован при входе, не «вписан мелким шрифтом».

**Keep your data.** Данные — не «в системе», не «принадлежат платформе». Notion воркспейс принадлежит тому, кто его создал. При правильной архитектуре (изолированные воркспейсы) уход не требует «попросить вернуть данные» — они изначально у тебя.

**Keep your share.** При Jetix overlay — вклад, сделанный до ухода, не аннулируется. Если было накоплено N points в Contribution Ledger — они остаются. Proportional share treasury при уходе — выплачивается.

**No penalty.** 30-day window при любом изменении Charter — уйти можно без штрафа. «Язык штрафа» в любом документе — fork_prevention_attempt, детектируется и блокируется при Jetix overlay.

**Cross-layer independence.** Уход из Layer 2 (команды) не влияет на Layer 1 (личный воркспейс). Уход из Layer 3 (бизнес-системы) не влияет на Layer 1 и Layer 2. Это не обещание — это архитектура.

## §8 Как форкнуть: пошаговый гайд по слоям

### Layer 1 (Personal Core)

1. Дублировать шаблон в свой Notion аккаунт.
2. Выбрать нишу (инженер / исследователь / предприниматель / методолог / гуманитарий).
3. Переименовать под себя.
4. Активировать нужные базы (можно начать с 3-4 центральных).
5. Внести первые реальные данные: текущие проекты, контакты, заметки.
6. Готово. Никаких дополнительных шагов.

### Layer 2 (Team Collaboration)

1. Выбрать путь: (A) generic baseline / (B) Brand pattern / (C) Jetix overlay / (D) свой overlay.
2. Для (A) и (B): дублировать нужный шаблон → адаптировать под команду → подключить участников.
3. Для (C) Jetix overlay: дополнительно → прочитать Charter template → согласовать с командой условия → подписать Charter → активировать R12 mechanics.
4. Для (D) свой overlay: взять generic Part A → написать свой Part B → документировать явно.
5. Подключить к Layer 1 если нужна личная / командная интеграция.

### Layer 3 (Universal Business Foundation)

1. Дублировать generic Layer 3 template.
2. Переименовать под бизнес.
3. Пройтись по 12 группам баз — включить нужные.
4. Для каждой базы активировать extension points под свой тип бизнеса.
5. Выбрать overlay (или не выбирать — generic достаточен).
6. Для Jetix overlay: всё как Layer 2 пункт 3 + экономические mechanics (Contribution Ledger, treasury, Charter).
7. Собрать Command Center из ключевых баз.
8. Первая неделя: внести реальные данные, откалибровать.

## §9 Почему R12 scoped именно так (а не шире)

Это не технический выбор — это ценностный.

**Вариант A (не выбрали)**: R12 в generic base для всех. Тогда стартап, берущий Layer 3 для cap table и investor reporting, получает встроенный Mondragón и роли кооператива. Его данные и структура «помечены» кооперативной моделью. Уйти от этого без переработки грамматики — нельзя. Это lock-in.

**Вариант B (не выбрали)**: Вообще никакого R12, даже как opt-in. Тогда кооперативные организации, которым R12 важен, вынуждены строить весь enforcement с нуля. Это лишает смысла shared substrate.

**Вариант C (выбрали)**: Generic = нейтральный, R12 = opt-in overlay. Каждый тип организации получает инструмент под свою модель. Кооперативы — полный R12 enforcement. Стартапы — cap table и VC-overlay. Агентства — свою модель. И все могут обновлять generic base из апстрима не теряя своего overlay.

Это одновременно технически правильно (модульность) и ценностно честно (не навязываем).

## §10 Constitutional posture Phase 8

- **R1 surface only** — какой overlay выбрать (Jetix/VC/agency/Brand pattern/none), включать ли R12 = выбор конкретной организации / Ruslan для Jetix instance.
- **R2 STRICT** — архитектурные изменения в Layer A (грамматика) требуют явного gate; overlay changes = Layer B, меньший blast-radius.
- **R6** — знания из апстрима обновляют только Layer A; Layer B (overlay) остаётся стабильным при обновлениях.
- **R11** — Ethereum on-chain enforcement = Phase 2+ opt-in, только при явном Charter flag; не auto для всех Jetix overlay.
- **R12 paired-frame** — scoped СТРОГО на Jetix cooperative overlay; generic baselines нейтральны; 8 вопросов + 4 action classes активны только при выбранном overlay; мета-R12 инвариант: не запираем даже в «правильную» модель.
- **IP-1 STRICT** — Jetix overlay = RUSLAN-LAYER instance, конкретная организация; roles в generic = abstract role-types; executor bindings вне этого документа.
- **Append-only** — Contribution Ledger, Charter change log, R12 Audit Log — только append, не редактировать прошлые записи.
