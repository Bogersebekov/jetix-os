# Phase 4 — Layer 3: Универсальный фундамент управления бизнесом

> **Что это.** Layer 3 — это **готовый нейтральный шаблон для управления любым бизнесом**:
> консалтинг, агентство, продукт, студия, маленькая команда — что угодно. Берёшь, копируешь
> себе в Notion, переименовываешь под свой бизнес — и у тебя сразу есть «пульт управления»:
> цели, деньги, проекты, люди, решения, брифинг. Всё на одной странице.
>
> **Тон.** Это **не «вот корпорация Jetix»**. Это «вот тебе чистый каркас, разворачивай свой
> бизнес». Универсальный, как шаблонный Excel-файл, только живой и связанный.
>
> **Главное отличие этой версии (v2) от v1:** сильно упростили. Раньше каждая база была расписана
> с кучей формул и расширений. Теперь — **минимальный базовый набор полей**, который работает
> сразу, плюс короткая заметка «🔧 что можно добавить, когда вырастешь». Меньше схем — больше
> объяснений, зачем это нужно.
>
> **STANDALONE.** Layer 3 работает сам по себе. Тебе **не нужны** Layer 1 (личная система) и
> Layer 2 (командная коллаборация), чтобы запустить управление бизнесом. Если хочешь — можно
> связать с Layer 1 (см. §4.X), но это **опция, а не требование**.

## §3.0 Обзор: 12 баз + новые страницы

Layer 3 — это **операционка для founder'а**: ты видишь весь бизнес с одной страницы и управляешь
им. Раньше всё держалось в голове, в заметках, в десяти разных табличках — теперь в одном месте,
где всё связано между собой.

| # | База | Зачем (одной строкой) | Ядро / опц |
|---|---|---|---|
| §3.1 | 🎯 Strategy & Goals | куда идём + по какой метрике мерим | ядро |
| §3.2 | 💰 Financial Overview | сколько денег / сколько горим / сколько осталось | ядро |
| §3.3 | 👥 Team / People | кто в команде + кто за что отвечает | ядро |
| §3.4 | 🚀 Projects Portfolio | что делается + на какой стадии | ядро |
| §3.5 | 🤝 Stakeholders / CRM lite | клиенты / партнёры / инвесторы | ядро |
| §3.6 | ⚖️ Decisions Log | что решили + почему + чем кончилось | опц |
| §3.7 | 🛡️ Risks Register | что может пойти не так + что делаем | опц |
| §3.8 | 📜 Compliance & Legal | юр / договоры / комплаенс | опц |
| §3.9 | 🧰 Tools Inventory | чем работаем + сколько стоит | опц |
| §3.10 | 📚 Documents / Knowledge Hub | где что лежит | опц |
| §3.11 | 📊 Executive Briefing | ежедневный/недельный обзор для founder | ядро |
| §3.12 | 🚨 Crisis Mode Playbook | что делать, когда всё горит | опц |

Плюс **три новых элемента** в этой версии:

- 🧪 **Hypotheses DB** (§3.13) — где ты записываешь свои предположения о бизнесе и проверяешь их.
- 📖 **Vision / Mission / Goals страницы** (§3.14) — текстовые документы, не таблицы: словами
  объясняют, зачем существует бизнес и куда он идёт.
- 🔗 **Layer 1 ↔ Layer 3 fast-connect** (§3.15) — если хочешь личную систему и бизнес в одном
  воркспейсе.

**Минимум для старта** (founder за 2-2.5 часа): §3.1 + §3.2 + §3.4 + §3.11 (стратегия + деньги +
проекты + брифинг). Остальное наслаивается по мере роста.

**Принцип «🔧 что можно добавить».** Каждая база описана как **простой скелет** + маленькая заметка
сбоку: «если у тебя вырастет X — добавь поле Y». Скелет без добавлений уже работает; добавления —
под конкретику твоего бизнеса. Не строй сразу всё. Построй минимум, поживи с ним неделю, добавь
то, чего реально не хватает.

## §3.1 🎯 Strategy & Goals (generic)

**Зачем.** Куда идёт бизнес и по какой метрике мерим. База **не навязывает фреймворк** — ты
выбираешь сам: OKR, V2MOM, EOS Rocks, или просто «3 цели на квартал». База даёт универсальный слот:
**цель → метрика → ответственный → срок → статус**. Это специально: разные founder'ы любят разные
системы, и шаблон не должен заставлять тебя жить по чужой методологии.

**Базовые поля (минимум):**

| Поле | Тип | Зачем |
|---|---|---|
| Title | Title | название цели |
| Type | Select | annual / quarterly / monthly |
| Target | Number/Text | целевое значение метрики |
| Current | Number | где сейчас |
| Progress % | Formula | `prop("Current")/prop("Target")*100` |
| Owner | Relation → People | кто отвечает |
| Horizon | Date | срок |
| Status | Select | on-track / at-risk / off-track / done |

**Views:** Текущий квартал, По ответственному, Под риском (`Status = at-risk/off-track`).

> 🔧 **Что можно добавить.** Если используешь OKR — добавь поля Objectives + Key Results (KR1-KR3
> с числами). Если V2MOM — добавь Values / Methods / Obstacles / Measures. Если EOS/Traction —
> добавь квартальные Rocks. Если хочешь «светофор» автоматом — добавь поле Health с формулой
> `if(прогресс >= ожидаемого_к_этой_дате, "🟢", "🟡"/"🔴")`. Не нужно сразу — добавь, когда целей
> станет больше пяти и захочется автоматики.

## §3.2 💰 Financial Overview (generic)

**Зачем.** Денежная картина: сколько зарабатываешь, сколько тратишь, и — главное — **на сколько
месяцев хватит денег** (runway). Это самая важная цифра для любого молодого бизнеса. Если ты не
знаешь свой runway, ты управляешь с закрытыми глазами.

**База Revenue (минимум):**

| Поле | Тип |
|---|---|
| Stream name | Title |
| Amount | Number (валюта) |
| Period | Date |
| Status | Select (projected / invoiced / received) |
| Customer | Relation → Stakeholders |

**База Expenses (минимум):**

| Поле | Тип |
|---|---|
| Name | Title |
| Category | Select (operating / people / tools / marketing / other) |
| Amount | Number |
| Recurring? | Checkbox |

**Runway gauge (главное число).** Один показатель на дашборде:

`Runway months = prop("Current cash") / prop("Monthly burn")` → подсветка 🔴 если меньше 3.

Где `Monthly burn` — сумма повторяющихся расходов за месяц. Эта одна цифра отвечает на вопрос
«сколько у меня времени». Всё остальное в финансах вторично, пока runway под контролем.

**Views:** P&L за месяц (доход vs расход), Runway gauge, Неоплаченные счета.

> 🔧 **Что можно добавить.** Invoices (счета): Number / Customer / Amount / Due date / Status +
> формула «просрочен» `and(prop("Due") < now(), prop("Status") != "paid")`. Pricing tiers, если у
> тебя тарифы. Cap table, если поднимаешь инвестиции. Multi-currency, если работаешь в нескольких
> валютах. Для старта достаточно: текущий кэш + повторяющиеся расходы = runway считается сам.

## §3.3 👥 Team / People (generic)

**Зачем.** Кто в команде и кто за что отвечает. База **не навязывает структуру** — иерархия,
плоская, матрица — выбираешь ты. Даже если ты один, эта база полезна: туда попадают подрядчики,
фрилансеры, советники.

**База Members (минимум):**

| Поле | Тип |
|---|---|
| Name | Title |
| Role | Relation → Roles |
| Status | Select (active / onboarding / paused / past) |
| Responsibilities | Text |
| Reports to | Relation → Members |

**База Roles (библиотека ролей — абстрактная, ты заполняешь сам):**

| Поле | Тип |
|---|---|
| Role title | Title |
| Responsibilities | Text |
| Decision rights | Text |

**Важный приём — роль ≠ человек.** Заметь: Roles (роли-контейнеры) и Members (конкретные люди)
разнесены. Это снимает проблему «незаменимости человека X»: когда роль описана отдельно от
человека, передать обязанности другому — это просто сменить relation, а не переписывать всё с
нуля. Полезно даже в команде из двух человек.

**Views:** Активная команда, По ролям, Оргструктура (через reports-to).

> 🔧 **Что можно добавить.** Comp band (вилка зарплат), если важно. Start date / контакты. Матрица
> «роль × человек», если один человек носит несколько шляп. Reporting chains, если структура
> растёт. Для старта: имя + роль + за что отвечает.

## §3.4 🚀 Projects Portfolio (generic)

**Зачем.** Что делается и на какой стадии. Это **высокоуровневый портфельный взгляд** — не
детальный трекинг задач (это уровень команды, Layer 2), а «какие у нас вообще проекты идут и как
они себя чувствуют».

**База Projects (минимум):**

| Поле | Тип |
|---|---|
| Name | Title |
| Status | Select (proposed / active / paused / done / killed) |
| Stage | Select (early / mid / late / wrap) |
| Owner | Relation → People |
| Target end | Date |
| Linked goals | Relation → Goals |

**Views:** Доска по стадиям (kanban), По ответственному, Активные vs завершённые.

> 🔧 **Что можно добавить.** Health-формула (сравнение Target end с сегодня → 🟢🟡🔴). Project value
> + margin, если проекты приносят деньги напрямую (тогда сумма течёт в §3.2). Sprint / velocity,
> если работаешь по Agile. Stage Gates, если у тебя формальные этапы с критериями.

## §3.5 🤝 Stakeholders / CRM lite (generic)

**Зачем.** Лёгкий CRM: клиенты, партнёры, инвесторы, советники, вендоры — все, с кем у бизнеса
отношения. Это не полноценная sales-CRM (для глубоких продаж — отдельный инструмент), а «список
важных людей и организаций + когда я с ними последний раз говорил».

**База Contacts (минимум):**

| Поле | Тип |
|---|---|
| Name | Title |
| Kind | Select (person / org) |
| Type | Select (customer / partner / investor / advisor / vendor / other) |
| Status | Select (lead / active / paused / past) |
| Last touch | Date |
| Days since touch | Formula `dateBetween(now(), prop("Last touch"), "days")` |
| Next action | Text |

**Главная польза — поле «Days since touch».** Оно автоматически показывает, с кем ты давно не
общался. Отношения «протухают» молча; это поле делает молчание видимым. View «Нужен follow-up»
(`Days since touch > N`) — твой еженедельный напоминальник.

**Views:** Pipeline (доска по стадии сделки), Нужен follow-up, По типу контакта.

> 🔧 **Что можно добавить.** Deal stage + amount + probability, если ведёшь продажи (тогда
> появляется forecast). Engagement health (strong/ok/weak). Relation контакт↔контакт (организация
> → её люди). Для старта: топ-10-20 важных контактов + дата последнего касания.

## §3.6 ⚖️ Decisions Log (generic)

**Зачем.** Что решили, почему, и чем это кончилось. Это **институциональная память бизнеса**. Через
полгода ты не вспомнишь, почему выбрал поставщика A, а не B — а здесь будет записано. И, что важнее,
поле «Outcome» заставляет тебя вернуться и оценить, было ли решение правильным.

**База Decisions (минимум):**

| Поле | Тип |
|---|---|
| Title | Title |
| Date | Date |
| Category | Select (strategic / financial / hiring / product / legal / other) |
| Status | Select (proposed / approved / executed / reversed) |
| Rationale | Text (почему) |
| Outcome | Text (чем кончилось — заполняешь позже) |
| Review date | Date (когда оценить) |

**Зачем поле Outcome + Review date.** Это **встроенная петля обучения**: бизнес учится на своих
решениях, а не повторяет ошибки. View «На ревью» (`Review date <= сегодня, Outcome пустой`)
напоминает вернуться и честно записать, сработало или нет. Большинство людей принимают решения и
никогда не оглядываются — эта база делает оглядку механической.

**Views:** Недавние, По категории, На ревью, Откаченные (учимся на откатах).

> 🔧 **Что можно добавить.** Impact assessment. RACI (Responsible / Accountable / Consulted /
> Informed). Linked projects/stakeholders (на что влияет решение — трассируемость). Board vote
> tracking, если есть совет директоров.

## §3.7 🛡️ Risks Register (generic)

**Зачем.** Что может пойти не так и что мы с этим делаем. Риск, записанный и с назначенным
ответственным, — управляемый. Риск в голове — нет.

**База Risks (минимум):**

| Поле | Тип |
|---|---|
| Title | Title |
| Category | Select (financial / operational / legal / market / people) |
| Severity (1-5) | Number |
| Likelihood (1-5) | Number |
| Score | Formula `prop("Severity")*prop("Likelihood")` |
| Mitigation | Text |
| Owner | Relation → People |
| Status | Select (open / mitigating / accepted / closed) |

**Views:** Топ рисков (sort по Score), По категории, Открытые высокоприоритетные.

> 🔧 **Что можно добавить.** Priority-формула с цветом (`Score>=15 → 🔴`). Review date. Sub-категории
> под индустрию (healthcare / finance) с привязкой к регулятору. Для старта: 5-10 главных рисков с
> оценкой важности × вероятности.

## §3.8 📜 Compliance & Legal (generic)

**Зачем.** Юр-статус, договоры, комплаенс, дедлайны по налогам. Скучно, но именно здесь бизнесы
получают неожиданные штрафы и проблемы. База — это «ничего не забыть».

**База Contracts (минимум):**

| Поле | Тип |
|---|---|
| Title | Title |
| Party | Relation → Stakeholders |
| Type | Select (client / vendor / employment / nda / partnership) |
| End / Renewal date | Date |
| Status | Select (draft / active / expired / terminated) |

**База Compliance checklist (минимум):** Item / Category (налоги / страховка / лицензии / privacy) /
Due date / Status / Owner.

**Views:** Активные договоры, Скоро истекают (`Renewal <= сегодня + 30д`), Комплаенс к сроку.

> 🔧 **Что можно добавить.** Legal Entity (форма юрлица / юрисдикция / налоговый ID). Contract value
> + формула «expiring soon». Привязка к локальной специфике (для DE — Steuerberater + выбор формы
> юрлица). Для старта: список действующих договоров + ближайшие дедлайны.

## §3.9 🧰 Tools Inventory (generic)

**Зачем.** Чем работаем и сколько это стоит. Подписки накапливаются незаметно и тихо съедают
бюджет. База показывает все инструменты и их стоимость — и какие из них никто не использует.

**База Tools (минимум):**

| Поле | Тип |
|---|---|
| Name | Title |
| Category | Select (dev / design / ops / marketing / ai / finance) |
| Status | Select (active / trial / deprecated) |
| Cost / period | Number |
| Usage | Select (daily / weekly / rare) |

**Views:** Активные, По категории, Стоимость (сумма → кормит §3.2 Expenses), Недоиспользуемые
(`Usage = rare & Cost > 0` — кандидаты на отмену).

> 🔧 **Что можно добавить.** Owner / renewal date. AI-tools sub-категория + версии моделей, если
> AI-heavy. Seat count, если remote-команда. Для старта: список платных подписок + стоимость в
> месяц = видишь общий tool-spend.

## §3.10 📚 Documents / Knowledge Hub (generic)

**Зачем.** Где что лежит — индекс документов бизнеса. Не сами документы, а **карта**: где найти
договор, где методичку, где финплан. Без этого через полгода «у меня это где-то было…» съедает часы.

**4 категории** (выровнено с внутренней классификацией DOCS-CLASSIFICATION — любой бизнес выигрывает
от такого деления):

- 🟢 **Пояснительные** — то, что можно показать наружу (клиентам, партнёрам): офферы, презентации,
  публичные тексты.
- 🛠️ **Шаблоны** — рабочие структуры: договоры-шаблоны, чек-листы, скрипты.
- 📚 **Substrate / глубина** — внутренний research, методология, база знаний (для своих, не наружу).
- ⚙️ **System / инфраструктура** — конфиги, доступы, внутренние регламенты (никогда наружу).

**База Document Index (минимум):**

| Поле | Тип |
|---|---|
| Title | Title |
| Category | Select (🟢 explanation / 🛠️ template / 📚 substrate / ⚙️ system) |
| Status | Select (draft / active / archived) |
| Link | URL |
| Access | Select (public / internal / restricted) |

**Зачем именно эти 4 категории.** Они решают вопрос «что можно показать человеку наружу?» за 2
секунды: берёшь только 🟢 (и при необходимости релевантный 🛠️), никогда не отдаёшь 📚 или ⚙️
напрямую. Это спасает от типичной ошибки — вывалить на нового клиента/партнёра внутренние документы,
после чего он тонет и закрывается.

**Views:** По категории, Что можно наружу (`Category = 🟢`), Недавно обновлённые, Устаревшие
(`Last updated > 180д`).

> 🔧 **Что можно добавить.** Owner / last updated / version control для SOP. Привязка к wiki, если у
> тебя база знаний. Для старта: список ключевых документов + ссылки + категория.

## §3.11 📊 Executive Daily/Weekly Briefing (generic)

**Зачем.** «Весь бизнес на одной странице». Брифинг — это **главный артефакт Layer 3**: он
превращает 12 баз в один управляемый поток внимания. Ты не открываешь 12 таблиц каждое утро — ты
смотришь один брифинг, где собрано самое важное.

**Важно: DRAFT-only.** Если у тебя есть ассистент (человек или AI), он **готовит черновик** брифинга
— собирает цифры из баз. Но **решения принимаешь ты**. Ассистент предлагает, не действует. Это
дисциплина: автоматика собирает данные, человек принимает решения.

**База Briefing (per период):**

| Поле | Тип |
|---|---|
| Period | Title (дата) |
| Cadence | Select (daily / weekly / monthly) |
| Critical attention | Text (≤5 пунктов) |
| Health snapshot | Text (метрики delta) |
| Progress | Text (закрыто / стартует / заблокировано) |
| People movement | Text |
| Strategic threads | Text (новые идеи / решения awaiting) |
| Reviewed? | Checkbox |

**5 секций брифинга:**

- 🚨 **Critical attention** — ≤5 пунктов, требующих твоего решения. Лимит в 5 — намеренный:
  если каждое утро 20 «срочных» дел, ты не управляешь, ты тушишь пожары. Пять — это столько,
  сколько человек реально удержит.
- 📊 **Health snapshot** — ключевые метрики и их изменение (runway / выручка / здоровье проектов).
- 🎯 **Progress** — что закрыто, что стартует, что заблокировано.
- 🤝 **People/Stakeholders movement** — кто пришёл, кто ушёл, кому нужен follow-up.
- 💡 **Strategic threads** — новые идеи, решения, ждущие тебя.

**Views:** Сегодняшний брифинг, История брифингов, Непросмотренные.

> 🔧 **Что можно добавить.** Если есть AI-ассистент — pipeline автогенерации (rollup из баз →
> черновик). Distribution rules, если брифинг рассылается команде. Для старта: один weekly-template
> с пятью секциями, заполняешь раз в неделю.

## §3.12 🚨 Crisis Mode Playbook (generic)

**Зачем.** Что делать, когда всё горит. Кризис — худшее время, чтобы думать «а что теперь?».
Заранее прописанные процедуры экономят критические часы и нервы.

**База Crisis Playbooks (минимум):**

| Поле | Тип |
|---|---|
| Crisis type | Title |
| Category | Select (financial-shock / key-person-departure / public-incident / legal / outage) |
| Trigger | Text (когда срабатывает) |
| Immediate actions | Text (чек-лист) |
| Escalation | Relation → People |

**5 типичных категорий:** финансовый шок (резкое падение выручки / неожиданный расход / сжатие
runway) / уход ключевого человека / публичный инцидент (репутация) / юр-триггер / операционный сбой.

**Views:** Все плейбуки, По категории, Активный кризис (если триггернулся).

> 🔧 **Что можно добавить.** Communication template (что сказать клиентам/команде). Debrief checklist
> (разбор после кризиса). Для старта: 2-3 самых вероятных для тебя кризиса + что делать в первый час.

## §3.13 🧪 Hypotheses DB (NEW — baseline)

**Зачем.** Бизнес — это набор предположений: «клиенты заплатят за X», «канал Y приведёт лидов»,
«цена Z не отпугнёт». Большинство людей держат эти гипотезы в голове как факты. База гипотез
заставляет **записать предположение явно** и потом проверить, оказалось оно правдой или нет. Это
превращает «мне кажется» в «мы проверили».

**База Hypotheses (минимум):**

| Поле | Тип | Зачем |
|---|---|---|
| Hypothesis | Title | формулировка предположения |
| Type | Select | market / product / pricing / channel / operational |
| Status | Select | proposed / testing / confirmed / refuted |
| Metric | Text/Number | по какому числу поймём, правда ли |
| Horizon | Date | к какому сроку проверим |
| Outcome | Text | что выяснили (заполняешь после) |

**Как пользоваться.** Записываешь гипотезу → выбираешь метрику, которая её подтвердит или опровергнет
→ ставишь срок → через срок возвращаешься и пишешь Outcome (confirmed / refuted). View «На проверке»
(`Status = testing & Horizon <= сегодня`) напоминает закрыть гипотезу честно.

**Views:** Активные (testing), На проверке к сроку, Подтверждённые, Опровергнутые (особенно ценны —
это то, во что ты больше не веришь зря).

> 🔧 **Что можно добавить (для продвинутых — Bayesian-lite).** Если ты используешь дисциплину
> гипотез из Method V2 §J — добавь поля: prior confidence (0-1, насколько верил до проверки) /
> posterior confidence (после) / evidence strength. Это даёт байесовский трекинг: видно, как
> убеждения сдвигаются под давлением реальности, а не остаются «я так и знал». Для обычного
> founder'а это избыточно — хватает proposed → testing → confirmed/refuted.

## §3.14 📖 Vision / Mission / Goals страницы (NEW — текстовые, не базы)

Здесь важный момент: **не всё должно быть таблицей**. Базы хороши для того, что считается и
фильтруется (цели с процентами, контакты с датами). Но **«зачем существует бизнес» нельзя засунуть в
ячейку**. Для этого нужны нормальные текстовые страницы, где ты пишешь словами, человеческим языком.

Почему это критично: таблица целей отвечает на «что мы делаем в этом квартале». Но она не отвечает на
«зачем мы вообще всем этим занимаемся». А именно ответ на «зачем» удерживает тебя (и будущую команду)
в трудные месяцы, помогает говорить «нет» хорошим, но не-твоим возможностям, и звучит честно, когда
ты объясняешь бизнес партнёру или клиенту. Голая Strategy DB этого не даёт — она про метрики, не про
смысл.

### 📄 Vision (страница)

**Что это.** 1-2 абзаца: каким ты видишь мир, если бизнес преуспеет. Не «мы заработаем денег», а
«что изменится для людей, которым мы служим». Это далёкий горизонт.

- **Зачем эта страница:** даёт направление, к которому сверяешься, когда непонятно, куда двигаться.
- **Как обновлять:** редко и осознанно. Vision не должен меняться от квартала к кварталу.
- **Когда ревизировать:** раз в год, или при крупном развороте (pivot) бизнеса.

### 📄 Mission (страница)

**Что это.** Чем именно ты занимаешься прямо сейчас, чтобы двигаться к Vision. Vision — это «куда»,
Mission — это «как мы туда идём сегодня». Конкретнее и приземлённее, чем Vision.

- **Зачем эта страница:** объясняет, что бизнес делает на практике, без жаргона. Это то, что ты
  говоришь, когда тебя спрашивают «а чем вы занимаетесь?».
- **Как обновлять:** при смене основного фокуса или модели.
- **Когда ревизировать:** раз в полгода, или при смене продукта / рынка.

### 📄 Goals strategic narrative (страница, связана со Strategy & Goals DB §3.1)

**Что это.** Здесь ты словами рассказываешь про цели: почему именно эти, как они связаны, что важнее.
Это **прозаическое дополнение** к таблице §3.1. Таблица показывает числа и статусы; narrative
объясняет логику за ними.

- **Зачем эта страница:** таблица целей выглядит как список, но не показывает связи и приоритеты
  между целями. Narrative связывает их в историю: «в этом квартале фокус на X, потому что без него
  Y не взлетит; Z пока подождёт». Это удобнее читать и легче объяснить.
- **Как обновлять:** в начале каждого планового периода (квартал), синхронно с обновлением §3.1.
- **Когда ревизировать:** каждый квартал; связывай страницу с базой §3.1 через ссылку, чтобы числа и
  нарратив жили рядом.

**Связка с базой.** На странице Goals narrative вставляешь linked view базы §3.1 (Strategy & Goals) —
получается одна страница, где сверху прозой объяснено «почему эти цели», а ниже живая таблица с
прогрессом. Слова и числа в одном месте.

## §3.15 🔗 Layer 1 ↔ Layer 3 fast-connect (NEW)

**Ситуация.** Некоторые founder'ы хотят держать **личную ежедневную систему** (Layer 1: дневник,
личные цели, личные проекты) и **управление бизнесом** (Layer 3) в одном воркспейсе — чтобы утром
смотреть и личное, и бизнесовое сразу. Другим это не нужно: бизнес отдельно, личное отдельно.

**Принцип: это опция, не требование (opt-in).** По умолчанию слои **не сливаются автоматически**.
Layer 3 работает сам по себе, Layer 1 — сам по себе. Связка включается, только если ты её осознанно
делаешь.

**Если хочешь связать — четыре быстрых linked view:**

| Связка | Как | Что даёт |
|---|---|---|
| Daily Log (L1) → Briefing (L3) | linked view личного дневника на странице бизнес-брифинга | утром видишь личное и бизнес рядом |
| Goals (L1 личные) → Strategy & Goals (L3) | linked view, но базы остаются раздельными | личные цели не путаются с бизнес-целями, но видны вместе |
| Projects (L1 личные) → Projects Portfolio (L3 бизнес) | две раздельные базы, linked view обеих на одной странице | видишь личную и бизнес-загрузку, не смешивая |

**Permissions (важно).** Если у тебя есть команда:

- **Founder видит оба слоя** (личное + бизнес).
- **Остальные видят только Layer 3** (бизнес). Личный Layer 1 закрыт — это твоё пространство.

Это делается через Notion-доступы: личные базы остаются в приватной части воркспейса, бизнес-базы —
в shared. Linked view'ы на твоей личной dashboard-странице видишь только ты.

**Если НЕ хочешь связывать** — просто не делай linked view'ы. Каждый слой работает автономно, без
авто-слияния. Никакой потери функциональности: Layer 3 полноценен сам по себе.

## §3.X 🟧 Jetix overlay — ОТЛОЖЕНО (только упоминание)

Существует **Jetix-специфичный overlay** для Layer 3: R12 anti-extraction дисциплина, Mondragón 5:1
ratio cap, 10 ролей кооператива, Stage Gates, программируемый Ethereum-слой, Charter. **Это —
следующая итерация, отдельным документом.** Здесь он намеренно **не строится**.

Base остаётся **нейтральным**: любой бизнес форкает чистый foundation (§3.1-§3.15) и при желании
пишет свой собственный overlay — VC-overlay, agency-overlay, cooperative-overlay (Jetix-style) или
никакой. Jetix-overlay — это **один пример из многих возможных**, не часть базового шаблона. Тон base
сознательно универсальный: это инструмент для любого founder'а, а не презентация конкретной компании.

## §3.16 Worked example — solo founder за 2-2.5 часа (иллюстративный)

> Это **описание пути**, не реальные данные. Никаких фейковых записей — только показываем, как
> founder разворачивает минимум.

Представь solo founder'а (один человек, небольшой бизнес — неважно какой). Он разворачивает минимум:
**Strategy + Financial + Projects + Briefing**.

| Шаг | Время | Что делает |
|---|---|---|
| 1 | 20 мин | дублирует Layer 3 template-воркспейс, переименовывает под свой бизнес |
| 2 | 30 мин | §3.14 — пишет Vision (1 абзац) + Mission (1 абзац) текстом; §3.1 — заносит 3 цели на квартал с метриками |
| 3 | 40 мин | §3.2 — вносит текущий кэш + повторяющиеся расходы → **runway считается сам** |
| 4 | 30 мин | §3.4 — заносит активные проекты + ответственных |
| 5 | 15 мин | §3.11 — настраивает первый weekly briefing template (5 секций) |
| 6 | 15 мин | собирает простую dashboard-страницу: linked view целей + runway gauge + доска проектов + сегодняшний брифинг |

**Итого ~2.5 часа → рабочий пульт управления бизнесом.** Founder открывает одну страницу утром и
видит: на сколько хватит денег, какие цели в работе, какие проекты активны, что требует внимания.

**Что наслаивается потом.** Decisions Log — как только начнёшь принимать важные решения и захочешь их
помнить. Stakeholders — когда появятся клиенты/партнёры, которых легко потерять из виду. Risks /
Legal — когда появятся договоры. Hypotheses — когда захочешь проверять предположения системно, а не
на интуиции. Crisis — когда команда вырастет и цена сбоя поднимется. Ничего из этого не нужно в
первый день.

## §3.17 Closing — STANDALONE + constitutional posture

**Layer 3 работает один.** Это ключевое: founder'у **не нужны** Layer 1 (личная система) и Layer 2
(командная коллаборация), чтобы получить полноценное управление бизнесом. Можно развернуть только
Layer 3, и он самодостаточен — 12 баз + новые страницы (Vision/Mission/Goals + Hypotheses) + брифинг
+ dashboard. Связка с Layer 1 (§3.15) — приятная опция для тех, кто хочет всё в одном месте, но не
требование. Если позже появится команда с общим котлом — снизу подключается Layer 2, и Projects
Portfolio начинает агрегировать проекты команды. Был standalone — стал вершиной стека. Но это уже по
желанию и по росту.

**Конституционная позиция (constitutional posture).**

- **R1 surface only** — какие из 12 баз ядро vs опц, какие добавления включить, связывать ли с Layer
  1, писать ли overlay — это **решения founder'а** (в случае Jetix — Ruslan). Документ только
  surfaces опции, не решает за владельца.
- **R2 STRICT** — никаких автономных архитектурных решений; base = нейтральный каркас, любые
  Jetix-specific дополнения идут отдельной итерацией через явное решение.
- **R6** — generic-знание (enterprise-OS общая практика) без аггрегации внутренней закрытой памяти в
  публичный base.
- **R11** — base ничего не enforce'ит автономно; brifing DRAFT-only (ассистент предлагает, founder
  решает).
- **IP-1 STRICT** — роли в §3.3 **абстрактны** (контейнеры); конкретных людей и конкретные 10 ролей
  Jetix в base нет. Привязка «роль → человек» = RUSLAN-LAYER, делается пользователем под свой бизнес.
- **No sample data** — все поля и формулы = паттерны; worked example (§3.16) = описание пути, не
  записи. Jetix-overlay только упомянут (§3.X), не построен.
