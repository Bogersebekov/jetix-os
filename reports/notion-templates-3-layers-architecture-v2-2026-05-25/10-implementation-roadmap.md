# Phase 9 — Дорожная карта внедрения (3 слоя, пересмотрено)

> **Что в этой фазе.** Порядок сборки трёх слоёв в 3-слойной архитектуре v2.
> Как каждый слой может быть начат независимо (standalone), когда их полезно
> соединить, и как это выравнивается с этапами платформы Build / Run / Scale.
> Плюс решения по плану Notion на каждом этапе. Без схем — только текст.

---

## §1 Напоминание: что изменилось в v2

В первой версии архитектуры было 4 слоя: Layer 1 (личное ядро), Layer 2 (расширенное
личное), Layer 3 (командное), Layer 4 (универсальный бизнес-фундамент). В v2 Layer 2
«Личное расширенное» убран как отдельный слой — его содержимое либо вошло в Layer 1
(дополнения и шаблоны), либо отложено на Run.

Теперь три слоя:

- **Layer 1 — Personal Core** (личное ядро): Daily Log, Projects, Contacts, Goals,
  Command Center. Самодостаточен для одного человека. Самый нужный первым.
- **Layer 2 — Team Collaboration** (командная работа): Generic-шаблоны совместной
  работы + Jetix-overlay как отдельный именованный слой поверх. Нужен, когда появляется
  первый партнёр или тестер.
- **Layer 3 — Universal Business Foundation** (универсальный бизнес-фундамент):
  Стратегический executive-вид — финансы, цели, проекты, briefing-цикл. Standalone:
  основатель может использовать независимо от Layer 1 и Layer 2.

**Companion (не слой):** AI Tools mega-page — отдельная страница с инструментами,
лайфхаками и практиками. Можно собрать в любой момент параллельно, не блокирует и не
зависит ни от одного слоя.

---

## §2 Порядок внедрения: рекомендованная очерёдность

Важно сразу сказать: **каждый слой можно начать независимо.** Это не waterfall, где
Layer 2 невозможен без завершённого Layer 1. Это три отдельных Notion-workspace-а (или
раздела), каждый из которых работает сам по себе. Порядок ниже — рекомендация под
конкретный контекст (тестер Дмитрий + Ruslan как founder + первый партнёр), не жёсткий
закон.

| Очередь | Слой | Почему сейчас |
|---|---|---|
| 1 | **Layer 1 — Personal Core** | Самый базовый; нужен Дмитрию на trial |
| 2 | **Layer 3 minimum** | Executive-вид для Ruslan; standalone, не ждёт L1 |
| 3 | **Layer 2 demo (Generic + Jetix overlay)** | Co-design с первым партнёром |
| — | **AI Tools mega-page** | Параллельно в любой момент |

## §3 Layer 1 — Personal Core: первым, потому что фундамент большинства

### Почему первым

Layer 1 закрывает самый частый запрос: один человек хочет систему «для себя» — что
делаю, что важно, с кем общаюсь, куда иду. Для тестера (Дмитрий) нужно именно это.
Нет смысла показывать ему командные или бизнес-шаблоны на первом касании.

Кроме того, Layer 1 — это та самая «Personal OS», с которой начинался продукт. Если
говорить языком продаж: Layer 1 — это ступень 1 воронки, точка входа для большинства
людей.

### Что строим внутри Layer 1

Пять баз + один Command Center хаб:

1. **Daily Log** — первая и самая важная. Здесь начинается ритуал дня (открываю →
   записываю → закрываю). Без Daily Log нет ощущения «система работает». Строится
   первой.
2. **Projects** — активные проекты (3-5 на старт). Связана с Daily Log через relation.
3. **Contacts** — топ-5 важных людей. Связана с Projects.
4. **Goals** — цели (Точка А / Точка B). Связана с Projects.
5. **Command Center** — хаб-страница, собирает всё в одном месте. Строится последней,
   после того как 4 базы работают.

Порядок строительства внутри Layer 1: Daily Log → Projects → Contacts → Goals →
Command Center. Это занимает 1-2 часа при первой сборке.

После сборки: «засев» — первые 3-5 реальных записей в каждую базу. Дмитрий пишет своё,
не демо-данные. Это критично: пустой шаблон не даёт ощущения «это моё».

### Add-ons Layer 1 (Week 2)

Дополнения, которые делают Layer 1 богаче, но не блокируют старт:

- Reviews (короткий шаблон еженедельного ревью — встроен в Daily Log или отдельная
  страница)
- Hypotheses (гипотезы как отдельная база — для тех, кто ведёт эксперименты)
- Reading List (книги, статьи, заметки)

Эти add-ons можно добавить на второй неделе, когда основной ритуал уже работает.

### Notion-план для Layer 1

Layer 1 работает на **Free**. Один человек, несколько баз, базовые шаблоны — это
укладывается в бесплатный план Notion без ограничений по функциям. Апгрейдиться не
нужно.

Если человек хочет больше гостей или больше блоков истории (history) → **Plus (~$10-12/мес)**,
но это уже личный выбор, не требование архитектуры.

## §4 Layer 3 — Universal Business Foundation: вторым, для founder-кейса

### Почему вторым

Layer 3 — это executive-вид на бизнес. Не «как я провёл день», а «где мой runway, что
за проекты, куда движется компания, что случилось за неделю». Этот слой нужен прежде
всего Ruslan как основателю: смотреть на всё с птичьего полёта.

Он **standalone** — не зависит от Layer 1. Основатель может начать с Layer 3 вообще
без Layer 1, если ему нужен только бизнес-обзор. Когда появятся партнёры и команда,
Layer 3 начнёт агрегировать данные снизу (из Layer 2), но на старте это опционально.

Порядок Layer 3 вторым — это не «сначала разберись с личным, потом с бизнесом». Это
просто: большинство пользователей начнут с Layer 1 (им нужен личный инструмент), а
Ruslan нужен executive-вид для себя. Поэтому Layer 3 = второй в очереди под founder-кейс.

### Что строим внутри Layer 3 minimum

Минимальный набор (примерно 2-2.5 часа):

1. **Финансы — runway** (первой): cash, burn rate, сколько месяцев осталось, когда
   нужен следующий доход. Это самое срочное для founder — без runway всё остальное
   теряет смысл.
2. **Стратегические цели** (3-5 больших): что хочу достичь за квартал / год. Не
   задачи — именно цели верхнего уровня.
3. **Активные проекты** (executive view): 5-8 проектов, статусы, энергия (не детальный
   backlog — это в Layer 1 или в проектных инструментах).
4. **Briefing-страница** (weekly briefing): шаблон еженедельного обзора для себя —
   что произошло, что дальше, где застрял. Первый briefing пишешь в конце первой недели.
5. **Command Center Layer 3** (хаб): страница, которая собирает runway, цели, проекты
   и последний briefing в одном месте.

Это minimum viable executive OS. Его хватит, чтобы каждое утро понимать, «где я и куда
иду».

### Layer 3 full (позже)

Расширенная версия Layer 3 добавляется по мере роста — это Run-задача:

- Детальная P&L (доходы, расходы, категории)
- Партнёрский трекер (кто в каком статусе, что предложено)
- Pipeline проектов и клиентов
- Ретроспективы (месяц, квартал)
- Jetix-overlay поверх generic (если нужен кооперативный вид)

Full Layer 3 добавляется в Run, когда пошли деньги и появились партнёры. В Build —
только minimum.

### Notion-план для Layer 3

Если Layer 3 используется только Ruslan — **Free или Plus**. Один пользователь, несколько
баз. Апгрейд до Plus ($10-12/мес) уместен если нужна расширенная история или больше
работы с API. Business-план не нужен до появления multi-tenant (Layer 2 с командой).

## §5 Layer 2 — Team Collaboration: третьим, при первом партнёре

### Почему третьим

Layer 2 не нужен для одного человека. Он начинает иметь смысл, когда появляется первый
партнёр: нужно показать ему систему, co-design сделать вместе, дать ему workspace.

В v2 важна структура Layer 2: **Generic-шаблоны** и **Jetix overlay** — это два разных
слоя внутри Layer 2. Generic-шаблоны — универсальные инструменты командной работы (Project
Catalog, Skills/Needs матрица, shared workspace, Charter). Jetix overlay — специфичная для
Jetix-кооператива надстройка (R12-компоненты, кооперативный слой, специфические шаблоны).

Для demo с первым партнёром достаточно Generic-части. Jetix overlay подключается позже,
когда партнёр принял Charter и понял базовую систему.

### Что строим внутри Layer 2 для demo

Минимум для первого партнёра (примерно 2-3 часа):

1. **Shared Teamspace** (или отдельный Notion workspace): отдельное пространство, где
   будет работать Layer 2. Не смешивается с личным Layer 1.
2. **Permissions matrix** (страница): кто что видит. Простая таблица — участник /
   менеджер / гость. Для demo: Ruslan = manager, партнёр = collaborator.
3. **Project Catalog** (база): 3-5 реальных проектов из существующих. Это точка входа
   для партнёра — «смотри, вот что мы делаем».
4. **Skills/Needs** (страница или база): что каждый участник может предложить и что
   ищет. Простой формат, не база данных.
5. **Charter draft** (страница): текст соглашения. На demo — драфт, не финальный. Показывает
   принципы, условия, R12-принципы простыми словами.

### Generic vs Jetix overlay: как объяснить партнёру

Разговор с первым партнёром идёт сначала через Generic Layer 2: «Вот система совместной
работы. Project Catalog — смотришь, что делается. Skills/Needs — видишь, кому что нужно.
Charter — принципы, как мы работаем.» Это понятно любому, кто работал в Notion или
в любом другом инструменте.

Jetix overlay добавляется как расширение: «А вот Jetix-специфические вещи — кооперативная
модель, условия распределения дохода, R12-принципы.» Это подключается после того, как
партнёр сказал «да, понятно, мне интересно».

Разделение Generic/Overlay важно по другой причине: Generic Layer 2 — это то, что можно
показывать любому потенциальному партнёру без упоминания кооператива. Jetix overlay —
это специфика, которую рано показывать до первого подтверждения интереса.

### Что в Layer 2 на Run

В Run Layer 2 начинает работать по-настоящему: когорта 5-50 человек, shared Teamspace
с real permissions, Charter подписан тремя и более участниками, partner-led доставка.
Matching helper (автоматический подбор пар из Skills/Needs) — тоже Run. На Build это
делается вручную.

### Notion-план для Layer 2

Как только в Notion появляется реальный multi-tenant (несколько людей активно работают
в одном workspace) — нужен **Business план (~$15-18/user/мес)**. Он даёт Teamspaces,
расширенные rights, admin-панель.

Тайминг апгрейда — решение Ruslan (R1): можно апгрейднуться в Build week 2-3 под demo
(чтобы показать Teamspace партнёру), или в Run week 4-5 (когда multi-tenant реально
нужен). Вариант по умолчанию: апгрейд в Build week 2-3 под Team OS demo, это часть
Build exit conditions.

Free/Plus вариант для Layer 2 — если используется как solo-demo без настоящего
multi-tenant: технически возможно, но лимиты по гостям и правам могут мешать.

## §6 AI Tools mega-page: companion в любой момент

AI Tools mega-page — это не слой архитектуры. Это companion-документ: одна большая
Notion-страница (или несколько связанных страниц) с инструментами, лайфхаками и практиками
работы с AI.

**Структура mega-page:**
- Инструменты по категориям (текст, изображения, код, аудио, видео, поиск)
- Лайфхаки и промпты (персональная коллекция)
- Практики и режимы работы
- Ресурсы (курсы, видео, статьи)
- Быстрые ссылки (daily-use tools)

Почему это companion, а не слой: mega-page не требует никакой базы данных и не связана
с остальными слоями. Её можно собрать за один сеанс (1-2 часа), она не зависит от того,
работает ли Layer 1, 2 или 3. Можно начать сразу сегодня, можно через месяц.

Полезно начать параллельно с Layer 1 — пока строятся базы, параллельно можно накидывать
AI-инструменты. Это не блокирует ничего и не требует решений по плану Notion.

## §7 Выравнивание с Build / Run / Scale

Сейчас (25.05.2026) — **Build, средняя часть.** Substrate готов, Notion-шаблоны на уровне
дизайна, первый тестер (Дмитрий) в работе. Наружу ещё не вышло — нет видео, нет внедрённых
баз, нет Charter.

Вот как слои ложатся на этапы:

### Build (недели 1-3, ~26.05 — 14.06)

**Build week 1-2 — Layer 1:**

Layer 1 собирается первым. Это пять баз + Command Center. Дмитрий получает свою копию,
начинает пользоваться, даёт отзыв. Параллельно Ruslan внедряет Layer 1 для себя, если
ещё не сделано — это одновременно и личный инструмент, и тест шаблона перед показом
тестеру.

Нотион-план: Free для одного человека. Апгрейда не нужно.

**Build week 3 — Layer 3 minimum:**

После того как Layer 1 засеян (есть первые записи, понятно что работает), Ruslan собирает
Layer 3 minimum для себя: финансы/runway, стратегические цели, активные проекты, первый
briefing. Это 2-2.5 часа. Не зависит от Layer 1 — можно было начать раньше, но разумнее
начать когда Layer 1 уже работает и есть первый фидбек.

Layer 3 также полезен для Build exit check: когда есть briefing-страница, видно «где я
по пяти критериям выхода из Build» (видео готово / Notion внедрён / Charter проверен /
Wave 1 отправлен / T1 confirmed).

Нотион-план: Free или Plus.

**Build week 2-3 (параллельно) — Layer 2 demo:**

Параллельно с Layer 3 или сразу после — Layer 2 demo для первого партнёра. Это shared
Teamspace + permissions + Project Catalog + Charter draft. Показывается первому партнёру
в co-design формате: «собираем вместе, я показываю принципы, ты предлагаешь что добавить».

Апгрейд Notion до Business — решение Ruslan. Если нужен реальный Teamspace для показа
партнёру, апгрейд имеет смысл в Build week 2-3.

### Run (недели 4-8, ~15.06 — июль)

В Run все три слоя работают, но на разных режимах:

**Layer 1 в Run:** тестеры → когорта. 5-15 человек имеют свои копии Layer 1. Начинается
сбор фидбека: что используют / что не используют / что непонятно. Add-ons (Reviews,
Hypotheses) добавляются по фидбеку, не заранее.

**Layer 3 в Run:** Layer 3 full начинает строиться. Добавляются детальные финансы (доходы,
расходы, pipeline), партнёрский трекер, ретроспективы. Если появились партнёры с Layer 2 —
Layer 3 начинает агрегировать данные снизу (Project Catalog Layer 2 → Projects Layer 3).

**Layer 2 в Run:** настоящий multi-tenant. Charter подписан. Первая когорта в Teamspace.
Matching helper работает вручную (или первые скрипты). Первые деньги потекли — Mondragón
5:1 применяется на каждом сплите.

Notion Business план в Run — уже необходим для multi-tenant Teamspace.

### Scale (~август 2026+)

Layer 3 full + overlay (если кооперативный вид). Layer 2 multi-cohort — несколько
Teamspace-ов для разных когорт. Enterprise-план — только когда нужны SSO, аудит, compliance
(Scale, не Build/Run). Ethereum overlay — текстовый принцип в Build/Run, механический
смарт-контракт только на Scale (~2027).

## §8 Зависимости между слоями: что на что влияет

Каждый слой standalone. Но есть точки соединения, которые активируются по мере роста:

**Layer 1 → Layer 2:** когда у партнёра есть Layer 1, его Daily Log и Projects можно
связать с Project Catalog в Layer 2. Это делается через relation/link. Не обязательно на
Build — удобно в Run.

**Layer 2 → Layer 3:** Project Catalog Layer 2 может агрегироваться в executive Projects
view Layer 3. Финансы Layer 3 можно связать с Charter Layer 2 (кто сколько принёс).
Это Run и Scale — не Build.

**Layer 3 → всё:** Layer 3 — это верхний уровень. Он агрегирует снизу, но не зависит
от того, работают ли нижние слои. Основатель может пользоваться Layer 3 в полном
одиночестве.

**Нет циклических зависимостей:** L1 → L2 → L3 (опционально). L3 standalone. L1 standalone.
Это важно — можно начать с любого слоя и не сломать остальные.

2. Enterprise — только Scale. Не раньше.
3. Layer 1+3 standalone = Free/Plus. Никакой срочности апгрейда до появления реального
   team-кейса.

## §10 Анти-паттерны: чего не делать

Несколько вещей, которые тормозят или ломают внедрение:

**Не собирать все три слоя одновременно.** Параллельная сборка трёх слоёв — это
паралич выбора и нет ничего готового к показу. Иди по очереди: Layer 1 работает → Layer 3
minimum → Layer 2 demo.

**Не перфекционировать шаблон до первого trial.** Layer 1 нужно показать Дмитрию в
состоянии «достаточно хорошо», а не «идеально». Итерация на реальном фидбеке — быстрее
и полезнее, чем шлифовка в вакууме.

**Не добавлять add-ons Layer 1 до засева.** Reviews и Hypotheses бесполезны, если в Daily
Log и Projects пусто. Сначала засей 5-10 реальных записей, потом добавляй расширения.

**Не апгрейдить Notion Business до появления реального multi-user.** Если Layer 2 нужен
только как demo для одного партнёра, можно показать через гостевой доступ на Plus. Business
нужен когда реально несколько человек работают в одном Teamspace каждый день.

**Не путать Layer 2 Generic и Jetix overlay.** Первый разговор с партнёром — через Generic
(понятно всем). Jetix overlay — после подтверждения интереса. Смешивание в первом показе
создаёт перегруз.

**Не строить Layer 3 full до minimum.** Начни с 5 ключевых блоков (runway, цели, проекты,
briefing, хаб). Full (P&L, pipeline, ретро) добавляй по мере реальной нужды.

## §11 STANDALONE — главный принцип

Порядок, описанный выше, — **рекомендация под конкретный контекст** (Дмитрий-trial + founder
executive view + первый партнёр в Build). Это не обязательство и не единственный путь.

Если тебе нужен только executive-вид прямо сейчас — начни с Layer 3. Если хочешь сначала
показать партнёру Team OS — начни с Layer 2. Если нужен только личный инструмент — Layer 1
и ничего больше.

Три слоя — три независимые системы. Каждая работает сама по себе. Вместе они дают больше,
но «вместе» — это Run и Scale, не обязательное условие для старта.

AI Tools mega-page — отдельно, в любой момент. Не блокирует ничего.

Принцип выбора прост: **начни с того слоя, который решает самую срочную задачу прямо
сейчас.** Для большинства людей — Layer 1. Для founder в Build — Layer 1 + Layer 3 minimum.
Для команды — добавить Layer 2.
