# Phase 6 — Как подключается Claude Code

> **Главная мысль фазы.** Claude Code — это помощник, который читает твой Notion,
> предлагает записи, подтягивает данные в ревью и держит файлы синхронными. Но он
> **ничего не решает и не перезаписывает сам** — он предлагает, ты подтверждаешь.
> И главное: **правда живёт в файлах на диске, Notion — это витрина**. Если они
> разошлись — прав файл.
>
> Эта фаза описывает: как идёт синхронизация в обе стороны, что Claude Code МОЖЕТ и чего
> НЕ может, какие нужны скрипты и команды. Сами скрипты тут не создаются — только ТЗ.

---

## §1 Кто главный: файлы или Notion?

**Файлы на диске = правда (source of truth). Notion = витрина (view).**

Почему так (Global Rule 4 + O-158 Notion-bypass):
- Notion — чужой сервис. Закроется, подорожает, забанит — данные не должны умереть с ним.
- Markdown-файлы — твои навсегда, читаются чем угодно, лежат в git (история всех версий).
- Notion удобен для глаз, голоса, телефона, комментариев — поэтому он витрина, не сейф.

**При конфликте: правит файл.** Если в Notion отредактировали поле, а в файле другое —
последнее изменение по времени решает направление синка, но при неоднозначности
**файл выигрывает**, Notion получает поправку.

## §2 Синхронизация в обе стороны

### §2.A Claude Code → Notion (push, «выкатить»)

| Когда | Что синкается |
|---|---|
| Ежедневно | изменения в файлах (коммиты) → обновление зеркала в Notion |
| Периодически | статусы проектов / обновления гипотез / новые записи |
| По запросу | массовый импорт / создание новой базы |

### §2.B Notion → Claude Code (pull, «забрать»)

| Когда | Что синкается |
|---|---|
| Голосовой ввод | DRAFT'ы из Notion → Claude Code обрабатывает → файлы |
| Ручные правки | отредактировал в Notion с телефона → дневной pull → в файлы |
| Комментарии | комментарии/добавления коллабораторов |

### §2.C Разрешение конфликтов

| Ситуация | Решение |
|---|---|
| Файл новее Notion | файл побеждает, push поправку в Notion |
| Notion новее файла | уведомить человека, спросить (не молча перезаписывать!) |
| Оба менялись (конфликт) | сохранить обе версии как DRAFT, человек выбирает |

**Принцип:** при любой неоднозначности — **не угадывать, а спросить**. Молчаливая
перезапись = потеря данных = главное табу.

## §3 Что Claude Code МОЖЕТ (allowed)

- ✅ Предлагать записи на основе голоса (DRAFT)
- ✅ Подсвечивать кандидатов на промоушен (Tier C → B → A в Ideas Bank)
- ✅ Предлагать связи между записями (эта гипотеза похожа на тот источник)
- ✅ Предлагать привязку к проекту по контексту
- ✅ Предзаполнять шаблоны ревью данными (из Daily Log, Hypotheses, Life Pulse)
- ✅ Находить застрявшее (проекты/контакты >14 дней) и всплывать
- ✅ Держать файлы и Notion синхронными
- ✅ Делать аудит расхождений (drift report)

## §4 Чего Claude Code НЕ может (never)

- ❌ Перезаписывать боевые записи сам (только DRAFT)
- ❌ Писать стратегическую прозу / цели (R1 — это человек)
- ❌ Менять поля без подтверждения человека
- ❌ Сам решать промоушен Tier (предлагает — ты решаешь)
- ❌ Сам архивировать/убивать проекты
- ❌ Сам менять POINT B / курс
- ❌ Молча разрешать конфликты синка
- ❌ Хранить данные ТОЛЬКО в Notion (всегда зеркало в файлах)

**Это и есть «AI предлагает, человек решает» (Pillar C правила 1, 2, 11) на personal
уровне.**

## §5 Нужные скрипты (ТЗ — не создаём здесь)

| Скрипт | Что делает |
|---|---|
| `tools/notion_export.py` | файлы → Notion (push зеркала) |
| `tools/notion_import.py` | Notion → файлы (pull правок и DRAFT'ов) |
| `tools/voice_intake_router.py` | голос → классификация → DRAFT в базы (Phase 4) |
| `tools/notion_audit.py` | поиск расхождений файл↔Notion + drift-отчёт |
| `tools/template_instantiate.py` | создать строку базы из шаблона (ревью/звонок) |
| `tools/personal_os_upgrade.py` | миграция Layer 1 при апдейте шаблона (Phase 7) |

**Общий принцип скриптов:** конфиг-driven (пути в `.claude/config/`, не захардкожены),
idempotent (безопасно перезапускать), git-native (откат через revert), аудит ключей
перед коммитом.

## §6 Нужные команды Claude Code (скиллы — ТЗ)

В дополнение к существующим (`/crm-*`, `/ingest`, `/ask`, `/plan-day`, `/close-day`):

| Команда | Что делает |
|---|---|
| `/notion-sync` | двусторонняя синхронизация (с отчётом что изменилось) |
| `/notion-audit` | найти расхождения / конфликты |
| `/notion-template <тип>` | создать ревью/звонок из шаблона (Phase 5) |
| `/notion-voice-process` | обработать пачку голосовых черновиков |
| `/notion-promote <draft-id>` | промоутнуть DRAFT в боевую запись (после ревью) |

## §7 Дневной ритм с Claude Code (как выглядит на практике)

| Время | Действие | Команда |
|---|---|---|
| Утро | загрузка контекста + план дня | `/plan-day` |
| Утро | разобрать ночные голос-черновики | `/notion-voice-process` |
| День | надиктовка на ходу (Wispr) → DRAFT'ы копятся | (авто) |
| Вечер | промоутнуть нужные черновики | `/notion-promote` |
| Вечер | итог дня + синк + коммит | `/close-day` + `/notion-sync` |
| Вс | недельное ревью | `/notion-template week-review` |

## §8 Анти-паттерны интеграции

- ❌ Ключ API в поле Notion (только в `.env`, никогда в базе)
- ❌ Хранение только в Notion без файлового зеркала
- ❌ Авто-действия Claude Code (заполнил/отправил/решил) — нет, только предлагает
- ❌ Сторонние интеграции без явной настройки человеком
- ❌ Синк, который молча перезаписывает при конфликте
- ❌ Зашитые page ID в Layer 1 (только в Layer 2 / `.env`)

## §9 Степень автоматизации — R1-решение

Для базового шаблона есть выбор (Руслан решает, §9.E промпта п.5):

| Вариант | Что значит | Кому |
|---|---|---|
| **A — ручной** | Notion заполняешь руками, Claude Code только подсказывает в чате | старт / нетехнический форк |
| **B — скрипты** | полный синк через `notion_*.py` + Notion API | технический форк / Руслан |
| **C — гибрид** | голос-ввод авто (DRAFT), остальное руками | рекомендуемый старт |

**По умолчанию для форка:** A (ручной) — чтобы не блокироваться на настройке API.
Скрипты — продвинутая надстройка.
