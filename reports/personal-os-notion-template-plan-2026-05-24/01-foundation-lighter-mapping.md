# Phase 1 — Фундамент → облегчённая Notion-версия

> **Главная мысль фазы.** Большая система Jetix стоит на 11 «кирпичах» (Parts) +
> конституция из 12 правил. Это спроектировано для **роя из 17 агентов**, которые
> работают сами. Personal OS — для **одного человека без агентов**. Поэтому многое
> можно облегчить: то, что в большой системе делают агенты и протоколы, в личной
> версии делает сам человек руками + Claude Code как помощник.
>
> Ниже — честный разбор каждого кирпича: **что переживает упрощение и остаётся даже
> в самой лёгкой версии**, **что выбрасываем** (для одного человека не нужно), и **что
> упрощаем** (адаптируем).

---

## §1 Три категории — как читать таблицы ниже

При переводе каждого кирпича в Notion я раскладываю его на три части:

- 🟢 **Что survives (остаётся)** — принцип настолько фундаментальный, что без него
  система перестаёт быть «системой» и превращается в свалку заметок. Это держим даже
  в самой лёгкой версии. Примеры: файлы = правда, голос = только черновик, у каждой
  записи есть источник.
- 🔴 **Что dropped (выбрасываем)** — нужно только когда внутри работают автономные
  агенты или большая команда. Для одного человека — лишний вес. Примеры: 17 ролей
  агентов, протокол согласования между агентами, формальные пакеты «жду одобрения».
- 🟡 **Что simplified (упрощаем)** — принцип остаётся, но в более лёгкой форме.
  Примеры: 4 Stage Gates → 3 чекпоинта; формальный F-G-R рейтинг → простое поле
  «насколько уверен».

## §3 Кирпич 1 — State Persistence (где живёт правда)

**В большой системе:** состояние разбросано по JSON-файлам, каждый агент держит своё,
файлы на диске — единственный авторитет (Notion не авторитетен).

🟢 **Survives.** Главное правило выживает целиком: **файлы на диске = правда, Notion =
витрина**. Если Notion и файл разошлись — прав файл (Global Rule 4). Frontmatter (шапка
с метаданными) обязателен у каждой записи. Логи — только дописываем сверху, не стираем.

🔴 **Dropped.** Распределённое состояние агентов, отдельные JSON state-файлы под
каждого агента — этого в personal-версии нет, потому что нет агентов.

🟡 **Simplified.** «Состояние» сворачивается в простые поля баз: статус проекта,
энергия дня, статус контакта. Не нужен отдельный механизм — Notion-свойства это и есть.

**Почему так.** Один человек не теряет состояние между «сессиями агентов» — он сам
носитель контекста. Но файлы-как-правда критичны: это защита от vendor lock-in (Notion
закроется/подорожает — данные остаются у тебя в markdown).

## §4 Кирпич 2 — Signal Ingestion (как входящее попадает внутрь)

**В большой системе:** входящие сигналы (голос, письма, события) автоматически
сортируются агентом-triage по правилам в коде.

🟢 **Survives.** Дисциплина **DRAFT-only**: всё входящее становится сначала черновиком,
никогда не перезаписывает «боевую» запись само. Единая точка приёма.

🔴 **Dropped.** Автоматический triage агентами, правила маршрутизации, зашитые в код.

🟡 **Simplified.** Маршрутизация превращается в **понятную матрицу** «тип голосовой
заметки → в какую базу» (это вся Phase 4). Claude Code предлагает, человек подтверждает.

**Почему так.** Голос — главный способ ввода у Руслана (диктует на ходу). Но автомат,
который сам решает и сам пишет в боевые записи — это риск мусора и потери данных.
DRAFT-only = страховка: сначала черновик, потом глаз человека, потом промоут.

## §5 Кирпич 3 — Knowledge & Methodology (библиотека знаний)

**В большой системе:** wiki из 9 типов сущностей + отдельная methodology library.

🟢 **Survives.** Знания накапливаются структурно (не свалкой). У каждого факта есть
источник. Типизация (концепт vs источник vs проверенное утверждение vs гипотеза).

🔴 **Dropped.** Полные 9 типов (concepts/entities/sources/topics/ideas/experiments/
claims/summaries/foundations) — для personal избыточно. Methodology как отдельная
подсистема тоже не нужна.

🟡 **Simplified.** 9 типов → **4 базы**: Concepts (понятия), Sources (книги/статьи/
видео), Claims (проверенные утверждения), Hypotheses («что хочу узнать»). Методы — это
тег или поле, не отдельная база. Ideas → отдельный Ideas Bank (Tier A/B/C).

**Почему так.** «Что хочу узнать» (Hypotheses) — прямой запрос Руслана из голоса 24.05
(«tracking что хочу узнать»). Это сердце познавательного цикла: гипотеза → источники →
проверенное утверждение → концепт. 4 базы покрывают этот цикл без перегруза.

## §6 Кирпич 4 — Role Taxonomy (кто что делает)

**В большой системе:** 17 агентов ROY-роя + brigadier как единый диспетчер + протокол
сообщений hub-and-spoke. Самый «тяжёлый» кирпич.

🟢 **Survives.** Идея разделения «режимов мышления» внутри одной головы: когда планирую
— я стратег; когда делаю — исполнитель; когда проверяю — критик. Это полезная привычка,
а не структура.

🔴 **Dropped — почти целиком.** 17 ролей агентов, brigadier-маршрутизация, протокол
сообщений между агентами, message-schema — **всё убираем**. В personal-версии нет
агентов: есть человек + Claude Code как ОДИН помощник.

🟡 **Simplified — опционально.** Продвинутый форк (инженер, который хочет мини-рой)
может вернуть несколько агентов Claude Code под свои задачи. Но это **не в базовом
Layer 1** — это надстройка.

**Почему так.** Это главное место, где personal ≠ company. Большая система — про
координацию многих агентов. Личная — про одного человека. Тащить сюда роевой протокол
= утопить шаблон в ненужной сложности. Поэтому Part 4 — самый «облегчённый» кирпич.

## §7 Кирпич 5 — Compound Learning (как система учится на себе)

**В большой системе:** каждый рабочий цикл агентов автоматически порождает методичку
(«вот как это лучше делать»), система компаундит знание.

🟢 **Survives.** Регулярная рефлексия: «что сработало, что нет, что повторить». Это
ядро роста. Без ревью система не учится — просто крутится.

🔴 **Dropped.** Авто-захват методичек из выходов агентских циклов (нет агентских циклов).

🟡 **Simplified.** Обучение через **ручные шаблоны ревью** (Phase 5): неделя / месяц /
квартал / год. Человек сам отвечает на 8-12 вопросов, Claude Code предзаполняет данными
(что было в Daily Log, какие гипотезы закрылись, динамика проектов).

**Почему так.** Руслан прямо просил «шаблон анализа недели + месяца». Compound learning
для одного человека = дисциплина ревью. Это самый ценный механизм после голосового
ввода: именно тут «прошивка» (способ думать) укрепляется.

## §8 Кирпичи 6a + 6b — Provenance + Human Gate (источники + подтверждение)

**В большой системе:** Provenance Officer (агент) маркирует каждое утверждение рейтингом
F-G-R (формальность / охват / надёжность). Human Gate — формальный пакет
«AWAITING-APPROVAL» перед важными действиями, с уровнями Halt-Log-Alert.

🟢 **Survives.** (а) У важного утверждения есть **источник** — «откуда я это знаю».
(б) **Человек подтверждает важное** — DRAFT не становится боевой записью без глаза.

🔴 **Dropped.** Роль Officer; формальная схема F-G-R на каждом claim; формальные пакеты
AWAITING-APPROVAL; уровни Halt-Log-Alert (≤1с/≤5с/≤60с).

🟡 **Simplified.** (а) F-G-R → одно поле **«уверенность: низкая / средняя / высокая»**
на записях в Claims/Hypotheses. (б) Human Gate → **чекбокс «проверено мной»** +
ручной промоут DRAFT → prod. Очередь решений живёт в Strategic DB.

**Почему так.** Provenance для personal — это защита от самообмана: «я правда это знаю
или придумал?». Но городить рейтинг F-G-R на каждую заметку — перебор. Одно поле
уверенности даёт 80% пользы за 5% сложности.

## §9 Кирпич 7 — Project Lifecycle (жизнь проекта)

**В большой системе:** проект проходит 4 типа (consulting/research/product/bets) и 4
Stage Gates (SG-1..SG-4) со строгими предикатами; есть скиллы de-morph (откат) и
promote (повышение типа).

🟢 **Survives.** Проект имеет **тип, приоритет, статус, стадию**. Есть **stuck-детект**:
активный проект без касания >14 дней — всплывает (не теряется).

🔴 **Dropped.** Строгие предикаты Stage Gates с banned-phrases; скиллы de-morph/promote;
формальный eval gate.

🟡 **Simplified.** 4 Stage Gates → **3 простых чекпоинта**: ① Старт (идея оформлена,
есть первый шаг) → ② В работе (есть движение, проверяем гипотезы) → ③ Готово или в
архив (закрыли / убили / запивотили). 4 типа проектов остаются как Layer 1, подтипы —
Layer 2 (под нишу).

**Почему так.** Stuck-детект — золото для одного человека: проекты тихо умирают от
заброшенности, и важно их видеть. А 4 формальных гейта — overkill для personal; 3
человеческих чекпоинта дают тот же ритм без бюрократии.

## §10 Кирпич 8 — Health Monitoring (здоровье)

**В большой системе:** мониторинг здоровья СИСТЕМЫ — SLI/SLO, integrity-нарушения,
алерты.

🟢 **Survives.** Идея регулярного замера + ранние сигналы («что-то идёт не так — вижу
заранее»).

🔴 **Dropped.** SLI/SLO системы, integrity-checks, технические алерты Part 8.

🟡 **Simplified — со сменой объекта.** Мониторим не систему, а **человека**: Life Pulse
DB — энергия / сон / здоровье / отношения / деньги / настроение. Ежедневный мини-замер
(одно число + заметка), недельная динамика.

**Почему так.** Самый красивый переход: машинный мониторинг → личный мониторинг.
Цифры здоровья человека — это «integrity» личной системы. Burnout = integrity-violation
на personal-уровне. Связка Daily Log ↔ Life Pulse даёт раннее предупреждение.

## §11 Кирпич 9 — Owner Interaction (единая точка входа)

**В большой системе:** scaffold того, как владелец общается с роем (брифинги, отчёты).

🟢 **Survives.** **Единая точка входа** — «всё видно с одного места», не надо помнить,
где что лежит.

🔴 **Dropped.** Scaffold общения с агентами; формат брифингов для роя.

🟡 **Simplified.** Превращается в **Command Center** — одну страницу-дашборд: фокус дня,
активные проекты, гипотезы в работе, тёплые контакты, какое ревью пора делать,
последний пульс жизни. (Полная схема — Phase 2 §3.C.)

**Почему так.** Для одного человека «точка входа» = утренняя страница, которую
открываешь первой. Не нужен протокол брифинга — нужна красивая сводка одним взглядом.

---

## §12 Кирпич 10 — External Touchpoints (связи с миром)

**В большой системе:** интерфейс к внешнему миру + сетевой протокол.

🟢 **Survives — почти целиком.** Сеть контактов (люди + организации), история касаний,
роли, статусы, stuck-детект. CRM уже спроектирован под personal-масштаб.

🔴 **Dropped.** Сетевой протокол между системами, внешние API-touchpoints.

🟡 **Simplified.** CRM наследуется **1:1**: People + Orgs, 24 роли в 6 группах, 13
статусов pipeline, append-only история. Это самый «готовый» кирпич — почти ничего не
меняем.

**Почему так.** CRM уже построен (`crm/` + 10 скиллов) и уже облегчён под одного
человека. В Notion-шаблоне это две базы с готовой структурой.

---

## §13 Кирпич 11 — Strategic Direction (куда движемся)

**В большой системе:** субстрат направления — 6 типов стратегических документов (Lock
Entry / North Star / Strategic Insight / Direction Card / Phase Plan / Reflection),
с трейлом авторства (стратегию пишет только человек).

🟢 **Survives.** **Стратегию пишет человек** (не AI) — это правило №1 конституции.
«Куда движемся» = POINT A (где я сейчас) → POINT B (куда хочу) + план на год/квартал.

🔴 **Dropped.** 6 формальных типов доков; формальный prose_authored_by трейл.

🟡 **Simplified.** 6 типов → **4 типа записи** в Strategic DB: POINT A (текущее
состояние) / POINT B (цель) / план периода / решение (очередь решений). Claude Code
может предзаполнить данные, но прозу цели пишет человек.

**Почему так.** Это прямая проекция правила «AI не стратегирует». Для personal POINT A →
POINT B — самая понятная рамка («где я и куда хочу»). 6 формальных типов — это про
company; 4 поля достаточно для личной навигации.

## §14 Кирпич C — Pillar C (конституция из 12 правил)

12 правил большой системы — это в основном **ограничения для агентов** («AI не делает
X»). Для personal-версии большинство неактуальны (нет агентов). Но 5 правил
переживают и становятся **памяткой Personal OS**:

| # из 12 правил | Большая версия | Personal-версия (survives?) |
|---|---|---|
| 1. AI не стратегирует | агент не пишет стратегию | 🟢 Claude Code предлагает, цель пишешь ты |
| 2. AI не меняет архитектуру без гейта | гейт перед изменением фундамента | 🟡 ты подтверждаешь важные структурные изменения шаблона |
| 4. AI не присваивает идентичность | agent acting_as | 🔴 неактуально (нет агентов) |
| 5. AI не претендует на долю | agent no skin-in-game | 🔴 неактуально |
| 6. AI не копит память бесконтрольно | знание только через явные файлы | 🟢 Claude Code не аккумулирует тайно; всё в файлах |
| 9. AI не самомодифицируется | agent не правит свой prompt | 🔴 неактуально |
| 10. AI не выдаёт себя за человека | дисклоуз | 🟡 если используешь авто-ответы — помечай, что это AI |
| 11. Default-Deny | непонятное действие = запрет | 🟢 Claude Code по умолчанию НЕ делает, а предлагает |
| 12. Нельзя запирать (anti-extraction) | fork-and-leave | 🟢 ты владеешь данными; экспорт в файлы; уйти можно всегда |
| + ключи API | не светим ключи нигде | 🟢 ключи только в `.env`, никогда в Notion-поле |
| + DRAFT-only голос | голос не перезаписывает prod | 🟢 переносим целиком |

**5 правил, которые держим в Personal OS** (короткая памятка для форка):
1. **Стратегию/цели пишешь ты** — Claude Code предлагает варианты.
2. **Claude Code по умолчанию НЕ делает действий сам** — предлагает, ты подтверждаешь.
3. **Голос → только черновик**, не перезаписывает боевые записи.
4. **Файлы = твоя правда**, ты владеешь данными, уйти можно в любой момент.
5. **Ключи API только в `.env`**, никогда в Notion.

**Почему так.** Конституция большой системы — про обуздание агентов. Для человека
остаётся то, что защищает ЕГО: контроль, владение данными, защита от мусора, защита
ключей. Это и есть «философия работает на personal level» — proof point из голоса 24.05.
