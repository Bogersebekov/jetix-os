# Phase 0 — Что у нас уже есть (substrate перед стройкой)

> **Простыми словами.** Прежде чем проектировать Notion-шаблон, я собрал в одном месте
> всё, на чём он стоит: фундамент из 11 кирпичей, существующие базы в Notion, как
> работает голосовой ввод, как устроена CRM. Это «инвентаризация склада» — чтобы не
> придумывать заново то, что уже построено два месяца на сервере.

---

## §0 Что мы вообще строим (одним абзацем)

Universal Personal OS — это **облегчённая Notion-версия большой системы Jetix**.
Большая система живёт в файлах на сервере (это «правда»), а Notion — это **красивая
витрина + место, куда удобно надиктовывать голосом и куда подключается Claude Code**.
Шаблон должен быть таким, чтобы **любой человек мог его скопировать к себе за 30-60
минут и настроить под свою нишу** — инженер, исследователь, предприниматель, методолог
или просто человек, который наводит порядок в жизни.

## §1 FPF lens scope (какой инструментарий применяю и где НЕ применяю)

FPF — это наш внутренний «набор линз», через которые смотрим на любой дизайн. Для
этого документа применяю узко и по делу (не вываливаю весь жаргон в текст):

| Линза | Что значит на человеческом | Где применяю в этом дизайне |
|---|---|---|
| **R1** (стратегию пишет человек) | Я предлагаю варианты, финальный выбор — за Русланом | Списки баз, набор ниш, порядок внедрения — это **пул, не приказ** (§9.E промпта) |
| **R2 STRICT** (не трогаем фундамент) | Читаю 11 Parts как материал, ничего в них не меняю | Фундамент — только источник, мэппинг идёт в новые файлы |
| **R6** (ссылки на источник) | Каждое нетривиальное утверждение — со ссылкой | Ссылки собраны в конце Main-доку, не размазаны по тексту |
| **R11** (по умолчанию запрет) | Никаких авто-действий, только план | НЕ создаю страницы в Notion, НЕ зову API |
| **R12** (нельзя запирать человека) | Можно уйти в любой момент со своими данными | Зашито в Layer 1: экспорт в файлы, fork-and-leave |
| **IP-1 STRICT** (роль ≠ конкретный человек) | Фундамент описывает абстрактные роли, не Руслана | Layer 1 — без имён; имена только в примерах Layer 2 |
| **EP-5** (формальность по уровням) | Фундамент = строгий (F2), Notion-мэппинг = производный (F3) | Помечаю каждый файл уровнем |
| **AP-6** (держим варианты открытыми) | Не выбираю заранее за человека | 3 варианта числа баз, 5 примеров ниш |

**НЕ применяю** (вне зоны этого документа): расчёт юнит-экономики, токеномика,
управление роем агентов. Это personal-уровень, не company-уровень.

## §4 Голосовой ввод — как работает сейчас (для Phase 4)

Текущий пайплайн (из `CLAUDE.md` Voice-Notes Pipeline + `voice-pipeline-canonical-2026-05-10`):

1. `transcribe.py` — OGG/MP3 → текст (Groq Whisper)
2. `extract.py` — текст → структурированные пункты (Claude)
3. `filter.py` — дедуп + мета-анализ
4. `review_report.py` — markdown-отчёт → `~/review-latest.md`
5. **СТОП.** Руслан читает, принимает решения вручную.

**Главное правило (которое переносим в шаблон):** голос → **только DRAFT (черновик)**,
никогда не перезаписывает «боевые» записи автоматически. Распределение по базам — после
ревью человеком. Это правило §4.2 CLAUDE.md «Voice-pipeline DRAFT-only».

В новом шаблоне добавляем Wispr Flow как фронт голосового ввода (быстрее, чем
надиктовка в мессенджер), но дисциплина DRAFT-only сохраняется без изменений.

## §5 CRM — структура (для Phase 3 + niche overlays)

Из `crm/README.md` + POINT-A §6:
- **24 роли в 6 группах:** sales / capital / partnership / advisory / team / network
- **13 статусов pipeline:** draft_from_voice → cold → warm → contacted → discovery_call
  → proposal → negotiation → closed_won/closed_lost + paused/active/past
- **14 секций** в каждом контакте, append-only история (§11)
- **Stuck detection:** active + >14 дней без касания → всплывает
- **Filesystem = source of truth**, Notion = view

Это готовая структура — в Notion-шаблоне CRM People + CRM Orgs наследуют её 1:1.
