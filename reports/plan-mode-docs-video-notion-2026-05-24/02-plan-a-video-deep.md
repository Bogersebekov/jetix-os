## §5 Recording order

### §5.1 Recommended order (per parent Phase 3 §1.5)

**1st: V1 anchor (Days 1-2)** — foundation; substrate для everything else; quality matters most.

**2nd: V2 deep (Days 5-7, after V1 feedback)** — informed by V1 reception; doubles down на resonant substrate area.

**3rd: V3-V7 shorts (Week 2, derivative from V1+V2)** — minimal new recording; max distribution coverage.

**4th: V_demo (Week 2-3, depends Plan C C1+C2)** — requires Notion template + Claude Code setup live.

**5th: V8+ topic-specific (Month 2+, demand-driven)** — only когда clear topic-demand emerges from V1/V2 feedback.

**6th: V_call_recording (opportunistic, consent-dependent)** — only when discovery call partner explicitly OKs.

### §5.2 Order rationale

- **V1 first because:** maximum reusability; foundation для derivatives.
- **V2 second NOT parallel because:** risk over-invest if V1 misses; feedback-driven scope.
- **V3-V7 derivative NOT fresh recording because:** energy efficiency; reuse footage.
- **V_demo dependent на Plan C because:** needs actual Notion template + Claude Code working.
- **V8+ demand-driven because:** avoid speculative content; let audience surface topics.
- **V_call_recording opportunistic because:** consent + R12 hardest; only когда natural fit.

### §5.3 Anti-patterns (NOT recommend)

❌ Record V1 + V2 in one day → energy depletion + persona inconsistency
❌ Bulk-record V8+ topic-specific upfront → speculative; audience may surface different topics
❌ Bulk-publish all formats simultaneously → channel-fatigue + cannot iterate
❌ Skip V1 → V2 directly → no anchor for derivatives
❌ Record V_call_recording without explicit consent → R12 violation
