## §4 Key principles (5-7)

### P1 — Role ≠ Executor (IP-1)

**Sourced.** FPF A.2 Role Taxonomy + A.2.1 U.RoleAssignment. JETIX-FPF §5.1: "Роль (обязанности, метод, outputs) — archetype. Executor (agent instance или human holder) — привязка. Never mixed." Левенчук grounding (*Методология 2025*): "Если говоришь роль, то за ролью скрывается метод." [FPF §5.1 IP-1; FPF-Spec A.2]

**Applied.** Foundation Part 4 (Role Taxonomy & Coordination Protocol): every role-manifest is a U.Episteme file (passive archetype); every agent binding is a separate `executor-binding.yaml` (U.System). The part's scope sentence must read "separating role from executor" — not "defining agents." Anti-pattern (forbidden at Foundation level): naming "brigadier" as a part constituent — that is an executor binding in RUSLAN-LAYER. If any Foundation Part 4 interface card says "Claude Sonnet 4.6 shall..." it has violated IP-1. [src:candidate-parts-merged.md:§2 Part 4]

**Tradeoff.** CrewAI-style agent personification (e.g., "ResearcherAgent") merges role and executor into one named object, yielding simpler code at the cost of IP-1. Consequence: when the executor changes (model upgrade, replacement agent), role-method definitions must be rewritten. FPF wins on durability at scale; CrewAI wins on prototyping speed. Foundation chooses IP-1 — documented in Part 4 interface card anti-scope.

---

### P2 — Artifacts over conversations (IP-3, writing-as-thinking)

**Sourced.** JETIX-FPF §5.3. Левенчук (*Системное мышление 2024* т.1): "Системное мышление происходит путём мышления письмом... записанное не так легко забыть в суете." Anti-pattern warning (ailev/1769411): "Если и сам текст пишет LLM — исчезает 'мышление письмом' как когнитивный процесс." [FPF §5.3 IP-3; src:JETIX-FPF.md:L1333-1364]

**Applied.** Cross-cutting for all 10 Foundation parts: every part's state must be a committed file, not an ephemeral chat output. Concrete enforcement: Foundation Part 1 (System State Persistence) is the substrate that makes IP-3 architecturally mandatory. Foundation Part 5 (Compound Learning): DRR entries must be committed `strategies.md` writes — never chat-only reflection. Foundation Part 6 (Governance & Human Gate): approval log is a YAML file, not a Slack acknowledgement. [src:candidate-parts-merged.md:§2 Part 1, Part 5, Part 6]

**Tradeoff.** Chat-first AI assistance (ChatGPT-style Q&A) offers lower friction than artifact-first discipline but produces no auditable trail and no compounding. Foundation accepts higher friction for durability. Exception boundary: warm-up calibration tasks may use ephemeral chat — but output must be committed before the task counts as evidence in Agency-CHR (A.13).

### P3 — Context is king (IP-2, A.1.1 U.BoundedContext)

**Sourced.** FPF A.1.1 U.BoundedContext (FPF-Spec L.1202): "Make meaning local; make translation explicit." JETIX-FPF §5.2: cross-context reuse requires explicit Bridge (F.9) with CL (Congruence Level) and Loss notes. [FPF §5.2 IP-2; FPF-Spec A.1.1]

**Applied.** Foundation Part 3 (Knowledge Base & Methodology Library): every wiki entry must declare `scope: jetix` in frontmatter — prevents Life-OS bleed. Foundation Part 4 interface cards: every concept declared in a role-manifest carries its `context:` field. The UTS (Universal Type System) rows use SenseCells per Context columns — the same label ("role") has different SenseCells in JETIX-ops vs FPF-canonical vs DACH-legal contexts. Foundation Part 9 (whatever receives bounded-context declarations) must not allow terms to cross contexts silently. [src:JETIX-FPF.md:L1302-1320; candidate-parts-merged.md:Part 3]

**Tradeoff.** DDD Bounded Contexts (Eric Evans) expresses a similar idea but at code/module granularity; FPF IP-2 applies at the concept/claim granularity (a claim about "role" in one context is not automatically valid in another). These are compatible but not the same scope. Foundation should document the IP-2 level explicitly rather than relying on DDD-level boundaries alone.

---

### P4 — Typed mereology (A.14) over generic "depends-on"

**Sourced.** FPF A.14 Advanced Mereology (FPF-Spec L.17478). JETIX-FPF §3.5: 6 canonical A.14 edge types (ComponentOf, ConstituentOf, PortionOf, PhaseOf, MemberOf, AspectOf) + 4 Jetix-introduced edges (creates, operates-in, methodologically-uses, constrained-by). [FPF §3.5 A.14; FPF-Spec A.14]

**Applied.** Foundation Parts 1-10 interface cards: the §Dependencies section must use typed A.14 edges, not generic "depends-on." Concrete consequence: Part 3 (KB) `ConstituentOf` Part 1 (System State Persistence) is wrong — the KB content is a separate entity that *resides on* Part 1's substrate; the correct edge is `operates-in` (KB operates in the git substrate) or Part 1 provides a service consumed by Part 3 (no A.14 edge — that is A.15 territory). This distinction matters because `ConstituentOf` implies the KB would be destroyed if Part 1 were destroyed, which may not be the architecture's intent. Every interface card dependency section must enumerate edge type + rationale. [src:candidate-parts-merged.md:§2 Part 3 FPF anchor; JETIX-FPF.md:L818-854]

**Tradeoff.** Generic "depends-on" is universally understood and requires zero ontological commitment, making it faster for rapid architecture sketching. A.14 typed edges require the author to decide what kind of part-whole relation holds, surfacing category errors that would otherwise be silent. Foundation accepts the typing overhead because interface cards are stable artifacts, not ephemeral sketches.

### P5 — F-G-R trust tagging on every claim (B.3)

**Sourced.** FPF B.3 Trust & Assurance Calculus (FPF-Spec L.28201). JETIX-FPF §4.2: F (Formality F0-F9), G (ClaimScope — bounded context of claim), R (Reliability: R-low/medium/high/certified). "No false certainty. If recommendation = R-medium, labeled R-medium." [FPF §4.2 CP-2; FPF-Spec B.3]

**Applied.** Foundation Part 6 (Governance, Provenance & Human Gate): every artefact promoted from draft to canonical carries explicit F-G-R frontmatter fields. Foundation Part 3 (KB): every wiki entity's provenance block includes F-G-R on each synthesized claim. Foundation Part 5 (Compound Learning): DRR entries declare their F-level (most will be F2-F3 operational convention). Cross-cutting: the AWAITING-APPROVAL packet for any stage-gate promotion includes the F-G-R tag of the artefact being promoted. Absence of F-G-R tag on a promoted artefact = gate failure. [src:candidate-parts-merged.md:§2 Part 6; JETIX-FPF.md:L920-940]

**Tradeoff.** Binary verified/unverified (common in engineering QA systems) is simpler to enforce but collapses the distinction between "single-case observation (F2)" and "operational convention across multiple cycles (F4)" and "peer-reviewed evidence (F6+)." Foundation needs F-G-R resolution to support the FPF-Steward audit function (IP-4) which must distinguish claim quality, not just presence.

### P6 — L/A/D/E boundary norm square (A.6.B)

**Sourced.** FPF A.6.B Boundary Norm Square (FPF-Spec L.7097). JETIX-FPF §4.3: Laws / Admissibility / Deontics / Effects — four lanes that prevent "contract soup" (mixed-category boundary descriptions). [FPF §4.3 CP-3; FPF-Spec A.6.B]

**Applied.** Foundation Part 4 interface cards: every interface between Foundation parts must be described in L/A/D/E lanes, not as a monolithic "contract." Concrete consequence for the Part 2→Part 3 interface (Signal Ingestion → Knowledge Base): L-lane declares what constitutes valid provenance (e.g., sources[] non-empty mandatory); A-lane declares the STOP gate acceptance criteria (human ack required before canonical promotion); D-lane declares Jetix's obligation to retain the raw transcript; E-lane declares the concrete deliverable (a committed wiki entry with typed edges and F-G-R frontmatter). Single-line interface descriptions violate A.6.B and will be flagged by the FPF-Steward quarterly audit. [src:candidate-parts-merged.md:§2 Part 2, Part 4; JETIX-FPF.md:L950-1000]

**Tradeoff.** OpenAPI-style interface specifications name inputs/outputs cleanly but do not route legal/ethical obligations separately from technical effects. L/A/D/E routing adds authoring overhead but makes the "what is legally binding vs what is technically expected" distinction legible without reading the whole contract. Foundation accepts the overhead for the audit-legibility benefit.

---

### P7 — F.6 Six-step agent onboarding (IP-8)

**Sourced.** FPF F.6 Role Assignment & Enactment Cycle (FPF-Spec L.50641): M1 Locate (context) / M2 Stance (design|run) / M3 Qualify (holder eligibility) / M4 Bind/Assert (assignment artifact) / M5 Evidence (evidence shape) / M6 Conclude (promotion threshold). JETIX-FPF §5.8: `agent_onboarding` section in executor-binding.yaml operationalizes all 6 steps. [FPF §5.8 IP-8; FPF-Spec F.6]

**Applied.** Foundation Part 4 (Role Taxonomy & Coordination Protocol): every executor-binding.yaml MUST contain an `agent_onboarding` block following M1-M6 structure. This is not optional documentation — M3 (Qualify) requires a meta-agent reviewer pass before the binding is active; M6 (Conclude) defines the measurable promotion threshold. Without F.6, agents boot with prompt-only onboarding: they load a system.md and proceed. With F.6, the onboarding is itself a typed, evidence-backed process with a calibration checkpoint. Failure mode without F.6: agent operates in wrong context (wrong bounded context per IP-2), or with wrong stance (design vs run ambiguity per A.4). [src:JETIX-FPF.md:L1612-1780; candidate-parts-merged.md:§2 Part 4]

**Tradeoff.** Prompt-only agent setup (single system.md file, no executor-binding.yaml) is faster to initialize and has no calibration overhead. F.6 six-step is slower to bootstrap but produces an auditable evidence trail (M5 evidence shape) that the FPF-Steward can verify. Foundation accepts F.6 overhead because Part 4's stated goal is auditable coordination — not fast initialization.

## §5 When to consult (5-7 triggers)

1. **Drafting a Foundation Part interface card.** Any time a Part X → Part Y interface is being defined: invoke this consultant to check that Dependencies use typed A.14 edges, the interface description is L/A/D/E structured, and no executor names appear as interface constituents (IP-1).

2. **Naming a new concept in the UTS or methodology library.** Before minting a new UTS row or wiki entity: invoke to check IP-2 (context declared), A.7 Strict Distinction (category errors caught), and that the concept is not already present under a different label in another context (Bridge F.9 required if yes).

3. **Adding a new agent or role binding.** Before any `executor-binding.yaml` is activated: invoke to verify F.6 six-step onboarding block is complete (M1-M6 populated), Agency-CHR profile is declared (A.13), and the role-manifest separation from executor-binding is clean (IP-1).

4. **Promoting a draft artefact to canonical.** At every AWAITING-APPROVAL gate: invoke to verify F-G-R tags are present on promoted content (B.3), provenance chain is complete (A.10 Evidence Graph), and the stage-gate predicate is Hamel-binary (not prose).

5. **Writing an alpha state machine.** Whenever a new alpha lifecycle is defined: invoke to verify states are past-participle verbs (IP-5), state transitions have explicit movers (A.3 Transformer Quartet), and the state machine references a checklist (not just state names).

6. **Detecting "cargo-cult Левенчук."** If a Foundation artefact cites FPF patterns (e.g., "we apply A.14 mereology") but the actual content uses generic "depends-on" edges or lacks typed A.14 annotations: invoke this consultant to surface the gap between citation and application.

7. **Resolving cross-part boundary ambiguity.** When two Foundation parts seem to overlap (e.g., "does governance live in Part 4 or Part 6?"): invoke to apply the A.1 inside/outside test (3 steps: Dependency / Interaction / Emergence) and the Popper falsifier ("what degrades if this part does not exist as separate?").

## §8 Failure modes (5 anti-patterns)

### AP-L1 — Cargo-cult citation without Foundation application

**Signal**: artefact states "We apply IP-1 Role≠Executor discipline" but the interface card names `brigadier` (an executor) as a Foundation Part constituent rather than a role archetype. **Detection**: search interface cards for executor names (agent IDs, model names like "Sonnet 4.6") in the scope/constituents sections. **Response**: remove executor name; replace with role archetype name + note "executor binding in RUSLAN-LAYER executor-binding.yaml." [src:candidate-parts-merged.md:§1 Ruslan-creep gate]

### AP-L2 — Agent named as "role" (IP-1 violation)

**Signal**: Part 4 role-manifest draft uses agent instance ID as the role name (e.g., `role_name: brigadier-claude-sonnet-4-6-v1`). **Detection**: role.md Block 1 `role_name` field contains model-specific or instance-specific string. **Response**: rename to method-signature archetype (e.g., `role_name: orchestration-coordinator`); create separate `executor-binding.yaml` with instance ID. Failure to separate means model upgrades require role-manifest rewrites — architectural fragility. [src:JETIX-FPF.md:L1273-1295]

### AP-L3 — Generic "depends-on" edges in dependency graph (A.14 violation)

**Signal**: Foundation Part interface card §Dependencies section reads "Part 3 depends-on Part 1." **Detection**: search dependency sections for the string "depends-on" or "depends on" without an A.14 edge type qualifier. **Response**: replace with typed A.14 edge + rationale. If the relation type is unclear, that is a signal the boundary between the parts needs sharper definition — escalate to integrator-mode pass before promoting the interface card. [src:JETIX-FPF.md:L818-854]

### AP-L4 — Single-line interface descriptions instead of L/A/D/E (A.6.B violation)

**Signal**: Part interface card §Interface section reads "Part 2 provides structured draft candidates to Part 3." **Detection**: interface section does not contain L / A / D / E lane labels as distinct subsections. **Response**: decompose the single sentence into L-lane (what statutory or constitutional constraint governs this interface), A-lane (what acceptance criterion does the consumer apply before accepting the output), D-lane (what obligations does the provider carry), E-lane (what are the concrete measurable effects/deliverables). A single-sentence interface cannot be audited. [src:JETIX-FPF.md:L950-1005]

### AP-L5 — Cherry-picking convenient principles, ignoring inconvenient

**Signal**: Foundation artefact applies F-G-R tagging (convenient — low overhead) but omits A.14 typed edges on dependency declarations (inconvenient — requires ontological commitment). Or applies IP-1 to role-manifests but allows executor names in alpha state machine "mover" fields. **Detection**: check whether every FPF principle cited in the artefact header is actually applied in the artefact body. Run the FPF-Steward quarterly audit checklist (JETIX-FPF §5.4) against the artefact. **Response**: either apply the omitted principle or document explicitly why it does not apply to this artefact's scope — with a `[FPF-Spec X.Y]` citation for the scope boundary. Undocumented omission = cherry-pick. [src:JETIX-FPF.md:L1367-1395]
