## §2 When to Consult This Consultant

Invoke this framework when the Foundation design decision involves:

- Decomposing a complex into parts — does the decomposition preserve system identity or does it destroy emergent properties?
- Choosing where to put control — should the controller be centralized or distributed? How much variety does it need?
- Diagnosing why an intervention keeps failing — is a balancing loop counteracting the push? Is there a reinforcing loop that should dominate?
- Projecting what happens at 10× scale — which loops will dominate? Which buffers will be overwhelmed?
- Designing health monitoring — what signals reveal the underlying structure, not just the surface events?
- Mapping Part-to-Part dependencies — are these causal (feedback) relationships or just co-presence?
- Choosing a governance gate — does it operate at the right VSM level? Is it trying to control variety it cannot match?

Do NOT invoke this framework for:
- Constitutional discipline (IP-1..IP-8, Role≠Executor) — that is Card #1 (Левенчук)
- Agent coordination patterns and MCP protocols — that is Card #3 (multi-agent)
- Capital allocation arithmetic — that is Card #10 (Buffett/Taleb)
- Code-level architecture quality (deep modules, refactoring) — that is Card #12 (clean code)

## §4 Key Principles (5 applied, each sourced + applied + tradeoff'ed)

### Principle 1 — Leverage Points: Goals > Rules > Information Flows >> Parameters

**Source.** Meadows (2008) Ch. 6 leverage-point list. In ascending order of leverage (L12 weakest to L1 strongest): L12 = numbers/constants, L11 = buffer sizes, L10 = stock-and-flow structure, L9 = delays, L8 = negative feedback loop strength, L7 = information flows, L6 = reinforcing loop gains, L5 = rules, L4 = self-organization, L3 = goals of the system, L2 = paradigms/mindsets, L1 = power to transcend paradigms. "Leverage points frequently are not intuitive. Or if they are, we too often use them backward." [src: meadows-thinking-in-systems-2008.md Ch. 6 p. 147]

**Applied to Foundation.** Foundation Parts decomposition decisions are primarily structural (L10) and rules choices (L5) — medium leverage. The highest-leverage decisions in Foundation are: (a) the system's goal — what the Foundation is FOR (L3); and (b) paradigm — whether Jetix Foundation is a designed machine or an evolving viable system (L2). Getting L3 and L2 wrong and then fiddling with L12 (parameters: which agent gets which API call budget) is the canonical trap. Specifically: **Part 8 (Health Monitoring)** should define SLI/SLO metrics at L7 (information flows), not L12 (numbers). If the health dashboard shows "number of API calls per day" but the system goal is "compound learning rate," the health signal is at L12 while the leverage is at L3 — measurement-goal mismatch. **Part 5 (Compound Learning)** is the L4 self-organization mechanism — it is the highest-leverage Foundation part by Meadows ordering. Underfunding or underspecifying Part 5 while polishing Part 1 (git discipline = L10-L12 territory) is a classic backward-leverage mistake.

**Tradeoff.** Meadows L3 (system goals) vs FPF IP-1 (role manifests are episteme = passive descriptions of goals, not system-level goal declarations). There is a genuine tension: FPF says the role's "purpose" is declared in the role.md file (U.Episteme, passive), but Meadows says the system's goal-setting function (L3) is where the highest leverage lives. Resolution: FPF IP-1 addresses the EXECUTOR binding (which agent carries the role), not the system's goal. The Foundation Part 6 (Governance Gate) is where L3 enforcement happens — it enforces that Ruslan's strategic goal (not the agent's local goal) is what AWAITING-APPROVAL packets test against. These are compatible, not in conflict, as long as Part 6's acceptance predicates reference FUNDAMENTAL §1 use-case categories (the system's goal), not local agent metrics.

### Principle 2 — Ashby's Law of Requisite Variety: Control Requires Matching Variety

**Source.** Ashby (1956) Ch. 11: "Only variety can destroy variety." The Law of Requisite Variety states that the number of distinct states a controller can take must be at least as large as the number of distinct states of the system being controlled. Beer summarizes (Ch. 3): "to cope with n things or events, each of which is or is not 'lit up' at any one moment, [the controller] may be presented in any one of the $2^n$ possible shapes — the variety is $n$." [src: ashby-introduction-to-cybernetics-1956.md Ch. 11; beer-brain-of-the-firm-1972.md Ch. 3]

**Applied to Foundation.** Every Part that contains a controller must declare its variety budget. Three specific applications:

1. **Part 4 (Role Taxonomy + Coordination Protocol)**: The brigadier hub is the primary coordinator. Its variety (what it can distinguish and route) must match the variety of incoming task types across all 20 matrix cells (5 experts × 4 modes). If the brigadier's routing table has fewer than 20 distinct states, some task types will be silently misrouted. Wave C design check: count distinct routing rules in `shared-protocols.md` and verify ≥ 20. If fewer — the controller has insufficient variety, and either (a) tasks collapse into undifferentiated "do something" requests, or (b) the role taxonomy itself must be simplified so controller variety ≥ system variety.

2. **Part 6 (Governance Gate)**: The AWAITING-APPROVAL gate's variety must match the variety of distinct decision classes it governs. FUNDAMENTAL §4.2 enumerates ~12 "never automate" hard rules. If the gate cannot distinguish between a J1-decision (auto-approve) and a J3-decision (strategic, human-only), it cannot regulate the system. Ashby predicts: under-variety gates produce either over-blocking (treating J1 as J3) or under-blocking (treating J3 as J1). Wave C must explicitly declare the gate's decision-class taxonomy and verify it covers the requisite variety of decision types.

3. **Part 8 (Health Monitoring)**: If the health monitor tracks 3 metrics but the system has 8 distinct failure modes (per FUNDAMENTAL §3 30+ SLI/SLO pairs), the monitor has insufficient variety to distinguish what is failing. "Variety reduction" is a legitimate strategy here (Ashby): filter signals down to the smallest set that still permits distinguishing all failure modes. The Part 8 Wave C task is: enumerate all failure modes → find minimal sensor set with variety ≥ failure-mode count.

**Tradeoff.** Ashby requisite variety vs Anthropic "context engineering — keep it simple." There is a genuine tension. Anthropic's building-effective-agents guidance (Card #3 territory) emphasizes: don't over-engineer agent context, keep system prompts simple. Ashby says: if the controller has insufficient variety, it cannot regulate the system. Resolution: variety must be matched at the FUNCTIONAL level, not the implementation level. A simple system prompt can carry high variety if it encodes distinct decision rules precisely (e.g., the J1/J2/J3 taxonomy in 3 sentences). Complexity of text ≠ variety. The operative discipline is: for each controllable domain, enumerate distinct states, then verify the controller can distinguish them — regardless of how the controller is implemented.

### Principle 3 — Beer's VSM: Every Viable System Needs 5 Functions

**Source.** Beer (Brain of the Firm, 1972) Parts 2-3. The Viable System Model maps 5 recursive control functions: System 1 (operations — produce); System 2 (coordination — dampen oscillations between S1 units); System 3 (audit/optimisation — resource bargaining + performance monitoring); System 4 (intelligence/futures — environment-scanning, adaptation); System 5 (identity/policy — ethos, ultimate authority). "Three vital systems are identified as prerequisites of all autonomous control." [src: beer-brain-of-the-firm-1972.md Ch. 9 p. 119] The key insight: Systems 1-3 maintain internal stability; System 4 maintains dynamic equilibrium with the external world; System 5 sets the meta-level constraints within which 1-4 operate. All 5 must be present in any viable system or the system will fail to survive environmental change.

**Applied to Foundation Parts.** The 10-part Foundation maps onto VSM imperfectly but instructively:

| VSM Level | Foundation Part mapping | Gap / observation |
|---|---|---|
| S1 — Operations | Parts 1-2-3 (State Persistence, Signal Ingestion, KB/Methodology) — these are the productive operations | S1 units currently lack explicit coordination protocol between them → S2 gap |
| S2 — Coordination | Part 4 (Role Taxonomy + Coordination Protocol) — the mailbox routing + shared-protocols IS the S2 coordination mechanism | S2 exists but is partly informal; Wave C must canonicalize |
| S3 — Audit/Optimisation | Part 8 (Health Monitoring) + Part 6 (Governance Gate, audit trail) — these jointly constitute S3 | S3 is SPLIT across two parts; Beer's VSM predicts this creates coordination overhead; Wave C should explicitly designate which Part owns S3 authority |
| S4 — Intelligence/Futures | Part 5 (Compound Learning) + Part 9 (Owner Interaction Scaffold) — compound cycles generate system-level learning; owner weekly/monthly reviews are S4 environment-scanning | S4 is currently weakly developed; methodology library + market research feeds are sparse |
| S5 — Identity/Policy | FUNDAMENTAL v1.0 (LOCKED) + JETIX-FPF (constitutional) + Ruslan's HITL authority | S5 exists and is stable; the LOCK mechanism is correct |

**Critical gap.** Beer's VSM requires that S4 (environment-scanning) has a structural channel to S5 (policy) — otherwise the system cannot adapt its identity in response to environmental change. In the current Foundation, the S4→S5 channel is: Ruslan reads health dashboards + owns AWAITING-APPROVAL acks. This channel is functional but thin. When Ruslan is unavailable for >7 days, S4 signals queue without propagating to S5 adaptation. Wave C design must declare the S4→S5 channel explicitly and specify a degraded-mode protocol when the channel is blocked (e.g., escalation timeout → brigadier opens a HITL packet proactively).

**Tradeoff.** VSM 5 systems vs Foundation 10 parts — different decomposition lenses. VSM decomposes by FUNCTION (what each level does in control terms); Foundation decomposes by ARTIFACT TYPE and LIFECYCLE (what is produced and when). They are not in conflict but they are not isomorphic. The danger: if Foundation parts are designed without VSM-awareness, S2 coordination and S3 audit may be accidentally merged (both inside Part 6) creating a controller that lacks the variety to distinguish coordination from audit — which Beer shows leads to oscillation between S1 operational units. Resolution: Wave C interface cards for Parts 4, 6, 8 must explicitly declare their VSM-level function. A part can serve multiple VSM levels but must NOT conflate their control logics.

### Principle 4 — Senge's Law 7: Cause and Effect Are Not Closely Related in Time or Space

**Source.** Senge (Fifth Discipline Fieldbook, 1994), Ch. 13-17 systems archetypes. From The Fifth Discipline (not the Fieldbook, but referenced throughout): "The bottom line of systems thinking is leverage — seeing where actions and changes in structures can lead to significant, enduring improvements." Law 7: "Cause and effect are not closely related in time and space." The Fieldbook tools section (Ch. 17): "Links and loops are the language of systems thinking." Reinforcing loops (R) are self-amplifying; balancing loops (B) are goal-seeking. The archetype "Eroding Goals" (Meadows Ch. 5 calls it "drift to low performance"): "When perceived performance slips, the goal is allowed to slip." [src: senge-fifth-discipline-fieldbook.md Ch. 13-17; meadows-thinking-in-systems-2008.md Ch. 5]

**Applied to Foundation.** Two specific feedback-loop identifications for Foundation Parts:

**R1 loop (Knowledge Compound, Part 3 + Part 5).** More KB entities → higher signal quality per /ask query → higher synthesis quality → better execution → more learning captured per cycle → more KB entities added. This reinforcing loop runs through Part 3 (KB substrate) + Part 5 (Compound Learning ritual). The DRR discipline in strategies.md is the conversion mechanism: execution experience → structured rule → KB entry. Senge Law 7 prediction: the full compounding effect of R1 will not be visible for 5-10 cycles. Interventions that look at KB entry count after 1 cycle and declare "not working" are confusing cause (DRR discipline) with effect (synthesized methodology leverage) across a time gap. Wave C health metric design must measure the LEADING indicator (DRR write rate per cycle) separately from the LAGGING indicator (synthesis quality per /ask query) — with explicit acknowledgment that the lag is 5-10 cycles.

**B1 loop (Governance Backpressure, Part 6 + Part 9).** More AWAITING-APPROVAL packets opened → more Ruslan attention consumed → less capacity for strategic work → owner interaction scaffold overloaded → decision quality declines → more AWAITING-APPROVAL opened retroactively to fix bad decisions → more packets opened. This is a Senge "fixes that fail" archetype: the governance gate is well-intentioned but generates backpressure that makes the system less governable. Law 7 diagnosis: the problem (owner overload) appears AFTER the solution (more gates) has been applied, making it look unrelated. Foundation design must explicitly declare the B1 loop and its safety valve: the J1/J2/J3 decision taxonomy (FUNDAMENTAL §4.2) is the variety-reduction mechanism that prevents B1 from dominating. Wave C must enforce that ≥70% of decisions route to J1 (auto-approve) to keep B1 in check.

**Tradeoff.** Senge "today's problems from yesterday's solutions" vs FPF IP-3 (artifacts over conversations — commit everything). FPF's discipline of making every state change a committed artifact is a design choice that generates audit overhead — which IS a "yesterday's solution" (version control) creating "today's problem" (commit fatigue). Resolution: the overhead is justified because audit resilience is L5 (rules leverage per Meadows), not L12 (parameter overhead). The cost is real but the leverage is higher than it appears. Wave C should standardize commit templates to reduce friction while preserving audit completeness.

### Principle 5 — Adjacent-Possible: The System Can Only Reach States Adjacent to Its Current Configuration

**Source.** Kauffman (At Home in the Universe, 1995) Ch. 1: "Thirty years of research have convinced me that... natural selection is important, but it has not labored alone to craft the fine architectures of the biosphere... Another source—self-organization—is the root source of order." The adjacent-possible: at any moment, the system can only evolve into states reachable from its current configuration — the set of states one transformation away from what exists now. Complexity explosion beyond the adjacent-possible requires structural change, not parameter adjustment. [src: kauffman-at-home-in-the-universe-1995.md Ch. 1] Beer (1972) Ch. 4 makes the same point via heuristics: "An heuristic specifies a method of behaving which will tend towards a goal which cannot be precisely specified because we know what it is but not where it is." [src: beer-brain-of-the-firm-1972.md Ch. 4 p. 53]

**Applied to Foundation.** The adjacent-possible principle constrains the realistic Wave C → Wave D → Phase B trajectory. From the current state (Phase A: 10-part Foundation, 6-agent ROY swarm, ~550 wiki entities, no canonical methodology library, no operational S3 audit function), the adjacent-possible next states are:

1. Canonicalize coordination protocol (Part 4) + health SLI/SLO schema (Part 8) → adjacent, achievable in Wave C without structural change.
2. Populate methodology library from DRR compound outputs (Part 3 + 5 integration) → adjacent, achievable if R1 loop is running.
3. Activate S3 audit function with quarterly meta-agent review → adjacent once Parts 6 + 8 are defined.

**NOT adjacent in Phase A** (require structural change first):
- Multi-client namespace isolation (needs D27 fork-and-merge operational + client-specific wiki root isolation active) — requires structural boundary change, not parameter change.
- Automated L3/L4 decision routing without HITL → requires J-level taxonomy fully validated across ≥50 real decisions, which takes 5-10 cycles from now.

Adjacent-possible discipline for Wave C: every per-Part work list should be checked against the adjacent-possible constraint. If a Wave C deliverable requires a structural precondition that is itself not yet operational, flag it as "Phase B, not Wave C."

## §5 Feedback-Loop Map — Foundation-Level

The systems-expert identifies 2 reinforcing loops + 2 balancing loops in the Foundation as designed (candidate-parts-merged.md):

**R1 — Knowledge Compound Loop** (+, Parts 3 + 5):
`DRR discipline (Part 5) → methodology entries (Part 3) → better retrieval quality → better execution → more DRR → ...`
Dominant in: long-cycle behavior (cycle 5+). Currently open-loop (insufficient conversion mechanism between DRR entries and wiki methodology entries). Wave C must close R1 by establishing the DRR → methodology-library → /ask retrieval path explicitly.

**R2 — Agent Self-Improvement Loop** (+, Part 5 + Part 4):
`strategies.md entries (Part 5) → better-dispatched modes (Part 4) → higher-quality artefacts → more learning extracted → strategies.md updates → ...`
Dominant in: medium-cycle behavior (cycle 2-5). Currently weakly operational (19 strategy entries across 8 agents per health.md). R2 will compound with R1 once the methodology library is populated.

**B1 — Governance Backpressure Loop** (−, Part 6 + Part 9):
`AWAITING-APPROVAL packets (Part 6) → owner attention consumed (Part 9) → decision quality declines under overload → more retroactive review needed → more packets → ...`
Safety valve: J1/J2/J3 taxonomy (FUNDAMENTAL §4.2). Must keep J1 ≥70% of all decisions. B1 becomes dangerous if governance gate is mis-scoped (treating L12 parameter choices as J3 decisions).

**B2 — Scope Boundary Loop** (−, Part 6 + Part 1):
`Canonical promotion (Part 1 git commits) → system state grows → new patterns emerge → new anti-scope rules needed → new gate conditions → more promotion friction → ...`
This is a Meadows "eroding goals" risk: as the system matures, the anti-scope rules multiply, and the friction of canonical promotion increases. Safety valve: regular `/lint --check-stage-gates` + quarterly anti-scope rule pruning (Part 8 immune-system function).

**Loop dominance hypothesis**: R1 dominates in cycles 5-50 (knowledge compounds faster than governance friction accumulates). B1 becomes dominant in cycles 50+ IF J1/J2/J3 taxonomy is not enforced. The Phase A target is to establish R1's dominance before B1 catches up.

## §7 Open Questions + Conflicts Surfaced

**TRADEOFF-01: VSM S3 Authority Split.**
Parts 6 (Governance Gate) and 8 (Health Monitoring) both serve VSM System 3 (Audit/Optimisation) function. Beer's VSM shows that splitting S3 across two controllers creates oscillation risk: neither Part has full authority to act on audit signals, so each waits for the other, introducing delay at exactly the point where fast response matters. Wave C must designate a clear S3 "lead Part" (likely Part 8) with Part 6 serving as its enforcement arm, not an independent audit function.

**TRADEOFF-02: Requisite Variety vs Context Engineering Simplicity.**
Ashby's Law demands the coordination controller (Part 4, brigadier) has variety ≥ task-type variety (≥20 distinct cells). Anthropic's building-effective-agents guidance (Card #3) demands: keep system prompts simple, avoid over-engineering. These pull in opposite directions at the implementation level. Resolution path: the variety must live in the ROUTING LOGIC (declarative YAML), not in the system prompt prose. Short system prompt + rich routing table = Ashby-compliant + Anthropic-simple. Wave C design constraint: brigadier system prompt ≤2000 tokens; routing table expresses all ≥20 distinct routing rules as structured YAML.

**TRADEOFF-03: R1 Time-Delay vs Cycle-Pressure.**
R1 loop (knowledge compound) has a 5-10 cycle delay before compounding becomes visible. ROY cycle pressure (brigadier wants to ship Wave C quickly) creates the Senge "fixes that fail" risk: if no compounding is visible after 2 cycles, someone will try to "fix" R1 by adding more DRR rules (L12 parameter change) rather than waiting for the feedback to propagate. Foundation Wave C must pre-declare the expected R1 lag time and protect the DRR discipline from premature optimization attempts.

**OQ-SYS-01:** Does the B1 loop (governance backpressure) have a measurable KPI that would indicate it is becoming dominant? Hypothesis: track `count(AWAITING-APPROVAL packets opened per cycle) / count(decisions made per cycle)`. If this ratio exceeds 0.3 (more than 30% of decisions require a gate), B1 is likely dominant. Ruslan should confirm this threshold or propose an alternative.

**OQ-SYS-02:** Kauffman's adjacent-possible constrains what is achievable in Wave C. The Wave C work-list for Part 5 (Compound Learning) includes "populate methodology library from DRR outputs." Is the R1 conversion mechanism (DRR entry → methodology library write → /ask retrieval) currently adjacent (one transformation from operational) or does it require a structural change (new tool, new pipeline step)? If structural: Part 5 Wave C scope must include the structural prerequisite, not just the compound output.
