## §1 Cross-claim syntheses

### §1.1 Shared architectural DNA

Both systems rest on the same three-layer structural spine:

- **Principles layer** (IWE: ZP → FPF → SPF; Jetix: Pillar C Tier-2 12 rules + Part 6b constitutional_never_list)
- **Knowledge substrate** (IWE: Pack as bounded context per domain; Jetix: wiki/ + 9 entity types + niches/)
- **Execution lifecycle** (IWE: WP state machine Open → Work → Close; Jetix: project 5-state scoped → staged → activated → under-review → closed)

`F: F4 | G: cross-read | R: refuted_if_either_system_lacks_these_three_layers`

[src: 02-iwe-deep-v2.md §2.3 generative hierarchy; swarm/wiki/foundations/part-7-project-lifecycle-substrate/architecture.md §0]

### §1.2 Divergent governance philosophies

**IWE** uses a single-owner exoskeleton pattern with explicit exoskeleton ≠ autopilot distinction. The user must be able to explain every result «без 'ИИ подсказал'». Intelligence amplification is the stated goal.

**Jetix** has a more complex governance: 12-agent hub-and-spoke, multi-role dispatch via brigadier, with AI doing NOT strategize (R1 hard rule). The intelligence amplification goal is present (F-G-R, IP-1, Pillar C) but the bulk of Foundation Parts (06-honest-self-audit.md §11) are memory + automation, not amplification.

`F: F4 | G: cross-read | R: refuted_if_Jetix_revises_governance_toward_single-agent`

[src: 02-iwe-deep-v2.md §1.3 «exoskeleton not prosthesis»; 06-honest-self-audit.md §12 honest answer]

### §1.3 FPF adoption mode

IWE: FPF as **generative base tier** — every DS artefact is generated by constraints flowing down ZP → FPF → SPF → FMT → Pack → DS. FPF is structurally upstream of all work.

Jetix: FPF as **tactical adoption** — 7 components are direct FPF-derivative (F-G-R schema, IP-1, Default-Deny, Pillar C, AWAITING-APPROVAL, executor-binding, Halt-Log-Alert); remaining ~27 components are memory + automation without formal FPF grounding.
