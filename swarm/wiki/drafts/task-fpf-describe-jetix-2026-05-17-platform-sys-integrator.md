## §2 Ashby requisite variety — platform-level analysis

**System in scope:** Jetix Platform as U.System (A.1 supersystem) hosting N U.BoundedContext workshop instances. The controller = platform coordination layer (5 components: C1–C5 per eng-integrator §3.4). The controlled system = the variety space generated by N heterogeneous workshops.

**Variety analysis:**

Each workshop produces a variety signal proportional to its specialisation depth. A single workshop (N=1, Phase A) is effectively a Self-OS instance (doc 01) — its variety is bounded by one deyatel's information-processing range: voice pipeline + wiki + git + one domain specialisation. The coordination layer (Phase A = ad-hoc, no platform code) does not need to manage variety beyond what one instance generates — this is trivially satisfied.

As N grows, the variety space of the platform system is:

```
V_system(N) ≈ N × V_single_workshop + V_cross_workshop_cooperation_events
```

The cross-workshop cooperation event variety (the number of distinct task-chunk × result-type × provenance combinations) grows super-linearly because each new workshop can cooperate with all prior workshops in novel combinations.

The coordination layer's variety budget is bounded by:
- C3 (cooperation event log): append-only JSONL — sequential, no aggregation variety; captures events but cannot pattern-match across them autonomously
- C5 (tools registry): static catalogue — no dynamic variety; knows what workshops have, cannot dynamically reallocate across workshops
- C4 (role declaration): static per-participant markdown — no dynamic role-reassignment based on load

**Verdict (Ashby):** Coordination layer variety is FIXED (5 static components); workshop system variety is GROWING (super-linear in N). The Ashby balance breaks at approximately N=3–5 workshops with heterogeneous specialisations. At that point, the coordination layer cannot track which workshop has capacity for which chunk type, cannot detect cross-workshop quality divergence, and cannot route cooperation events to the optimal workshop — because it has no sensor variety to cover the system variety.

**AP-SYS-10 flag (Ashby requisite variety violated at N>5):** `controller_variety (fixed 5-component coordination layer) < system_variety (N×domain-specific workshop signals)`. No compensating mechanism is named in the eng-integrator draft.

```yaml

F: F3  # multi-source pattern: Ashby book-as-frame + doc 01 S3 stub precedent + cooperation event log design
ClaimScope: |
  Holds for the Phase B target (multiple workshop network). NOT applicable to Phase A (N=1).
  NOT applicable if all N workshops are homogeneous (same domain) — homogeneous workshops
  reduce combined variety substantially. Claim applies to the heterogeneous-specialist network
  described in Workshop Concept §5.
R: |
  Refuted if coordination layer variety is demonstrated to handle ≥5 heterogeneous workshops
  without quality degradation across cooperation events (measurable by cooperation-event
  result-acceptance-rate metric across ≥10 events per workshop pair).
  Accepted while platform code = 0 and S3 platform component is absent.
```

## §4 Feedback loops — platform ↔ workshop

Eng-integrator names cooperation events and types them (§5 sequence diagram). Cooperation events are the SUBSTRATE of loops, not the loops themselves. A feedback loop requires: a signal, a comparator, an effector, and a closed path back. The eng-integrator draft has the signal path (cooperation events go out and come back) but no comparator and no effector.

**Platform-level feedback loops identified:**

| Loop | Polarity | Substrate | Platform component(s) | Current state |
|---|---|---|---|---|
| **R1 Task-complexity growth** | + reinforcing | As platform successfully handles complex tasks, it attracts more complex tasks (reputation compound) | None — requires S4 intelligence function + external reputation signal | **OPEN.** No closing mechanism exists in Phase A or B design. Aspirational. |
| **R2 Capability-accumulation** | + reinforcing | Each cooperation event produces DRR entries; DRR entries improve workshop capabilities; improved capabilities attract more complex tasks | Cluster 6 (Эволюция) per-workshop DRR → meta-evolution concept | **PARTIALLY CLOSED per-workshop (doc 01 R2).** Platform-level meta-evolution loop = OPEN (mechanism undesigned per §3 S4 aspirational). |
| **B1 Quality-correction** | − balancing | Cooperation event result quality below threshold → reject → re-route to different workshop → quality improvement | NOT DESIGNED. Requires S3 platform health monitoring to detect below-threshold results. | **SEVERED.** No quality comparator at platform level. Eng-integrator §3.3 states «result A + provenance» returns to coordination layer; coordination layer has no quality comparator (C1–C5 have no comparator logic). |
| **B2 Specialisation-drift brake** | − balancing | Workshop specialises too narrowly → becomes unable to absorb cooperation events outside its domain → workshop signals its constraint (role declaration C4) → coordination layer routes elsewhere | C4 role declaration (static) — partial only | **WEAKLY CLOSED.** C4 exists in concept; drift detection (when a workshop's declared roles no longer match its actual capability) has no monitoring mechanism. |
| **B3 Isolation-preservation** | − balancing | Workshop internal state leaking across BoundedContext boundary (invariant 3 violation) → isolation enforcement | Invariant 3 in eng-integrator §2.2: «workshop isolation default» | **CONSTITUTIONALLY DECLARED** but not mechanically enforced. Relies on Pillar C rule application, not on a monitoring loop. |

**Loop dominance (Phase A):** No platform-level loops are closed. Phase A = zero platform code; platform dynamics are entirely determined by single-workshop dynamics (doc 01 loops: R1 externalisation, R2 compound, B1 severed, B2 weak). The platform as a distinct dynamic system does not yet exist.

**Loop dominance (Phase B target, N=3–5):** R2 capability-accumulation is the highest-gain loop IF the meta-evolution mechanism is designed (S4 functional). Without S3, B1 quality-correction is severed — the platform runs on reinforcing loops only (R1 + R2), which is fragile to quality drift from any workshop (Senge law: «behaviour grows better before it grows worse»).

```yaml
ClaimScope: |
  Holds for Phase B (N=2–5 workshops). NOT applicable to Phase A (N=1; platform loops moot).
  Changes if platform S3 component is designed and operational before first multi-workshop event.
R: |
  Refuted if: platform quality-correction loop closes before first multi-workshop cooperation event
```

## §5 Federation topology — boundary analysis

**Topology per eng-integrator §2.2:** Each workshop = U.BoundedContext. Platform = coordination layer outside all workshops. Cross-workshop interaction = cooperation event routed through coordination layer. This is a HUB topology: coordination layer is the hub; workshops are spokes.

**Clean boundaries or overlap?**

The eng-integrator declares four invariants (§2.2). Systems analysis of each:

| Invariant | Clean or risk | Systems finding |
|---|---|---|
| Filesystem-as-SoT preserved | Clean | Each workshop's filesystem is its SoT. Platform adds a coordination log (C3) that is also filesystem-based. No conflict. |
| Method ownership stays with workshop (R12) | Clean | Ownership is declared. Enforcement depends on Pillar C rule application — same constitutional mechanism as individual-workshop R12 enforcement. |
| Workshop isolation (no cross-BC read without cooperation event) | **Risk** | This is the weakest boundary. The mechanism for preventing cross-BC reads is constitutional (Pillar C rule 9 — no self-modification at runtime). But at platform scale, two workshops could share a tools registry entry (C5) or a typed edge endpoint (Cluster 5 Связи) that implicitly couples their internal state. Typed edges = cross-BC pointers; if edge endpoints are read without a cooperation event wrapper, invariant 3 is soft-violated. |
| Foundation fork-portability | Clean | Structural — each fork is independent. No coupling risk. |

**Weakest boundary: Cluster 5 (Связи) — typed edges as implicit coupling channel.**

Eng-integrator §3.2 Cluster 5: «cross-BoundedContext edges с явным провенансом. Каждый edge = cooperation event». This is the intended design. But the typed edge schema (wiki/graph/edges.jsonl) does not enforce that every edge creation is wrapped in a cooperation event. An agent in workshop A could write a typed edge pointing to an artefact in workshop B's BoundedContext without triggering a cooperation event — this is a LOW-COST isolation violation that the current architecture does not prevent.

**D-PLAT-3 resolution (typed edges sufficiency):**

D-PLAT-3 (eng-integrator §8.2): «typed edges = coordination mechanism; sys-alternative: typed edges = feedback loop substrate; platform viability depends on feedback loop architecture».

Systems resolution: BOTH positions are correct AND insufficient alone.

- Typed edges ARE the feedback loop substrate (eng position confirmed).
- Typed edges alone are NOT sufficient for platform viability (sys position confirmed).
- Missing: a COMPARATOR that reads the typed edge patterns and generates corrective signal. Without the comparator, typed edges are a write-only log — they accumulate provenance but generate no corrective feedback.
- The needed addition is a platform-level pattern-detection function (S3 or S4 level) that reads cooperation event edges and surfaces: divergent workshop quality, missing coverage areas, duplicate work patterns.

```yaml
claim: "D-PLAT-3 integrated resolution: typed edges are necessary AND insufficient; the missing element is a pattern-detection comparator at S3/S4 that closes the coordination feedback loop"
F: F4  # operational convention: Ashby comparator requirement + VSM S3 function definition + Senge information-flows rung L6/L7
ClaimScope: |
  Holds for the Phase B platform architecture. NOT applicable in Phase A (N=1; no cross-workshop
  edges exist). Applies specifically to the heterogeneous-workshop network described in
  Workshop Concept §6.
R: |
  Refuted if: (a) cooperation events alone (without pattern-detection) produce acceptable
  coordination quality across ≥10 inter-workshop events (N≥3 workshops); OR
  (b) human steward (Platform Steward role, eng-integrator §2.2 Roles) manually performs
  the comparator function and this proves sustainable at N=5.
  Accepted if S3 comparator is absent and cooperation event quality degrades in first multi-
  workshop project.
```

## §6 Scalability — Phase A → Phase B transition

**What holds at N=1 (Phase A):**
- S1 (workshop operations): evidenced — doc 01 R1 externalisation loop operational.
- S5 (constitutional layer): operational — Pillar C + R12 LOCKED.
- Typed edges: operational within single BoundedContext (wiki/graph/edges.jsonl).
- 6-cluster topology: validated conceptually (Workshop Concept F5 LOCKED).

**What breaks at N=2 (first phase transition):**

The FIRST break is S2 absence — not S3. When the second workshop joins, the platform needs:
1. A way to signal that workshop A has taken chunk X (so workshop B does not duplicate it).
2. A way to signal that result A is ready for aggregation.
3. A way to route chunk Y to the correct workshop (B vs any future C).

None of C1–C5 provides these three functions in a loop-closed way. C3 (cooperation event log) provides a signal of what happened, but not in real-time, and not with routing intelligence. C4 (role declaration) provides a static capability list, but has no mechanism to signal current load vs capacity.

**The antifragility check (Brief §5.1 ≤30% refactor at 10×):**

10× from Phase A (N=1) → N=10 workshops. Structural-change estimate:
- S2 component: must be designed (currently 0%) → adds ~15–20% structure (cooperation routing + load signalling).
- S3 component: must be designed (currently 0%) → adds ~15–20% structure (quality monitoring + pattern detection).
- Typed edge enforcement: must add cooperation-event-wrapper discipline → ~5% structure.
- Total structural change at N=10: ~35–40%.

**Verdict: FRAGILE at 10× (>30% threshold).** The platform as described in Phase A design does not achieve antifragility at 10× scale. The 30%+ restructuring is driven by two absent VSM layers (S2, S3), not by internal refactoring of existing components.

**Antifragility condition (how to achieve at-least-not-fragile):** Pre-invest in S2 design before N=2 and S3 design before N=3. The analogy from strategies.md `rule: G3-pre-investment-mandatory-for-G4-antifragility` applies directly: the platform must pre-specify S2+S3 components before the load of N=2+ workshops arrives.

```yaml
claim: Platform is FRAGILE at 10× (N=10 workshops) requiring ~35–40% structural change; pre-investment in S2+S3 component design before N=2 is the antifragility-enabling intervention
F: F3  # multi-source: Ashby variety analysis §2 + VSM gap table §3 + strategies.md G3-pre-investment rule + antifragility-check formula
ClaimScope: |
  Holds for the heterogeneous-workshop network. NOT applicable if platform is deliberately kept
  homogeneous (single domain). Pre-investment claim holds specifically for Phase B target design;
  post-N=3 is the BOSC-A-T-X S+C gate (scale + composition change).
R: |
  Refuted if: platform launches at N=2+ WITHOUT pre-designed S2+S3 and maintains acceptable
  cooperation quality across ≥20 events (empirical antifragility disproof).
  Accepted if structural change at N=10 is verified ≥30% by engineering review.
```
