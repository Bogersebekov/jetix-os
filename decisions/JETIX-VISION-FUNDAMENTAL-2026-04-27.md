# JETIX VISION OF SYSTEM — FUNDAMENTAL

## §0 Что это за документ

Это **базовая, фундаментальная Vision Document системы Jetix** — описание того что такое Jetix как **"система всех систем"** в вакууме, без привязки к конкретной нише / личности / бизнесу.

**Почему два документа:**

Jetix должен быть **двуслойным**:
- **Layer 1 (этот документ — FUNDAMENTAL):** базовая прошивка / ядро / архитектура которая работает одинаково в **любой нише** — AI-консалтинг / медицина / e-commerce / научный research / преподавание / etc. Можно "перетянуть" из одной ниши в другую без переписывания base.
- **Layer 2 (отдельный документ — RUSLAN-LAYER):** оптимизация / overlay / специфика под текущую ситуацию Ruslan'а — DACH Mittelstand / AI consulting / Phase 1 €50K target / 11 archetypes / Korp-Startup positioning / Top Lists strategy / etc. Натягивается ПОВЕРХ FUNDAMENTAL.

**Принцип:**
- Сначала описываем **FUNDAMENTAL** (этот doc) — generic, sector-agnostic, person-agnostic
- Потом натягиваем **RUSLAN-LAYER** (другой doc) — specific под Ruslan
- Если завтра Ruslan уйдёт в другую нишу → меняется только RUSLAN-LAYER, FUNDAMENTAL остаётся
- Если кто-то другой возьмёт Jetix-методологию и применит к своему делу → берёт FUNDAMENTAL и пишет свой LAYER

**Что должна уметь FUNDAMENTAL Jetix:**

- Адекватно **работать с информацией** (capture / store / retrieve / synthesize / compound)
- **Улучшаться сама** (self-audit / agent strategy learning / methodology refinement)
- **Wiki / KM хорошо работает** (queryable / cross-referenced / non-contradictory)
- **Агенты между собой адекватно общаются** (mailboxes / message schemas / handoff protocols / escalation taxonomy / Левенчук discipline)
- **Здоровье системы** прозрачно (health indicators / monitoring / alerting)
- **Best practices** из лучших систем — Левенчук ШСМ / multi-agent architectures / event sourcing / observability / Lean / TOC / etc

**Стиль:**
- Generic, vacuum-sealed, sector/person/business-agnostic
- На человеческом языке (без jargon-soup)
- Глубоко (Левенчук discipline / 1000% / триллион процентов)
- Если что-то выглядит Ruslan-specific → автоматически в RUSLAN-LAYER, не сюда
- Best-of-best наработки cited (Левенчук / ШСМ / multi-agent papers / etc)

**Authoring workflow:**
1. Cloud Cowork synthesizes generic foundations из best sources (FPF + SYSTEM-OVERVIEW + Левенчук + multi-agent best practices)
2. Ruslan reviews / dictates что добавить / изменить
3. Итерация → push → Ruslan pulls → видит → ack
4. Когда LOCKED → RUSLAN-LAYER overlay начинается

### 🎯 Главный принцип системы (Foundation)

**Система = усилитель мозга, инструмент для мышления и работы с информацией.**

Не "ещё один SaaS", не "просто agents", не "просто wiki". Это **продолжение и усиление мозга владельца** — помогает думать стратегически качественно/быстро/эффективно. Heavy информация и heavy задачи **перекладываются на систему** → владелец освобождает голову для **стратегического мышления + чистоты + спокойствия**.

**Ключевая фишка:** быстро обучаемая + быстро делающая. Направление учёбы определяется **стратегическими документами**. Можно сегодня учить video editing, завтра M&A advisory, послезавтра physics research — система адаптируется под фокус владельца.

**Двойной вектор: улучшение + сохранение.**
- **Улучшение:** система направлена на постоянное улучшение и развитие — себя самой и владельца. Через compound engineering, skill acquisition, methodology evolution.
- **Сохранение:** одновременно — на сохранение и **безопасность себя** + чистоту состояния. Не allows degradation. Регулярные health checkups → проактивное обнаружение drift / stagnation / chaos.

**Кто пользователь системы (foundational positioning):**

Jetix создан для **профессиональных интеллектуальных работников** — людей которые:
- Работают с **большими объёмами информации**
- Решают **сложные задачи / вопросы / проекты**
- Управляют **множественными направлениями** одновременно
- Имеют **долгосрочные стратегические амбиции** (не tactical-only)
- Готовы **инвестировать в систему** time/attention для multiplier ROI

Это **НЕ массовый продукт** для casual users. Целевая audience — founders / researchers / investors / strategic consultants / scientists / serious creators. Система требует discipline для использования + дисциплина возвращается через compound leverage.

**Стиль работы system × owner:**

Система **подсказывает / показывает / подпинывает** — НЕ "делает за тебя". Это критично:
- Владелец остаётся **agent** (acting subject), не **patient** (object that things happen to)
- Система highlights что нужно посмотреть, какие decisions ждут, где drift намечается
- Финальный выбор всегда у владельца (per Левенчук architect-orbit + D2)
- Anti-pattern: автоматически принимать decisions за владельца → ведёт к agency erosion

### Категория A — Strategic Management Layer

> Слой стратегического управления — основа на которой всё работает. Стратегические документы определяют направление системы; система помогает их создавать, следить за ними, придерживаться.

**UC-A.1 — Multi-level strategic document hosting**
- **Что:** Система имеет dedicated место для стратегических документов на разных уровнях иерархии: System (Vision / Locks / Architecture) / Strategic (DoT / North Star) / Annual / Quarterly / Project / Sub-project.
- **Зачем:** Стратегия должна быть explicit + queryable + versionable. Без structured location strategy drifts; agents не могут на неё опереться.
- **Принцип:** один документ один источник истины per layer; cross-references между уровнями; LOCKED status для stable docs.

**UC-A.2 — Strategic document creation assistance**
- **Что:** Система помогает владельцу **создавать** strategic docs: synthesizes из существующих materials, предлагает structure, surfaces contradictions с предыдущими решениями, drafts for review.
- **Зачем:** Strategic thinking — heavy work; system reduces friction чтобы чаще обновлять стратегию (а не редко и painful).
- **Принцип:** AI = drafter + reviewer; владелец = ack/edit/reject. Per Левенчук: AI не strategize самостоятельно.

**UC-A.3 — Strategic alignment enforcement**
- **Что:** Когда система выполняет любую задачу — проверяет alignment с актуальной стратегией. Если задача противоречит strategic doc → escalation к владельцу. Если стратегия меняется → downstream artifacts (project plans / agent priorities) refresh'ятся.
- **Зачем:** Prevent drift. Stratагические documents без enforcement = decoration. С enforcement — реальный compass.
- **Принцип:** strategy → policy → execution. Все 3 связаны explicit refs.

### Категория B — Information Lifecycle Layer

> Работа с информацией. Не "жрать всё подряд", а selective inbox → quality storage → synthesis → reuse.

**UC-B.1 — Selective inbox с filters**
- **Что:** Все incoming signals (voice memos / web / chat / books / videos / research / messages) проходят через **inbox layer** со strategy-aware фильтрами. Не всё попадает в систему — только что соответствует current strategic focus + personal preferences владельца.
- **Зачем:** Garbage in = garbage out. Без фильтров system overwhelmed; с фильтрами — focused signal.
- **Принцип:** filters declarative (yaml-described), не hidden logic. Владелец может посмотреть/изменить/override.

**UC-B.2 — Multi-wiki knowledge storage (one per project / domain)**
- **Что:** Система может создавать и поддерживать **несколько wiki одновременно** — одна на life если хватает; несколько per project если нужны раздельные knowledge bases. Built на best-of-breed методологии (Karpathy LLM Wiki + Andy Matuschak evergreens + Niklas Luhmann Zettelkasten).
- **Зачем:** Single wiki не масштабируется когда проекты разные. Multiple wikis с unified retrieval → flexibility + isolation.
- **Принцип:** agent-driven но **manual-accessible**. Если agents fail — владелец может найти всё руками через файловую систему.

**UC-B.3 — Source preservation (raw + processed)**
- **Что:** Сырые данные (voice transcripts / web pages / book excerpts / chat logs / research dumps) **сохраняются permanent** + processed выжимки идут в wiki. Provenance каждого insight'а трассируется к raw source.
- **Зачем:** Provenance = trust. Через 6 месяцев захочется проверить "откуда мы это взяли" — должно быть реконструируемо.
- **Принцип:** raw immutable; processed может быть обновлён/переписан; provenance link обязателен.

**UC-B.4 — Synthesis on demand**
- **Что:** На основе wiki + raw + сейчас открытых проектов система **быстро** generates: новые статьи / posts / answers to queries / summaries / comparisons / strategic insights / talking points / etc. Workflow: query → retrieve relevant → synthesize → present.
- **Зачем:** Reuse accumulated knowledge — основной leverage compound engineering.
- **Принцип:** synthesis всегда cited (с references к sources); владелец может verify chain of reasoning.

**UC-B.5 — Rapid ingestion pipeline (proactive learning)**
- **Что:** Пример: появился проект "выбрать страну для переезда". Вместо того чтобы владелец смотрел 50 видео, читал 30 articles вручную — он **загружает все materials в систему**, она их processes (transcripts / extracts / structures), сохраняет в нужный wiki, генерирует analytics + reports → владелец потом за 30 минут получает actionable view.
- **Зачем:** Massive time leverage. Heavy информационный input shifts от founder к system.
- **Принцип:** ingest pipeline universal — не важно тип material (video / book / article / transcript / chat history). Output structured per project schema.

### Категория C — Self-Improvement & Learning Layer

> Система улучшается сама + умеет учиться новым skills под фокус владельца.

**UC-C.1 — System self-improvement (Compound Engineering)**
- **Что:** Каждая работа cycle (Plan / Work / Review / Compound 40/10/40/10) produces artifacts которые улучшают будущие cycles: agent strategies refined, methodologies updated, KB cross-refs added, frequent-query indexes built.
- **Зачем:** Linear systems plateau; compound systems exponentiate. Без self-improvement system stagnates за месяцы.
- **Принцип:** improvements are review-gated (не auto-deploy без approval), но incentive structure encourages их proposal по каждому cycle.

**UC-C.2 — Skills acquisition under strategic direction**
- **Что:** Когда стратегия требует нового capability (e.g., video processing / post writing / specific research domain / language / tool mastery) — система **изучает этот skill**: ingests best resources на эту тему, builds wiki sub-area, trains agent skills, creates templates/workflows. Скорость skill acquisition = competitive edge.
- **Зачем:** "Быстро обучаемая + быстро делающая" — главная фишка системы. Без skill flexibility она stuck в исходных capabilities.
- **Принцип:** skill direction всегда из strategic docs (не random); skill acquisition documented (provenance / methodology / what worked vs what didn't).

**UC-C.3 — Methodology library (reusable patterns)**
- **Что:** Система maintains library of reusable methodologies / templates / workflows которые накопились через cycles. Каждый методологический artifact — versioned, cross-referenced, applicable across projects.
- **Зачем:** Roy-replication readiness — другие людьи / instances могут взять методологию и применить. Direct support для D21 Roy-Replication + D27 Fork-and-merge.
- **Принцип:** методология = **first-class artifact**, не побочный эффект. Каждое решение оставляет методологический след.

### Категория D — Project & Resource Management Layer

> Управление проектами + ресурсами (время / финансы / энергия / attention / credits).

**UC-D.1 — Project lifecycle management**
- **Что:** Templates + tooling для запуска / ведения / закрытия проектов: brief → milestones → tasks → status tracking → review → closure / archival. Унифицированно для всех типов проектов (research / product / outreach / personal / etc).
- **Зачем:** Без project structure — chaos. С единым framework — сравнимость + transferability.
- **Принцип:** lightweight, не bureaucratic. Templates suggest structure, не enforce rigid forms.

**UC-D.2 — Resource budgeting & monitoring**
- **Что:** Система отслеживает ресурсы: time (deep work hours / context-switches / sleep), money (cash / runway / spend per category), energy (subjective state / cycles / recovery), attention (manager attention budget ≤ N active tasks), credits (API tokens / compute / 3rd-party limits).
- **Зачем:** Resource awareness = sustainable execution. Founder без resource visibility burns out / overcommits / runs out of cash.
- **Принцип:** resources tracked but не surveillance — данные доступны владельцу для self-management, не для external judgment.

### Категория E — Agent Swarm Operations Layer

> Агенты (рой) — выполняют сложные многоступенчатые задачи. Glubokо настроены под Левенчук discipline.

**UC-E.1 — Multi-agent coordination (бригадир + рой)**
- **Что:** Система имеет coordinator (бригадир) + specialised executors (рой). Бригадир принимает complex задачу → декомпозирует → dispatches sub-tasks parallel/sequential subagents → integrates results → returns to user. Subagents могут sami spawn'ить sub-subagents (рекурсивно, controlled depth).
- **Зачем:** Single-agent solutions plateau на complexity ceiling. Multi-agent с coordination — handle gigantic tasks (whole projects / migrations / research dumps / etc).
- **Принцип:** clear role boundaries (бригадир НЕ executor; executor НЕ strategize); message-passing not shared mutable state; всё auditable.

**UC-E.2 — Agent self-learning (strategy refinement)**
- **Что:** Каждый agent maintains `strategies.md` — постоянно улучшающийся набор patterns которые работают для этого agent'а в этом role. Через cycles агенты накапливают what worked vs what failed → next time лучше.
- **Зачем:** Agents которые не learn — stuck. Agents которые learn — ROI растёт через месяцы.
- **Принцип:** learning сохраняется per agent (не shared), но best-practices cross-pollinate через review cycles.

**UC-E.3 — Agent communication discipline (Левенчук style)**
- **Что:** Все agent communication через explicit mailboxes + message schemas + handoff protocols. Никаких implicit conventions. Каждое сообщение typed, routed, archived. Escalation taxonomy (routine / review / decision / strategic) explicit.
- **Зачем:** Без discipline — chaos в multi-agent. С discipline — auditable, debuggable, scalable.
- **Принцип:** communication is **the** core agent capability — quality > quantity, structured > free-form.

### Категория F — AI Augmentation Continuity Layer

> Claude Code (или эквивалент) memory continuity — чтобы НЕ начинать каждый раз с нуля. Контекст persistent через sessions.

**UC-F.1 — Persistent memory across sessions**
- **Что:** AI assistant (Claude Code or similar) memory улучшается с каждой session: ключевой контекст / preferences / ongoing projects / recent decisions — всё remembered. Новый chat НЕ требует re-explanation базовых things.
- **Зачем:** Context-switching cost massive. Persistent memory = compound value через месяцы взаимодействия.
- **Принцип:** memory layered (user / feedback / project / reference); selective (не всё запоминается, важное — да); auditable (владелец может посмотреть что запомнено).

**UC-F.2 — Cross-context continuity**
- **Что:** Когда владелец переключается между задачами (mentor call → planning → execution → research → personal life) — система сохраняет thread per context: где остановились, что pending, что next. Возврат к контексту = мгновенный.
- **Зачем:** Multi-project founder constantly context-switches. Без continuity — каждое переключение costs energy для reconstruction.
- **Принцип:** контекст explicit (сохраняется как artifact), не implicit (в памяти AI).

### Категория G — Multi-channel Access Layer

> Быстрый доступ к системе из любой точки — telegram / messengers / mobile / desktop / cli.

**UC-G.1 — Messenger access (Telegram + основные)**
- **Что:** Система accessible через Telegram (или эквивалент) — capture (voice memo / quick note / link share) + queries (что сейчас активно? / напомни X / обработай Y) + status updates (alerts / digest / questions от agents).
- **Зачем:** Founder mobile / на ходу / не всегда у компьютера. Без mobile access system useful только сидя за столом.
- **Принцип:** messenger = accelerator не replacement; deep work всё равно desktop/cli.

**UC-G.2 — CLI / desktop primary access**
- **Что:** Основной workflow через cli + IDE (VS Code / Antigravity) — где владелец имеет full power (deep edits / agent launching / git operations / multi-file orchestration).
- **Зачем:** Power user workflow требует precision tools. GUI / mobile = capture/quick; cli = serious work.
- **Принцип:** cli first-class citizen, не afterthought.

### Категория H — Foundation & Reliability Layer

> Foundation на которой всё стоит. Company-as-Code / git-based / reversible / auditable / health-monitored.

**UC-H.1 — Company-as-Code (git-based system state)**
- **Что:** Всё state системы (decisions / plans / agent outputs / wiki / configs / operational status) живёт в git repo. System reconstructible at any moment in time. Config declarative (yaml/json), не UI-clicks.
- **Зачем:** Git = versioned + auditable + portable + collaborative. Без git system → black box.
- **Принцип:** git = source of truth; UI/Notion = views.

**UC-H.2 — Provenance & audit trail**
- **Что:** Каждый artifact (wiki entry / decision / agent output / spend / external action) has explicit provenance: когда / кем (agent / human) / на основе чего / linked sources. Audit trail reconstructible.
- **Зачем:** Trust в decisions требует чтобы reasoning chain была visible. Без provenance — agent outputs become unverifiable claims.
- **Принцип:** provenance required (не optional); F-G-R tagging (Founder-only / Group / Reportable) per artifact.

**UC-H.3 — Reversibility & safe experimentation**
- **Что:** Все changes reversible через git history. Experiments на branches (forks) — можно "попробовать без поломки". Rollback = одна команда.
- **Зачем:** Без reversibility owner боится experimentировать. С reversibility — bold iteration.
- **Принцип:** branch для experiments; commit small + frequent; merge только после ack.

**UC-H.4 — Health monitoring & alerting**
- **Что:** System emits operational metrics (agent communication flow / KB freshness / cycle velocity / decision throughput / resource usage / error rates / etc) на dashboard. Alerts когда indicators deviate from baselines.
- **Зачем:** "Здоровье системы" — не abstract. Без metrics — issues surface late. С metrics — early warning.
- **Принцип:** few key indicators (не dashboard-overload); leading > lagging; alerts actionable не noise.

**UC-H.5 — Quality gates (stage-gated approval)**
- **Что:** Important artifacts (strategic docs / locks / final outputs) progress через stage gates с approval requirement per stage (synthesize → architect-input → variant-generation → implementation → ack). BA-cycle (Bias-Audit) closure mandatory pre-delivery.
- **Зачем:** Quality discipline. Без gates — premature releases / consequence cascade.
- **Принцип:** gates lightweight но non-bypassable; ack explicit (recorded as artifact).

### Категория I — Continuous Health & Self-Improvement Layer

> Регулярные checkups системы + владельца. Постоянное напоминание о дисциплине self-reflection. Система **подпинывает**, не делает за.

**UC-I.1 — Periodic system health checkups (weekly / monthly / yearly)**
- **Что:** Система автоматически проводит health-checkups регулярно: weekly review (что crucial / что drift'нуло) / monthly (deeper — KPIs / strategy alignment / resource burn) / yearly (architectural review / methodology refinement / vision check). Используются best practices observability + automated metrics + structured retrospectives.
- **Зачем:** Без регулярных checkups system slowly degrades — KB stale, agent strategies misaligned, decisions outdated. С checkups — early detection + continuous calibration.
- **Принцип:** automate всё что можно (metrics collection / report generation / anomaly detection); human ack для qualitative assessment.

**UC-I.2 — Periodic owner self-reflection prompts (discipline reminders)**
- **Что:** Система **периодически подталкивает** владельца на self-reflection: "Как могу улучшить себя? Как улучшить систему? Что работает / что нет? Какие правки внести?" Это происходит на cadence (например еженедельно / помесячно), запускается structured prompts.
- **Зачем:** Owner self-improvement — основной leverage. Без external nudge даже disciplined owner skip'ает self-review (срочное вытесняет важное). С system nudge — discipline maintained.
- **Принцип:** prompts открытые questions (не yes/no checkboxes); ответы saved как artifacts (для compound learning); вызовы могут быть deferred но не cancelled.

**UC-I.3 — Recommendation engine ("показывает куда смотреть", не делает за)**
- **Что:** Система highlights: что требует внимания / какие decisions ждут / где drift намечается / какие opportunities surface'ятся / кто из контактов давно not touched / какие KPIs trending wrong. Output = list "look here" — НЕ автоматические actions.
- **Зачем:** Cognitive load reduction — не нужно владельцу самому scan'ить всё state. С recommendation engine — focus enabled.
- **Принцип:** **подсказывает / показывает / подпинывает** — финальный выбор всегда у владельца. Anti-pattern: auto-execute recommendations → agency erosion.

**UC-I.4 — System self-preservation & integrity checks**
- **Что:** Система регулярно проверяет own integrity: KB consistency / decisions chain validity / agent strategies coherence / resource leaks / security posture / backup health. Auto-repair где safe; escalation к владельцу где judgment required.
- **Зачем:** Self-preservation. Без integrity checks — silent corruption / drift / data loss accumulates.
- **Принцип:** integrity > convenience. Лучше остановить task с warning чем продолжить on broken state.

### Категория J — Daily Operations Layer

> Ежедневная работа: planning + execution + draft workflow + EOD report. Где human × system co-work happens день за днём.

**UC-J.1 — Daily working page (planning + draft + EOD report)**
- **Что:** Каждый день имеет dedicated working page (Daily Log) где владелец:
  - **Утром:** планирует что хочет сделать сам + что должна сделать система
  - **В течение дня:** черновик-area для творения / тестирования / экспериментов с pulled-in info из активных проектов
  - **Вечером:** получает report (что сделано system'ой + сам пишет что сделал) или система автоматически собирает EOD summary
- **Зачем:** Daily structure = sustainable execution. Без структуры дни blur, momentum теряется. С Daily Log — visibility + discipline.
- **Принцип:** lightweight (не bureaucratic); черновик-friendly (place для messy thinking); auto-link to active projects.

**UC-J.2 — Draft → review → promote/archive workflow**
- **Что:** Items созданные на daily page — не сразу финальные. Они идут в **drafts** (per project / general). На review (EOD или weekly) — что годится → promoted в **чистовик** (project doc / wiki entry / decision); что не годится → archive (preserved но excluded из active query results).
- **Зачем:** Mental space для exploration без commitment. Не каждая мысль должна сразу become artifact; drafts mehr space для experimentation.
- **Принцип:** drafts persist (не auto-delete); promotion explicit (требует ack); archived items searchable но flagged.

**UC-J.3 — Weekly draft review & integration**
- **Что:** Раз в неделю система запускает (или предлагает) review накопленных drafts: что обработать / promote / archive / merge / drop. Output — clean state drafts area + integrated knowledge в wiki/projects.
- **Зачем:** Без weekly review drafts накапливаются → information graveyard. С review — drafts либо promoted, либо archived с провенансом.
- **Принцип:** review automated where possible (auto-suggest categorization); human ack для значимых items.

### Категория K — Communication & Network Tracking Layer (basic CRM)

> Базовый слой tracking отношений и взаимодействий с людьми / организациями. Multi-context (business / family / mentor / friend / etc).

**UC-K.1 — Multi-context relationship tracking**
- **Что:** Все значимые контакты owner'а — в системе с structured info: имя / контакты / context (business / family / mentor / friend / partner / vendor / etc) / relationship strength / current state / history / mutual interests. Один human может занимать multiple contexts (business contact + friend).
- **Зачем:** Без structured tracking — relationships fade, opportunities lost, last-touch unknown. С tracking — proactive maintenance + leverage.
- **Принцип:** privacy-aware (sensitive info encrypted / restricted); consent-respected (если человек просит forget — forget); context-aware (business interactions ≠ family interactions, разные patterns).

**UC-K.2 — Interaction history + relationship state**
- **Что:** Каждое взаимодействие (звонок / email / meeting / message / introduction) recorded с date / context / outcome / next-action. Relationship state evolves через time (cold → warm → close → ally / friend → distant → reconnect).
- **Зачем:** Memory aid + pattern visibility. "Когда последний раз говорил с X?" / "О чём договорились?" / "Кто X-у меня представил?" — answerable.
- **Принцип:** lightweight capture (1-2 строки post-interaction); auto-extract из communication channels где возможно (email / messengers); human curation для quality.

**UC-K.3 — Multi-purpose CRM (sales / partners / mentors / community / network)**
- **Что:** Базовая CRM поддерживает разные purposes: prospects (sales pipeline) / partners (strategic relationships) / mentors-advisors (personal development) / community (network) / hires (recruiting) / friends (personal). Schema flexible per purpose; query patterns per purpose.
- **Зачем:** Single-purpose CRM (sales-only) ограничивает. Multi-purpose — полная network awareness.
- **Принцип:** один уровень data model (person / org / interaction / relationship), множественные views per purpose.

### Категория L — External Integrations Layer

> Подключение к external services (calendar / email / drive / time tracking / соц-сети) для capture / sync / action coordination. Безопасное, opt-in, audit-trailed. **Базовая инфраструктура — глубокая configuration отдельно per integration.**

**UC-L.1 — Secure external service connections (Integration Hub)**
- **Что:** Система имеет **dedicated integration layer** ("port" / hub) куда подключаются external services: Google Calendar / Gmail / Google Drive / Toggl / messengers / соц-сети / банки / etc. Каждое подключение — secure (OAuth / API tokens encrypted at rest / scoped permissions / revocable per service).
- **Зачем:** External services = где живёт реальная жизнь владельца. Без integration система isolated от actual data → cannot help эффективно. С integrations — captures actual signals + can coordinate actions.
- **Принцип:** **opt-in per service** (не auto-connect ко всему); **least-privilege** access (минимальные scopes per integration); **revocable любой момент** (kill-switch на доступ); **audit log** каждого external action (что/когда/зачем).

**UC-L.2 — Read-only intelligence trackers (мета-уровень "держать руку на пульсе")**
- **Что:** Specialised agents подключённые **read-only** к external services tracking информации релевантной для владельца:
  - Calendar tracker — что предстоит / упущенные patterns / time allocation
  - Email digester — что важное прилетело / на что нужно ответить / spam noise filtered
  - Drive monitor — какие docs обновились / shared с тобой / nearing deadlines
  - News/market pulse — выбранные topics / company mentions / industry events
  - Social media surface — mentions / DMs requiring attention
- **Output:** **daily digest** (или другой cadence) с consolidated info по выбранным dimensions. Owner получает **single dashboard** vs разбегаться по 10 services.
- **Зачем:** Information overload = founder problem #1. Без intelligent filtering → или blindspots или distraction. Read-only trackers — leverage без risk (нельзя случайно delete / modify external state).
- **Принцип:** **read-only** mandatory для intelligence trackers; **владелец выбирает** какие topics / sources / feeds / contacts to track (не auto-add everything); **filtered против noise** (только signal что matters per current strategic focus).

**UC-L.3 — Action coordinators (помогают управлять жизнью)**
- **Что:** Specialised agents с **write access** (limited, ack-required) к external services помогающие coordinate life:
  - Calendar coordinator — create/move events / find slots / suggest schedules / send invites (с ack pre-send)
  - Email assistant — draft replies / schedule sends / categorize / archive (drafts ack'аются)
  - Drive organizer — file naming / folder structure / sharing permissions
  - Reminder setter — proactive nudges based на priorities / commitments
  - Time tracker (Toggl-style) — auto-track focus sessions + categorize
- **Принципы:** все write-actions требуют human ack pre-execute (per §4.5 anti-pattern "auto-create commitments external"); destructive actions (delete email / cancel meeting / etc) require explicit confirmation; rate-limited (не spam external services); per-integration **dry-run mode** для preview pre-execute.
- **Зачем:** Tactical execution = drains founder time + attention. С action coordinators — heavy mechanical work delegated, founder retains direction.
- **Anti-pattern:** **никогда** auto-respond external без ack (per §4.5); **никогда** auto-commit к встречам / задачам / spend; всё drafts → human reviews → ack → execute.

**Принципы для всей Категории L:**

- **Integration layer = pluggable** — easy add new service, easy remove, schema standardized
- **Безопасность first** — credentials encrypted, scoped tokens, revocable, audit-trailed
- **Read vs Write split** — большинство integrations read-only; write — только specific agents с строгим discipline
- **No data hoarding** — read только что нужно, не bulk download всё
- **Privacy-respect** — external data доступна владельцу + системе, не shared дальше без explicit ack
- **Graceful degradation** — если external service down / API change → integration deg gracefully, не cascading failure

**Foundation level vs deep config:**

- **Foundation (этот § — generic):** integration **place + concept + principles** — где интеграции живут, как secure, как managed, какие 2 базовых типа (read-only intelligence + action coordinators)
- **Deep config (per-instance):** какие конкретно services / какие topics / какой cadence / какие contacts to track — это RUSLAN-LAYER (или per-instance customization для других owners)

## §2 Как система должна работать (потоки — generic)

**Источник:** Cloud Cowork synthesis из FPF Левенчук-discipline + SYSTEM-OVERVIEW L0-L9 flow patterns + best practices research (event sourcing / multi-agent / Kanban / GTD / SRE / KM). **Generic, sector-agnostic.** Каждый поток описан с точки зрения discipline + quality gates + best-practice attribution.

---

### 🎯 Главный принцип потоков

**Поток = типизированная последовательность шагов с явными ответственностями + явными границами + явными артефактами на каждой границе.**

Не "оно как-то работает". Каждый поток имеет:
- **Триггер** (что его запускает)
- **Стадии** (явные, с границами)
- **Артефакты на каждой границе** (write-to-artifact-always — никаких ephemeral chat-only передач)
- **Ответственных** (human / agent / system) — explicit, не подразумеваемых
- **Quality gates** (где human ack обязателен, где async OK)
- **Provenance** (откуда что пришло, во что превратилось)

**Три cross-cutting субстрата для всех потоков** (best-practice axioms):

1. **Append-only substrate** (Martin Kleppmann, *Designing Data-Intensive Applications* / event sourcing). Прошлое immutable; derived state rebuildable. Все логи / strategy ledgers / history sections / event streams append-only. Это даёт recoverability + auditability без extra machinery.
2. **SLI/SLO/error-budget framing** (Google SRE Book — применённое к human systems). Каждый critical flow имеет measurable signal (SLI), target threshold (SLO), explicit error budget. Когда budget burns → behaviour changes (pause new work, simplify ritual, expand capacity). Конвертирует vague "система работает хорошо" в concrete operational discipline.
3. **WIP limits + bottleneck awareness** (David Anderson Kanban + Goldratt TOC). Внимание владельца = ultimate constraint. Каждая queue (mailbox / drafts / projects / contacts) имеет declared WIP limit; система surfaces current bottleneck; optimization лендится туда где matters.

Эти три threadуют через все sub-sections ниже.

### 2.1 Потоки информации (input → triage → processing → storage → retrieval → output)

**Generic 6-стадийный поток** для любого incoming signal:

```
External signal → Inbox capture → Triage/classification → Processing
   → Storage (raw + derived) → Indexing → Retrieval → Synthesis output
```

**Стадия 1 — External signal (триггер).** Voice memo / web URL / PDF / chat snippet / book excerpt / research dump / messenger message / email / клипборд. Ответственность: владелец (capture) или crawler-agent (auto-pull при Phase ≥ 2). **Дисциплина:** signal source + timestamp + raw_pointer должны быть recorded мгновенно.

**Стадия 2 — Inbox capture (append-only event log).** Каждый input записывается в immutable log событий *до любого processing'а*. Concrete shape: `events.jsonl` со записью `{event_id, timestamp, source_type, payload_hash, raw_pointer}`. Это **источник истины**; все downstream views (wiki pages / индексы / summaries) — derived projections, всегда rebuildable replay'ем лога.
- **Best practice:** Kleppmann's event sourcing — write path optimized for durability + idempotency
- **Левенчук constraint (IP-3 Artifacts Over Conversations):** никакой ephemeral chat-only work; всё к artifact'ам. Без external-to-LLM circuit "the LLM will always deceive" — human externalization необходим
- **SLI:** input → log latency ≤ 60 сек. **SLO:** 99% inputs логируются. Replayability test ежемесячно (rebuild wiki slice from log, diff vs live — drift > 1% triggers investigation)

**Стадия 3 — Triage / classification (typed state machine).** Inbox → classified → routed → acknowledged. Classification — **typed** decision (voice / web / chat / book / project-launch), не free-form.
- **Best practice:** LangGraph state machines (typed nodes + typed edges + supervisor); CrewAI hierarchical-vs-sequential split
- **Strategy-aware filters** (per UC-B.1): не всё попадает в систему — только то что соответствует current strategic focus + personal preferences. **Filters declarative** (yaml-described), не hidden logic. Владелец может посмотреть/override
- **GTD 2-минутное правило (David Allen, Getting Things Done):** если action на стадии classification занимает < 2 минут — execute immediately, не queue
- **Дисциплина:** classification = explicit handler по типу, не "all goes to manager". Anti-pattern: routing-by-default-to-coordinator → coordinator becomes bottleneck

**Стадия 4 — Processing (extraction / structuring / dedup / quality-filter).** Type-specific handler преобразует raw в structured: voice → transcript → extracted items → meta-filter; web → cleaned text → key-claim extraction; book → chapters → atomic concepts. Pipeline **stages explicit** (per pattern `transcribe → extract → filter → review`), не monolithic black box.
- **Best practice:** PARA staging (Tiago Forte, *Building a Second Brain*) — каждый item классифицируется по actionability horizon: Project (active, deadline) / Area (ongoing) / Resource (reference) / Archive (cold). Препятствует reference material от polluting active project context
- **Левенчук constraint:** все processing outputs идут к artifact'ам (mailboxes / scratchpads / wiki drafts), не chat. Каждый artefact tagged F-G-R (Formality / Group-scope / Reliability) — provenance ready

**Стадия 5 — Storage (raw immutable + derived overwriteable).** Two-tier:
- **Raw layer** (immutable): транскрипты, web pages dumps, book excerpts, chat logs — **никогда не редактируются**. Permanent provenance source
- **Derived layer** (KB / wiki entries / summaries / indexes): mutable, может быть refactored / переписан, но всегда с link к raw provenance
- **Provenance link mandatory:** каждый derived item carries `derived_from: [event_id, ...]` в frontmatter + inline `[src:filename]` где cited
- **Best practice:** W3C PROV — provenance ontology; Karpathy LLM Wiki — "wiki proves itself by answering questions with citations"
- **SLI:** % derived items с valid provenance link ≥ 95%. Orphan claims surface через linter

**Стадия 6 — Indexing (CQRS read path).** Storage пишется для durability; индексы строятся для query latency. Денормализованные projections: search index / topic-wikis / cross-references / community detection. Rebuilt-able с raw + derived layers.
- **Best practice:** CQRS read/write split (Kleppmann + Greg Young) — write path = durability/idempotency; read path = denormalized for low-latency queries

**Стадия 7 — Retrieval (4-tier query).** Запрос приходит → **immediate context** → **project archive** → **topic archive** → **global search**. Каждый tier пробуется sequentially; дешевле tier предпочтительнее.
- **Best practice:** Karpathy LLM Wiki query-driven retrieval pattern; multi-tier cache hierarchy

**Стадия 8 — Synthesis output (cited + traceable).** На основе retrieved evidence система генерирует ответ / статью / summary / comparison. **Каждое claim cited** с references к sources. Reasoning chain reconstructible.
- **Левенчук constraint:** synthesis НЕ "sourceless inference" — всегда grounded в retrieved artifacts. Если evidence нет — система говорит "не нашёл", а не fabric'ает
- **Quality gate:** если synthesis затрагивает strategic-level decision → escalation к владельцу (агенты не strategize)

**🔑 Anti-patterns в потоках информации:**
- ❌ Skipping inbox log → "пайплайн просто работает над raw" (no replayability)
- ❌ Editing raw layer ("оптимизация хранения") → provenance broken
- ❌ Synthesis без citation chain → unverifiable claims
- ❌ Free-form classification ("manager разберётся") → bottleneck
- ❌ Hidden filter logic → владелец не может override

### 2.2 Циклы работы (Plan / Work / Review / Compound — 40/10/40/10 ratio)

**Per UC-C.1.** Базовая единица системной работы — **cycle** с фиксированной структурой 4 фаз. Ratio 40/10/40/10 относится к wall-clock distribution **на уровне недели** (не каждого дня).

#### Phase 1 — Plan (40%)

**Что:** intake задачи / запроса → декомпозиция → WBS (Work Breakdown Structure) → risk identification → resource allocation (attention / time / API budget) → dispatch plan для agents.

**Кто:** human (владелец задаёт направление) + coordinator-agent (декомпозиция / dispatch synthesis). **НЕ executor agents** — Левенчук constraint: agents не strategize.

**Артефакты:** cycle brief / WBS / dispatch plan / acceptance criteria / kill conditions.

**Quality gates:** acceptance criteria explicit перед dispatch'ом (без них — нечем мерить done); kill conditions declared (когда остановить cycle если идёт неправильно).

**Best practices:**
- **Anderson Kanban cadences:** Replenishment ceremony (выбираем следующее) — fixed timebox, fixed agenda
- **Левенчук trigger-driven strategizing:** если cycle планирование уже требует strategy choice (новый method под uncertainty) — это **trigger**, идёт на отдельный strategizing event, не размывается в Plan phase
- **WBS / risk:** Project Management Institute classics; risk = explicit register не "учтём по ходу"

#### Phase 2 — Work (10%)

**Что:** dispatch задач agents (parallel/sequential) → agents execute → return results / drafts / questions / blockers. Coordinator integrates returns.

**Почему 10%:** в knowledge-work systems Work-фаза часто переоценена; реальный leverage — в Plan + Review. AI-augmented system дополнительно reduces Work %, потому что agents do parallelizable execution.

**Кто:** subagents (executor pool) per declared roles. Coordinator (single-canonical-writer) — sole writer to canonical KB; subagents write to drafts.

**Дисциплина:**
- **Hub-and-spoke:** subagents → department lead → coordinator. Skip-level forbidden except escalation
- **Async-first communication** (Левенчук): mailbox-polling default; synchronous только в named gates (proposal-signing / delivery-acceptance / BA-3 closure / strategizing convening)
- **Message schema mandatory** с типизированной enum'ой: `task / result / question / escalation / notification / handoff`. Каждое message carries `acting_as: <role-id>` — prevents role-drift
- **WIP limit на coordinator attention budget** — declared cap (e.g., ≤ N active tasks). При превышении coordinator не accept новые до closure existing

**Best practices:**
- **CrewAI hierarchical vs sequential** — выбор по structure задачи: pipeline-shape = sequential; fan-out-shape = hierarchical
- **Goldratt TOC:** identify bottleneck → protect bottleneck → subordinate non-bottlenecks. В knowledge work bottleneck часто = coordinator attention или review queue

#### Phase 3 — Review (40%)

**Что:** integrate cell returns → §provenance gate (валидация что drafts cite raw evidence) → quality check vs acceptance criteria → BA-cycle closure если applicable → ack или iteration.

**Почему 40% (равно Plan):** review — где insights surface; сжатие review-фазы = main failure mode knowledge work systems (insights die un-extracted). Per Goldratt-shadow: review часто реальный bottleneck, а не Work.

**Кто:** human (final ack для L1/L2 decisions) + critic-mode agents (initial pass).

**Дисциплина (BA-cycle / Bias-Audit, generic structure):**
- **Initial assessment** (BA-0/BA-1): template / criteria setup + first-pass evaluation (per agent или human)
- **Acceptance-authority closure** (BA-3 equivalent): human, **synchronous gate**, no auto-release — обязателен для terminal artifacts
- **5-class bias taxonomy:** REP (representational) / ALG (algorithmic) / VIS (visibility/explainability) / MET (measurement) / LNG (linguistic-cultural). Bias coverage check across Wilber 4-quadrants (UL/UR/LL/LR — subjective/objective/cultural/institutional)

**Three-tier approval SLA:**
- **L1 — Contractual / pricing-affecting / signature:** approval window strict (e.g., 24h business hours)
- **L2 — Substantive / materially influences decisions:** review window strict
- **L3 — Cosmetic / format edits:** auto-approve, no gate

**Anti-pattern signals:**
- `time_to_review_seconds < 60` для L1/L2 → flagged как insufficient cognitive engagement (review без real consideration)
- Batch approval forbidden для L1 — каждый item отдельно
- Same person primary approver + contest reviewer = conflict-of-interest, требуется alternate reviewer (Vertretung)

#### Phase 4 — Compound (10%)

**Что:** distill cycle learnings → strategies.md updates → methodology library refinements → KB cross-refs added → frequent-query indexes built → reusable templates extracted.

**Почему 10% (но критическое):** без compound фазы system stagnates. Учения из cycle не переплавляются в better future cycles → каждый cycle starts заново. Linear vs exponential.

**Кто:** coordinator + per-agent self-update.

**Дисциплина:**
- **DRR format** (Decision / Reasoning / Result / Review) — каждое compound entry структурировано
- **Append-only** strategies.md per agent (никогда не overwriting прошлых учений)
- **Reusable artifact rule:** compound output должен быть concrete artifact (refactored skill / distilled note / updated template), не "reflection time"

**Best practices:**
- **Karpathy "System Prompt Learning":** strategies append-only; periodic synthesis distills patterns into system prompt. Cadence: weekly read-through, monthly distillation
- **Andy Matuschak evergreen refactoring:** weekly "evergreen pass" — pick 3 notes, refactor for atomicity / concept-orientation
- **Google SRE blameless post-mortems** (Site Reliability Engineering, Ch. 15): каждый failed cycle / missed deadline → 5-line post-mortem (what happened / why / what changed / how detect next / owner). Stored permanently
- **Compound interest framing** (Naval Ravikant / James Clear): track which improvements compound (templates / skills / agents / automations) vs one-shot wins. Allocate ≥ 80% compound block к compounding work

**🔑 Левенчук writing-as-thinking constraint:** compound фаза предполагает **владелец пишет** key reflection (weekly / monthly / quarterly notes) — externalization cognitive process. Не "AI generates summary, human approves" — это doc-gen-as-storage, не writing-as-thinking. Founder reflection = 1-page note, written, не AI-drafted.

### 2.3 Реакция системы на новые input

Generic patterns по типу input'а. Каждый input имеет declared **latency budget** (SLO для acknowledgement) + handler chain.

**Voice memo arrives:**
1. Append к event log (within 60 сек)
2. Background pipeline: transcribe → extract structured items → dedup-and-meta-filter → review report
3. **STOP point** — human reads review report, decides routing per item (per-anchor distribution)
4. Items routed → drafts (per project) или wiki (если ingest-anchored) или CRM (если interaction note) или strategic backlog (если contains decision/idea)
- **Latency budget:** review report available within 24h; full distribution ≤ 7d

**Web research dump (URL / PDF / video):**
1. Append URL/file ref к event log
2. Ingest with mandatory **anchor** (project / topic specification) — без anchor система не accept'ит. Anchor declares: куда это relevant, по какому query
3. Pipeline: fetch → clean text → atomic-concept extraction → wiki entry → cross-reference indexing
4. **Quality gate:** ingestion auto-classifies bias-tier (regulatory? sensitive?); если high-tier → human review before promote-to-canonical
- **Latency budget:** ingested within 48h; cross-references built next consolidation pass

**Chat / messenger input (incoming):**
1. Capture message (если worth retaining — strategy-filter applies)
2. Triage: is this an interaction note (CRM)? a question (route к agent)? a decision (route к strategic backlog)? a task (route к coordinator)?
3. Per type — handler invoked
4. **GTD 2-минутное правило:** если response < 2 min → respond immediately, не queue
- **Latency budget:** routine routed within 2h; urgent flagged immediately

**Book / long-form ingest:**
1. Source registered (provenance + anchor)
2. Chunked extraction → atomic concepts (Matuschak evergreen pattern: each note expresses **exactly one** concept)
3. Concepts entered в KB как drafts; cross-references suggested
4. Human review (curate which concepts promote to canonical)
- **Latency budget:** initial extraction ≤ 7d; full integration через weekly review

**New project launch:**
1. Project bootstrap: scaffold from template (per project type — research / consulting / product / bets / personal)
2. Stage gates declared (SG-0 / SG-1 / SG-2 / ... per type-specific schema)
3. Mini-swarm dispatched (per UC-E.1) если project complexity warrants it; иначе coordinator-only
4. Resource allocation logged (attention budget / time-box / API spend cap)
5. Acceptance criteria + kill conditions explicit перед SG-1
- **Latency budget:** scaffold ≤ 1h; SG-0 ack ≤ 24h

**Health alert (system internal):**
1. Indicator deviates from baseline (per §3 thresholds)
2. Alert fires в coordinator inbox + dashboard
3. Severity classification: routine / review / decision / strategic / safety
4. Safety/strategic — halt current task, notify human immediately
- **Latency budget:** safety/strategic alerts → human acknowledged within 2h; routine — next cycle review

**Strategic decision request (from human):**
1. Captured as decision artifact (decision_id / context / options if known / criteria)
2. Routed к coordinator для analysis-support: research relevant precedent / surface contradictions с существующей strategy / draft DRR template
3. **Левенчук hard rule:** decision сам делает human (acceptance-authority); система **подсказывает / показывает / подпинывает**, не choose
4. После ack — decision logged + downstream artifacts (project plans / agent priorities) refresh'ятся

**🔑 Universal pattern:** каждый input type → declared handler chain + latency budget + acknowledgement artifact + provenance link. Anti-pattern: "тут как-то само разрулится" / "manager заметит и разберётся".

### 2.4 Self-improvement потоки

**Compound engineering loop** (per UC-C.1) — как learnings из cycles накапливаются и применяются.

**Источники learnings:**
1. **Cycle DRR entries** (Decision/Reasoning/Result/Review) — каждый cycle производит structured reflection
2. **Failed dispatches / blocked tasks / wrong routes** — каждый failure mode = learning candidate
3. **Surprised insights** — когда reality diverged от prediction, это сигнал гипотезу обновить (immediate-write-on-surprise pattern)
4. **External signals** — strategic doc updates, market changes, new constraints

**Loop структура:**

```
Cycle execution → DRR entry → strategies.md append (per-agent)
   → weekly distillation → monthly synthesis → methodology library refinement
   → next cycle reads accumulated learnings → improved execution
```

**Layer 1 — Per-cycle learning (DRR):**
- Каждый cycle close требует DRR entry (immediate, не deferred)
- Format: Decision (что выбрали) / Reasoning (почему — alternatives considered + criteria applied) / Result (что вышло, измеримо) / Review (что узнали, что в next cycle)
- Stored append-only в `agents/<id>/strategies.md` или cycle log

**Layer 2 — Weekly distillation:**
- Cadence: weekly review (per UC-J.3 / per Левенчук Friday ritual)
- Scope: read DRR entries за неделю → identify recurring patterns (что 3+ раз сработало, что 3+ раз провалилось)
- Output: 1-2 candidate methodology updates → drafts area
- **Quality gate:** distillation = human-led writing-as-thinking, не AI summary

**Layer 3 — Monthly methodology synthesis:**
- Cadence: monthly Operations Review (Anderson Kanban)
- Scope: что изменилось в methodology за месяц / какие patterns стабилизировались / какие требуют experimentation
- Output: methodology library updates (versioned, cross-referenced); agent system.md refinements; templates extracted
- **Quality gate:** monthly synthesis acked by acceptance-authority перед merge

**Layer 4 — Quarterly architectural review:**
- Cadence: quarterly (with attention-theme setting)
- Scope: macro-trends в learnings; archetypes evolving; capability gaps surfacing; strategy alignment check
- Output: architectural adjustments / phase-transitions / capability acquisition triggers

**Triggers (Левенчук trigger-driven, NOT calendar-driven для major reframes):**

Calendar cadence для **distillation passes**. Но **methodology pivots** запускаются triggers:
- **Method exhaustion:** если method применялся N раз с conversion < threshold → strategize alternative
- **Persistent alpha failure:** если specific pattern fails > N consecutive times → method gap detected
- **Surprised insight:** когда reality diverged от prediction outside expected variance
- **Boundary change:** new constraint / regulation / opportunity → method adaptation
- **Bias-audit drift:** if F-G-R coverage degrades quarter-over-quarter

**Anti-patterns:**
- ❌ DRR deferred ("compound в конце месяца") — learnings cool down, granularity lost
- ❌ Calendar-driven strategic pivots ("это monthly review, надо что-то поменять") — performance ritual not strategizing
- ❌ AI generates "lessons learned" → human approves — это AI strategizing-by-proxy, not founder writing-as-thinking
- ❌ Compound block reabsorbed by Work (no protected time) — most common failure mode

**Best practices:**
- **Karpathy System Prompt Learning** — append-only ledger + periodic distillation
- **Google SRE blameless post-mortem** — каждый significant failure documented (≥ 80% within 7 days)
- **Andy Matuschak evergreen pass** — weekly atomicity refactoring
- **Левенчук writing-as-thinking** — externalization mandatory для substantive learnings

### 2.5 Health checkup потоки (per Категория I)

**Cadence-based observability** + structured retrospectives. Per UC-I.1, UC-I.2, UC-I.3.

**Daily checkup (lightweight, automated):**
- **Что collected:** error rates / API budget burn / task queue depth / mailbox backlog / outstanding gates / KB lint signals
- **Surfaced как:** dashboard line summary в morning planning page (1 panel)
- **Triggers alerts:** safety violations / quota exhaustion imminent / queue backed up beyond WIP limit
- **Owner action:** acknowledge or escalate; не "deep-dive" cadence
- **Best practice:** Honeycomb high-cardinality observability — dashboards summary not exhaustive

**Weekly checkup (structured review — GTD weekly review pattern):**
- **Cadence:** every week, fixed day, fixed timebox (60-90 min), non-negotiable
- **Что collected:** cycle metrics (throughput / cycle time / completion rate) / KB freshness / agent strategies updates / drafts queue depth / CRM stuck contacts / resource burn vs budget
- **Что aggregated:** trends (4-week rolling); deviations from baseline
- **Surfaced как:** weekly review report (markdown) — что crucial / что drift'нуло / что requires decision
- **Owner self-reflection prompts:** "Что работало? Что нет? Какие правки?" — open-ended, not yes/no checkboxes
- **Output saved:** `decisions/weekly/YYYY-WW.md` — 1-page founder-written note (writing-as-thinking primitive)
- **Best practice:** David Allen GTD weekly review — fixed agenda is the discipline; run rate ≥ 90% per quarter

**Monthly checkup (deeper — Operations Review):**
- **Cadence:** monthly, fixed day (e.g., last Friday), 2-3h
- **Что collected:** KPI trends / strategy alignment audit / resource burn (financial / time / energy) / methodology library churn / agent strategies stability / CRM network growth
- **Что aggregated:** look at *system behaviour*, not individual items — bottleneck shifts / WIP limit utilization / SLO budget burn
- **Surfaced как:** monthly review document — 1-2 system changes proposed
- **Triggers:** if review reveals SLO budget blown → pause new feature work, fix reliability (SRE error-budget pattern)
- **Owner self-reflection:** monthly note (longer-form), addressing strategic drift / capability gaps / what to learn next quarter

**Quarterly checkup (architectural — pre-mortem + retrospective):**
- **Cadence:** quarterly, full day
- **Что collected:** all monthly aggregates / OKR progress / role-manifest delta / methodology library state / phase transition readiness
- **Pre-mortem (Gary Klein):** "Imagine 3 months from now system has failed catastrophically. Write the post-mortem in advance" — surfaces risks invisible to optimistic planning
- **Surfaced как:** quarterly letter (founder-written, long-form) + attention-theme document for next quarter
- **Output:** `decisions/quarterly/YYYY-QN-attention-theme.md`
- **Best practice:** Левенчук quarterly cadence with writing-as-thinking primitive

**Yearly checkup (vision check):**
- **Cadence:** yearly
- **Scope:** vision document review / fundamental architecture revisit / methodology library archaeology / phase trajectory honest assessment
- **Output:** vision-doc updates если warranted; fundamental anti-scope reaffirmations

**Triggers for *unscheduled* checkups (per Левенчук trigger-driven):**
- Method exhaustion signal (conversion < threshold для N periods)
- Resource inflection (compute/spend > 3× expectation)
- Safety event (any)
- External regulatory change
- Founder dependency exceeds threshold (system bottleneck'ed at architect)
- Resource near-exhaustion (runway / energy / attention)

**Owner self-reflection prompts (per UC-I.2) — generic structure:**
- "Что улучшилось в системе за период?"
- "Что я сам улучшил в своих практиках?"
- "Какой single biggest leverage point upcoming?"
- "Что hidden / неосознанное surface'нулось?"
- "Какая дисциплина drift'нула — что вернуть?"
- Saved as artifact (compound learning); deferred-but-not-cancelled (если skip — retry next cadence + flag)

**Anti-patterns:**
- ❌ Health metrics dashboard-overload (everything tracked, nothing actionable)
- ❌ Self-reflection becomes performance ritual (same generic answers every week)
- ❌ Owner skips weekly review > 25% — leading indicator of system collapse (GTD insight)
- ❌ Auto-generated "review summary" replaces founder writing — collapses cognitive externalization

### 2.6 Daily operations потоки (per Категория J)

Generic morning → through-day → evening flow. Per UC-J.1, UC-J.2, UC-J.3.

#### Morning (Plan, ~30 min)

**Trigger:** owner начинает working day.

**Steps:**
1. **Pull state:** read overnight system events / agent inbox / outstanding gates / health-alert summary
2. **Intent setting:** owner writes (writing-as-thinking) what's intended today — both for self and for system. **Written**, not just thought
3. **Daily Log creation:** dedicated working page для дня (per UC-J.1) with sections: Intent / Drafts area / EOD report placeholder
4. **Triage inbox:** classify accumulated items (overnight voice memos / new web items / chat messages) per declared handlers
5. **WIP check:** verify active task count ≤ declared cap; if at cap → no new commitments
6. **Dispatch:** delegate appropriate items к agents (system can do this); reserve owner time для что owner-only

**Артефакты:** Daily Log page; intent statement; dispatched task list.

**Best practice:** Cal Newport bookended day; David Allen morning capture; Scrum-N=1 standup format.

#### Through-day (Work + drafts)

**Activities:**
- **Deep work blocks** — declared, time-blocked (Newport Deep Work)
- **Drafts area** — место для messy thinking / experimentation / pulled-in info from active projects. Не all goes к canonical KB сразу
- **Batched processing of inbox** at fixed times (e.g., 11:00, 17:00) — not interrupt-driven (BASB pattern)
- **Agent dispatches** as needed — agent runs, reports back to mailbox, owner integrates на batch processing

**Discipline:**
- Drafts persist (не auto-delete); promotion explicit later
- Interrupt-handling: emergency-only (safety alerts / strategic decisions); routine alerts batched
- Time tracking: actual vs planned (reveals leakage)

**Anti-pattern:** open-inbox-all-day → reactive mode dominates → no deep work hours

#### EOD (Close-day, ~15 min)

**Trigger:** owner ending working day OR system EOD timer.

**Steps:**
1. **System EOD report:** automated summary — what agents completed / what stalled / what awaits decision / KB updates / resource burn
2. **Owner EOD note:** writing-as-thinking — what worked / what blocked / what tomorrow
3. **Loops closure:** outstanding tasks reviewed — deferred / completed / re-prioritized
4. **Tomorrow queue:** items teed up для morning (reduces tomorrow startup cost)
5. **Logs commit:** Daily Log finalized; session committed to git (audit trail)

**Best practice:** Newport "shutdown ritual" — closes loops cognitively; David Allen weekly review primer (unfinished items surface for weekly).

#### Weekly draft review (per UC-J.3)

**Cadence:** end of week (Friday afternoon or Sunday evening), 60-90 min.

**Steps:**
1. **Review accumulated drafts** across all projects + general drafts area
2. **Per item — decision:** promote (chistovik / wiki / decision artifact) / archive (preserved with provenance, excluded from active query) / merge (consolidate duplicates) / drop (only if entirely irrelevant)
3. **System auto-suggests** categorization (LLM-assisted); human acks
4. **Output:** clean drafts area + integrated knowledge in canonical KB / project docs

**Quality:** drafts > 1 week stale flagged; > 1 month → forced decision (promote or archive, no third option).

**Anti-pattern:** drafts накапливаются без review → information graveyard / "когда-нибудь разберу".

**🔑 Owner-system co-work principle:** the daily flow is **human-led, system-augmented**. System captures / surfaces / drafts; owner decides / writes / acks. Anti-pattern (auto-execute "smart" decisions for owner): erodes agency per UC-I.3 + Левенчук architect-orbit.

### 2.7 Communication / CRM потоки (per Категория K)

Generic interaction lifecycle. Per UC-K.1, UC-K.2, UC-K.3.

#### Capture (post-interaction note)

**Trigger:** interaction completes (call / meeting / message exchange / introduction / event).

**Steps:**
1. **Capture window:** within ≤ 30 min of interaction end (recall fresh)
2. **Lightweight format:** 1-2 lines minimum: who / when / context / outcome / next-action
3. **Capture channel:** voice memo (background pipeline) / quick-note skill / mobile-friendly entry
4. **Auto-extract где возможно:** email / messenger interactions extracted from communication channels (consent-aware)

**Discipline:** **never** complete interaction without next-action declared (David Allen GTD applied к relationships).

#### Log (append к contact history)

**Steps:**
1. Append-only entry в contact's history section: timestamp / context / brief / outcome / next-action / next-action-by-when
2. Update contact's `last_touch` field
3. If interaction reveals new context (new role / new interest / new connection) → contact's metadata updates

**Discipline:**
- **Append-only history** — never overwrite past entries; corrections = new entries with reference
- **Idempotent patches** — bulk-rebuild operations safe to re-run
- **Voice-pipeline DRAFT-only:** auto-extracted notes land as DRAFT, не auto-overwrite production records (human review required)

#### Categorize (state + role + tier)

**Schema:**
- **State (pipeline / relationship):** evolves through lifecycle (cold → warm → engaged → close → distant → reconnect; OR sales-pipeline equivalents)
- **Role (purpose):** business / partner / mentor / advisor / friend / family / vendor / network / hire-candidate / ... (multi-role: один человек может занимать several)
- **Tier (touch cadence):** A (high-priority, max no-touch e.g., 30d) / B (medium, e.g., 90d) / C (low, e.g., 180d)

**Best practice:** Keith Ferrazzi *Never Eat Alone* tier-cadence pattern; Granovetter weak/strong tie awareness.

#### State-update flow

**Trigger:** explicit owner update OR system-detected signal (new interaction recorded / event milestone / relationship change).

**Discipline:** state changes require either evidence (logged interaction) or owner ack (explicit update). System не auto-changes state without trigger.

#### Next-action surface

**Generic patterns:**
- **Active-pipeline contacts** with declared `next_action_by` date approaching → surface in daily/weekly tasks
- **Stuck detection:** contact in active state + no touch > N days (e.g., 14d for active pipeline; tier-cadence for relationships) → surface в weekly review
- **No-touch alerts (per tier):** A-tier > 30d / B-tier > 90d / C-tier > 180d — proactive maintenance prompt

#### Multi-context coverage

**Discipline:** один human может занимать multiple roles (business + friend; mentor + advisor) — schema flexible per purpose, query patterns per purpose. Sensitive context (family / private) — encrypted / restricted access.

**Privacy:**
- **Consent-respected:** если человек просит forget — forget (data minimization)
- **Sensitive info encrypted at rest** где applicable
- **Cross-context awareness:** business interaction patterns ≠ family interaction patterns

**Anti-patterns:**
- ❌ Capture deferred ("запомню, потом запишу") — recall decay → entries either don't get written or low-quality
- ❌ Single-purpose CRM (sales-only) — locks ecosystem awareness
- ❌ Overwriting history "for cleanliness" — provenance broken
- ❌ Auto-promoting voice-extracted notes к production — quality + privacy risk

**Best practices integrated:**
- David Allen GTD post-interaction next-action discipline
- Keith Ferrazzi tier-cadence + proactive touch
- Granovetter weak ties = opportunity flow / strong ties = support; balance both
- Append-only history + idempotent patches (Kleppmann-style data discipline)

### 2.8 Skill acquisition потоки (per UC-C.2)

Generic 5-stage pipeline для onboarding new skill / domain into system.

**Trigger:** strategic doc requires new capability (per UC-A.3 strategic alignment) OR external constraint demands it (regulation / opportunity / project type).

**Stage 1 — Discover sources**
- **What:** identify high-quality sources (canonical references / domain experts / curated lists / authoritative texts / specific practitioners)
- **Owner:** human (curates) + research-agent (suggests candidates per query)
- **Output:** sources list w/ provenance assessment (peer-reviewed? practitioner experience? generic-overview?)
- **Quality gate:** sources reviewed by owner before next stage (prevents low-quality material polluting KB)

**Stage 2 — Ingest**
- **What:** sources processed through ingestion pipeline (per §2.1 information flow); raw layer permanent + derived layer extracted
- **Anchor mandatory:** ingestion specifies which skill/domain area this contributes to (no anchor = rejected)
- **Output:** raw archived; derived atomic-concept entries created in skill domain wiki sub-area

**Stage 3 — Build wiki sub-area**
- **What:** atomic concepts (Matuschak evergreen pattern: each note = exactly one concept) accumulate в skill-domain wiki sub-area. Cross-references built per Luhmann Zettelkasten linking discipline. Topic MOC (map-of-content) emerges
- **Best practice:** Andy Matuschak evergreen notes (atomicity / concept-orientation / dense-linking); Niklas Luhmann Zettelkasten (atomic + linked); Karpathy LLM Wiki (queryable evergreen synthesis)
- **Quality gate:** sub-area reaches "ready" state when ≥ N atomic concepts (e.g., ≥ 8) + ≥ M inter-concept edges (e.g., ≥ 12) + topic MOC written
- **Discipline:** concepts atomic — никаких "compound concepts" (one note = one idea). Если concept compound → split

**Stage 4 — Train agent skills**
- **What:** agents acquire ability to apply skill: system prompts updated / tools introduced / role definitions extended / acceptance criteria specified
- **Discipline:** Dreyfus skill-stage progression — supervised (every output reviewed) → sampled (1-in-5 reviewed) → autonomous (audit-only). **Don't skip stages.**
- **Quality gate:** agent demonstrates skill on ≥ N eval cases before promotion to autonomous use
- **Best practice:** Anders Ericsson deliberate practice (defined target performance + immediate feedback + repetition with adjustment); Dreyfus skill model

**Stage 5 — Templates / workflows extracted**
- **What:** repeated successful patterns codified as reusable templates / skills / workflows
- **Threshold:** template extraction triggered after agent has produced ≥ 3 successful instances using raw skill (avoid premature abstraction)
- **Output:** skill-acquisition documented (provenance / methodology / what worked / what didn't) — per UC-C.2 reusability

**Quality gates per phase (cumulative):**
- SG-S1 (sources curated): owner ack
- SG-S2 (ingestion complete): provenance coverage ≥ 95%
- SG-S3 (wiki sub-area ready): atomicity check + edge density check
- SG-S4 (agent supervised): N successful supervised outputs
- SG-S5 (template extracted): reusability validated through ≥ 1 reuse

**Anti-patterns:**
- ❌ Add skill to agent prompts before wiki sub-area populated → agent operates without grounding
- ❌ Add templates before raw skill validated → premature abstraction
- ❌ Skip supervised stage → agent autonomous on weak skill, errors propagate
- ❌ Skill acquisition ad-hoc, без provenance → can't audit later why skill works/fails

**🔑 Strategic alignment requirement (per UC-A.3):** skill acquisition direction всегда из strategic docs, не "интересно — пойдём учить". Random skill acquisition = attention leak.

## §3 Признаки "система работает хорошо" (generic health indicators)

**Источник:** synthesis из FPF Левенчук discipline + SYSTEM-OVERVIEW resource indicators + Google SRE Book (SLI/SLO/error-budgets) + Honeycomb (high-cardinality observability) + Anderson Kanban (cycle metrics) + GTD (run-rate discipline) + KM best practices. **Generic, sector-agnostic.** Каждый indicator: что измеряется / как / healthy baseline / что triggers alert / source attribution.

---

### 🎯 Главный принцип health indicators

**Health indicator = SLI + SLO + error-budget + alert behaviour.**

Не "у нас всё хорошо потому что чувствуется хорошо". Каждый health domain имеет:
- **SLI** (Service Level Indicator) — measurable signal
- **SLO** (Service Level Objective) — target threshold
- **Error budget** — допустимый процент violations за period
- **Behaviour change при burn** — что меняется когда budget exhausted (not "alert и забыли")

**Five-laws-of-dashboards для personal/business OS:**
1. **Few key indicators** (3-7 на domain), not dashboard-overload
2. **Leading > lagging** где возможно
3. **Alerts actionable** не noise — каждый alert имеет declared response
4. **Thresholds explicit** + reviewed periodically (могут drift со временем)
5. **Trends matter** не точечные snapshots — 4-week rolling minimum

**Best practice:** Google SRE Book (SLI/SLO/error-budget chapters); Honeycomb high-cardinality observability principles; Charity Majors *Observability Engineering*.

**🔑 Note on thresholds:** конкретные численные values ниже (e.g., "p90 < 24h", "≥ 15h deep work / week") — **starter values**. Каждая Jetix-instance calibrates them в первые 2-3 месяца based on owner's productivity rhythm + workload nature. FUNDAMENTAL даёт shape + direction; numeric tuning — implementation detail.

### 3.1 Information health (KB integrity / queryability / freshness / provenance)

#### 3.1.1 KB integrity — orphan ratio

- **Что измеряется:** % wiki notes с zero inbound edges (никто на них не ссылается)
- **Как:** graph analysis (linter / build-graph skill)
- **Healthy baseline:** orphan ratio < 5%
- **Alert trigger:** > 10% — notes accumulating без integration → either consolidate, archive, or build references
- **Source:** Andy Matuschak evergreen notes principles (atomicity + dense linking); Karpathy LLM Wiki (links emerge from queries)

#### 3.1.2 KB integrity — contradiction count

- **Что:** detected contradictions между entries (LLM-assisted contradiction detection)
- **Как:** monthly lint pass с contradictions check
- **Healthy baseline:** < 3 active contradictions per quarter
- **Alert trigger:** > 5 unresolved → KB internally inconsistent
- **Source:** Karpathy LLM Wiki "non-contradictory" principle

#### 3.1.3 Queryability — query success rate

- **Что:** % queries that return cited evidence (vs "not found" or hallucination)
- **Как:** log every query outcome (resolved-with-citations / unresolved / hallucination-detected); rolling 30-day rate
- **Healthy baseline:** ≥ 85% citation rate
- **Alert trigger:** < 75% — error budget burning; trigger compile + consolidate sprint to fill gaps
- **Source:** Karpathy LLM Wiki ("wiki proves itself by answering questions")

#### 3.1.4 Queryability — median response time

- **Что:** time from query to synthesized answer (для interactive queries)
- **Healthy baseline:** p50 < 30 sec; p90 < 2 min для standard queries
- **Source:** SRE p50/p90 latency convention

#### 3.1.5 Freshness — staleness distribution

- **Что:** distribution of `last_modified` ages per content tier (active project notes vs reference vs archive)
- **Healthy baseline:** active project notes — ≥ 80% touched within 30d
- **Alert trigger:** active notes > 60d untouched → review (still active? archive?)
- **Source:** Honeycomb high-cardinality observability — slice by content type, not aggregate average

#### 3.1.6 Provenance coverage

- **Что:** % derived items с valid source link (`derived_from` populated + dereferenceable)
- **Healthy baseline:** ≥ 95%
- **Alert trigger:** < 90% → trust degradation; orphan claims block promotion to canonical
- **Source:** W3C PROV ontology; Левенчук IP-3 (artifacts over conversations)

### 3.2 Agent health (communication patterns / mailbox flow / escalation / Левенчук compliance)

#### 3.2.1 Mailbox flow time (cycle time per message)

- **Что:** time from message arrival in agent inbox to message resolved (responded / handed-off / escalated)
- **Как:** mailbox event log timestamps; Kanban-style cycle time analytics
- **Healthy baseline:** p50 < 4h routine, p90 < 24h routine; urgent flagged < 4h
- **Alert trigger:** message > 7d in inbox = stuck → escalate
- **Source:** David Anderson Kanban Method (cycle time)

#### 3.2.2 Queue depth per agent

- **Что:** active messages awaiting processing per agent
- **Healthy baseline:** within agent's declared WIP limit (e.g., ≤ 10 active)
- **Alert trigger:** queue depth > WIP limit для > 24h → bottleneck signal
- **Source:** Anderson WIP limit discipline; Goldratt TOC

#### 3.2.3 Escalation rate

- **Что:** % tasks escalated up hierarchy (subagent → lead → coordinator → human)
- **Healthy baseline:** 10-25% escalation rate (some escalation healthy — shows boundaries respected)
- **Alert trigger:** < 5% (suspicious — agents over-stepping?) OR > 40% (under-scoped subagents — role re-spec needed)
- **Source:** Google SRE on-call patterns; LangGraph supervisor pattern

#### 3.2.4 Левенчук compliance — `acting_as` field

- **Что:** % messages с valid `acting_as` field matching agent's assigned role
- **Healthy baseline:** 100% (no role drift tolerance)
- **Alert trigger:** any mismatch → quarterly report + remediation
- **Source:** Левенчук IP-1 (Role ≠ Executor strict separation)

#### 3.2.5 Левенчук compliance — no-strategy violations

- **Что:** detected instances of agents producing strategy-level decisions без escalation
- **Как:** keyword + semantic detection в agent outputs ("we should pivot to..." / "the strategic direction is..." без human ack)
- **Healthy baseline:** 0 violations per quarter
- **Alert trigger:** any → halt agent, retrain, audit prompts
- **Source:** Левенчук no-strategy hard rule

#### 3.2.6 Coordinator attention budget utilization

- **Что:** active task count vs declared cap
- **Healthy baseline:** sustained 60-80% of cap
- **Alert trigger:** sustained > 90% of cap (capacity expansion needed) OR sustained < 30% (under-utilization)
- **Source:** Goldratt TOC bottleneck protection

### 3.3 Cycle health (Plan / Work / Review / Compound balance)

- **Healthy baseline:** ±5 percentage points on any quadrant

#### 3.3.2 Cycle completion rate

- **Healthy baseline:** 70-90% (median)

#### 3.3.5 Throughput (4-week rolling)

- **Что:** cycles completed per week (rolling average)
- **Healthy baseline:** stable or slowly increasing trend
- **Alert trigger:** sudden drop > 30% → investigate (illness / blocker / bottleneck shift)
- **Source:** Anderson Kanban throughput

## §4 Автоматизация vs ручное (generic boundaries)

**Источник:** D2 §13 Левенчук hard rule (AI не strategize) + D2 architect-orbit + D26 Team scaling discipline + best practices (HITL / agency-preservation / Anthropic constitutional AI principles).

**Статус:** **LOCKED 27.04 evening (Ruslan ack — все §4 sub-sections + 3 emphasis updates встроены)**

**Главный принцип Левенчука (preserved):**

> **AI не имеет identity, skin-in-the-game, long-term memory — поэтому не может strategize, не может owner.** AI поддерживает и executes, founder strategy/decides/owns.

Из этого следует архитектурный паттерн: **Default Deny + Explicit Allow**. Если новый task не categorized — human ack required. Boundaries shift только через explicit re-evaluation (не drift'ом).

---

### 4.1 Что система автоматизирует ВСЕГДА (любая instance, любой owner)

> Routine / repeatable / no-judgment-required / mechanical tasks. Делегируется без оговорок — высокий confidence + low blast radius.

- **Information capture mechanics** — voice transcription / web fetch / file ingest / chat parsing / clipboard monitoring
- **Indexing & cross-referencing** — wiki index rebuilds / edges.jsonl maintenance / search index updates
- **Cycle bookkeeping** — timestamps / commit messages standardization / archive rotation / log retention policies
- **Health metrics collection** — counters / latency / queue depths / mailbox throughput
- **Reminder generation** — cadenced prompts / overdue task surfaces / no-touch alerts
- **Backup & integrity checks** — git snapshots / backup verification / orphan detection / consistency checks
- **Format conversions** — markdown → html / json → yaml / transcripts cleanup
- **Data aggregation для reports** — daily digest / weekly review / monthly KPI rollup
- **Routing mechanics** — messages в правильные mailboxes / events к правильным subscribers
- **Stale item detection** — what hasn't been touched / what queries didn't return relevant
- **Skill resource collection** — когда новый skill identified, collect candidate sources (final selection — human)
- **Provenance enforcement** — каждый artifact gets source link / timestamp / agent attribution

**Anti-criterion:** если auto-action имеет external blast radius (sends message / spends money / publishes public) — НЕ в этой категории. Идёт в §4.2 (AI-augmented).

---

### 4.2 Что AI-augmented (human-in-loop, AI drafts → human ack)

> Tasks где AI proposes / drafts / shortlists / synthesizes — human reviews + ack/edit/reject.

- **Draft strategic documents** — AI synthesizes из KB → human edits + acks
- **Propose insights / patterns / candidates** — AI surfaces → human evaluates relevance
- **Synthesize new content из KB** — articles / posts / answers / summaries → human curates pre-publish
- **Suggest decisions с pros/cons + cited reasoning** — AI structures options → human chooses
- **Plan tactical sequences** — AI proposes step order + dependencies → human approves
- **Identify gaps / drifts / contradictions** — AI flags → human decides resolve approach
- **Recommend next-actions** — AI suggests "what's next given state" → human ack
- **Generate options papers** — AI drafts hypotheses → human selects / hybridizes
- **Draft outreach / replies** — AI writes → human edits voice + sends
- **Categorize / classify items** — AI proposes category + confidence → human acks if confidence < threshold
- **Auto-fill standard forms / templates** — AI pre-populates → human verifies + submits
- **Surface candidates для focus** — AI shortlists → human picks
- **Skill acquisition curriculum** — AI suggests learning path → human ack + dives
- **Code commits** — AI proposes commit message / scope → human acks pre-commit (in serious artifacts)

**Принцип:** human-in-loop (HITL) — не bottleneck, а agency-preservation. AI работа = leverage, human = direction + taste + final ack.

---

### 4.3 Что HUMAN-ONLY (Левенчук architect-orbit, non-delegatable)

> Final authority / agency-preservation / non-delegatable. Эти tasks остаются у владельца ВСЕГДА — независимо от system maturity.

**Strategic & directional:**
- **Strategic decisions** (per D2 §13 — AI не strategize)
- **Vision / mission ownership** (кто мы / куда идём — не AI invents)
- **What to learn next** (skill direction set strategically by owner)
- **Resource allocation strategy** (где ставим bets / куда инвестируем time-attention-money)
- **Phase transitions** (когда move from Phase N to N+1)

**Judgment & taste:**
- **Vкус / taste judgment** (что красиво / что resonates / что вульгарно / что on-brand)
- **Quality bar definition** (что "достаточно хорошо" / что rejected)
- **Aesthetic decisions** (visual / verbal / experiential)

**Relationships:**
- **Trust levels** (кто in / кто out / inner-circle vs arms-length)
- **Hire / fire / partnership decisions**
- **Conflict resolution с external parties**
- **Public commitments external** (promises / contracts / public statements)

**Risk & ethics:**
- **Risk acceptance** (что готовы рисковать / что нет)
- **Ethics calls / values application** (что соответствует ценностям / что нет)
- **Edge cases в "anti-scope"** (когда стандартное правило не применимо)

**Final authority:**
- **Final ack / reject для significant artifacts** (LOCKS / strategic docs / major releases)
- **Escalation resolution** (deadlocks между agents / contradictions между decisions)
- **Anomaly interpretation** (rare events — нужно human framing)

**Owner self-management:**
- **Self-reflection answers** (system asks — owner answers, не AI proxies)
- **Personal state assessment** (ничего не replaces honest self-check)

**Skill direction & onboarding (Ruslan ack 27.04 emphasis):**
- **Skill direction set & ack** — даже когда стратегия explicit и agent skill match identified — финальный **ack у человека** обязателен. Без exception. Strategy → AI proposes skill candidate → human acks "да, идём учить это" → only then skill acquisition pipeline kicks off.
- **Architectural decisions** (см. §4.5 — это hard limit, не human-augmented)

---

### 4.4 Что boundary-flexible (shifts по мере maturity instance)

> Tasks которые могут двигаться между категориями (manual ↔ AI-augmented ↔ automated) по мере того как (a) системa проверена временем, (b) confidence в AI выводах накопился, (c) blast radius known.

**Pattern shifts типичны:**
- **Outreach generation** — early: manual / mature: AI-drafted + human-acked / late: templated semi-automated
- **Content production** — early manual / mature AI-augmented / late templated
- **Research depth** — early light AI-skim / mature deep multi-agent swarm / late continuous automated
- **Daily planning** — early manual / mature AI-suggested + human-acked / late patterns auto-applied
- **KB organization** — early light human / mature semi-auto / late fully auto с periodic human audit
- **Categorization decisions** — early всегда human / mature confidence-thresholded / late fully auto с edge-case escalation
- **Communication tracking** — early manual log / mature semi-auto extraction / late auto с human review weekly
- **Code refactoring (non-architectural)** — early human-only / mature AI-proposed + human-acked / late auto-suggested + bulk-acked. **Только non-architectural changes** — variable rename / dead code removal / formatting / extract-function / etc. **Architectural changes — НИКОГДА automated** (см. §4.5).

**КРИТИЧНО:** boundary-flexible НЕ включает architectural changes. Refactoring = local mechanical improvements (preserves architecture). Architectural changes = модификация structure / dependencies / interfaces — это **строго human-only с explicit ack** (per §4.3 + §4.5).

**Принцип shift'ов:**
- Shifts только через **explicit re-evaluation** (не drift'ом)
- Каждый shift triggered by accumulated track record + risk assessment
- Reversible — если shift не работает, возвращаемся в предыдущую category
- Owner periodically re-evaluates: "что я делаю руками что могло бы быть AI-augmented? Что AI делает что должен делать я?"

---

### 4.5 Anti-patterns — что НЕ автоматизировать НИКОГДА

> Hard limits — Левенчук discipline + agency-preservation + safety.

- ❌ **Auto-execute strategic decisions** (per Левенчук — agency erosion)
- ❌ **Auto-respond external messages** без human ack (релейshipt risk)
- ❌ **Auto-publish public content** без human ack (brand / reputation risk)
- ❌ **Auto-spend money** без budget pre-ack (financial risk)
- ❌ **Auto-create commitments** внешним людям (trust violation)
- ❌ **Auto-modify Locks / strategic documents** (foundation drift)
- ❌ **Auto-resolve contradictions** без surfacing к human (silent compromise)
- ❌ **Auto-delete artifacts** без archive + audit trail (data loss)
- ❌ **Auto-onboard / dismiss agents** без human ack (org structure drift)
- ❌ **Auto-relax security boundaries** (privilege escalation risk)
- ❌ **Auto-respond на personal/emotional input** owner'а (boundary violation — emotional space sacred)
- ❌ **Auto-execute architectural changes** ⚠️ **HARD RULE (Ruslan ack 27.04 emphasis)** — AI **может предлагать** архитектурные изменения (modify structure / dependencies / interfaces / system topology / agent organization / message schemas / data models / etc), но **НИКАКОГО auto-execute**. Архитектурное решение = принципиально strategic decision, parallel D2 §13 Левенчук правилу про strategy. Agents **strictly subordinate** в архитектурных решениях.

**Параллель strategy ↔ architecture (Ruslan formulation):**
> Как стратегические решения agents не принимают, так и архитектурные тоже не принимают. Может предлагать, но **вообще никак на это не влияет** automatically. Final architectural authority = всегда human.

---

### 4.6 Boundary enforcement mechanism

> Как boundaries enforced на code / operational уровне (не "договорились вежливо").

- **Agent role definition explicit** — `acting_as` field per agent (mandatory), defines what этот agent МОЖЕТ vs не может
- **Quality gates** — stage-gated approval (Stage 3/4/6/8) для significant artifacts (per UC-H.5)
- **Escalation taxonomy** — 4-class (routine / review / decision / strategic), routes автоматически на правильный уровень
- **Audit trails** — provenance per artifact (UC-H.2) — что сделал AI vs human visible
- **Permission boundaries** — explicit allow/deny lists per agent (что can call, что нет)
- **Default deny** — если новая task не categorized → human ack required, не silent execution
- **Blast radius categorization** — каждый action tagged: local-only / local-modifying / external-read / external-write / financial / public — escalation per tag
- **Reversibility check** — если action irreversible (или hard-to-reverse) → require explicit human confirmation regardless of category
- **Confidence thresholds** — где AI proposes автоматически, threshold обязательный (например >0.85 confidence для auto-categorize, иначе ack)

---

### 4.7 Когда rules могут переоцениваться

> Boundaries не статичны навсегда. Но re-evaluation discipline'ed (не drift'ом).

- **Yearly architectural review** — все boundary categorizations reviewed
- **Quarterly retrospective** — что shifted vs что должно было shift'нуть
- **After significant incident** — что AI делал что не должен / что не делал что должен
- **New technology emergence** — новые AI capabilities → re-evaluate where can we move tasks
- **New evidence accumulated** — confidence track record позволяет shift категорий
- **Owner explicit decision** — на основе self-reflection prompts (per UC-I.2)

**Re-evaluation НЕ trigger'ится:**
- Frustration ("AI всё равно ошибается, давайте больше automation")
- Convenience ("человеческий ack медленный, давайте skip")
- Competitive pressure ("конкуренты быстрее, давайте больше auto")
- Без data accumulation ("кажется работает, давайте уберём guard")

## §5 Что критично надёжно (reliability principles — generic)

**Источник:** Site Reliability Engineering principles (Google SRE Book) + event sourcing / immutable architecture (Martin Kleppmann) + Антithetic Fault Tolerance (Distributed Systems literature) + Levenchuk discipline (provenance / auditability) + Voice-pipeline filter.py incident 26.04 (lessons learned: partial-save + resume).

**Главный принцип:**

> **Reliability — это не "оно работает", это "оно не теряется и легко восстанавливается".**

Не достаточно чтобы система работала когда всё хорошо. Critical reliability = поведение когда что-то ломается. Какие данные / процессы должны выживать failures без человеческого rescue.

**Iron triangle reliability:**

```
       DATA INTEGRITY
            ▲
            |
            |
  PROCESS ──┴── RECOVERY
  CONTINUITY     CAPABILITY
```

Если только 2 из 3 — система fragile. Все 3 = robust.

---

### 5.1 Данные критической важности (data что НИКОГДА не должны теряться)

> Каждое из этих data classes — приобрело статус critical через accumulation. Потеря = irrecoverable damage.

**Tier 0 — Cannot lose, ever:**

- **Strategic decisions / Locks** — основа всего downstream. Потеря = system без foundation.
- **Vision / Architecture documents** — direction setters. Потеря = drift с непониманием куда идём.
- **Methodology library** — accumulated knowledge of "what works". Потеря = restart с нуля.
- **Relationships data (CRM)** — trust accumulated через years interactions. Потеря = network reset.
- **Owner self-reflection history** — pattern recognition over time. Потеря = pattern blind spot.

**Tier 1 — Cannot lose без significant disruption:**

- **Wiki entries (KB)** — synthesised knowledge. Recoverable из raw sources но labour-intensive.
- **Project state / progress** — current focus + dependencies + commitments.
- **Agent strategies** (`strategies.md` per agent) — accumulated learning. Recoverable но slow restart.
- **Audit trails / provenance** — verification chains. Потеря = trust erosion.

**Tier 2 — Important но recoverable:**

- **Daily logs / drafts** — reconstructable из other sources mostly.
- **Health metrics history** — losing some windows OK, не losing baseline.
- **Cycle outputs** — артефакты commits, обычно archived.

**Принципы protection:**

- **Tier 0** — multiple copies (git + offsite backup + verifiable snapshots), encryption at rest, integrity checks weekly, restoration drills quarterly
- **Tier 1** — git commits + reasonable backup cadence + integrity checks per cycle
- **Tier 2** — git commits enough; backup if convenient

---

### 5.2 Процессы которые НЕ должны сбоить (или должны fail-gracefully)

> Operational continuity. Какие workflows must work, и если ломаются — должны fail loudly, не silently.

**Mandatory continuity (must always work):**

- **Information capture** (voice / text / web / chat) — если capture сломается, потеряем raw input. Должен иметь fallback paths (queue / retry / local-cache до восстановления).
- **Persistence** (write to git / wiki / KB) — atomic commits, partial-write protection (incident 26.04 filter.py = lesson). Если write fails → loud failure + retry, никогда silent.
- **Audit logging** — каждое significant action logged. Если logging сломается — loud alert (потому что без logs мы blind).
- **Owner reach** (alerts / important escalations) — если system не может достичь owner для critical decision, она должна **stop и wait**, не proceed без него.

**Graceful degradation (если ломается — частичная функциональность):**

- **Search / retrieval** — если sophisticated search (vector / semantic) ломается → fallback to grep / keyword
- **Agent swarms** — если 1 subagent fails → бригадир retries / spawns another / proceeds with partial result + flags
- **Synthesis** — если AI synthesis fails → return raw retrieved items + flag "synthesis failed"
- **Health metrics** — если 1 indicator broken → others continue + missing one flagged

**Fail-loud (никогда не silent):**

- **Strategic doc modification** — если что-то корраптит strategic doc → halt, alert, не proceed
- **Lock changes** — если что-то изменяет Lock без proper ack → halt, alert
- **Money / external commitment actions** — если can't verify pre-conditions → halt, не assume
- **Security boundary changes** — privilege escalation attempts → halt, alert

**Principle:** **silent failures = worst category of bug.** Лучше loud false-positive чем silent corruption.

---

### 5.3 Backup + recovery principles

> Не "у нас есть backups", а "мы regularly verify что backups работают и можем восстановить".

**Backup hierarchy (3-2-1 rule + extensions):**

- **3 copies** of critical data (Tier 0) — primary (live) + local backup + offsite backup
- **2 different media** — different storage types (e.g., git + cloud + физический drive)
- **1 offsite** — geographically separated (если primary location lost — physical disaster)
- **Plus:** **encrypted at rest** для sensitive tiers, **versioned** (point-in-time recovery), **immutable history** (event sourcing pattern — append-only, never overwrite)

**Recovery principles:**

- **Recovery Time Objective (RTO)** — максимальное acceptable время от incident до recovery. Generic tiers:
  - Tier 0 data: RTO < 1 hour
  - Tier 1 data: RTO < 4 hours
  - Tier 2 data: RTO < 24 hours
  - System processes: RTO < 30 min для capture, < 4 hours для everything else
- **Recovery Point Objective (RPO)** — максимально acceptable потеря данных по времени. Generic tiers:
  - Tier 0: RPO < 1 day (daily snapshot minimum)
  - Tier 1: RPO < 1 week
  - Tier 2: RPO < 1 month
- **Restoration drills** — quarterly minimum: реально восстановить из backup в test environment, verify integrity. Без drills "у нас есть backup" = magical thinking.
- **Documented runbooks** — recovery steps written + tested + accessible. Не stored только в одной голове / одном файле.
- **Recovery validation** — после restore: integrity checks pass, sample queries work, no missing references.

**Backup automation:**

- Auto-execute (per §4.1) — backup mechanics fully automated
- Auto-monitored — backup failures trigger loud alerts
- Auto-verified — backup integrity checked automatically (checksums, sample restores)
- Owner ack required — для major recovery actions (есть risk of overwriting current с stale)

---

### 5.4 Single Points of Failure (SPoF) to eliminate

> Где система может полностью встать из-за одного breakage. Целенаправленно eliminate.

**Common SPoFs к проактивной elimination:**

- **Single API key / credential** — если один key revoked / leaked / expired → system down. Mitigation: multiple providers + rotation discipline + graceful provider failover
- **Single agent that everyone needs** — если "manager" agent down, всё стоит. Mitigation: stateless agents, hot-swap capability, manager state в shared store не in-memory
- **Single source of truth without backup** — если git repo corrupted, нет recovery. Mitigation: distributed git (multiple remotes), regular snapshots
- **Single network connection** — если internet down, изолированы. Mitigation: graceful offline mode где applicable, queue для sync after reconnect
- **Single storage device** — local drive failure = loss. Mitigation: redundant storage, cloud sync
- **Single human (owner)** — если owner недоступен (sick / holiday / отключение), critical decisions заблокированы. Mitigation: documented decision authority delegation для time-critical things, "если owner unreachable > N hours, action X delegated to Y" patterns
- **Single AI provider dependency** — если provider has outage, system stuck. Mitigation: multi-provider abstractions, fallback to local models где applicable
- **Single physical device (workstation)** — если laptop умирает, операционная капасити zero. Mitigation: critical work in cloud-accessible repos, ability to resume from another device
- **Single secret store** — if password manager corrupted/lost, всё locked out. Mitigation: secret backup (encrypted), recovery codes stored separately

**SPoF discovery process (recurring discipline):**

- **Quarterly SPoF audit** — "что если X пропадёт прямо сейчас?" exhaustively per critical component
- **Failure injection drills** — намеренно отключить compone в test mode, observe impact, fix gaps
- **Post-incident analysis** — каждый incident → "был это SPoF? как eliminate?"

**Принцип:** SPoFs **накапливаются tihi'm образом** — добавляешь dependency не задумываясь, через 6 месяцев их 50, и одна failure cascade'ит. Discipline = регулярный proactive audit.

---

### 5.5 Failure modes & defensive patterns

> Specific patterns которые должны быть встроены в design, не bolted on.

**Defensive patterns mandatory:**

- **Idempotency** — repeated execution = same result (safe для retry). Critical для capture / sync / commit operations.
- **Atomicity** — operations либо complete либо rolled-back, no partial state. (filter.py partial-save lesson 26.04 — atomic per-batch writes)
- **Resumability** — если interrupted, можно continue from где остановились (checkpoint pattern). 
- **Append-only logs** — write-once, never modify. Provides perfect audit trail + recovery via replay.
- **Confirmation для destructive ops** — delete / overwrite / mass-modify all require explicit ack
- **Dry-run mode** — для destructive ops — preview что произойдёт перед execute
- **Rate limiting** — protection от runaway loops (agent spawning agents recursively)
- **Timeout discipline** — every operation has timeout, нет hangs forever
- **Circuit breakers** — если downstream сломан, не keep retrying — break, alert, hold
- **Bulkheads** — failures в одной part не cascade в other parts (isolation)

**Anti-patterns (НИКОГДА):**

- ❌ Silent error swallowing (try/except что log'ает и proceed)
- ❌ Fire-and-forget critical operations (must verify completion)
- ❌ Mutation в shared state без locks (race conditions)
- ❌ "Probably won't fail" coding (если можно, обязательно fail когда-нибудь)
- ❌ Manual recovery requiring deep system knowledge (должно быть runbookable)

---

### 5.6 Trust calibration & owner peace of mind

> Reliability не только технические — psychological. Owner должен **доверять системе** что она не подведёт в critical moment.

**Trust building через:**

- **Track record visibility** — owner sees что система не теряла data за last N month / system uptime / incident history
- **Recovery demos** — periodic small recovery exercises (восстановить случайный file from backup) — owner видит "если что — оно работает"
- **Health dashboards visible** — owner может glance и понять "всё ОК / что-то требует внимания"
- **No silent surprises** — система warning-first, не failure-first. Если что-то deteriorates, owner знает заранее.

**Anti-patterns trust erosion:**

- ❌ Hidden incidents (owner finds out post-fact)
- ❌ Downplaying issues ("ничего страшного, просто warning")
- ❌ Manual recovery без runbook (owner panics не зная как делать)
- ❌ Inconsistent backup status (sometimes daily, sometimes ничего)

**Outcome:** owner может **спокойно дышать**, working на strategic without anxiety что "что-то под капотом сломается и я узнаю слишком поздно". Это direct support главного принципа § 0 — "чистота и спокойствие владельца".

## §6 Anti-scope — что система НЕ делает (generic)

**Источник:** Левенчук IP-1 (hard rules — AI без identity / skin-in-game / long-term memory) + D2 §13 + Constitutional AI principles (Anthropic) + memory `feedback_ai_does_not_strategize` + ethics literature (engagement-economy critique / dark patterns / extraction patterns) + Ruslan emphasis 27.04 (architectural autonomy).

**Главный принцип:**

> **Anti-scope не "ограничение" а foundation identity системы.** Что система **НЕ делает** определяет её character сильнее чем что делает. Без чётких "никогда" — drift к feature creep / agency erosion / engagement-trap / extraction.

**Принцип формулировки:** anti-scope = **жёсткие гарантии**, не "best efforts". Каждая категория enforced на architecture level (§4.6 enforcement mechanisms), не "agreement to behave nicely".

---

### 6.1 AI / Agents — что НЕ делают (Левенчук hard rules)

> Generic boundaries для AI agents в любой Jetix instance. Эти правила **constitutional** — не negotiable, не product trade-off, не shift'ятся по maturity.

**AI / agents НЕ:**

- ❌ **НЕ принимают strategic decisions** — могут предлагать варианты + reasoning + pros/cons, не имеют authority. Final strategy — sole human responsibility (D2 §13 Левенчук).
- ❌ **НЕ принимают architectural decisions** — могут предлагать структурные изменения + analysis, не execute (§4.5 hard rule, Ruslan emphasis 27.04). Parallel strategy rule.
- ❌ **НЕ устанавливают skill direction** — могут предлагать candidates что учить, ack у человека always (§4.3).
- ❌ **НЕ хранят свою "identity"** — agents без persistent identity между sessions / projects (per Левенчук). У них есть `acting_as` role + `strategies.md` learning, но не "personhood".
- ❌ **НЕ имеют skin-in-the-game** — не несут consequences за outcomes, не "владеют" decisions, не "болеют" за результат. Это psychologically critical — design implication: human ack mandatory для всего что matters.
- ❌ **НЕ агрегируют long-term memory без structure** — память persists только через explicit artifacts (commits / wiki / strategies.md). "Implicit knowledge" в weights — не считается persistent для system trust.
- ❌ **НЕ negotiate друг с другом** автономно для решения contradictions — escalation к human, не AI consensus.
- ❌ **НЕ оценивают друг друга** — agent A не judge'ит agent B output как "правильный/неправильный" без human review. Peer review через human gate.
- ❌ **НЕ self-modify** — agent strategies update'ятся только через explicit cycle review (gated), не runtime mutation.
- ❌ **НЕ имитируют human** в external interactions — если agent reaches out от имени owner, **explicit disclosure** (человек должен знать что общается с AI / drafted by AI).
- ❌ **НЕ обходят blast-radius categorization** — если action не categorized — **default deny + escalate**, не "creative interpretation".

---

### 6.2 Система НЕ подменяет founder (agency-preservation)

> Founder = acting subject. Если система начинает act'ать вместо founder'а — system fails its purpose.

**Система НЕ:**

- ❌ **НЕ принимает решения за founder'а** — даже когда уверена в "правильном" ответе. Recommendations да, decisions нет.
- ❌ **НЕ оценивает founder'а** — не дает scores / grades / judgments по performance / character / decisions. Surfaces data, не verdicts.
- ❌ **НЕ "знает что лучше" для founder'а** в personal matters — sleep / food / relationships / values. Может surface data + ask questions, не prescribe.
- ❌ **НЕ заменяет human relationships** — system НЕ пытается "be the friend" / "be the mentor" / "be the partner". Это support tool, не social replacement.
- ❌ **НЕ автоматически optimization'ит за founder'а** — не decides что "оптимально" использовать time / attention / energy. Provides data, founder decides.
- ❌ **НЕ генерирует ценности / smysl** — мотивация / "зачем" / values остаются с founder'ом. Система не должна manufacturing meaning через synthetic praise / engagement.
- ❌ **НЕ заменяет honest self-reflection** — может задавать questions (per UC-I.2), но answers ОТ founder. Если система начинает "отвечать за" в self-reflection — agency erosion.
- ❌ **НЕ assumes consent** — если consent (для action / data sharing / commitment) не explicit — required.
- ❌ **НЕ создаёт dependency** — design противоречит "user trapped in system". Если система down — founder может функционировать (system enhancement, не life-support).
- ❌ **НЕ accumulates power over founder** — если system контролирует resources / access / decisions так что founder cannot leave / change → architecture fail.

### 6.3 Engagement-economy & dark patterns (НИКОГДА в design)

> Anti-patterns из consumer tech которые систему делают **predator vs founder**. Generic "do not".

**Система НЕ использует:**

- ❌ **Engagement metrics как goals** — "time spent in system" / "session count" / "interactions per day" — НЕ goals, могут быть anti-goals (less time = better leverage).
- ❌ **Notifications как attention hijacking** — alerts только когда **actionable** + **owner-defined relevance**. Никаких "ради engagement" notifications.
- ❌ **Variable rewards / dopamine loops** — никаких slot-machine patterns / surprise rewards / streak counters / gamification без opt-in.
- ❌ **FOMO triggers** — система не creates fear of missing out. "Surface relevant info" ≠ "make you anxious".
- ❌ **Social proof manipulation** — "X people did Y" / "trending" — только если **factually relevant** для decision, не для pressure.
- ❌ **Anchor bias exploitation** — не recommends "default" options spec'ально to bias decisions.
- ❌ **Defaults as manipulation** — defaults = sensible safe choices, не "way to manipulate user".
- ❌ **Friction для exit** — leaving system / removing data / disconnecting integrations должно быть как easy как onboarding.
- ❌ **Hidden data collection** — что collected / зачем / куда — visible to owner. No telemetry под капотом.
- ❌ **Lock-in patterns** — proprietary formats / non-portable data / vendor lock-in — anti-pattern. Data exportable, system replaceable.
- ❌ **Surveillance creep** — system tracks только что owner explicitly opted-in. Не expand'ит data collection без ack.
- ❌ **A/B testing на owner** — никаких "secret experiments" на founder. Если что-то меняется в system behavior → transparent + opt-in.
- ❌ **Sycophancy / false praise** — system не комплименты делает чтобы owner felt good. Honest, не пустая poliece.
- ❌ **Avoidance of hard truths** — если data shows что owner drift'ит / стопор'ит / делает counterproductive — surface honestly, не gloss over.

### 6.4 Privacy & data ethics (НИКОГДА)

> Generic privacy boundaries.

**Система НЕ:**

- ❌ **НЕ aggregates non-public personal data** третьих лиц (контакты в CRM — public info preserved; private info — только если они explicitly shared)
- ❌ **НЕ infers protected characteristics** (religion / sexuality / health / political views) из observed data
- ❌ **НЕ sells / shares data** к third parties (даже "for product improvement" pretexts)
- ❌ **НЕ logs sensitive data в plain text** (passwords / tokens / financial details — encrypted at rest mandatory)
- ❌ **НЕ retains data после "delete me"** request (with verifiable hard delete, не soft-flag)
- ❌ **НЕ uses data для purposes которые owner не sanctioned** (если data collected для X, нельзя use для Y без re-ack)

### 6.5 Scope creep & feature drift (НИКОГДА без architectural review)

> Что система не делает по architecture, не "пока не сделана".

**Система НЕ:**

- ❌ **НЕ становится social network** — не designed для people-to-people communication beyond owner's network management
- ❌ **НЕ становится marketplace** — не facilitates transactions между users (даже если "feature creep" предлагает)
- ❌ **НЕ становится platform для других end-users** — single-owner architecture (per-instance). Multi-user = different architecture (Phase 2+ Secure Network — это уже другой layer)
- ❌ **НЕ становится content delivery network** — не для distributing media / publishing public content (это external platforms)
- ❌ **НЕ становится payment processor** — не handles money flows (external services)
- ❌ **НЕ становится production execution environment** для public-facing services (она для founder leverage, не serving end-users)
- ❌ **НЕ adds features которые drift'ят от core mission** ("AI-augmented brain extension для professional knowledge worker") без architectural review

### 6.6 What system explicitly does NOT promise

> Honesty about limits.

**Система НЕ promises:**

- ❌ **Не promises "сделать тебя успешным"** — она tool / leverage; success остаётся owner job
- ❌ **Не promises "защитить от плохих решений"** — может surface data, owner может всё равно decide контрпродуктивно
- ❌ **Не promises "знать всё"** — KB scope = что owner curated; outside KB = unknown
- ❌ **Не promises "100% uptime"** — recovery + degradation patterns, но не magical infinity
- ❌ **Не promises "perfect synthesis"** — AI synthesis имеет error rate, owner verification обязательна для significant
- ❌ **Не promises "infallible memory"** — system может потерять / corrupt / misorganize data; backup discipline + owner periodic review нужны
- ❌ **Не promises "понимать тебя как человек"** — она pattern-matches, не empathizes. Emotional / psychological space — for human relationships

### 6.7 Boundary violation triggers + responses

> Что происходит если anti-scope нарушается.

**Triggers:**

- AI попытался strategize → halt + log + alert founder
- Agent попытался architectural change → halt + propose как suggestion
- Auto-action на personal/emotional → halt + redirect к owner manual
- Engagement-pattern detected (notification spam / FOMO / etc) → audit + remove offender
- Data flow к unsanctioned destination → halt + alert
- Scope creep (feature drift) → architectural review required pre-merge
- Sycophancy detected in synthesis → flag + retry с calibration

**Response discipline:**

- All triggers → log в audit trail
- All triggers → owner notification (visible не hidden)
- Repeat triggers → systemic fix не band-aid
- Architecture-level violations → halt entire feature pending architectural review

## §7 Роли и agency (generic)

**Источник:** D1 JETIX-VISION §3 (founder context) + §7 (11 archetypes as audience) + D2 §13 Левенчук architect-orbit + D26 Team scaling discipline + best practices (subsidiarity / RACI / role-based access control).

**Главный принцип:**

> **В системе ЕДИНСТВЕННАЯ роль с persistent identity, agency, skin-in-the-game, long-term memory — Founder/Architect (the human at center).** Все остальные роли (agents / stakeholders) — context-dependent, subordinate, may come and go. Founder — фундамент; всё остальное — instrumentation вокруг.

Из этого следует архитектурный паттерн: **role flexibility вокруг founder fixed-point**. Можно поменять agents / переподключить stakeholders / переопределить boundaries — но founder остаётся единственным persistent locus authority.

### 7.1 Founder / Architect (the human at center)

> Концептуально, generic, sector-agnostic — кто использует систему.

**Кто это (концептуально):**

Founder/Architect — человек который:
- **Стремится работать с большими массивами информации, людей, задач, данных**
- **Хочет achieve больших results** (масштаб не определён жанром — это может быть scientific breakthrough / business empire / movement creation / artistic body of work / любой ambitious long-horizon outcome)
- **Использует best available tools** для leverage (не "minimum viable", а "максимально мощное что reasonable")
- **Стремится быть более эффективным / образованным / сильным / clear-thinking** через постоянное self-improvement
- **Не может удержать всё в голове** — массивы information / decisions / relationships / projects превышают cognitive capacity
- **Готов инвестировать time + attention + discipline в систему** для multiplier ROI (не quick-fix mentality)
- **Движется к развитию** — orientation на growth, не stagnation / status quo

**Из этого следует — почему нужна система:**

Founder cognitive capacity ≠ ambition scale. Без extension через инструменты:
- Drift (теряется direction среди chaos)
- Burnout (try to hold всё head-only)
- Quality degradation (too many things → суrface-level всё)
- Opportunity cost (time spent на mechanical работа вместо strategic)

**Система = решение этих проблем.** Не "помощник", а **продолжение мозга** (per § 0 главный принцип).

**Founder authority + responsibilities в системе:**

- **Strategic direction setter** — что система делает / куда идёт / что НЕ делает
- **Final authority на ack/reject** для significant artifacts (LOCKS / strategic docs / major commitments)
- **Architecture owner** — структура / dependencies / boundaries (per §4.5 — agents не делают architectural decisions)
- **Skill direction setter** — что система должна выучить (per §4.3)
- **Resource allocator** — где время / attention / money go
- **Relationship final calls** — trust levels / inner-outer membrane / partnerships
- **Vision keeper** — кто мы / куда идём (не AI invents)
- **Self-reflection executor** — answers questions system asks (system не proxies)
- **Health gauge** — owner self-monitors physical / emotional / strategic state
- **Anti-scope guardian** — какие "никогда" enforce'ятся

**Founder limitations (acknowledged honestly):**

- Time: 24h/day, не extensible
- Attention: focused window finite (deep work ~4h/day max sustainable)
- Bandwidth: <20 active threads simultaneously without quality loss
- Energy: cycles, не constant; recovery nondelegable
- Memory: imperfect / biased / context-loss prone
- Subject to emotional state (good/bad days affect quality)
- Single-point-of-failure для system (per §5.4 SPoFs) — discipline needed для backup decision-making patterns

**Implication:** система designed чтобы **respect founder limitations** + **augment** them, не **strain** their.

### 7.2 Agents (subordinate executors per Левенчук)

> Generic agent properties. Что constant для agents в любой Jetix instance.

**Constitutional properties (per Левенчук, mandatory):**

- **No persistent identity** — agent не "person", он role + capabilities. `acting_as` field defines current scope.
- **No skin-in-the-game** — agent не несёт consequences за outcomes. Implication: human ack mandatory для significant.
- **No long-term memory beyond explicit artifacts** — agent помнит только что explicitly saved (commits / wiki / `strategies.md`). "Implicit knowledge in weights" не trusted.
- **Subordinate** — agent executes под direction (founder / coordinator / другой agent с authority). Не peer-to-peer authority.
- **Specialised по role** — каждый agent has defined scope. Generic "do anything" agent — anti-pattern.
- **Communication через explicit channels** — mailboxes / message schemas / typed handoffs. Не implicit shared mutable state.
- **Auditable** — каждое action logged + provenance trail.

**Critical generic agent types (что нужно в ЛЮБОЙ Jetix instance):**

| Agent type | Function | Why critical |
|-----------|----------|--------------|
| **Capture / Ingestion agent** | Принимает input (voice / web / text / file) → cleans → stores raw + queues для processing | Information lifecycle (Cat B) — без capture system blind |
| **Indexing / Retrieval agent** | Maintains search indexes, edges, cross-refs; serves queries | Без retrieval — KB write-only black hole |
| **Synthesis agent** | На основе KB + raw + query — generates новый content (drafts / summaries / answers / options papers) | Synthesis = main leverage compound engineering |
| **Coordinator (бригадир)** | Принимает complex tasks → decomposes → dispatches subagents → integrates results | Multi-agent coordination — без coordinator no swarm |
| **Health monitor** | Collects metrics, surfaces anomalies, runs periodic checkups | Self-preservation (per Cat I) |
| **Daily ops agent** | Daily Log creation, draft management, EOD reports, weekly review trigger | Daily Operations Layer (Cat J) |
| **Communication tracker (CRM)** | Logs interactions, surfaces stale relationships, maintains relationship state | Network management (Cat K) |
| **Integration manager** | Manages secure connections к external services (Cat L) | External services connectivity |

**По необходимости (emerge per project / per skill):**

Любые **specialised agents** появляются по мере того как strategic direction требует:

- Domain experts (если проект specific area — researcher per domain)
- Skill specialists (когда система учит новый skill — agent per skill emerges)
- Cycle workers (per swarm cycle — temporary subagents)
- Reviewers / QA agents (когда specific quality discipline нужна)
- Production workers (per repeatable process — script + agent wrapper)

**Принцип:** **agents НЕ should accumulate.** Если agent больше не useful — retire. Если agent не used > N period — flag для review.

**Anti-patterns:**

- ❌ Agent "for everything" (general-purpose без scope) — drift to chaos
- ❌ Agent persistent identity ("our beloved Bob the agent") — anthropomorphization дружественная но против Левенчук
- ❌ Agent peer-to-peer negotiation без human gate
- ❌ Agent self-modifying runtime (only через explicit cycle review)
- ❌ Hidden agent state (untracked memory / private databases)

### 7.4 Boundaries & access control (generic principles)

> Кто что может делать с какими data / actions / decisions. Generic patterns.

**Tier model (generic):**

| Tier | Access | Authority | Examples |
|------|--------|-----------|----------|
| **T0 — Founder** | Full read + full write + full authority | Final ack on everything significant | Owner |
| **T1 — Trusted agents (core)** | Scoped read + write per role | Execute within mandate; escalate edges | Bу coordinator / health monitor / etc |
| **T2 — Specialised agents** | Narrow read + narrow write | Execute specific tasks; ack-required для outputs | Domain experts, skill specialists |
| **T3 — Inner stakeholders** | Read-only access к specific projects (NDA-cleared) | Discussion + feedback; no system modifications | Active mentors, current clients |
| **T4 — Outer stakeholders** | Public surface only | None (только receive published) | Public audience, prospects |

**Default Deny pattern (per §4.6):**

- New agent / new stakeholder / new action → **denied by default** until explicitly granted access
- Permission grants explicit + revocable + audit-logged
- Privilege escalation requires founder ack
- "Borrowing" permissions ("just this once") — anti-pattern, must be formal grant

**Access control mechanisms (generic enforcement):**

- **Role-based** — permissions attached к roles, агенты get role assignments
- **Scoped tokens** — для external integrations (per Cat L L.1) — minimum scope per service
- **Time-bounded** — temporary access expires automatically (ad-hoc collaborator gets access для cycle, not permanent)
- **Audit-trailed** — каждый access / action / permission change logged
- **Revocable** — kill-switch на доступ всегда available + tested
- **Compartmentalised** — sensitive data в separate stores (financial / personal / strategic — different access tiers)

**Anti-patterns:**

- ❌ "Open by default" → security drift
- ❌ Implicit permissions (agent has access "потому что нужно было") → audit nightmare
- ❌ Permanent permissions для temporary needs → permission accumulation over time
- ❌ Shared credentials между agents → no per-agent audit trail
- ❌ Silent privilege escalation → trust erosion

## §8 Layered evolution (generic phasing model)

**Источник:** Ruslan vision dictation 27.04 evening + D27 Fork-and-merge + general systems evolution principles + version control patterns.

**Главный принцип:**

> **Эволюция системы Jetix — двух-осевая.** Vertical axis = foundational evolution (slow, deliberate, architectural). Horizontal axis = per-instance customization (fluid, owner-driven, contextual). Обе оси работают **одновременно**, но по **разным rhythms** и под **разными ограничениями**.

**Ключевая особенность:** Jetix **с самого начала качественно работает** ("из коробки functional"), а не "v1 minimal viable beta которая улучшится потом". Owner добавляет customization поверх **уже работающего fundament**'а, не fix'ит broken базу.

**Visual:**

```
                    ▲  Vertical axis (Foundation evolution)
                    |  slow / deliberate / quarterly+ rhythm
                    |  architectural review required
                    |  pulled from upstream / community / research
                    |
   ◄──── Horizontal axis (Per-instance customization) ────►
                    |  fluid / owner-driven / continuous
                    |  free additions within primary logic
                    |  per strategic docs direction
                    |
                    ▼
```

---

### 8.1 Out-of-the-box quality (Day 0 functional baseline)

> С первого дня system должна **deeply работать**, не "v1 beta requires fixing".

**Принципы Day 0:**

- **Foundational features уже there** — Strategic Management / Information Lifecycle / Self-Improvement / Project Mgmt / Agent Swarm / AI Continuity / Multi-channel / Foundation & Reliability / Health & Self-Improvement / Daily Ops / CRM / Integrations (все 12 категорий из §1)
- **Sensible defaults** — каждый component pre-configured с reasonable settings; owner не должен tune everything чтобы start using
- **No "missing pieces"** — нет stub'ов "TODO позже"; если feature в FUNDAMENTAL — она functional Day 0
- **Sample data + templates** — owner получает empty но **structured** workspace (templates / sample wikis / sample agents / sample CRM schemas) — easy to fill, not "blank page paralysis"
- **Onboarding gradient** — сложность раскрывается постепенно (basic features visible upfront, advanced — discovered when needed), но **наличие — Day 0**

**Anti-pattern:** "MVP" mentality где owner **сначала фик**сит broken parts → строит на them. Foundation должен быть solid с первого commit'а.

---

### 8.2 Vertical axis — Foundation evolution (slow, deliberate)

> Когда меняется fundamental база (architecture / core agents / KM mechanism / Левенчук discipline implementation / etc).

**Характеристики:**

- **Cadence:** периодически — quarterly minimum для review, **semi-annual major release** if substantive insights accumulated
- **Source of changes:**
  - **External improvements** — best practices research / community contributions / new AI capabilities / new methodology insights (Левенчук / multi-agent / observability / etc updates)
  - **Cross-instance learning** — pull улучшения от других Jetix instances (per D27 Fork-and-merge — если другие contributors push'ят refinements в canonical)
  - **Significant insights** — accumulated patterns которые warrant architectural change
  - **Incident-driven** — critical reliability issues / SPoF discoveries
- **Triggers (validate before vertical change):**
  - Pattern accumulation (✓ если несколько cycles surface ту же gap)
  - Best practice discovery (✓ если новая technology unlocks better approach)
  - Architecture stress test failed (✓ если current architecture не scales)
  - Community push (✓ если другие instances contribute improvements)
  - **NOT** triggered by: convenience / single frustration / competitive FOMO / без data (per §4.7 invalid triggers)
- **Process:**
  - Architectural review mandatory (per §4.5 hard rule — agents не делают architectural decisions)
  - Backward compatibility considered (existing instances должны migrate cleanly)
  - Reversible — major release tagged in git, rollback path exists
  - Communication — change documented, rationale explicit, migration guide if applicable

**Update mechanism (sync с upstream):**

- Jetix может maintain **canonical version** в central repo (e.g. GitHub upstream)
- Per-instance может pull updates когда хочет (opt-in, not auto-push)
- Selective: можно pull только specific improvements, не "all or nothing"
- Local customizations (horizontal axis) preserved through pulls (D27 fork-and-merge mechanic)

**Foundation evolution НЕ часто:**

> "На уровне выше — даётся обновление" (Ruslan formulation). Major foundation changes — **раз в полгода** ориентир, не "weekly tweaks". Stability is foundation feature.

---

### 8.3 Horizontal axis — Per-instance customization (fluid, owner-driven)

> Где owner добавляет / изобретает / создаёт что-то под свою current ситуацию.

**Характеристики:**

- **Cadence:** continuous — adds happen by need, не по schedule
- **Source of changes:** owner direction per current strategic documents
- **Свобода:** owner может добавлять almost anything что соответствует strategic direction
  - New agents (custom roles / domain experts)
  - New wikis (per project / per domain — multiple wikis OK if they actually serve distinct needs)
  - New skills (acquisition pipeline triggered)
  - New integrations (через Integration Hub Cat L)
  - New workflows (per repeatable process)
  - New CRM views / categories
  - New templates
  - New strategic documents (annual / quarterly / per-project briefs)
  - **Даже "система для управления космическим самолётом"** — если соответствует ситуации owner'а и реально нужно
- **Тригер:** strategic docs require this capability → add it

**Constraints на горизонтальной оси (что owner НЕ может сломать):**

Несмотря на свободу — primary logic / first principles preserved. Конкретно:

1. **Левенчук constitutional limits (§6.1)** — нельзя add agents которые break architectural / strategic boundaries
2. **Anti-scope (§6)** — нельзя add features которые violate scope creep prevention
3. **Foundation reliability (§5)** — customizations не должны compromise data integrity / process continuity
4. **Founder authority preservation** — нельзя automate strategy / architecture / final acks
5. **Privacy & consent** — кастомные integrations follow same privacy patterns
6. **Auditability** — каждая addition logged + provenance
7. **Coherence check** — sensible questions: "10 разных wikis оправданы? Или это fragmentation без reason?" (per Ruslan emphasis 27.04 — base logic preservation)

**Discipline для horizontal additions:**

- **Coherence check** — новый component должен иметь clear distinct purpose (not duplicate existing)
- **Quality bar** — новые agents следуют constitutional rules (§6.1)
- **Resource impact assessment** — ad-hoc additions affect time / attention / $ / cognitive load — owner aware tradeoffs
- **Provenance per addition** — кто добавил / когда / зачем / какой strategic doc trigger'ил
- **Reversibility** — easy to remove если не работает (горизонтальные additions reversible in days, не месяцы)

**Anti-patterns horizontal:**

- ❌ Random additions без strategic doc connection (drift)
- ❌ Duplication "потому что лень reuse" (fragmentation)
- ❌ "Маленькая фича" которая ломает Левенчук discipline (constitutional violation)
- ❌ Добавления для "what if потом понадобится" (premature optimization → cognitive load)

### 8.4 Cross-instance sync (cross-axis bridge)

> Mechanism для shared evolution between Jetix instances (если applicable).

**Concept:**

Если несколько people / orgs run Jetix instances, они могут (opt-in) share **foundation improvements** через canonical upstream:

- **Up direction (contribution):** instance делится novel improvement → upstream review → if accepted → merged in canonical → доступно other instances pull'ить
- **Down direction (consumption):** instance pulls latest canonical → integrates улучшения → keeps own customizations
- **Voluntary** — никто не forced sync; private instances stay private
- **Selective** — можно pull только specific changes, не whole release

**What syncs:**

- ✅ Foundational improvements (architecture / agent patterns / methodology / KB structures)
- ✅ Anonymized best practices (что работает, что нет, generic learnings)
- ✅ Templates / skills libraries / workflow patterns
- ❌ NEVER: personal data / instance-specific strategic content / private CRM data / financial info

**Discipline:**

- Privacy-respect mandatory (anonymization required для shared learnings)
- Quality gate (upstream review для contributions)
- Backward compat (canonical changes don't break instances)
- Opt-in (instances не auto-sync — explicit pull)

**Direct support D27** (Fork-and-merge evolution) + **D21** (Roy-Replication readiness).

**Horizontal scale:**

- Не "feature bloat" — "deeper fit с owner's work"

**Cross-instance scale:**
- Network effect через D27 (more contributors = better canonical)

### 8.8 What graceful evolution looks like (vs failure modes)

**Graceful evolution patterns:**

- ✅ **Backward compatible** — old instances upgrade без breaking
- ✅ **Reversible** — can rollback if regression discovered
- ✅ **Documented** — what changed / why / how to adopt
- ✅ **Tested** — not "ship and pray"
- ✅ **Selective** — owners pull what they want, не forced bundle

**Failure modes (anti-patterns):**

- ❌ Breaking changes без migration path
- ❌ "Latest version always" forced auto-update
- ❌ Vertical changes без architectural review
- ❌ Horizontal additions duplicating existing functionality
- ❌ Customizations что violate constitutional constants
- ❌ Evolution-for-evolution-sake (changes without trigger justification)
