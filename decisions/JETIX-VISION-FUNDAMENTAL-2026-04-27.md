# JETIX VISION OF SYSTEM — FUNDAMENTAL

## §0 Что это за документ

Это **базовая, фундаментальная Vision Document системы Jetix** — описание того что такое Jetix как **"система всех систем"** в вакууме, без привязки к конкретной нише / личности / бизнесу.

**Почему два документа:**

Jetix должен быть **двуслойным**:
- **Layer 1 (этот документ — FUNDAMENTAL):** базовая прошивка / ядро / архитектура которая работает одинаково в **любой нише** — AI-консалтинг / медицина / e-commerce / научный research / преподавание / etc. Можно "перетянуть" из одной ниши в другую без переписывания base.
- **Layer 2 (отдельный документ — RUSLAN-LAYER):** оптимизация / overlay / специфика под текущую ситуацию Ruslan'а — DACH Mittelstand / AI consulting / Phase 1 €50K target / 11 archetypes / Korp-Startup positioning / Top Lists strategy / etc. Натягивается ПОВЕРХ FUNDAMENTAL.

**Принцип:**
- Сначала описываем **FUNDAMENTAL** (этот doc) — generic, sector-agnostic, person-agnostic
- Потом натягиваем **RUSLAN-LAYER** (другой doc) — specific под Ruslan
- Если завтра Ruslan уйдёт в другую нишу → меняется только RUSLAN-LAYER, FUNDAMENTAL остаётся
- Если кто-то другой возьмёт Jetix-методологию и применит к своему делу → берёт FUNDAMENTAL и пишет свой LAYER

**Что должна уметь FUNDAMENTAL Jetix:**

- Адекватно **работать с информацией** (capture / store / retrieve / synthesize / compound)
- **Улучшаться сама** (self-audit / agent strategy learning / methodology refinement)
- **Wiki / KM хорошо работает** (queryable / cross-referenced / non-contradictory)
- **Агенты между собой адекватно общаются** (mailboxes / message schemas / handoff protocols / escalation taxonomy / Левенчук discipline)
- **Здоровье системы** прозрачно (health indicators / monitoring / alerting)
- **Best practices** из лучших систем — Левенчук ШСМ / multi-agent architectures / event sourcing / observability / Lean / TOC / etc

**Стиль:**
- Generic, vacuum-sealed, sector/person/business-agnostic
- На человеческом языке (без jargon-soup)
- Глубоко (Левенчук discipline / 1000% / триллион процентов)
- Если что-то выглядит Ruslan-specific → автоматически в RUSLAN-LAYER, не сюда
- Best-of-best наработки cited (Левенчук / ШСМ / multi-agent papers / etc)

**Authoring workflow:**
1. Cloud Cowork synthesizes generic foundations из best sources (FPF + SYSTEM-OVERVIEW + Левенчук + multi-agent best practices)
2. Ruslan reviews / dictates что добавить / изменить
3. Итерация → push → Ruslan pulls → видит → ack
4. Когда LOCKED → RUSLAN-LAYER overlay начинается

### 🎯 Главный принцип системы (Foundation)

**Система = усилитель мозга, инструмент для мышления и работы с информацией.**

Не "ещё один SaaS", не "просто agents", не "просто wiki". Это **продолжение и усиление мозга владельца** — помогает думать стратегически качественно/быстро/эффективно. Heavy информация и heavy задачи **перекладываются на систему** → владелец освобождает голову для **стратегического мышления + чистоты + спокойствия**.

**Ключевая фишка:** быстро обучаемая + быстро делающая. Направление учёбы определяется **стратегическими документами**. Можно сегодня учить video editing, завтра M&A advisory, послезавтра physics research — система адаптируется под фокус владельца.

**Двойной вектор: улучшение + сохранение.**
- **Улучшение:** система направлена на постоянное улучшение и развитие — себя самой и владельца. Через compound engineering, skill acquisition, methodology evolution.
- **Сохранение:** одновременно — на сохранение и **безопасность себя** + чистоту состояния. Не allows degradation. Регулярные health checkups → проактивное обнаружение drift / stagnation / chaos.

**Кто пользователь системы (foundational positioning):**

Jetix создан для **профессиональных интеллектуальных работников** — людей которые:
- Работают с **большими объёмами информации**
- Решают **сложные задачи / вопросы / проекты**
- Управляют **множественными направлениями** одновременно
- Имеют **долгосрочные стратегические амбиции** (не tactical-only)
- Готовы **инвестировать в систему** time/attention для multiplier ROI

Это **НЕ массовый продукт** для casual users. Целевая audience — founders / researchers / investors / strategic consultants / scientists / serious creators. Система требует discipline для использования + дисциплина возвращается через compound leverage.

**Стиль работы system × owner:**

Система **подсказывает / показывает / подпинывает** — НЕ "делает за тебя". Это критично:
- Владелец остаётся **agent** (acting subject), не **patient** (object that things happen to)
- Система highlights что нужно посмотреть, какие decisions ждут, где drift намечается
- Финальный выбор всегда у владельца (per Левенчук architect-orbit + D2)
- Anti-pattern: автоматически принимать decisions за владельца → ведёт к agency erosion

### Категория A — Strategic Management Layer

> Слой стратегического управления — основа на которой всё работает. Стратегические документы определяют направление системы; система помогает их создавать, следить за ними, придерживаться.

**UC-A.1 — Multi-level strategic document hosting**
- **Что:** Система имеет dedicated место для стратегических документов на разных уровнях иерархии: System (Vision / Locks / Architecture) / Strategic (DoT / North Star) / Annual / Quarterly / Project / Sub-project.
- **Зачем:** Стратегия должна быть explicit + queryable + versionable. Без structured location strategy drifts; agents не могут на неё опереться.
- **Принцип:** один документ один источник истины per layer; cross-references между уровнями; LOCKED status для stable docs.

**UC-A.2 — Strategic document creation assistance**
- **Что:** Система помогает владельцу **создавать** strategic docs: synthesizes из существующих materials, предлагает structure, surfaces contradictions с предыдущими решениями, drafts for review.
- **Зачем:** Strategic thinking — heavy work; system reduces friction чтобы чаще обновлять стратегию (а не редко и painful).
- **Принцип:** AI = drafter + reviewer; владелец = ack/edit/reject. Per Левенчук: AI не strategize самостоятельно.

**UC-A.3 — Strategic alignment enforcement**
- **Что:** Когда система выполняет любую задачу — проверяет alignment с актуальной стратегией. Если задача противоречит strategic doc → escalation к владельцу. Если стратегия меняется → downstream artifacts (project plans / agent priorities) refresh'ятся.
- **Зачем:** Prevent drift. Stratагические documents без enforcement = decoration. С enforcement — реальный compass.
- **Принцип:** strategy → policy → execution. Все 3 связаны explicit refs.

### Категория B — Information Lifecycle Layer

> Работа с информацией. Не "жрать всё подряд", а selective inbox → quality storage → synthesis → reuse.

**UC-B.1 — Selective inbox с filters**
- **Что:** Все incoming signals (voice memos / web / chat / books / videos / research / messages) проходят через **inbox layer** со strategy-aware фильтрами. Не всё попадает в систему — только что соответствует current strategic focus + personal preferences владельца.
- **Зачем:** Garbage in = garbage out. Без фильтров system overwhelmed; с фильтрами — focused signal.
- **Принцип:** filters declarative (yaml-described), не hidden logic. Владелец может посмотреть/изменить/override.

**UC-B.2 — Multi-wiki knowledge storage (one per project / domain)**
- **Что:** Система может создавать и поддерживать **несколько wiki одновременно** — одна на life если хватает; несколько per project если нужны раздельные knowledge bases. Built на best-of-breed методологии (Karpathy LLM Wiki + Andy Matuschak evergreens + Niklas Luhmann Zettelkasten).
- **Зачем:** Single wiki не масштабируется когда проекты разные. Multiple wikis с unified retrieval → flexibility + isolation.
- **Принцип:** agent-driven но **manual-accessible**. Если agents fail — владелец может найти всё руками через файловую систему.

**UC-B.3 — Source preservation (raw + processed)**
- **Что:** Сырые данные (voice transcripts / web pages / book excerpts / chat logs / research dumps) **сохраняются permanent** + processed выжимки идут в wiki. Provenance каждого insight'а трассируется к raw source.
- **Зачем:** Provenance = trust. Через 6 месяцев захочется проверить "откуда мы это взяли" — должно быть реконструируемо.
- **Принцип:** raw immutable; processed может быть обновлён/переписан; provenance link обязателен.

**UC-B.4 — Synthesis on demand**
- **Что:** На основе wiki + raw + сейчас открытых проектов система **быстро** generates: новые статьи / posts / answers to queries / summaries / comparisons / strategic insights / talking points / etc. Workflow: query → retrieve relevant → synthesize → present.
- **Зачем:** Reuse accumulated knowledge — основной leverage compound engineering.
- **Принцип:** synthesis всегда cited (с references к sources); владелец может verify chain of reasoning.

**UC-B.5 — Rapid ingestion pipeline (proactive learning)**
- **Что:** Пример: появился проект "выбрать страну для переезда". Вместо того чтобы владелец смотрел 50 видео, читал 30 articles вручную — он **загружает все materials в систему**, она их processes (transcripts / extracts / structures), сохраняет в нужный wiki, генерирует analytics + reports → владелец потом за 30 минут получает actionable view.
- **Зачем:** Massive time leverage. Heavy информационный input shifts от founder к system.
- **Принцип:** ingest pipeline universal — не важно тип material (video / book / article / transcript / chat history). Output structured per project schema.

### Категория C — Self-Improvement & Learning Layer

> Система улучшается сама + умеет учиться новым skills под фокус владельца.

**UC-C.1 — System self-improvement (Compound Engineering)**
- **Что:** Каждая работа cycle (Plan / Work / Review / Compound 40/10/40/10) produces artifacts которые улучшают будущие cycles: agent strategies refined, methodologies updated, KB cross-refs added, frequent-query indexes built.
- **Зачем:** Linear systems plateau; compound systems exponentiate. Без self-improvement system stagnates за месяцы.
- **Принцип:** improvements are review-gated (не auto-deploy без approval), но incentive structure encourages их proposal по каждому cycle.

**UC-C.2 — Skills acquisition under strategic direction**
- **Что:** Когда стратегия требует нового capability (e.g., video processing / post writing / specific research domain / language / tool mastery) — система **изучает этот skill**: ingests best resources на эту тему, builds wiki sub-area, trains agent skills, creates templates/workflows. Скорость skill acquisition = competitive edge.
- **Зачем:** "Быстро обучаемая + быстро делающая" — главная фишка системы. Без skill flexibility она stuck в исходных capabilities.
- **Принцип:** skill direction всегда из strategic docs (не random); skill acquisition documented (provenance / methodology / what worked vs what didn't).

**UC-C.3 — Methodology library (reusable patterns)**
- **Что:** Система maintains library of reusable methodologies / templates / workflows которые накопились через cycles. Каждый методологический artifact — versioned, cross-referenced, applicable across projects.
- **Зачем:** Roy-replication readiness — другие людьи / instances могут взять методологию и применить. Direct support для D21 Roy-Replication + D27 Fork-and-merge.
- **Принцип:** методология = **first-class artifact**, не побочный эффект. Каждое решение оставляет методологический след.

### Категория D — Project & Resource Management Layer

> Управление проектами + ресурсами (время / финансы / энергия / attention / credits).

**UC-D.1 — Project lifecycle management**
- **Что:** Templates + tooling для запуска / ведения / закрытия проектов: brief → milestones → tasks → status tracking → review → closure / archival. Унифицированно для всех типов проектов (research / product / outreach / personal / etc).
- **Зачем:** Без project structure — chaos. С единым framework — сравнимость + transferability.
- **Принцип:** lightweight, не bureaucratic. Templates suggest structure, не enforce rigid forms.

**UC-D.2 — Resource budgeting & monitoring**
- **Что:** Система отслеживает ресурсы: time (deep work hours / context-switches / sleep), money (cash / runway / spend per category), energy (subjective state / cycles / recovery), attention (manager attention budget ≤ N active tasks), credits (API tokens / compute / 3rd-party limits).
- **Зачем:** Resource awareness = sustainable execution. Founder без resource visibility burns out / overcommits / runs out of cash.
- **Принцип:** resources tracked but не surveillance — данные доступны владельцу для self-management, не для external judgment.

### Категория E — Agent Swarm Operations Layer

> Агенты (рой) — выполняют сложные многоступенчатые задачи. Glubokо настроены под Левенчук discipline.

**UC-E.1 — Multi-agent coordination (бригадир + рой)**
- **Что:** Система имеет coordinator (бригадир) + specialised executors (рой). Бригадир принимает complex задачу → декомпозирует → dispatches sub-tasks parallel/sequential subagents → integrates results → returns to user. Subagents могут sami spawn'ить sub-subagents (рекурсивно, controlled depth).
- **Зачем:** Single-agent solutions plateau на complexity ceiling. Multi-agent с coordination — handle gigantic tasks (whole projects / migrations / research dumps / etc).
- **Принцип:** clear role boundaries (бригадир НЕ executor; executor НЕ strategize); message-passing not shared mutable state; всё auditable.

**UC-E.2 — Agent self-learning (strategy refinement)**
- **Что:** Каждый agent maintains `strategies.md` — постоянно улучшающийся набор patterns которые работают для этого agent'а в этом role. Через cycles агенты накапливают what worked vs what failed → next time лучше.
- **Зачем:** Agents которые не learn — stuck. Agents которые learn — ROI растёт через месяцы.
- **Принцип:** learning сохраняется per agent (не shared), но best-practices cross-pollinate через review cycles.

**UC-E.3 — Agent communication discipline (Левенчук style)**
- **Что:** Все agent communication через explicit mailboxes + message schemas + handoff protocols. Никаких implicit conventions. Каждое сообщение typed, routed, archived. Escalation taxonomy (routine / review / decision / strategic) explicit.
- **Зачем:** Без discipline — chaos в multi-agent. С discipline — auditable, debuggable, scalable.
- **Принцип:** communication is **the** core agent capability — quality > quantity, structured > free-form.

### Категория F — AI Augmentation Continuity Layer

> Claude Code (или эквивалент) memory continuity — чтобы НЕ начинать каждый раз с нуля. Контекст persistent через sessions.

**UC-F.1 — Persistent memory across sessions**
- **Что:** AI assistant (Claude Code or similar) memory улучшается с каждой session: ключевой контекст / preferences / ongoing projects / recent decisions — всё remembered. Новый chat НЕ требует re-explanation базовых things.
- **Зачем:** Context-switching cost massive. Persistent memory = compound value через месяцы взаимодействия.
- **Принцип:** memory layered (user / feedback / project / reference); selective (не всё запоминается, важное — да); auditable (владелец может посмотреть что запомнено).

**UC-F.2 — Cross-context continuity**
- **Что:** Когда владелец переключается между задачами (mentor call → planning → execution → research → personal life) — система сохраняет thread per context: где остановились, что pending, что next. Возврат к контексту = мгновенный.
- **Зачем:** Multi-project founder constantly context-switches. Без continuity — каждое переключение costs energy для reconstruction.
- **Принцип:** контекст explicit (сохраняется как artifact), не implicit (в памяти AI).

### Категория G — Multi-channel Access Layer

> Быстрый доступ к системе из любой точки — telegram / messengers / mobile / desktop / cli.

**UC-G.1 — Messenger access (Telegram + основные)**
- **Что:** Система accessible через Telegram (или эквивалент) — capture (voice memo / quick note / link share) + queries (что сейчас активно? / напомни X / обработай Y) + status updates (alerts / digest / questions от agents).
- **Зачем:** Founder mobile / на ходу / не всегда у компьютера. Без mobile access system useful только сидя за столом.
- **Принцип:** messenger = accelerator не replacement; deep work всё равно desktop/cli.

**UC-G.2 — CLI / desktop primary access**
- **Что:** Основной workflow через cli + IDE (VS Code / Antigravity) — где владелец имеет full power (deep edits / agent launching / git operations / multi-file orchestration).
- **Зачем:** Power user workflow требует precision tools. GUI / mobile = capture/quick; cli = serious work.
- **Принцип:** cli first-class citizen, не afterthought.

### Категория H — Foundation & Reliability Layer

> Foundation на которой всё стоит. Company-as-Code / git-based / reversible / auditable / health-monitored.

**UC-H.1 — Company-as-Code (git-based system state)**
- **Что:** Всё state системы (decisions / plans / agent outputs / wiki / configs / operational status) живёт в git repo. System reconstructible at any moment in time. Config declarative (yaml/json), не UI-clicks.
- **Зачем:** Git = versioned + auditable + portable + collaborative. Без git system → black box.
- **Принцип:** git = source of truth; UI/Notion = views.

**UC-H.2 — Provenance & audit trail**
- **Что:** Каждый artifact (wiki entry / decision / agent output / spend / external action) has explicit provenance: когда / кем (agent / human) / на основе чего / linked sources. Audit trail reconstructible.
- **Зачем:** Trust в decisions требует чтобы reasoning chain была visible. Без provenance — agent outputs become unverifiable claims.
- **Принцип:** provenance required (не optional); F-G-R tagging (Founder-only / Group / Reportable) per artifact.

**UC-H.3 — Reversibility & safe experimentation**
- **Что:** Все changes reversible через git history. Experiments на branches (forks) — можно "попробовать без поломки". Rollback = одна команда.
- **Зачем:** Без reversibility owner боится experimentировать. С reversibility — bold iteration.
- **Принцип:** branch для experiments; commit small + frequent; merge только после ack.

**UC-H.4 — Health monitoring & alerting**
- **Что:** System emits operational metrics (agent communication flow / KB freshness / cycle velocity / decision throughput / resource usage / error rates / etc) на dashboard. Alerts когда indicators deviate from baselines.
- **Зачем:** "Здоровье системы" — не abstract. Без metrics — issues surface late. С metrics — early warning.
- **Принцип:** few key indicators (не dashboard-overload); leading > lagging; alerts actionable не noise.

**UC-H.5 — Quality gates (stage-gated approval)**
- **Что:** Important artifacts (strategic docs / locks / final outputs) progress через stage gates с approval requirement per stage (synthesize → architect-input → variant-generation → implementation → ack). BA-cycle (Bias-Audit) closure mandatory pre-delivery.
- **Зачем:** Quality discipline. Без gates — premature releases / consequence cascade.
- **Принцип:** gates lightweight но non-bypassable; ack explicit (recorded as artifact).

### Категория I — Continuous Health & Self-Improvement Layer

> Регулярные checkups системы + владельца. Постоянное напоминание о дисциплине self-reflection. Система **подпинывает**, не делает за.

**UC-I.1 — Periodic system health checkups (weekly / monthly / yearly)**
- **Что:** Система автоматически проводит health-checkups регулярно: weekly review (что crucial / что drift'нуло) / monthly (deeper — KPIs / strategy alignment / resource burn) / yearly (architectural review / methodology refinement / vision check). Используются best practices observability + automated metrics + structured retrospectives.
- **Зачем:** Без регулярных checkups system slowly degrades — KB stale, agent strategies misaligned, decisions outdated. С checkups — early detection + continuous calibration.
- **Принцип:** automate всё что можно (metrics collection / report generation / anomaly detection); human ack для qualitative assessment.

**UC-I.2 — Periodic owner self-reflection prompts (discipline reminders)**
- **Что:** Система **периодически подталкивает** владельца на self-reflection: "Как могу улучшить себя? Как улучшить систему? Что работает / что нет? Какие правки внести?" Это происходит на cadence (например еженедельно / помесячно), запускается structured prompts.
- **Зачем:** Owner self-improvement — основной leverage. Без external nudge даже disciplined owner skip'ает self-review (срочное вытесняет важное). С system nudge — discipline maintained.
- **Принцип:** prompts открытые questions (не yes/no checkboxes); ответы saved как artifacts (для compound learning); вызовы могут быть deferred но не cancelled.

**UC-I.3 — Recommendation engine ("показывает куда смотреть", не делает за)**
- **Что:** Система highlights: что требует внимания / какие decisions ждут / где drift намечается / какие opportunities surface'ятся / кто из контактов давно not touched / какие KPIs trending wrong. Output = list "look here" — НЕ автоматические actions.
- **Зачем:** Cognitive load reduction — не нужно владельцу самому scan'ить всё state. С recommendation engine — focus enabled.
- **Принцип:** **подсказывает / показывает / подпинывает** — финальный выбор всегда у владельца. Anti-pattern: auto-execute recommendations → agency erosion.

**UC-I.4 — System self-preservation & integrity checks**
- **Что:** Система регулярно проверяет own integrity: KB consistency / decisions chain validity / agent strategies coherence / resource leaks / security posture / backup health. Auto-repair где safe; escalation к владельцу где judgment required.
- **Зачем:** Self-preservation. Без integrity checks — silent corruption / drift / data loss accumulates.
- **Принцип:** integrity > convenience. Лучше остановить task с warning чем продолжить on broken state.

---

### Категория J — Daily Operations Layer

> Ежедневная работа: planning + execution + draft workflow + EOD report. Где human × system co-work happens день за днём.

**UC-J.1 — Daily working page (planning + draft + EOD report)**
- **Что:** Каждый день имеет dedicated working page (Daily Log) где владелец:
  - **Утром:** планирует что хочет сделать сам + что должна сделать система
  - **В течение дня:** черновик-area для творения / тестирования / экспериментов с pulled-in info из активных проектов
  - **Вечером:** получает report (что сделано system'ой + сам пишет что сделал) или система автоматически собирает EOD summary
- **Зачем:** Daily structure = sustainable execution. Без структуры дни blur, momentum теряется. С Daily Log — visibility + discipline.
- **Принцип:** lightweight (не bureaucratic); черновик-friendly (place для messy thinking); auto-link to active projects.

**UC-J.2 — Draft → review → promote/archive workflow**
- **Что:** Items созданные на daily page — не сразу финальные. Они идут в **drafts** (per project / general). На review (EOD или weekly) — что годится → promoted в **чистовик** (project doc / wiki entry / decision); что не годится → archive (preserved но excluded из active query results).
- **Зачем:** Mental space для exploration без commitment. Не каждая мысль должна сразу become artifact; drafts mehr space для experimentation.
- **Принцип:** drafts persist (не auto-delete); promotion explicit (требует ack); archived items searchable но flagged.

**UC-J.3 — Weekly draft review & integration**
- **Что:** Раз в неделю система запускает (или предлагает) review накопленных drafts: что обработать / promote / archive / merge / drop. Output — clean state drafts area + integrated knowledge в wiki/projects.
- **Зачем:** Без weekly review drafts накапливаются → information graveyard. С review — drafts либо promoted, либо archived с провенансом.
- **Принцип:** review automated where possible (auto-suggest categorization); human ack для значимых items.

---

### Категория K — Communication & Network Tracking Layer (basic CRM)

> Базовый слой tracking отношений и взаимодействий с людьми / организациями. Multi-context (business / family / mentor / friend / etc).

**UC-K.1 — Multi-context relationship tracking**
- **Что:** Все значимые контакты owner'а — в системе с structured info: имя / контакты / context (business / family / mentor / friend / partner / vendor / etc) / relationship strength / current state / history / mutual interests. Один human может занимать multiple contexts (business contact + friend).
- **Зачем:** Без structured tracking — relationships fade, opportunities lost, last-touch unknown. С tracking — proactive maintenance + leverage.
- **Принцип:** privacy-aware (sensitive info encrypted / restricted); consent-respected (если человек просит forget — forget); context-aware (business interactions ≠ family interactions, разные patterns).

**UC-K.2 — Interaction history + relationship state**
- **Что:** Каждое взаимодействие (звонок / email / meeting / message / introduction) recorded с date / context / outcome / next-action. Relationship state evolves через time (cold → warm → close → ally / friend → distant → reconnect).
- **Зачем:** Memory aid + pattern visibility. "Когда последний раз говорил с X?" / "О чём договорились?" / "Кто X-у меня представил?" — answerable.
- **Принцип:** lightweight capture (1-2 строки post-interaction); auto-extract из communication channels где возможно (email / messengers); human curation для quality.

**UC-K.3 — Multi-purpose CRM (sales / partners / mentors / community / network)**
- **Что:** Базовая CRM поддерживает разные purposes: prospects (sales pipeline) / partners (strategic relationships) / mentors-advisors (personal development) / community (network) / hires (recruiting) / friends (personal). Schema flexible per purpose; query patterns per purpose.
- **Зачем:** Single-purpose CRM (sales-only) ограничивает. Multi-purpose — полная network awareness.
- **Принцип:** один уровень data model (person / org / interaction / relationship), множественные views per purpose.
