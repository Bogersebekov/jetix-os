# Jetix Conceptual Frame v2 — Мастерская для работы с информацией

> **Status:** LOCKED conceptual frame. Заменяет метафору «дом» из master overview 29.04.
> **Method:** Ruslan-dictated 2026-04-30. AI = scribe per memory rule (никаких добавлений от AI кроме structuring).

---

## §0 TL;DR (one paragraph)

**Jetix = мастерская для работы с информацией.** Мастерская с кучей профессиональных работников и кучей станков/инструментов любого калибра. Ключевая фишка — **continuous capability expansion**: быстро внедряет новые станки/инструменты, работники быстро изучают новое, переиспользует наработки с других направлений. Владелец может играть **любую роль** — управляющий (главная), мастер за станком, исследователь нового инструмента, фильтр входящей информации, перенастройщик workflow. **Вход = информация, выход = информация** (другая, переработанная, ценная). Эволюция: одна мастерская одного человека → команда работает с одной системой → community мастеров с мастерскими совместно решают задачи слишком сложные для одной мастерской.

## §1 Ядро концепции

### Что Jetix есть

**Мастерская для работы с информацией.**

Не «дом» (статичный объект). Не «оркестр» (один дирижёр, исполнители играют по нотам). Не «город» (слишком большой). **Мастерская**, потому что:

1. **Что-то делается** — постоянный активный процесс, не сидячий объект.
2. **Есть мастер + работники + инструменты** — структура понятная.
3. **Есть конкретные deliverables** — заказы выполняются, ценность отгружается.
4. **Adaptive** — можно переучиться, поставить новый станок, переориентировать на другую нишу.
5. **Knowledge accumulates** — мастер становится опытнее с каждым проектом.

### Главная фишка — adaptability / continuous expansion

Это **не статичная мастерская с фиксированным набором станков**. Это мастерская которая:

- Может **быстро внедрять новые станки** (новые tools, новые agent skills, новые MCP integrations)
- Работники могут **быстро изучать новое** (compound learning, DRR ledger, methodology promotion)
- Может **использовать наработки с других направлений** (cross-niche knowledge transfer, fork-portable Foundation)
- Переориентируется **под изменяющиеся задачи владельца**

**Эта adaptability — главный moat.** Конкуренты с фиксированной capability отстают; Jetix эволюционирует с каждым cycle.

## §2 Что такое «информация» в Jetix

**«Информация» — это почти всё**, что осмысленно попадает в систему. Не строгое CS-определение, а функциональное:

- Информация **о рынке** (анализ ниш, конкуренты, opportunities)
- Информация **о личном самочувствии** (Life-OS, health log, energy)
- Информация **о финансах** (счета, runway, cash flow)
- Информация **о возможностях в мире** (партнёрства, новые tech, события)
- Информация **«как нужно что-то делать»** (методологии, best practices, playbooks)
- Информация-**инсайт** (новые идеи, синтезы, observations)
- Информация **о задачах / целях** (когда даёшь системе задачу — ты передаёшь информацию)

**Принцип:** всё что попадает в систему, что-то значит — это **информация**. У каждого мастера / владельца **свой набор релевантной информации** — у инвестора одно, у психолога другое, у маркетолога третье.

## §3 Роль владельца — adaptive

Владелец (Ruslan в Phase 1) **может играть любую роль** в своей мастерской:

| Роль | Что делает |
|---|---|
| **Управляющий** (default, главная) | задаёт стратегию, распределяет ресурсы, определяет направление |
| **Мастер за станком** | сам сидит и делает работу (когда хочется глубоко вовлечься) |
| **Исследователь нового станка** | разбирается в новом tool / framework / agent (D2 Studio, ROY swarm, MCP) |
| **Фильтр входящей информации** | решает что попадает в систему (Part 2 STOP-gate human ack) |
| **Перенастройщик workflow** | меняет процессы, добавляет hooks, переписывает agent prompts |

**Это даёт свободу.** Не «AI делает всё, я только смотрю» (неприемлемо per memory: AI=scribe). Не «я делаю всё, AI помогает скорости» (не масштабируется). **Гибкость роли в зависимости от задачи и настроения.**

## §4 Input → System → Output — всё информация

### Input

**На входе только информация**, прошедшая через **фильтр**:
- Гипотезы (от мозга / разговора / звонка)
- Воспоминания / opservations
- Запросы / задачи
- Сырые данные (transcripts, документы, web)
- Идеи

**Фильтр (Part 2 STOP-gate)** = «всякое дерьмо в систему не попадает». Только то что **ценно** и с чем **хочется работать**.

### Internal

Внутри системы **всё устроено** для того чтобы информация:
- **Адекватно работала** (provenance, F-G-R, structure)
- **Переливалась между частями** (typed edges, cross-references)
- **Не терялась / не засыхала** (compound learning, regular review)
- **AI не трогал важное / не обосрал** (constitutional rules, default-deny, human gate)

### Output

**На выходе тоже только информация**, но **переработанная**:

| Вход | Что система делает | Выход (тоже информация) |
|---|---|---|
| Гипотезы | анализирует, синтезирует | **План действий** |
| Сырые данные + контекст | находит паттерны | **Новый инсайт** |
| «Хочу научиться X» + материалы | структурирует, прорабатывает | **Новый навык в голове** |
| Specs + cycles | строит, тестирует | **Рабочая программа** (= информация о том как программа работает) |
| Task | делегирует, координирует | **Готовый deliverable** (= информация в форме артефакта) |

**Ключевая мысль:** «программа» / «инсайт» / «навык» — всё это **информация в разной форме**. Внутри системы всё проходит как информационный поток.

## §5 Specialization — каждый мастер свой

**У каждого мастера / владельца — своя специализация:**

- Один мастер настроил мастерскую под **инвестиции** (информация о рынках, активах, портфелях)
- Второй мастер — под **психологию** (информация о клиентах, методологиях, sessions)
- Третий — под **исследования / R&D** (литература, эксперименты, протоколы)
- ...

**Foundation v1.0 — fork-portable.** Структура одна. RUSLAN-LAYER (специализация) — заменяема. Каждый мастер берёт Foundation как базу, накладывает свою специализацию.

## §6 Эволюция фаз — от одного к community

### Phase 1 — мастерская одного владельца (текущее)

- **Один владелец** + **AI агенты** = команда мастерской
- Может делать **сложные задачи**: исследования, анализ, syntеsis, generation
- Накапливает knowledge, специализируется на нише
- **Это уже довольно охуенная мастерская** (цитата Ruslan'а 30.04)

### Phase 2 — команда работает с одной системой

- **Несколько человек** + AI + одна Jetix-мастерская
- Workflow тот же (информация-centric)
- Цель командная
- Система чуть-чуть улучшается под team scenarios

### Phase 3 — community мастеров с мастерскими

**Mega-system = network мастеров с своими мастерскими.**

Когда задачи становятся слишком сложными для одной мастерской:

1. Мастер 1 берёт **кусок задачи** в свою мастерскую → разбирает → выдаёт **одну качественную деталь**
2. Мастер 2 берёт **другой кусок** → выдаёт **другую качественную деталь**
3. ...
4. Все детали собираются на верхнем (community) уровне в **finished product**

**Это тоже информационный flow:** каждый мастер получил информацию (часть задачи) → выдал информацию (своя качественная часть результата). На community-уровне информация агрегируется в готовый комплексный артефакт.

**Платформа как раз для этого.** Phase 3 = «community of workshops» — не просто люди, а **люди со своими мастерскими**, и инфраструктура для координации между мастерскими.

## §7 Visual / View principle

Jetix не показывает **физическую реальность**. Он показывает **view**.

Аналогия: **программирование с визуальным редактором**. У тебя:
- **Код** (физическая реальность — реальные дома, реальные люди, реальные машины, реальные деньги)
- **Visual view** в IDE (концентрированное представление того что важно)

Jetix даёт **view** на:
- Свои дома (если есть недвижимость — учёт)
- Своих людей (партнёры, контакты, family — CRM)
- Свои машины (assets — учёт)
- Свои проекты, задачи, ресурсы

Удобно **навигировать**, **передвигать ресурсы между задачами**, **использовать**.

**Не смотришь на всё физическое одновременно** — это перегрузка. Смотришь на **view** который система собрала — релевантное, структурированное, удобное.
