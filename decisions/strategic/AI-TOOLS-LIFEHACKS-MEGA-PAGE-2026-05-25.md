# 🧰 AI Tools & Lifehacks — Mega-page

> **Инструменты для быстрой работы с информацией. От голоса до решения — за минуты, а не часы.**
> Companion-страница к любому из 3 слоёв Notion-архитектуры (Personal Core / Team Collaboration / Universal Business Foundation).

---

## §1 Назначение страницы

Эта страница — **живой каталог инструментов и лайфхаков**, которые превращают сырьё (голос, PDF, ссылку, мысль на бегу) в structured-запись в системе за минуты. Идея простая: ты не должен «садиться писать» — ты говоришь, кидаешь ссылку, фоткаешь страницу книги, и система сама раскладывает это по полочкам. Это не теория — почти всё здесь **уже работает внутри Jetix OS** (54 skill-команды Claude Code, реальные Python-пайплайны в `tools/`, ROY-рой из 17 экспертов). Где инструмент пока внешний или планируется — помечено явно. Страница — **reference, не кнопка**: ничего здесь не запускается автоматически, никаких API-ключей в открытом виде, никаких автозаписей в прод без твоего ревью (R11 + R6 + voice-DRAFT discipline). Бери любой инструмент, читай «Что делает / Как использовать / Когда / Где живёт», и подключай к тому слою Notion, который тебе сейчас нужен.

---

## §2 15+ инструментов в 4 кластерах

Кластеры по жизненному циклу информации: **захватил → визуализировал → синтезировал → скоординировал**. Для каждого инструмента: что делает, как запустить (quick start), когда применять, и где он живёт в реальной системе Jetix.

---

### 🟦 Кластер A — Capture & Process (захват и обработка)

#### 📱 1. Telegram заметки processor → автозанос в Wiki

- **Что делает.** Ты кидаешь мысль/ссылку/голосовуху в свой Telegram-канал (или Saved Messages), а система забирает это и раскладывает в wiki как structured entry. Один inbox для всего «на бегу».
- **Как использовать.** Скинул сообщение в канал → периодический забор тянет его в `inbox/` → дальше идёт через тот же extract-пайплайн, что и голос → попадает в `wiki/concepts/` или `ideas/`. Quick start: просто пиши в канал как в блокнот, без форматирования.
- **Когда.** Когда мысль пришла не за столом — в метро, на прогулке, между встречами. Telegram всегда под рукой.
- **Где живёт в Jetix.** Inbox-механика: `inbox/` каталоги (`inbox/voice/`, `inbox/processed/`). Полноценный Telegram-bot забор — **(планируется)**; сейчас рабочая дорожка — голос + ручной drop в `inbox/`. Конечная точка — `wiki/` через `/ingest`.

#### 🎙️ 2. Voice notes pipeline (голос → транскрипт → structured items)

- **Что делает.** Наговариваешь голосовую заметку — система транскрибирует её, вытаскивает structured-пункты (идеи, задачи, контакты, факты), дедуплицирует и делает markdown-отчёт тебе на ревью. **Сердце системы захвата.**
- **Как использовать.** Записал OGG/MP3/M4A → положил в `inbox/voice/` → запустил `bash tools/run_pipeline.sh`. Пайплайн делает: (0) `sync_context.py`, (1) `transcribe.py` через Groq Whisper (`whisper-large-v3`, язык `ru`), (2) `extract.py` через Claude Code headless (`claude -p`, без отдельного API-ключа — на Max-подписке), (2b) CRM voice-routing в DRAFT, (3) `summarize.py` → `~/summary-latest.md`. **СТОП** — ты читаешь отчёт и принимаешь решения, потом `bash tools/run_filter.sh`.
- **Когда.** Каждый день — это основной способ заносить мысли. Лучшая практика: один длинный voice-dump вечером вместо набора текста.
- **Где живёт в Jetix.** `tools/transcribe.py` (Groq Whisper), `tools/extract.py` (CC-headless по умолчанию), `tools/run_pipeline.sh` (оркестратор), `tools/voice-pipeline-orchestrator.py`. Внешний слой записи: Wispr Flow (диктовка) **(внешний)** → но любой рекордер подойдёт. Ground-факт: extract emit'ит `inbox/processed/extract-items-latest.yaml`.

#### 📥 3. Mistral OCR pipeline (PDF/скан → searchable Markdown)

- **Что делает.** Берёт PDF, скан, фото страницы — и превращает в searchable Markdown с сохранением структуры и извлечёнными картинками. Книги и бумажные документы становятся grep-friendly текстом.
- **Как использовать.** `python3 tools/mistral_ocr.py <input_dir_or_pdf> [--out <output_dir>]`. Пример: `python3 tools/mistral_ocr.py inbox/failed-books/`. Идемпотентно — пропускает уже сконвертированные файлы. Атомарная запись (`.md.tmp` → rename), Ctrl+C не оставит половинок. Стоимость ~$0.001/страница (~$1 за 1000 страниц).
- **Когда.** Когда нужно загнать книгу/статью/контракт в KB, чтобы потом искать и цитировать. Особенно для сканов, где обычный copy-paste не работает.
- **Где живёт в Jetix.** `tools/mistral_ocr.py` (Mistral Files API, `mistral-ocr-latest`), вывод в `raw/books-md/`. Альтернатива без OCR-сервиса: `tools/convert_pdfs_to_md.py` (pymupdf4llm → docling → pdftotext fallback-цепочка). Реокр-помощник: `tools/marker_reocr.sh`, `tools/REOCR-README.md`.

#### ✂️ 4. Web clipper (браузер → сохранёнка)

- **Что делает.** Кнопка в браузере «сохранить эту страницу» → статья/тред летит в saved-for-later, откуда её потом можно прогнать через ingest.
- **Как использовать.** Поставил расширение (Notion Web Clipper или аналог) → клик на интересной странице → она падает в Notion-базу «Прочитать/Saved». Quick start: настрой одну базу-приёмник и кидай туда всё подряд, разбор — потом.
- **Когда.** Когда наткнулся на статью/тред, но читать сейчас некогда. Снижает FOMO — клипнул и забыл, вернёшься на разборе.
- **Где живёт в Jetix.** Notion Web Clipper **(внешний)** — Notion-сторона. Мост в систему: сохранённую ссылку прогоняешь через `/ingest <url>` (поддерживает URL как тип источника) → попадает в `wiki/`.

### 🟩 Кластер B — Visualize & Communicate (визуализация и коммуникация)

#### 📊 5. Mermaid quick setup (попросил CC → готовая схема)

- **Что делает.** Просишь Claude Code «нарисуй схему X» — и получаешь готовую mermaid-диаграмму (flowchart, sequence, граф) прямо в markdown. Никакого ручного рисования.
- **Как использовать.** Внутри Jetix есть полноценный mermaid-тулчейн как skills: `/mermaid-create` (создать), `/mermaid-iterate` (доработать по фидбеку), `/mermaid-validate` (проверить синтаксис), `/mermaid-export` (отрендерить в картинку). Quick start: `/mermaid-create процесс онбординга клиента` → правишь через `/mermaid-iterate`.
- **Когда.** Когда нужно объяснить процесс/архитектуру за 30 секунд. Схема доходит лучше абзаца текста.
- **Где живёт в Jetix.** `.claude/skills/mermaid-create/`, `mermaid-iterate/`, `mermaid-validate/`, `mermaid-export/` (4 директорийных skill). Плюс утилита `tools/fix_mermaid_text_color.py`. (В этой странице mermaid нет намеренно — диаграммы централизованы в главном архитектурном документе.)

#### 🎨 6. Miro / Mural для схем + общения с людьми

- **Что делает.** Бесконечный whiteboard для живого мышления вдвоём/командой: стикеры, стрелки, кластеры. Хорошо для созвонов, где надо «порисовать вместе».
- **Как использовать.** Открыл доску → накидал стикеры → расставил связи. Quick start для созвона: заранее сделай 3-4 зоны (Проблема / Варианты / Решение / Next steps), на звонке заполняй вживую.
- **Когда.** Воркшопы, brainstorm с командой/клиентом, мэппинг сложной системы вдвоём. Когда нужен co-creation, а не готовая картинка.
- **Где живёт в Jetix.** Miro **(внешний)** — есть MCP-коннектор (`mcp__claude_ai_Miro`) для интеграции, но как авторитетный источник — нет; результат воркшопа заносится в систему вручную через voice/ingest. Mural — аналог **(внешний)**.

#### 🖼️ 7. Excalidraw embed в Notion (быстрые наброски)

- **Что делает.** Лёгкие «от руки» наброски прямо внутри Notion-страницы. Эстетика whiteboard-скетча, ноль настройки.
- **Как использовать.** В Notion вставляешь Excalidraw embed (`/embed` → excalidraw.com ссылка) → рисуешь скетч → он живёт прямо в странице. Quick start: для одноразовой схемы «на коленке» это быстрее mermaid.
- **Когда.** Когда нужен быстрый рисунок-идея внутри заметки, а не строгая диаграмма. Наброски архитектуры, наброски UI, mind-map на 5 узлов.
- **Где живёт в Jetix.** Excalidraw **(внешний)** — Notion-сторона. В файловой системе аналога нет; для версионируемых схем используй mermaid skills (см. #5).

#### 📈 8. Notion native charts

- **Что делает.** Встроенные графики Notion поверх баз данных: bar/line/donut по полям. Не нужен внешний BI для простой аналитики.
- **Как использовать.** В базе → добавь Chart view → выбери поле для оси и агрегацию. Quick start: на базе задач сделай donut по статусам — мгновенный обзор «что в работе».
- **Когда.** Лёгкий дашборд по своим же базам: прогресс проекта, распределение задач, динамика касаний в CRM. Когда хватает «на глаз», без экспорта в таблицы.
- **Где живёт в Jetix.** Notion native **(внешний)** — слой представления. Авторитетная аналитика живёт в файловой системе (`reports/`, `tools/aggregate_activity.py`); Notion-чарты — удобный view поверх.

### 🟨 Кластер C — Synthesize & Decide (синтез и решения)

#### 🧠 9. Claude / GPT для synthesis (готовые промпты)

- **Что делает.** LLM как со-мыслитель: сжать длинный материал, разложить решение по критериям, накидать варианты в brainstorm. Главное — не «свободный чат», а **готовые промпты под задачу**.
- **Как использовать.** Держи 3 шаблона под рукой: **review** («прочитай это и дай 3 риска + 3 сильные стороны»), **decision** («вариант A vs B по критериям: стоимость / время / обратимость / риск, дай таблицу»), **brainstorm** («10 непохожих идей про X, потом отбери 3 лучшие и обоснуй»). Quick start: скопируй шаблон, вставь материал, получи структуру.
- **Когда.** Перед решением (decision), после сбора материала (review), на старте проблемы (brainstorm). Не для «поболтать», а для конкретного выхода.
- **Где живёт в Jetix.** Claude Code сам по себе — основной движок (`extract.py` ходит в `claude -p` headless). ROY-рой (#13) — это и есть multi-perspective synthesis built-in. GPT/прочие LLM — **(внешний)**, для cross-check. Конституция: AI **surface'ит варианты, не решает за тебя** (R1 — Ruslan = единственный стратег).

#### 🔍 10. OFFLINE_MODE structured-excerpt (паттерн /ask)

- **Что делает.** Поиск по своей KB **без inference-воды**: вместо «AI сочинил ответ» ты получаешь structured-выдержки с цитатами и путями к источникам. Честно, проверяемо, дёшево.
- **Как использовать.** `OFFLINE_MODE=1 /ask "<запрос>"` → вернёт релевантные куски wiki со ссылками `[src:...]`, без галлюцинаций. Quick start: используй когда нужен факт из своих заметок, а не творчество.
- **Когда.** Когда нужна точность и provenance: «что я уже знаю про X», подготовка к звонку, проверка «не противоречит ли это прошлым записям». Когда не доверяешь свободному синтезу.
- **Где живёт в Jetix.** `.claude/skills/ask.md` — флаг `OFFLINE_MODE=1` даёт structured-excerpt вместо inference. Конституция R6: знание персистится только через явные artefact-пути, не через «память AI».

#### 💡 11. Hypothesis tracker (Method V2 §J — Bayesian-lite)

- **Что делает.** Ведёт твои гипотезы как живые объекты: формулировка, текущая уверенность, доказательства за/против, апдейты. Не «вериши/не веришь», а «насколько уверен и почему» — Bayesian-lite.
- **Как использовать.** `/hypothesis-add` (завести), `/hypothesis-update` (обновить уверенность при новом факте), `/hypothesis-link` (связать с источником), `/hypothesis-dash` (обзор), `/hypothesis-stuck` (что висит без движения), `/hypothesis-close` (закрыть). Quick start: заведи 3 ключевые гипотезы бизнеса и обновляй при каждом новом сигнале.
- **Когда.** Когда строишь бизнес/проект на предположениях и хочешь честно отслеживать, что подтверждается, а что разваливается. Антидот к самообману.
- **Где живёт в Jetix.** 11 skills `hypothesis-*` в `.claude/skills/`, каталог `hypotheses/`, билдеры `tools/build-hypothesis-views.py`. Метод-якорь: `METHOD-V2.md` §J (`decisions/strategic/METHOD-LIFE-DEVELOPMENT-V2-2026-05-21.md`).

#### 🗂️ 12. Wiki ingest (новый источник → wiki + log + edges)

- **Что делает.** Любой источник (ссылка, PDF, YouTube, голосовая, email, буфер обмена) → структурированные страницы wiki + запись в лог + типизированные связи в граф знаний. Один вход для всего, что хочешь «запомнить системно».
- **Как использовать.** `/ingest <path-or-url>` → создаёт страницы в `wiki/concepts/` (или нужном entity type), дописывает `wiki/index.md` и `wiki/log.md`, добавляет рёбра в `wiki/graph/edges.jsonl`. Поддерживает 6 типов источников. Quick start: `/ingest https://...` на любой статье.
- **Когда.** Когда контент должен стать частью KB, а не утонуть в закладках. Граф связей потом окупается на `/ask`.
- **Где живёт в Jetix.** `.claude/skills/ingest.md`, целевая структура `wiki/` (9 entity types), граф `wiki/graph/edges.jsonl`. Связка с pipeline: `tools/wiki_integration/`.

### 🟥 Кластер D — Coordinate & Track (координация и трекинг)

#### 🤖 13. ROY swarm — multi-perspective analysis

- **Что делает.** Рой из 17 экспертов-агентов смотрит на вопрос с разных углов: инженер, инвестор, менеджер, философ, системщик (+ книжно-driven эксперты). Получаешь не одно мнение, а спектр — как мини-совет директоров.
- **Как использовать.** Бригадир (единый диспетчер, hub-and-spoke) маршрутизирует вопрос к нужным экспертам. Quick start: формулируешь решение → рой даёт 5 перспектив → ты синтезируешь. Базовая пятёрка: `engineering-expert`, `investor-expert`, `mgmt-expert`, `philosophy-expert`, `systems-expert`.
- **Когда.** Перед серьёзным решением, на ревью стратегии, когда чувствуешь, что у тебя «тоннельное зрение». Multi-perspective ловит слепые зоны.
- **Где живёт в Jetix.** 17 агентов в `.claude/agents/` (9 original + 8 book-driven, MAX-8 ack 2026-05-24). Маршрутизация: `swarm/lib/routing-table.yaml`. R12 paired-frame: `influence-ethics-expert` авто-фаером при influence-агентах. Конституция IP-1: роли ≠ исполнители.

#### 📅 14. CRM voice integration (голос → draft-контакты → ручная промоция)

- **Что делает.** Из голосовых заметок вытаскивает упоминания людей и создаёт DRAFT-карточки контактов — но **никогда не перезаписывает прод автоматически**. Ты промоутишь вручную после ревью.
- **Как использовать.** В обычном voice-пайплайне шаг 2b (`crm voice-route`) превращает items с `target:crm` в `<slug>-DRAFT.md` (статус `draft_from_voice`). Quick start: наговорил «созвонился с Церэном, обсудили X» → появляется драфт → ты проверяешь и `/crm-update` промоутишь в прод.
- **Когда.** После каждого нетворкинга/звонка — наговорил итоги, система подготовила черновики карточек. Экономит ручной занос контактов.
- **Где живёт в Jetix.** `crm/_scripts/voice_router.py`, шаг 2b в `tools/run_pipeline.sh`. 10 CRM skills (`/crm-add`, `/crm-touch`, `/crm-stuck`, `/crm-weekly` и др.). Voice-DRAFT discipline — конституционное (никаких авто-перезаписей прода).

#### ⏱️ 15. Toggl pipeline (авто-записи + дневной отчёт)

- **Что делает.** Синхронизирует time-entries из Toggl и строит markdown-отчёт с разбивкой по проектам/тегам. Плюс — заносит время из надиктованного текста через skill `log-time`.
- **Как использовать.** `python3 tools/toggl_sync.py` (сегодня) / `--week` (неделя). Токен лежит в `~/.config/jetix/toggl_token` (mode 600). Или диктуешь «работал 9-12 над outreach, потом обед» → skill `/log-time` сам разложит в Toggl + Daily Log. Quick start: вечером один `/log-time` вместо ручного таймтрекинга весь день.
- **Когда.** Для честной картины «куда ушло время». Особенно если хочешь привязать часы к проектам и видеть фокус.
- **Где живёт в Jetix.** `tools/toggl_sync.py`, `tools/toggl_log_entries.py`, история в `tools/toggl-entries-*.json`. Skill: `.claude/skills/log-time/SKILL.md` (Wispr Flow → CC → entries).

#### 📊 16. ActivityWatch timeline (дневной отчёт для self-review)

- **Что делает.** Пассивно трекает, в каких приложениях/окнах ты провёл время, и собирает это в дневной markdown-отчёт. Объективное зеркало дня без ручного ввода.
- **Как использовать.** Windows-экспорт (`tools/export_activitywatch.ps1`) кладёт JSON в `raw/activitywatch/json/` → `python3 tools/aggregate_activity.py` (сегодня / дата / `--last-7`) → отчёт в `reports/activity_YYYY-MM-DD.md`. Cron-able. Quick start: на `/close-day` сверь план с фактом из ActivityWatch.
- **Когда.** Для честного self-review: где реально утекало внимание, сколько deep work vs переключений. Объективные данные против ощущений.
- **Где живёт в Jetix.** `tools/aggregate_activity.py`, `tools/export_activitywatch.ps1`, `tools/stage1_aggregate.py`, данные в `raw/activitywatch/`.

### ➕ Bonus — ещё 4 инструмента (из реальной системы)

#### 📚 17. Mistral OCR для книг (raw книги → searchable MD)

- **Что делает.** Тот же OCR-движок (#3), но как осознанный workflow для библиотеки: загоняешь книги пачкой и получаешь grep-friendly корпус для ROY book-driven экспертов.
- **Как использовать.** `python3 tools/mistral_ocr.py inbox/failed-books/` → `raw/books-md/<topic>/`. Идемпотентно, посчитает стоимость (~$1.50 за 5 книг по ~300 стр). Quick start: положил PDF книги → один запуск → книга в KB.
- **Когда.** Когда строишь корпус под экспертизу (8 book-driven ROY-экспертов опираются именно на такой корпус).
- **Где живёт в Jetix.** `tools/mistral_ocr.py` + `tools/convert_pdfs_to_md.py`, вывод `raw/books-md/`. Связь: book-driven agents (`decisions/RUSLAN-ACK-BOOK-DRIVEN-AGENTS-2026-05-24.md`).

#### ✅ 18. Schema validation / lint (консистентность)

- **Что делает.** Проверяет здоровье KB и конституционную консистентность: orphan-страницы, противоречия, протухшие claims, missing frontmatter, дубли slug, sync CLAUDE.md ↔ principles.
- **Как использовать.** `/lint` (общий health-check) + специализированные: `/lint --check-stage-gates`, `--validate-predicate`, `--check-claude-md-sync`. Quick start: гоняй `/lint` раз в неделю как «техосмотр».
- **Когда.** Регулярно (еженедельно) и перед важными изменениями Foundation. Ловит дрейф и тихую порчу данных рано.
- **Где живёт в Jetix.** `.claude/skills/lint.md` + 8 специализированных `lint-check-*` skills. `tools/lint-frontmatter.py`. Конституция: fail-loud (FUNDAMENTAL §5.5).

#### 🧪 19. Mermaid validation tool (надёжность диаграмм)

- **Что делает.** Проверяет mermaid-синтаксис до публикации — чтобы диаграмма не сломалась в Notion/markdown. Reliability-страховка для визуалки.
- **Как использовать.** `/mermaid-validate <файл-или-блок>` → говорит, валиден ли синтаксис; `/mermaid-export` рендерит в картинку для проверки глазами. Quick start: после `/mermaid-iterate` всегда прогоняй `/mermaid-validate` перед коммитом.
- **Когда.** Перед тем как вставить диаграмму в документ/Notion. Дешевле поймать ошибку до, чем чинить сломанный рендер после.
- **Где живёт в Jetix.** `.claude/skills/mermaid-validate/`, `mermaid-export/`. Утилита цвета: `tools/fix_mermaid_text_color.py`.

#### 🌅 20. AI-augmented review (CC читает логи → wrap-of-day)

- **Что делает.** Claude Code читает твои дневные логи/активность и предлагает итог дня: что сделано, что висит, что предложить на завтра. Утром — загрузка контекста и план.
- **Как использовать.** `/plan-day` (утро: контекст + план), `/close-day` (вечер: итог + обновление логов проектов + git commit/push), `/focus` (фокус-сессия), `/company-status` (git-native снапшот всей системы ≤80 строк, zero network). Quick start: рамка дня = `/plan-day` утром, `/close-day` вечером.
- **Когда.** Каждый день как ритуал открытия/закрытия. Превращает разрозненные логи в осмысленный нарратив.
- **Где живёт в Jetix.** `.claude/skills/plan-day/`, `close-day/`, `focus/`, `company-status.md`, `knowledge-diff.md`. Связь с ActivityWatch (#16) и Toggl (#15) для факта дня.

## §3 Discovery — как находить новые инструменты

Каталог не должен застывать. Где искать новое, не утонув в хайпе:

- **Twitter / X.** Подпишись на 15-20 практиков (build-in-public, AI-tooling, indie-hackers). Лучший сигнал — не анонсы, а «я полгода юзаю X, вот что реально работает». Клипай треды через web clipper (#4) → `/ingest` на разборе.
- **Hacker News.** Раздел "Show HN" и комменты под Launch HN — там инструменты с технической рефлексией. Фильтр качества: читай не пост, а критику в комментах. Раз в неделю — 15 минут на топ-страницу.
- **Reddit.** Сабреддиты `r/productivity`, `r/Notion`, `r/LocalLLaMA`, `r/ArtificialIntelligence` для нишевых лайфхаков. Сигнал — посты «my setup» с конкретикой, а не «top 10 tools 2026».
- **Community recommendations.** Самый сильный канал — рекомендация от человека, который решает похожую задачу. Заноси такие в CRM (§7 offers / §8 asks) и через voice → CRM draft (#14). Когда несколько людей независимо называют один инструмент — это Schelling-сигнал, стоит попробовать.
- **Правило фильтра (важно).** Не внедряй инструмент сразу в прод-флоу. Сначала — гипотеза в hypothesis tracker (#11): «инструмент X сократит время на Y». Тест 1-2 недели. Подтвердилось → добавляешь сюда (append-only, новая запись). Не подтвердилось → закрываешь гипотезу, инструмент не трогаешь. Так каталог растёт только проверенным.
- **Discovery как ingest.** Любую находку прогоняй через `/ingest` — пусть инструмент-кандидат станет страницей wiki со связями. Тогда `/ask "чем заменить таймтрекинг"` потом сам поднимет твои же находки.

## §4 Как companion стыкуется с 3 слоями

Не каждый инструмент нужен на каждом слое. Маленькая таблица — что где полезнее всего:

| Инструмент | Layer 1 Personal Core | Layer 2 Team Collaboration | Layer 3 Universal Business |
|---|:---:|:---:|:---:|
| 🎙️ Voice pipeline (#2) | ★★★ ядро захвата | ★★ командные заметки | ★★ заметки со встреч |
| 📱 Telegram processor (#1) | ★★★ мысли на бегу | ★★ быстрый командный inbox | ★ |
| 📥 Mistral OCR (#3, #17) | ★★ книги/доки | ★ общие материалы | ★★★ контракты/документы |
| ✂️ Web clipper (#4) | ★★★ личный read-later | ★★ shared reading list | ★★ research-материалы |
| 📊 Mermaid (#5, #19) | ★★ личные схемы | ★★★ объяснить команде | ★★★ процессы/архитектура |
| 🎨 Miro / Mural (#6) | ★ | ★★★ co-creation на созвонах | ★★★ воркшопы с клиентом |
| 🖼️ Excalidraw (#7) | ★★★ быстрый скетч | ★★ | ★ |
| 📈 Notion charts (#8) | ★★ личный дашборд | ★★★ командные метрики | ★★★ бизнес-аналитика |
| 🧠 Claude/GPT synthesis (#9) | ★★★ daily синтез | ★★ командные решения | ★★★ стратег. решения |
| 🔍 OFFLINE /ask (#10) | ★★★ «что я знаю» | ★★ командная KB | ★★ |
| 💡 Hypothesis tracker (#11) | ★★ личные ставки | ★★ командные гипотезы | ★★★ бизнес-гипотезы |
| 🗂️ Wiki ingest (#12) | ★★★ личная KB | ★★★ общая KB | ★★ |
| 🤖 ROY swarm (#13) | ★★ multi-angle для себя | ★★ ревью команды | ★★★ стратег. ревью |
| 📅 CRM voice (#14) | ★★ личная сеть | ★★★ командный CRM | ★★★ клиенты/партнёры |
| ⏱️ Toggl (#15) | ★★★ личный фокус | ★★ командные часы | ★★ биллинг по часам |
| 📊 ActivityWatch (#16) | ★★★ self-review | ★ | ★ |
| ✅ Lint (#18) | ★★ здоровье личной KB | ★★ командная консистентность | ★★ |
| 🌅 AI review / plan-day (#20) | ★★★ рамка дня | ★★ командный standup | ★★ |

**Как читать.** ★★★ = ядро этого слоя, бери первым; ★★ = полезно; ★ = опционально. Логика: **Layer 1 (Personal Core)** живёт на захвате (voice, telegram, ingest) и self-review (ActivityWatch, plan/close-day). **Layer 2 (Team)** усиливается коммуникацией (Miro, mermaid, charts) и общим CRM. **Layer 3 (Universal Business)** опирается на документы (OCR), стратегические инструменты (ROY swarm, hypothesis, synthesis) и клиентский CRM.

> **Принцип companion'а.** Эта страница не привязана к одному слою — её подключают рядом с любым. Взял Layer 1 отдельно (standalone) — берёшь левую колонку. Соединил Layer 1 + Layer 3 (founder personal + business) — берёшь обе. Инструменты переносимы; слой определяет только приоритет.
