# Над бизнес-проектами можно работать теми же принципами, что и над кодом

## Точная формулировка

Принципы совместной разработки кода (версионирование, ветвление, review, merge, курированный доступ) применимы к бизнес-проектам — стратегиям, стартапам, запускам продуктов, исследованиям — и дают сопоставимые эффекты: прозрачность, скорость, накопление знаний.

## Контекст

Утверждение возникло в рамках размышлений о том, как Jetix может построить сообщество предпринимателей с работой над проектами, а не просто чат. Отсылает к успеху GitHub и Linux как доказательству того, что такая модель масштабируется.

## Аргументы за

- Прецедент: GitHub + open source сделали разработку ПО массово-коллаборативной.
- Audit trail и review перед merge — то, чего требуют инвесторы, регуляторы, партнёры.
- Версионирование стратегии / offer / pricing уже неявно делается в документах; git делает это явным.
- Параллели в знании-менеджменте: wiki-driven engineering → wiki-driven business (Karpathy LLM Wiki).

## Аргументы против

- Бизнес-артефакты — не чистый текст: много контекста, live-переговоры, конфиденциальность.
- Низкий процент предпринимателей, готовых работать в git-like интерфейсе.
- Часть решений принимается offline/verbal — git не улавливает.

## Что опровергнуло бы это утверждение

- Серия попыток (Notion, Slab, Coda) не смогла создать массовый коллаборационный паттерн для бизнеса — если эта область уже «сгорела», утверждение слабее.
- Эмпирически: Jetix-команды, которые пробовали project-as-repo, продемонстрировали худшую скорость, чем обычный чат + Notion.
