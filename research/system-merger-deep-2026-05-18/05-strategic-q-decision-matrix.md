## §3 Per-option deep analysis

### §3.1 Option A — Outsource (consultancy)

**Pros:**
1. **Predictable revenue** Day-1; cash flow positive within 6-12 months of first engagement.
2. **Lower constitutional risk:** FPF stays internal, partial disclosure per engagement; R12 enforced per contract.
3. **Faster learning cycle:** 3-5 engagements compress pattern-recognition vs. waiting for platform critical mass.
4. **Lower upfront investment:** no need to build full platform tooling Day-1.
5. **Defensible vs. AI-disruption:** consultancy = embedded human judgment; harder to commoditise.

**Cons:**
1. **Linear scaling:** capacity-bottlenecked; Jetix team = scaling ceiling.
2. **Lock-in to consultancy positioning:** harder to pivot to platform later if A becomes identity.
3. **Lower margin long-term:** consultancy margins compress with competition.
4. **Less optionality value:** doesn't compound on FPF as universal language; FPF stays as Jetix-internal tool.

**Use case examples:**
- "AcmeCorp (€500M revenue) wants to acquire BetaAI (€20M) — hire Jetix to design FPF-mediated integration."
- "Two mid-size SaaS companies discussing merger — Jetix consults on day-1 / 100-day plan with FPF constraint catalog."

**First-customer profile:**
- Mid-to-large enterprise (€100M+ revenue), recent failed integration scar, M&A pipeline ≥1 deal/year, DACH/EU geography (Ruslan's network).
- Sponsor: Head of M&A or Chief Strategy Officer.
- Pain: "Our last 2 mergers underdelivered synergy; we need a different approach."

**Implementation complexity:** LOW — needs 1 case study + sales pitch + 1 reference architecture. 3-6 months to first paid engagement plausible.

**Cross-precedent reference (Phase 1):** Option A ≈ **McKinsey M&A practice + Cisco internal M&A team model** — proven business model; scaling laws well-understood; commoditization risk known.

### §3.2 Option B — Platform (Jetix-FPF-as-platform-companies-come-to)

**Pros:**
1. **Exponential scale potential:** network effect; each merger event compounds (n² connections).
2. **Higher margin long-term:** software margins > consultancy margins.
3. **R12 enforced systemic:** Ethereum smart contracts enforce constitutional discipline platform-wide — qualitative differentiation vs. consultancy.
4. **FPF as moat:** more usage → more F-G-R provenance → more authority → more usage (compounding flywheel).
5. **Optionality maximum:** platform supports many merger types; can specialize later.
6. **Conway's-Law-friendly:** FPF embedded in communication infrastructure → mergers naturally adopt FPF structure.

**Cons:**
1. **Chicken-and-egg adoption:** platform value low until critical mass; critical mass requires platform value.
2. **Higher upfront investment:** full tooling (lint, audit, smart contracts, SDK, mock environment) before revenue.
3. **Slower revenue ramp:** 12-24 months to material revenue; capital intensive.
4. **Regulatory risk:** programmable enforcement = potential securities / financial-instrument regulation depending on jurisdiction.
5. **Trust burden:** "platform that mediates mergers" requires high trust threshold — slower B2B sales.

**Use case examples:**
- "Anthropic and Sber AI agree on FPF-mediated research collaboration — both onboard Jetix-FPF platform; smart contracts handle IP-sharing royalty distribution; audit trail public."
- "10 mid-size AI startups form federation via Jetix-FPF — each opts into namordnik baseline; QF revenue distribution applied."

**First-customer profile:**
- Visionary tech founder (10-100 employees), open-source/crypto-aligned, R12 / Mondragón / Plurality conceptually aligned.
- Sponsor: founder or CTO.
- Pain: "We want to collaborate / merge but trust mechanisms are insufficient; cooperative governance is what we want."

**Implementation complexity:** HIGH — needs full SDK, smart contract layer, governance UI, audit infrastructure, 5+ pilots for critical mass. 12-24 months investment.

**Cross-precedent reference (Phase 1):** Option B ≈ **USB-C 10y adoption + GitHub GraphQL anchor + Mondragón federation 68y** — high upside; long ramp; needs anchor tenant + open governance + tooling ecosystem.

### §3.3 Option C — Hybrid (broker + platform; phase-gated)

**Pros:**
1. **Maximum optionality preservation:** can collapse to A if B fails, or transition to B if A produces enough learnings.
2. **Phase 1 cash flow positive** (broker model) funds Phase 2 platform investment.
3. **Risk-managed:** broker mergers de-risk platform claims (constraint catalog, translation rules, R12 enforcement) before scaling.
4. **Anchor-tenant pattern aligned with USB-C / GitHub GraphQL precedent:** first 3-5 broker mergers become reference cases for platform.
5. **R12 phase-evolves:** per-contract (broker) → systemic (platform) — same constitutional commitment, different enforcement infrastructure.
6. **Repeat-acquirer pattern:** Jetix accumulates merger expertise per Phase 2 §3.1 success pattern, then productizes.

**Cons:**
1. **Strategy hedge = strategy clarity dilution:** "we're both consultancy and platform" can confuse market positioning.
2. **Resource split:** consulting team + platform engineering = higher fixed-cost than either pure A or pure B.
3. **Transition moment fraught:** decision "when to scale platform" = high-stakes call without clear trigger.
4. **Brand ambiguity:** consultancy brand and platform brand have different cultures (services vs. SaaS DNA).

**Use case examples:**
- Phase 1 (months 0-12): "Jetix consults on first 3-5 mergers using FPF protocol. Open-sources tooling progressively as pattern stabilizes."
- Phase 2 (months 12-36): "Jetix launches Jetix-FPF platform with first 5 reference mergers as case studies. Platform tier subscriptions + QF distribution layer."
- Phase 3 (months 36+): "Platform reaches 50+ users; consultancy de-emphasized; QF flywheel operational."

**First-customer profile:**
- Phase 1: same as Option A.
- Phase 2: same as Option B.
- **Transition trigger:** 5 successful broker mergers OR Workshop Phase 3 ≈ Network State substrate emergence (per memory `project_balaji_outreach_target.md`).

**Implementation complexity:** MEDIUM — phased; first phase = Option A complexity; second phase = Option B complexity layered on learned foundation.

**Cross-precedent reference (Phase 1):** Option C ≈ **AWS evolution (managed services 2006 → platform), Stripe (engineering consulting → API platform), Cisco (internal M&A → standardized playbook → ecosystem partner)** — proven hybrid pattern.

## §7 Anti-pattern warnings

1. **"Hybrid" as procrastination:** Option C can mask indecision. Set explicit Phase 1 → Phase 2 trigger criteria; honor them.
2. **Pure-A under-investment in FPF tooling:** if A persists too long, FPF doesn't develop platform-grade affordances → cannot transition later.
3. **Pure-B over-investment without revenue:** platform-first without broker learning = high risk of building wrong abstractions.
4. **R12 dilution under cash flow pressure:** in any option, R12 must remain mandatory baseline. Negotiating-away R12 to close a deal = constitutional violation per FUNDAMENTAL §6.1 candidate rule 12.
5. **Brand confusion in C:** mitigate by clear separation of "Jetix Advisory" (Option A frame) vs "Jetix-FPF Platform" (Option B frame); shared FPF substrate underneath.
