## §1 Setup

### §1.1 Company A — "AlphaResearch AI" (hypothetical)

- **Archetype:** Western open-source research-focused (Anthropic-like / Mistral-like / Cohere-like).
- **Geography:** US / EU; HQ likely SF / Paris / London.
- **Size:** 80-200 employees; 40% researchers + 30% engineers + 30% rest.
- **Culture:** flat structure; long-horizon thinking; OSS release default; English-language; high autonomy expected.
- **Strategic goal:** publish foundational research; build defensible models; OSS partial release as community contribution.
- **R12 alignment:** moderate-high (many such orgs have public benefit / non-profit components).
- **Decision rights:** founder-led + research lab heads; Slack-async + weekly all-hands.

### §1.2 Company B — "BetaAppliedAI" (hypothetical)

- **Archetype:** RU/CIS or DACH enterprise applied AI (Sber AI / Yandex AI / Tinkoff AI / SAP AI / Deutsche Telekom).
- **Geography:** Moscow / Berlin / Munich; large parent company subsidiary.
- **Size:** 300-1000 employees; 20% researchers + 50% engineers + 30% domain/sales.
- **Culture:** hierarchical; quarterly OKR; closed-source default; Russian/German-language; strong execution focus.
- **Strategic goal:** deploy AI products into enterprise customers; revenue from B2B contracts; regulatory compliance critical.
- **R12 alignment:** low-to-moderate (extractive corporate incentives present; large parent organization).
- **Decision rights:** CEO + senior leadership; quarterly steering; less async.

### §1.3 Common ground

- Both = information-processing systems.
- Both have ML/AI engineers (per `research/ml-ai-engineers-2026-05-18`) who share 7-step workflow.
- Both interested in synergy: A wants enterprise revenue + applied data; B wants research depth + OSS credibility.
- **Hypothetical merger motivation:** joint venture / federation / strategic partnership (not full acquisition) — preserves both identities; FPF symbiosis mode (per Phase 2 §4.1 Haspeslagh-Jemison matrix).

## §2 5-phase merger protocol (Jetix-FPF)

### §2.1 Phase 1 — Discovery (months 0-2)

**Goal:** establish compatibility surface; declare mode (FULL-FPF / PARTIAL / OPAQUE); identify behavioural variety V_I.

**Activities:**
- Joint Discovery Workshop (3-5 days; hosted by Jetix advisor in neutral location e.g. Vienna / Lisbon).
- Cultural audit (Hofstede + Trompenaars dimensions; per Phase 2 §3.2).
- Strategic logic declaration (Symbiosis mode per Haspeslagh-Jemison).
- Information-system inventory: data assets, role taxonomy, decision rights, key processes.
- Mode selection (FULL / PARTIAL / OPAQUE per Phase 3 §B.4).
- Variety mapping V_I (behavioural variety per Ashby).

**Deliverable:** Discovery Report (~20 pages) signed by both CEOs.

**Acceptance criteria:**
- Mode declared explicitly (with rationale).
- ≥80% of relevant V_I categorized.
- Mutual acceptance of FPF as protocol substrate.
- Strategic-logic statement signed.

**Falsifier:** Cultural distance audit shows incompatibility insurmountable (Hofstede composite > threshold T1; see §3 below) OR Strategic logic cannot be agreed → halt merger; consider OPAQUE-BRIDGE downgrade or de-merge.

**Strategic Q application:**
- **Option A (Outsource):** Jetix runs Discovery as €50-100K consulting engagement; deliverable owned by both companies.
- **Option B (Platform):** Both companies self-onboard Jetix-FPF platform; Discovery is platform workflow (auto-generated); minimal Jetix human time.
- **Option C (Hybrid):** Jetix advisor leads; platform tooling supports (Discovery Report templated; cultural audit dashboarded).

### §2.2 Phase 2 — Constraint negotiation (months 2-4)

**Goal:** Both parties commit to constraint catalog (намордник per Phase 3 §A).

**Activities:**
- Constraint catalog session (host = Jetix-FPF baseline C1-C10; both parties = incoming relative to FPF).
- Per-constraint negotiation: scope, enforcement, violation resolution.
- Smart contract deployment to testnet (if Option C / B).
- Exit token issuance (C10) — both parties receive non-revocable fork-and-leave tokens.
- Constitutional posture signing ceremony (cryptographic + legal).

**Deliverable:** Signed namordnik catalog + smart contract testnet deployment + exit tokens issued.

**Acceptance criteria:**
- All 10 baseline constraints accepted (or principled exceptions documented + Ruslan ack).
- Smart contract testnet operational; on-chain compliance monitor active.
- Exit tokens cryptographically held by both parties.
- Legal review of constraint catalog by both parties' counsel.

**Falsifier:** Either party rejects R12 anti-extraction (C1) or fork-and-leave (C10) → STOP. These are non-negotiable per FUNDAMENTAL §6.1 candidate rule 12.

**Strategic Q application:**
- **Option A:** Jetix custom-crafts constraint catalog per merger; tier-3 enforcement (governance) emphasized over tier-1 (cryptographic).
- **Option B:** Standardized constraint catalog from platform; both parties opt-in/opt-out per item.
- **Option C:** Baseline standardized; per-engagement extensions custom.

### §2.3 Phase 3 — Bridge layer setup (months 4-7)

**Goal:** Build USB-C port translation rules (per Phase 3 §B); operational interoperability.

**Activities:**
- Per-entity-class translation rules authored (Person, Project, Decision, Artefact, Customer, etc.).
- ACL implementation in each system (Anti-Corruption Layer per Evans DDD).
- Audit hook deployment (append-only event log; chain anchoring).
- Mode-specific affordances:
  - FULL-FPF: both adopt FPF-native data models.
  - PARTIAL-FPF: translation rules + ACL on both sides.
  - OPAQUE-BRIDGE: minimal API contract + black-box wrapper.
- Translation rule library reviewed by FPF-Steward (Левенчук or delegate).

**Deliverable:** Operational bridge with audit trail flowing; ≥5 entity-class translation rules attested at F3.

**Acceptance criteria:**
- Round-trip translation test passes for ≥5 entity classes (per Phase 3 §B.8 FM-U3 mitigation).
- Translation overhead measured <15% (per H-SM-12).
- Audit trail anchoring on-chain functional.
- FPF-Steward attestation on rule library.

**Falsifier:** Translation overhead >20% (FM-U4) OR round-trip lossy beyond acceptable threshold (FM-U3) OR ACL leaks contaminate host model (FM-U7) → re-architect bridge or downgrade mode.

**Strategic Q application:**
- **Option A:** Jetix engineers build rules custom; rules become engagement deliverable (open-source or proprietary per contract).
- **Option B:** Rules drawn from platform library; new rules authored contribute to platform corpus (gated by F-G-R).
- **Option C:** Platform library seeded; rule extensions Jetix-supported.

### §2.4 Phase 4 — Pilot integration (months 7-12)

**Goal:** Execute 1 joint project; validate merger protocol operationally; collect empirical evidence for synergy claim.

**Activities:**
- Pilot project selection (small scope; reversible; both parties contribute resources).
- Pilot team composition (3-5 people from each side; FPF roles assigned).
- Operational cadence (weekly cross-team standup under FPF protocol; quarterly retro).
- Audit event review (continuous monitoring of namordnik compliance + bridge correctness).
- Synergy measurement: time-to-deliverable, quality metrics, retention.

**Deliverable:** Pilot project deliverable + retrospective report + audit findings.

**Acceptance criteria:**
- Pilot delivered within ±20% original estimate.
- Retention 100% during pilot (no attrition).
- Zero F8 (constitutional) violations.
- ≤1 F4 (high) violation per quarter; remediated.
- Both parties' satisfaction ≥7/10 (qualitative survey).

**Falsifier:** Multiple F8 violations OR retention <90% OR pilot delivers <50% planned OR either party invokes fork-and-leave within 12 months → de-merge candidate.

**Strategic Q application:**
- **Option A:** Jetix embedded as integration lead; ~30% Jetix time on pilot.
- **Option B:** Self-serve; Jetix monitors via platform dashboards; ~5% Jetix time.
- **Option C:** Hybrid; ~15% Jetix time; platform-supported.

### §2.5 Phase 5 — Scale-or-de-merge decision (month 12)

**Goal:** Empirical decision based on Phase 4 evidence + audit findings + both parties' strategic review.

**Decision options:**
1. **SCALE:** Expand merger scope; additional pilots / teams / domains; commit to multi-year integration.
2. **MAINTAIN:** Keep current scope; refine bridge + constraints; do not expand or contract.
3. **PARTIAL DE-MERGE:** Reduce integration scope (e.g., end pilot, retain federation membership only).
4. **FULL DE-MERGE:** Exercise fork-and-leave; both parties exit cleanly; preserved artefacts archived.

**Decision criteria (weighted):**
- Synergy realized (Phase 4 metrics) — 30%.
- Cultural integration progress (qualitative + Hofstede re-audit) — 20%.
- Constraint compliance (audit trail) — 20%.
- Strategic logic still valid (no major external change) — 15%.
- Cost vs. value (financial + opportunity cost) — 15%.

**Deliverable:** Decision document; if scale → Phase 6 (new scope); if de-merge → exit ceremony + archive.

**Falsifier:** Decision deadlocked (parties disagree) → arbitration via FPF-Steward + Ruslan + neutral third party; if unresolvable → default to PARTIAL DE-MERGE (preserves optionality).

**Strategic Q application:**
- **Option A:** Decision moment = end of Jetix engagement; future engagements separately scoped.
- **Option B:** Decision integrated into platform workflow; scaling = additional platform tier.
- **Option C:** Decision documented + platform-shared; case-study published (with permission).

## §6 Anti-patterns to avoid (per Phase 2 §8)

1. **Skip Phase 1 cultural audit** = AOL-Time Warner failure pattern repeated.
2. **Compress timeline** (e.g., 6 months instead of 12) = retention failures + violation spike.
3. **Make R12 negotiable** to close deal = constitutional violation per FUNDAMENTAL §6.1 candidate rule 12.
4. **Centralize integration in single Jetix person** = bus factor + IP-1 violation.
5. **Treat first merger as production-grade** (P0) — designate as R&D / reference implementation (acceptable failures).
