## §A. Cross-precedent hypotheses (H-SM-1..10)

### H-SM-1 — USB-C 10y adoption velocity adapts к FPF
**Claim:** FPF System Merger Protocol can reach ubiquity (≥50% of relevant merger events using FPF) within 10 years if anchor-tenant pattern + EU regulatory push + open spec mirror USB-C trajectory.
**F:** F2 (extrapolative; informed by USB-C precedent).
**G:** EU + DACH AI/info-system M&A market.
**R:** refuted_if (a) no anchor tenant identified within 24 months OR (b) FPF adoption <5% by year 5 OR (c) no regulatory push emerges supporting protocol-mediated merger standards.
**Test design:** quarterly count of FPF-protocol-attested mergers; benchmark vs. USB-C adoption curve year-by-year.
**Acceptance:** adoption curve matches USB-C ±50% in years 2-5.
**Refutation:** adoption flat or declining year 3+.
**Cross-ref:** Phase 1 §A.

### H-SM-2 — TCP/IP layered architecture analog applies к FPF
**Claim:** FPF 5-primitive stack (A.1 / U.Method / A.6.B / A.14 / E.5) provides replaceability per layer analogous to TCP/IP — primitives can be revised without breaking entire protocol.
**F:** F2.
**G:** FPF protocol evolution over 5-10 years.
**R:** refuted_if a single primitive change (e.g., A.6.B revised) requires breaking changes к ≥2 other primitives in production.
**Test design:** plan + execute 1 primitive revision; measure breaking-change blast radius.
**Acceptance:** ≤1 primitive affected per revision.
**Cross-ref:** Phase 1 §B.

### H-SM-3 — HTTP critical mass threshold for FPF platform mode ≈ 5-10 mergers
**Claim:** Mosaic-era HTTP needed ~100 sites; FPF being higher-per-instance complexity needs only 5-10 critical mass for platform mode network effect.
**F:** F2.
**G:** Strategic Q Option B platform mode.

**Acceptance:** 5-10 mergers + ≥60% report counterfactual unavailable.
**Cross-ref:** Phase 1 §C.
