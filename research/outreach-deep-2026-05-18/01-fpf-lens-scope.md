# Phase 0 — FPF Lens Scope Definition (Outreach System Scalable)

> **R1 surface.** Phil critic + sys cybernetics dispatch. Define WHAT we research'им + ON WHICH FPF layer ДО Phase 1.

## §1 Surface 4 candidate scopes simultaneously (NOT pick)

Per FPF lens (per `00-JETIX-FPF-MASTER-2026-05-17.md` + concept doc D §2), Outreach System Scalable can be researched через 4 distinct FPF lenses. Brigadier surfaces ALL 4; Ruslan picks operational primary.

### §1.1 Scope A — Outreach-as-system (U.System holonic, A.1)

- **FPF primitives:** A.1 + U.System + U.Capability + U.Function.
- **Treatment:** outreach engine = sub-system of Jetix platform; holonic — has internal parts (operators / scripts / CRM / metrics) and is part of larger Jetix substrate (cohort funnel → Workshop → Clan).
- **What we research:** internal subsystem boundaries; external interfaces (CRM, Workshop, hackathon platform); throughput / latency / quality metrics; failure modes; capacity limits.
- **Output emphasis:** Phase 3 (10-team) + Phase 4 (100-trained) as system architecture; Phase 6 (target audience) as input load characterisation.
- **Strength:** scalability discipline (Ashby requisite variety; VSM). Suitable for engineering review.
- **Weakness:** risks reifying outreach as black-box; under-emphasises craft + relationship qualities.

### §1.2 Scope B — Outreach-as-method (U.MethodDescription + U.Method)

- **FPF primitives:** U.MethodDescription (canonical doc) + U.Method (per-execution instance).
- **Treatment:** outreach = methodology canonical document (one of N Jetix methodologies, parallel to ML 7-step / FPF Workshop / Voice-pipeline / CRM).
- **What we research:** method stages; per-stage acceptance predicate; method portability across target classes; method version evolution; method documentation discipline (Karpathy LLM Wiki pattern).
- **Output emphasis:** Phase 1 (cross-precedent — what methods exist) + Phase 2 (6-resources framework as method-input typology) + Phase 7 (hypothesis bank as method-test design).
- **Strength:** transferable + teachable; aligns with Workshop apprenticeship + Master Workshop training.
- **Weakness:** risks over-formalisation; outreach has tacit/craft layer (per `research/deepening-2026-05-18/14-tacit-explicit-tps-mechanism.md`) that pure method-canonical under-captures.

### §1.3 Scope C — Outreach-as-cohort-mechanism (A.2 Role + U.SpeechAct + U.PromiseContent)

- **FPF primitives:** A.2 U.Role (per-target role-type per voice anchor Thread 15) + U.SpeechAct (per-touch invocation primitives) + U.PromiseContent (per-engagement commitment).
- **Treatment:** outreach = per-target invocation mechanism; each engagement is a speech-act with promise-content + role assertion (operator) + role recognition (target).
- **What we research:** speech-act typology (opening / value / urgency / CTA per concept doc D §7.2); per-target-class role assertion variance; promise-content commitment ladder; relationship state-machine (cold → warm → discovery → cohort-entry).
- **Output emphasis:** Phase 5 (personalisation mechanism) + Phase 6 (per-class script variance) + Phase 7 (per-touch hypothesis testing).
- **Strength:** authenticity preservation; R12 anti-extraction natural fit (speech-act = consent boundary explicit).
- **Weakness:** harder to operationalise at scale (3000+ targets) without flattening to template.

### §1.4 Scope D — Outreach-as-process (A.16 Work-as-process)

- **FPF primitives:** A.16 + U.Work + U.WorkProduct + B.5.1 Explore state.
- **Treatment:** outreach = process documentation; per-stage outputs; per-operator daily/weekly workflow; pipeline stages explicit.
- **What we research:** pipeline stage taxonomy (research → script → record → personalise → send → follow-up → close); per-stage work-product; per-stage acceptance gate; cycle-time per stage; bottleneck identification.
- **Output emphasis:** Phase 3 (per-role daily workflow) + Phase 4 (per-trainee process discipline) + Phase 5 (per-engagement process audit).
- **Strength:** measurable; CRM-substrate-aligned (status pipeline tracking); throughput-monitorable.
- **Weakness:** process-discipline can mask quality drift (Ashby variety attenuation if process over-specified).
