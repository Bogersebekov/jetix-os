## §1 Scope 1 — Hackathon-as-system (U.System holonic; A.1)

### §1.1 FPF mapping
- **A.1 System** (primary): Jetix hackathon platform = systemic entity Phase 0 inventory O-25 candidate. Holonic decomposition: platform → events → teams → participants → artefacts.
- **U.System** (holonic): each event = sub-system; platform = super-system holding events; participant = sub-sub-system.
- **A.0 Substrate**: Cloud Cowork compute + FPF protocol + Ethereum DAO substrate.

### §1.2 Implications для Jetix
- Platform = stable substrate; events = ephemeral instances.
- Sponsor relationships, mentor pool, IP discipline, R12 anti-extraction = system-level invariants applicable к all events.
- Versioning: platform v1.0 → v2.0; events tagged по platform version.
- Health monitoring (Part 8 Foundation): platform-level SLIs (event count, participant retention, sponsor renewal, mentor density).

### §1.3 Cross-precedent fit
- **High fit:** MLH (Mike Swift; ~150 events/year; платформа-level substrate). [src: `research/hackathon-deep-2026-05-18/05-mike-swift-mlh-deep-profile.md`]
- **High fit:** Devpost (~1M users; platform-as-substrate). [src: concept doc A §4]
- **Med fit:** ETHGlobal (5-6 cities/year; flagship-driven).

### §1.4 Per-scope FPF load
A.1 system + U.System holonic + A.0 substrate = high primitive load (12+ Foundation parts touched per cycle).

## §2 Scope 2 — Hackathon-as-method (U.Method + A.3.1)

### §2.1 FPF mapping
- **A.3.1 Method** (primary): hackathon = canonical method для compressed-time problem-solving с specific participant + mentor + sponsor protocols.
- **U.MethodDescription**: canonical hackathon method spec (e.g. day-rhythm spec, month-rhythm spec).
- **A.3.2 Practice** (instance per event): each event = practice instantiation of method.

### §2.2 Implications для Jetix
- Multi-rhythm method taxonomy (day / weekend / week / month / quarter / year) per text_009 ¶10. [src: text_009:¶10]
- Method library: reusable templates; brigadier dispatches «execute day-rhythm method instance».
- Method evolution: capture lessons → patch method canonical → next instantiation incorporates.
- FPF B.3 F-G-R applied to method: F2 surface for new variant, F3 при single successful run, F5 при 5+ successful runs.

### §2.3 Cross-precedent fit
- **High fit:** Kaggle (competition method; ML-specific). [src: concept doc A §4]
- **High fit:** Startup Weekend (54-hour method canon; replicated в 700+ cities).
- **Med fit:** ETHGlobal flagship hackathons (method evolution через iterations).

### §2.4 Per-scope FPF load
A.3.1 method + U.MethodDescription + A.3.2 practice = medium primitive load. Compatible с Foundation Part 5 (compound learning).

## §3 Scope 3 — Hackathon-as-event-class (A.15 Work × A.16 Process)

### §3.1 FPF mapping
- **A.15 Work-as-artefact** (primary): per-event artefact = solution / project / prize allocation / IP package.
- **A.16 Work-as-process** (primary): event orchestration = initiate → match → solve → reward → recurse cycle.
- **A.17 Trajectory**: per-team journey from registration через submission.
- **U.PromiseContent** (sponsor commitment): pre-event sponsor promise; mid-event mentor promise.

### §3.2 Implications для Jetix
- Per-event success criteria = artefact quality × process quality.
- Falsifiers per event (per concept doc A §6 HP-T1..T5): clear refutation_conditions.
- Operational blueprint (Phase 5) = event-class instance spec.
- A.15 + A.16 separation: artefact retention by participant; process owned by Jetix platform.

### §3.3 Cross-precedent fit
- **High fit:** Y Combinator Demo Day (event-bounded; Demo Day = process culmination + artefact crystallisation).
- **High fit:** ETHGlobal events (per-event prize allocation + winners showcase).
- **High fit:** NeurIPS competitions (per-challenge artefact + leaderboard process).

### §3.4 Per-scope FPF load
A.15 + A.16 + A.17 + U.PromiseContent = medium-high. Strong fit для Part 7 (project lifecycle substrate).

## §4 Scope 4 — Hackathon-as-cohort-mechanism (A.2 Role + U.SpeechAct)

### §4.1 FPF mapping
- **A.2 Role** (primary): participant / mentor / sponsor / investor / organizer / audience / blogger / customer.
- **U.SpeechAct** (recruitment + announcement): «we invite you» speech act; «we award» speech act.
- **A.11 Commons** (cohort = commons): cohort members access shared knowledge pool + connections.
- **E.7 Civic norms** (R12 anti-extraction applied to cohort relations).

### §4.2 Implications для Jetix
- Cohort recruitment = primary growth lever (text_008 «100× multiplier»; text_009 ¶1 «дотянуться до любого человека»).
- 5 cohort segments (Phase 3): bloggers / engineers / investors / mentors / customers.
- Per-cohort outreach script template = U.SpeechAct canonical.
- Retention = cohort-level invariant (Foundation Part 4 §H role taxonomy).
- IP-1 Role≠Executor: cohort role-types abstract; executor bindings RUSLAN-LAYER.

### §4.3 Cross-precedent fit
- **High fit:** MLH (university student cohort recruitment substrate). [src: `research/hackathon-deep-2026-05-18/05-mike-swift-mlh-deep-profile.md`]
- **High fit:** Y Combinator (batch cohort retention model).
- **High fit:** Encode Club (cohort-based Web3 education).

### §4.4 Per-scope FPF load
A.2 + U.SpeechAct + A.11 + E.7 = medium. Cross-cuts с Part 4 (role taxonomy) + Part 10 (external touchpoints).
