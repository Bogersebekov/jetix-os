## §1 Hypothesis A — Jetix во всех хакатонах

### §1.1 Description

**Claim:** Jetix мастера + Jetix methodology присутствуют в major hackathon ecosystems (MLH / Devpost / ETHGlobal / Lablab / Devfolio / etc.) — as participants / mentors / sponsors / observers.

**Forms of presence:**
- **A1 — Participate** — Jetix-staffed teams compete (signal: «we ship»)
- **A2 — Mentor** — Jetix мастера mentor across tracks (signal: «we transfer knowledge»)
- **A3 — Sponsor** — Jetix sponsors specific tracks / API integrations (signal: «we platform-curate»)
- **A4 — Observe** — Jetix staff attend без active role (signal: «we listen / learn»)

Typically combine A1+A2+A4 simultaneously; A3 requires budget commitment.

### §1.2 Information-flow angle (Nexus lens)

Per `01-fundamentals.md` §5.1 six info streams:

| Stream | Jetix benefit at hackathon (A1+A2+A4) |
|---|---|
| Ideas | Inbound: spotted emerging vertical / pattern / problem-class |
| Methods | Inbound: best-practice от sponsor APIs + competing teams; Outbound: Jetix methodology demoed in pitch + team artefacts |
| Connections | Bidirectional: meet potential partners / hire / mentor / sponsor candidates |
| Hiring | Bidirectional: spot Jetix hire candidates AND signal Jetix как employer |
| Investment signal | Inbound: see what investors fund; Outbound: signal Jetix presence в ecosystem |
| Cultural memes | Bidirectional: absorb community language; spread «FPF / Workshop / R12» frames |

**Velocity advantage realized:** Jetix learns + signals в 24-48h what would take weeks/months через other channels.

[src: 01-fundamentals.md §5.1 + this doc empirical]

### §1.3 Dataism risk (Homo Deus lens)

**Risk:** Jetix participation could collapse в «brand impression metrics» — counting hackathons attended without learning compound. Become «hackathon-tourism» (visible но shallow).

**Mitigation:**
- Track explicit post-hackathon learnings (knowledge artifacts в wiki)
- Limit annual hackathon count к manageable depth (e.g., 4-8 events Year-1 not 20+)
- Senior Jetix presence (não junior-only delegation)

[src: 02-homo-deus-jetix-lens §5 critique-1 + this doc]

### §1.5 Pros / Cons

**Pros:**

- ✓ Fastest learning velocity (compound A.16 lineage across events)
- ✓ Lowest IP-exposure (Jetix retains everything)
- ✓ Immediate feedback на FPF / Workshop / R12 messaging

**Cons:**

- ✗ Risk of hackathon-tourism without depth tracking

### §3.2 Information-flow angle ⭐ direct match Harari Nexus thesis

Per Harari Nexus framework: information networks self-amplify when they become substrate for next iteration of info-flow. **Jetix self-bootstrapping hackathon = textbook example of Nexus recursive amplification.**

```
Hackathon-1 (build Workshop UI)
  └─> Workshop UI deployable
       └─> Workshop UI хosts Hackathon-2 (build FPF tooling)
            └─> FPF tooling enables better Workshop curriculum + Hackathon-3 efficiency
                 └─> Hackathon-3 (Pattern Language artefact) more productive
                      └─> Pattern Language artefact = Jetix substrate signal к external
                           └─> Hackathon-4 attracts wider talent
                                └─> ...
```

**Recursive amplification properties:**
- Each event adds substrate; substrate enables next event;
- Compound info-flow about Jetix itself (substrate signaling = self-referencing meta-info)
- Participants become substrate-builders + substrate-users simultaneously
- Cohort builds = «human-army» multiplication (per text_006 «снежный ком 10→100→1000→100k»)

**This is literal text_002 vision implementation** — «новый internet layer для инженеров» bootstrapped via хакатон cycle.
