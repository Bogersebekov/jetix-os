## §1 Plan-mode Primitive 1: Surface candidate

### §1.1 Definition

**Surface candidate** = brigadier-orchestrated cell dispatch produces a direction option (concept doc / AWAITING-APPROVAL packet / §extension / research deep / strategic insight) for Ruslan review. Output never auto-executes; surface is intentional R1 (Ruslan = sole strategist).

### §1.2 FPF mapping

- `U.MethodDescription` — surface candidate = method instance
- `A.2.8 Commitment` — surface = brigadier commitment to deliverable
- `A.2.9 SpeechAct` — surface = brigadier speech-act «here is candidate X»
- `B.3 F-G-R` — surface grade declared explicit (F2/F3 typical)

### §1.3 Operational mechanism (current Jetix implementation)

1. Voice anchor or open-question trigger
2. brigadier reads context (CLAUDE.md + concept parents + voice anchor verbatim)
3. brigadier dispatches relevant cell from matrix (Primitive 2)
4. Cell outputs draft → brigadier-scribe authors prose tagged R1
5. F-G-R declared in frontmatter (Primitive 4)
6. Provenance per claim (Primitive 6)
7. Output to: `decisions/strategic/` OR `research/<run-id>/` OR `swarm/awaiting-approval/`

### §1.4 Failure modes

- **F1:** Strategic prose without voice anchor → R1 violation (rule 1 — AI does NOT strategize)
- **F2:** Candidate auto-promoted to LOCK без ack → IP-1 violation
- **F3:** F-grade overstated (F2 published as F4+) → EP-5 disclosure violation
- **F4:** Cell dispatch mismatch (wrong expert / mode for problem type) → low quality surface

### §1.5 Test design

- Per-cycle: count surface candidates produced; Ruslan ack rate; clarification rate
- Per-week: surface count distribution by category (concept / packet / research / extension)
- Refutation predicate per surface: explicit in frontmatter `R:` field

### §1.6 Refinement candidate (F3+ Phase 8 candidate)

Surface-candidate template extension: add `executor_binding:` explicit field at draft stage (currently implicit via author convention). Phase 8 AWAITING-APPROVAL packet candidate.

## §2 Plan-mode Primitive 2: Cell dispatch matrix (5 × 4 = 20)

### §2.1 Definition

**Cell dispatch matrix** = ROY swarm 5 experts × 4 role-modes = 20 invocation cells. brigadier selects cells per problem domain + role-mode for plan-mode surface generation.

### §2.2 FPF mapping

- `U.System` (holonic sub-system) — matrix = decomposition of cognitive labor
- Foundation Part 4 §H — hub-and-spoke dispatch
- `A.4 Temporal Duality` — plan-mode invocation distinct from execute-mode

**5 experts:** engineering / investor / mgmt / philosophy / systems
**4 role-modes:** critic / optimizer / integrator / scalability (engineering); analogous for others per ROY-ALIGNMENT §3

### §2.3 Operational mechanism

Per problem type, brigadier picks 2-4 cells (rarely all 20). Typical dispatch:
- Strategic concept surface: phil × critic + eng × scalability + mgmt × integrator
- Operational implementation surface: eng × integrator + sys × cybernetics
- Capital allocation surface: investor × roi-frame + phil × critic
- Cross-domain transfer surface: sys × cybernetics + phil × critic + eng × scalability

### §2.4 Failure modes

- **F1:** Cell over-dispatch (all 20 cells for trivial question) → cost + context bloat
- **F2:** Cell under-dispatch (1 cell for complex multi-domain) → biased surface
- **F3:** Wrong expert for domain → low-quality output
- **F4:** AP-6 dissent collapsed (averaging instead of preserving)

### §2.5 Test design

- Per-cycle: average cells dispatched per surface; cost per cell
- Per-week: cell utilisation balance (no cell starved)
- Refutation: if 80%+ surfaces use single cell → matrix not actually utilised

### §2.6 Refinement candidate

Dispatch heuristic table (which cells for which problem type) — currently brigadier judgement-based. Phase 5 Hansei + Kaizen ritual может formalize.

## §3 Plan-mode Primitive 3: Dissent preservation (AP-6)

### §3.1 Definition

**AP-6 dissent preservation** = when cells disagree on candidate framing, **dissent is preserved verbatim** (NOT averaged into consensus). brigadier-scribe authors prose marking dissenting positions as `Counter 1/2/3/4` sections.

### §3.2 FPF mapping

- `B.3 F-G-R` — F grade explicit for each Counter position
- Pillar C Tier 2 rule 7 — agents do NOT negotiate contradictions autonomously
- Pillar C Tier 2 rule 8 — agent does NOT evaluate peer agent without human review

### §3.3 Operational mechanism

Every research doc in this namespace has `## §X Counter-positions (AP-6 dissent preserved)` section. Counter positions tagged with surface resolution (e.g., «Surface: correct» / «Surface: legitimate; further test needed» / «Resolution via §Y framing»).

### §3.4 Failure modes

- **F1:** Counter positions omitted («consensus reached») → AP-6 violation
- **F2:** Counter position dismissed («not worth surfacing») → dissent suppression
- **F3:** Counter position auto-resolved by brigadier → rule 7 violation
- **F4:** Counter position labeled «wrong» without Ruslan review → rule 8 violation

### §3.5 Test design

- Per surface: count Counter positions ≥1
- Per cycle: dissent preservation rate ≥ 95% (target)
- Refutation: any auto-resolved Counter without Ruslan ack note

### §3.6 Refinement candidate

Counter-position metadata schema: add explicit `resolution_required: yes/no` field. Phase 8 candidate.

### §4.2 FPF mapping

- Part 6a Provenance Officer §I.1 F8 schema

### §4.3 Operational mechanism

### §4.4 Failure modes

### §4.5 Test design

- Per doc: F + G + R fields present
- Per cycle: F-grade distribution histogram (most claims F2-F3 expected; F4+ rare)
- Refutation predicate explicitness test: can a 3rd party identify exactly what would refute?

## §6 Plan-mode Primitive 6: §5.5.5 provenance gate (per-claim [src:...])

### §6.1 Definition

**Provenance gate** = every promoted claim carries `[src:<file>:<location>]` citation. brigadier-scribe enforces during prose authoring; brigadier validates before commit.

### §6.2 FPF mapping

- R6 provenance (canonical operational rule)
- Part 6a §I.1 F-G-R Reliability dimension
- FPF B.3 (provenance required for F3+)

### §6.3 Operational mechanism

Inline `[src:filename:section_or_paragraph]` after each promoted claim. Frontmatter `parents:` field at doc level. Per-cycle commit hooks (`swarm/lib/hooks/`) validate.

### §6.4 Failure modes

- **F1:** Claim без `[src:...]` → R6 violation
- **F2:** Source cited but doesn't exist → Halt-Log-Alert F4 + reproducibility fail
- **F3:** Source cited but doesn't support claim → audit gap
- **F4:** Verbatim quote modified silently → R6 + EP-5 dual violation

### §6.5 Test design

- Per doc: grep `[src:` count vs claim count ratio
- Per cycle: claim-without-src count (target = 0)
- Refutation: any orphan citation OR uncited promoted claim

### §6.6 Refinement candidate

Auto-validation hook: `swarm/lib/hooks/provenance-validate.py`. Phase 8 candidate (extending existing `/lint` skill).

## §7 Plan-mode Primitive 7: Word budget enforcement

### §7.1 Definition

**Word budget enforcement** = phase summaries / concept docs / research outputs adhere to declared `word_budget:` frontmatter field. Over-budget = warning; under-budget = OK; over-budget by >20% = halt or compress.

### §7.2 FPF mapping

- `A.4 Temporal Duality` — read-time vs write-time constraint
- WORD COUNT 10K-20K constitutional schema (shared/schemas)
- Compounding-Engineering principle (terse beats verbose)

### §7.3 Operational mechanism

Frontmatter: `word_budget: <N>`. Final line of doc: «Word count: ~<actual> / <budget> budget. Compliant.» (или non-compliant note).

### §7.4 Failure modes

- **F1:** Word budget missing → no enforcement signal
- **F2:** Over-budget by >20% без compression attempt → Compounding violation
- **F3:** Under-budget by >50% → may signal under-developed surface
- **F4:** Word count reported inaccurate → audit gap

### §7.5 Test design

- Per doc: actual word count ≤ budget × 1.1
- Per cycle: budget compliance rate
- Refutation: doc claiming compliance but >budget × 1.2

### §7.6 Refinement candidate

Auto word-count via commit hook. Phase 8 candidate.

## §8 Plan-mode Primitive 8: Halt-Log-Alert detection

### §8.1 Definition

**Halt-Log-Alert** = F8/F4/F2 grade integrity violation → halt ≤1s/≤5s/≤60s respectively; emit to `swarm/approvals/log.jsonl` + Part 8 SLI alert.

### §8.2 FPF mapping

- Part 6b §I.2 LOCKED (constitutional)
- `halt-log-alert.schema.json` (shared/schemas)
- FPF §5.5 fail-loud (no silent swallowing)

### §8.3 Operational mechanism

Detection triggers (plan-mode-relevant):
- F8 — Foundation file modification without ack trail
- F4 — Default-Deny novel action attempted
- F2 — F-grade overstated (post-hoc audit)

Action: halt immediately within grade window; append to `swarm/approvals/log.jsonl` (JSONL append-only); Part 8 SLI alert.

### §8.4 Failure modes

- **F1:** Violation silently swallowed → §5.5 violation + corrigibility breach
- **F2:** Halt delayed beyond grade window → SLI degradation
- **F3:** Log append fails → reproducibility loss
- **F4:** False positive halt (legitimate action mis-categorised) → throughput loss

### §8.5 Test design

- Per-week: zero F8 violations
- Per-month: F4/F2 violations <5 each
- Refutation: any silent swallow detected in audit
