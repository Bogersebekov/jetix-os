## <span id="id2962084"></span>Don't Just Do Something, Stand There!

</div>

</div>

The most powerful optimization technique in any programmer's toolbox is
to do nothing.

This very Zen advice is true for several reasons. One is the exponential
effect of Moore's Law — the smartest, cheapest, and often
<span class="emphasis">*fastest*</span> way to collect performance gains
is to wait a few months for your target hardware to become more capable.
Given the cost ratio between hardware and programmer time, there are
almost always better things to do with your time than to optimize a
working system.

We can get mathematically specific about this. It is almost never worth
doing optimizations that reduce resource use by merely a constant
factor; it's smarter to concentrate effort on cases in which you can
reduce average-case running time or space use from
O(<span class="emphasis">*n*</span><sup>2</sup>) to
O(<span class="emphasis">*n*</span>) or
O(<span class="emphasis">*n*</span> log
<span class="emphasis">*n*</span>),<sup>\[<a href="ch12s01.html#ftn.id2962140" id="id2962140">112</a>\]</sup>
or similarly reduce from a higher order. Linear performance gains tend
to be rapidly swamped by Moore's
Law.<sup>\[<a href="ch12s01.html#ftn.id2960862" id="id2960862">113</a>\]</sup>

Another very constructive form of doing nothing is to not write code.
The program can't be slowed down by code that isn't there. It can be
slowed down by code that <span class="emphasis">*is*</span> there but
less efficient than it could be — but that's a different matter.

<div class="footnote">

<sup>\[<a href="ch12s01.html#id2962140" id="ftn.id2962140">112</a>\]</sup>
For readers unfamiliar with O notation, it is a way of indicating how
the average running time of an algorithm changes with the size of its
inputs. An O(1) algorithm runs in constant time. An
O(<span class="emphasis">*n*</span>) algorithm runs in a time that is
predicted by `A`<span class="emphasis">*`n`*</span>` + C`, where `A` is
some unknown constant of proportionality and `C` is an unknown constant
representing setup time. Linear search of a list for a specified value
is O(<span class="emphasis">*n*</span>). An
O(<span class="emphasis">*n*</span><sup>2</sup>) algorithm runs in time
`A`<span class="emphasis">*`n`*</span><sup>`2`</sup> plus lower-order
terms (which might be linear, or logarithmic, of any other function
lower than a quadratic). Checking a list for duplicate values (by the
naïve method, not sorting it) is
O(<span class="emphasis">*n*</span><sup>2</sup>). Similarly,
O(<span class="emphasis">*n*</span><sup>3</sup>) algorithms have an
average run time predicted by the cube of problem size; these tend to be
too slow for practical use. O(log <span class="emphasis">*n*</span>) is
typical of tree searches. Intelligent choice of algorithm can often
reduce running time from
O(<span class="emphasis">*n*</span><sup>2</sup>) to O(log
<span class="emphasis">*n*</span>). Sometimes when we are interested in
predicting an algorithm's memory utilization, we may notice that it
varies as O(1) or O(<span class="emphasis">*n*</span>) or
O(<span class="emphasis">*n*</span><sup>2</sup>); in general, algorithms
with O(<span class="emphasis">*n*</span><sup>2</sup>) or higher memory
utilization are not practical either.
