# Режим `ultrathink` в Claude Code: как это работает на самом деле

> Отчёт актуален на апрель 2026. Многое из того, что пишут в гайдах 2024–2025 года, уже устарело: модель эволюционировала от «магических ключевых слов» к адаптивному рассуждению. Ниже — то, как режим работает сегодня, а не в исторических версиях.

---

## 1. Что такое extended thinking (и почему `ultrathink` — это надстройка над ним)

Claude умеет «думать про себя» перед ответом — генерировать внутренние токены рассуждения, которые потом не показываются пользователю как основной ответ, но влияют на его качество. Это называется **extended thinking** (расширенное мышление). Представь это как блокнот для черновиков: модель сначала рассуждает в нём, потом пишет чистовик.

В Claude Code расширенное мышление **включено по умолчанию**. Это важная точка: тебе не нужно ничего активировать, чтобы получить базовое рассуждение. Когда модель «думает», в терминале крутится спиннер с подсказками типа «still thinking», «almost done thinking». Чтобы увидеть сами рассуждения серым курсивом, нажми `Ctrl+O` (verbose mode).

`ultrathink` — это **дополнительная инструкция поверх дефолтного режима**, которая говорит модели: «на этом конкретном ходу подумай ещё глубже». То есть это не отдельный «режим» в смысле включаемого тумблера — это сигнал в промпте, который Claude Code распознаёт.

---

## 2. Как это устроено технически

Тут нужно разделить два мира, потому что они работают по-разному:

### 2.1 Современные модели (Opus 4.7, Opus 4.6, Sonnet 4.6) — адаптивное рассуждение

На моделях, которые поддерживают **effort levels**, бюджет мышления не фиксированный. Модель сама решает, сколько ей думать, опираясь на:

1. Уровень `effort`, который ты выставил (`low`, `medium`, `high`, `max`)
2. Сложность конкретной задачи
3. Прямые подсказки в промпте — в частности, ключевое слово `ultrathink`

Это и есть **adaptive reasoning**. На простой вопрос вроде «переименуй переменную» модель почти не думает и отвечает мгновенно. На сложный архитектурный вопрос — сама поднимает бюджет рассуждений.

`ultrathink` в этом контексте работает как **in-context instruction**. Это инструкция в самом промпте, которая говорит модели «именно на этом ходу не экономь — подумай как следует». Важно: `ultrathink` **не меняет глобальный effort level**. Он действует только на текущий ход.

> **Тонкость, которую путают:** Anthropic в официальной доке прямо пишет, что фразы вроде «think», «think hard», «think more» в новых версиях интерпретируются просто как обычные слова в промпте — они **не выделяют дополнительные токены на рассуждение**. Магические слова из старых гайдов («megathink», «think harder») фактически не работают как триггеры. Из всего набора реально работает только `ultrathink`.

Opus 4.7, кстати, **всегда использует адаптивное рассуждение** — у него вообще нельзя задать фиксированный бюджет.

### 2.2 Старые модели (Opus 4.5, Sonnet 4.5, Haiku 4.5 и ниже) — фиксированный бюджет

В старых моделях extended thinking работал по-другому: ты выделял конкретный бюджет токенов, и модель использовала ровно столько. Исторически это и были те самые легендарные уровни:

| Триггер           | Бюджет (старые модели)        |
|-------------------|-------------------------------|
| (без триггера)    | 0 токенов                     |
| `think`           | ~4 000 токенов                |
| `think hard`      | ~10 000 токенов               |
| `ultrathink`      | ~31 999 токенов (максимум)    |

На современных моделях эта таблица **больше не работает буквально**. Адаптивное рассуждение само распределяет бюджет, а `ultrathink` просто сдвигает его в большую сторону.

Если хочешь принудительно вернуть старое поведение на 4.6 моделях, есть переменная окружения `CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1`. Для Opus 4.7 это уже не сработает.

## 3. Чем `ultrathink` отличается от обычного режима

| Параметр                | Обычный режим                              | `ultrathink`                                       |
|-------------------------|--------------------------------------------|----------------------------------------------------|
| Сколько модель думает   | Минимум, нужный под задачу (адаптивно)     | Сильно больше — по верхней границе                 |
| Скорость ответа         | Быстро (особенно на простых задачах)       | Медленнее — модель буквально дольше «молчит»       |
| Стоимость токенов       | Экономно                                   | Заметно выше (думающие токены оплачиваются)        |
| Глубина анализа         | Достаточная для рутины                     | Глубокая: альтернативы, edge-cases, trade-offs     |
| Когда выбирает модель   | По умолчанию                               | Только когда ты явно попросил                      |

Главная мысль: `ultrathink` — это **не «лучший режим всегда»**. Это инструмент под конкретный класс задач. На простых задачах он не даст прироста качества, зато даст потерю времени и денег. Эмпирическое правило: если человек, столкнувшись с задачей, сделает её хуже, если будет «слишком долго думать» — модель тоже. Для рефакторинга по шаблону или мелкой правки `ultrathink` бесполезен или вреден.

## 4. Какие задачи реально оправдывают `ultrathink`

Список — по убыванию обоснованности:

1. **Архитектурные решения** — выбор между микросервисами и монолитом, проектирование границ модулей, дизайн API под несколько одновременных ограничений.
2. **Сложный дебаг** — баги, которые не воспроизводятся стабильно, гонки, утечки памяти, невидимые dependency-конфликты. Случаи, где «очевидная» гипотеза почти всегда неправильная.
3. **Миграции** — план перехода JS → TS, MongoDB → PostgreSQL, монолит → микросервисы, со всеми вопросами обратной совместимости и стратегии данных.
4. **Code review глубокого уровня** — не «найди опечатки», а «проанализируй модуль с точки зрения безопасности, производительности и поддерживаемости одновременно».
5. **Незнакомая кодовая база** — когда тебе нужно, чтобы Claude провёл связный анализ нового проекта, а не выдал поверхностное описание.
6. **Trade-off анализ** — «выбери БД для real-time чата на 100K пользователей и обоснуй».

И список того, где `ultrathink` **не нужен** и часто вреден:

- Простые правки и переименования
- Шаблонный рефакторинг по уже понятному паттерну
- Креативное написание текстов (тут модель «передумывает» в плохом смысле)
- Любое pattern matching — поиск по регэкспам, форматирование

## 5. Как использовать: пошаговая инструкция

### Базовое применение

Просто включи слово `ultrathink` в промпт — где угодно, в начале, в середине или в конце:

```
ultrathink: проанализируй архитектуру этого сервиса и предложи план миграции на event-driven подход
```

```
у нас деградирует latency в P99 после обновления — ultrathink, найди возможные причины и проранжируй их по вероятности
```

```
проведи глубокий code review модуля auth/, ultrathink, фокус на безопасности и race conditions
```

### Связка с Plan Mode (рекомендуемый паттерн)

Лучшая практика — **сначала исследовать, потом планировать с `ultrathink`, и только потом писать код**. Это методология «explore → plan → code → commit», которую советует сама Anthropic:

```
1. EXPLORE → "прочитай файлы в src/auth/ и опиши, что там происходит. Не пиши код."
2. PLAN    → "ultrathink. На основе того, что ты видел, предложи план рефакторинга. Не пиши код."
3. CODE    → "теперь реализуй план"
4. COMMIT  → "сделай коммит и создай PR"
```

Чтобы войти в Plan Mode: `Shift+Tab` в активной сессии (циклически переключает permission modes), или запуск с флагом:

```bash
claude --permission-mode plan
```

### Управление через effort level

Если хочешь, чтобы Claude всегда думал глубже без необходимости каждый раз писать `ultrathink`:

```bash
# В сессии
/effort high      # глубокое рассуждение по умолчанию
/effort max       # максимум (только Opus 4.6)

# Через переменную окружения
export CLAUDE_CODE_EFFORT_LEVEL=high
```

`ultrathink` тут не противоречит — он работает поверх любого effort level и просит «именно сейчас подумать ещё».

### Полезные команды и шорткаты

| Действие                                           | Как сделать                                  |
|----------------------------------------------------|----------------------------------------------|
| Посмотреть рассуждение модели                      | `Ctrl+O` (toggle verbose mode)               |
| Включить/выключить thinking в сессии               | `Option+T` (macOS) / `Alt+T` (Win/Linux)     |
| Поменять effort level                              | `/effort low|medium|high|max`                |
| Глобально включить thinking всегда                 | `/config` → toggle `alwaysThinkingEnabled`   |
| Полностью отключить thinking                       | `export MAX_THINKING_TOKENS=0`               |
| Вернуть фиксированный бюджет на 4.6                | `export CLAUDE_CODE_DISABLE_ADAPTIVE_THINKING=1` |

### Где `ultrathink` НЕ работает

Это критически важная деталь, которую часто упускают:

- ❌ **claude.ai в браузере** — слово `ultrathink` там просто часть промпта, никаких триггеров
- ❌ **API напрямую** — нужно явно передавать параметр `thinking` в запросе
- ✅ **Только в Claude Code CLI** — там работает препроцессор, который ловит ключевое слово

Если ты пишешь в API напрямую, эквивалентом будет:

```python
import anthropic

client = anthropic.Anthropic()

# Современный способ (Opus 4.6, Sonnet 4.6, Opus 4.7)
response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=16000,
    thinking={"type": "adaptive"},
    output_config={"effort": "max"},
    messages=[
        {"role": "user", "content": "проведи архитектурный анализ ..."}
    ]
)

# Старый способ для моделей 4.5 и ниже
response = client.messages.create(
    model="claude-sonnet-4-5-20241022",
    max_tokens=16000,
    thinking={"type": "enabled", "budget_tokens": 31999},
    messages=[...]
)
```

## 6. Примеры на «человеческом» языке

**Сценарий 1: разобраться, почему сайт тормозит после релиза**

```
ultrathink: у нас в проде после вчерашнего релиза P99 latency вырос с 200мс
до 1.2с. До релиза было нормально. Логи и метрики я приложил.
Не предлагай очевидные гипотезы — сначала структурируй, что мы знаем,
что не знаем, и какие наблюдения противоречат друг другу.
```

Без `ultrathink` Claude скорее всего сразу выдаст список из 5 типичных причин. С `ultrathink` он проведёт настоящее расследование: разделит факты и гипотезы, найдёт нестыковки, попросит конкретных данных, которые отсекут половину вариантов.

**Сценарий 2: спланировать миграцию**

```
Я хочу мигрировать наш Node.js монолит на событийную архитектуру с Kafka.
ultrathink: построй план в три фазы, где каждая фаза самодостаточна
и не ломает прод. Учти: команда из 4 человек, у нас нет опыта с Kafka,
и мы не можем останавливать сервис больше чем на 30 минут.
```

**Сценарий 3: код-ревью без поверхностности**

```
ultrathink: проведи ревью модуля payments/. Не ищи опечатки и стиль.
Меня интересуют только три вопроса: (1) есть ли места, где можно
проиграть деньги клиента из-за race condition, (2) есть ли утечки
секретов в логи, (3) что произойдёт при таймауте внешнего API
в середине транзакции.
```

## 7. Примеры на техническом языке

**Связка с subagent для параллельного анализа:**

```
ultrathink + используй три subagent параллельно:
- security-reviewer на src/auth/
- perf-analyzer на src/api/
- test-coverage на src/billing/
Потом сведи их выводы в единый отчёт с приоритезацией по риску.
```

**Headless-режим в CI/CD пайплайне:**

```bash
# В GitHub Actions можно прогонять глубокий ревью на критичных PR
claude --permission-mode plan -p "ultrathink: проанализируй diff
этого PR на предмет breaking changes в публичном API. Верни JSON
со списком обнаруженных breaking changes." --output-format json
```

**Многораундовая критика для качества:**

```
ultrathink + plan mode: спроектируй deployment-стратегию для blue-green
на нашем k8s. Затем сам скритикуй свой план: где edge-cases, где
single point of failure, где ты сделал неявные допущения.
```

**Через API для автоматизации:**

```python
def deep_architecture_review(codebase_summary: str) -> dict:
    response = client.messages.create(
        model="claude-opus-4-7",
        max_tokens=32000,
        thinking={"type": "adaptive"},
        output_config={"effort": "max"},
        messages=[{
            "role": "user",
            "content": f"""ultrathink: проведи архитектурный анализ.

Контекст: {codebase_summary}

Верни структурированный JSON со схемой:
- bottlenecks: [...]
- coupling_issues: [...]
- migration_risks: [...]
- recommendations: [...]

Только JSON, без preamble."""
        }]
    )
    return json.loads(response.content[0].text)
```

## 8. Главное, что стоит запомнить

1. **Extended thinking включён по умолчанию** — на адаптивных моделях ты уже получаешь умное распределение бюджета без всяких триггеров.
2. **`ultrathink` — это in-context инструкция**, а не глобальный режим. Действует на один ход.
3. **Работает только в Claude Code CLI**, не в веб-чате и не в API напрямую (там нужны явные параметры).
4. **Используй точечно** — на архитектуре, дебаге, миграциях, глубоком ревью. На рутине это вредит.
5. **Лучшая связка**: `ultrathink` + Plan Mode + цикл explore→plan→code. Это даёт максимальный прирост качества.
6. **Стоит токенов** — следи через `/cost`. Отдельно дорогая глубокая сессия — это нормально, но «ultrathink на каждый чих» — антипаттерн.
