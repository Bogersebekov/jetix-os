# Plan Mode в Claude Code: как использовать на полную

> Отчёт актуален на апрель 2026. Plan Mode за последний год превратился из «приятной фичи» в один из главных рабочих режимов Claude Code — особенно после появления Opus Plan Mode и улучшений в 4.6 поколении моделей.

---

## 1. Что такое Plan Mode и зачем он вообще нужен

Главная проблема при работе с агентским кодинг-инструментом — **слишком быстрая имплементация**. Claude видит запрос и сразу начинает писать код. И пишет его быстро, уверенно, и часто — решает не ту задачу, которую ты имел в виду. Ты просишь «добавь soft delete», а через 15 минут у тебя 14 изменённых файлов, сломанные эндпоинты и глобальный фильтр, конфликтующий с миграциями.

Plan Mode решает это структурно: он **физически блокирует Claude от изменения чего-либо**. Никаких записей в файлы, никаких bash-команд, никаких MCP-инструментов с побочными эффектами. Только чтение, поиск, анализ — и построение плана. Ты сначала **видишь, что Claude собирается сделать**, утверждаешь это, и только потом он начинает писать код.

Это полная аналогия с разницей между «джун начал кодить» и «сеньор сначала подумал, потом написал». Изменить план дешевле, чем изменить код.

---

## 2. Что Claude может и не может делать в Plan Mode

В Plan Mode инструменты делятся на две группы:

**✅ Доступные (read-only и research):**

- `Read` — чтение файлов
- `LS` — листинг директорий
- `Glob` — поиск файлов по паттернам
- `Grep` — поиск по содержимому
- `Task` — research-агенты
- `TodoRead`/`TodoWrite` — управление задачами
- `WebFetch` — забор контента с URL
- `WebSearch` — поиск в интернете
- `NotebookRead` — чтение Jupyter-ноутбуков
- `AskUserQuestion` — уточняющие вопросы тебе

**❌ Заблокированные (всё, что меняет состояние):**

- `Edit` / `MultiEdit` — правка файлов
- `Write` — создание файлов
- `Bash` — команды в терминале
- `NotebookEdit` — правка ноутбуков
- Все MCP-инструменты, которые модифицируют что-либо

Если ты в Plan Mode попросишь Claude «сделай быстрый фикс» — он откажется. Это намеренная архитектура: режим даёт **гарантию** того, что код не изменится, пока ты явно не разрешишь.

## 3. Как включить Plan Mode

### Способ 1: На лету в сессии (самый частый)

Нажми **`Shift+Tab`** в активной сессии. Шорткат циклически переключает permission modes:

```
Normal Mode → Auto-Accept Mode → Plan Mode → Normal Mode → ...
```

Из обычного режима это **два нажатия Shift+Tab подряд**. Внизу терминала увидишь индикатор:
- `⏵⏵ accept edits on` — Auto-Accept
- `⏸ plan mode on` — Plan Mode

Чтобы выйти из Plan Mode обратно — снова `Shift+Tab` до Normal.

### Способ 2: Стартовать сессию сразу в Plan Mode

```bash
claude --permission-mode plan
```

Это мой рекомендуемый способ для нетривиальных задач: ты заранее знаешь, что будешь планировать, и сразу попадаешь в правильный режим.

### Способ 3: Headless для CI/CD и скриптов

```bash
claude --permission-mode plan -p "Проанализируй authentication и предложи улучшения"
```

Полезно, когда нужен план как артефакт — например, в pipeline, который автоматически создаёт RFC по PR.

### Способ 4: Сделать Plan Mode дефолтом в проекте

В файле `.claude/settings.json`:

```json
{
  "permissions": {
    "defaultMode": "plan"
  }
}
```

Каждая сессия в этом проекте теперь будет стартовать в Plan Mode. Полезно для production-проектов, где «джамп в код» опасен.

## 4. Главное отличие: 4-фазный workflow Explore → Plan → Code → Commit

Anthropic в официальных best practices выделяет именно эту схему. Она простая, но именно её регулярно нарушают, и именно её соблюдение даёт основной прирост качества.

```
1. EXPLORE  →  Claude читает код, без правок и без планирования
2. PLAN     →  Claude строит план (в Plan Mode), без кода
3. CODE     →  Выходим из Plan Mode, имплементируем по плану
4. COMMIT   →  Git commit + PR
```

**Фаза 1 — Explore.** Ты говоришь Claude изучить нужные части кода. Никаких просьб «сразу предложи решение» — только понять текущее состояние:

```
прочитай /src/auth и разберись, как мы работаем с сессиями и логином.
ещё посмотри, как у нас устроены переменные окружения для секретов.
не предлагай решение — просто опиши, что видишь.
```

**Фаза 2 — Plan.** Включаешь Plan Mode и просишь конкретный план:

```
я хочу добавить Google OAuth. какие файлы нужно изменить? как пойдёт
поток сессий? составь подробный план. без кода.
```

Claude в Plan Mode задаст уточняющие вопросы через `AskUserQuestion` — про неоднозначные требования, архитектурные предпочтения, варианты реализации. Это и есть момент, где ты ловишь все скрытые допущения **до** того, как они превратились в код.

**Фаза 3 — Code.** Выходишь из Plan Mode, говоришь имплементировать утверждённый план. Claude уже не блуждает — у него есть контракт.

**Фаза 4 — Commit.** Коммит и PR.

Ключевое в этой схеме — фазы **разделены**. Если попытаться смешать «исследуй и сразу планируй» в одном запросе, Claude начнёт планировать на половине информации.

## 5. Фишки, которые удваивают полезность

### 5.1 `Ctrl+G` — открыть план в текстовом редакторе

Когда Claude сгенерировал план, нажми **`Ctrl+G`**. План откроется в твоём $EDITOR (vim/nvim/code/whатever) — можешь редактировать его напрямую: уточнить шаг, удалить лишнее, добавить ограничение. Сохранил → Claude видит изменённую версию и идёт по ней.

Это сильнее, чем «обсудить план чатом». Чат тратит токены и контекст; правка плана — точечная и дешёвая.

### 5.2 Планы сохраняются как файлы

Каждый сгенерированный план Claude сохраняет как markdown-файл в `~/.claude/plans/` (со случайным именем типа `dreamy-orbital-7a3f.md`). Это **переживает** `/clear` и компакцию контекста — можно вернуться к плану после перезагрузки сессии.

Если хочешь, чтобы планы лежали внутри проекта (для git-версионирования), задай в `settings.json`:

```json
{
  "plansDirectory": "./plans"
}
```

Тогда планы попадают в репозиторий, и:
- Их можно ревьюить через PR-диффы — план как артефакт
- Команда может править план совместно
- План становится living spec, а не одноразовым черновиком

Для долгих рефакторингов это маст-хэв.

### 5.3 Opus Plan Mode (планируешь Opus-ом, кодишь Sonnet-ом)

С появлением Opus 4.6 (а теперь и Opus 4.7) появилась элегантная конфигурация: **планирование на самой умной модели, имплементация на быстрой**.

Запуск: `/model` → выбери опцию **"Use Opus in plan mode, Sonnet 4.6 otherwise"**.

Что ты получаешь:
- Plan фаза идёт на Opus с его 1M контекстом — он видит весь репозиторий и строит реально продуманный план
- Code фаза автоматически переключается на Sonnet — быстрее, дешевле, и для механической имплементации этого хватает с запасом

По стоимости это часто выгоднее, чем «всё на Opus»: тяжёлая когнитивная работа происходит один раз, остальное — конвейер.

### 5.4 Связка с extended thinking / ultrathink

Plan Mode отлично сочетается с глубоким мышлением. Два варианта:

**Через шорткат:** В Plan Mode нажми `Option+T` (Mac) / `Alt+T` (Win/Linux), чтобы включить extended thinking для текущей сессии.

**Через промпт:** Просто добавь `ultrathink` в запрос на план:

```
ultrathink: построй план миграции с REST на GraphQL для нашего API.
учти: команда из 4 человек, нельзя ронять прод дольше 30 минут,
обратная совместимость нужна минимум на 6 месяцев.
```

Это связка, которую я рекомендовал в предыдущем отчёте: `ultrathink` + Plan Mode даёт принципиально другое качество планирования. Plan Mode гарантирует, что Claude не сорвётся в код; `ultrathink` гарантирует, что он подумает над альтернативами и trade-offs.

### 5.5 `@` для контекста без блуждания

Вместо того чтобы Claude искал нужные файлы поиском, используй `@`:

```
прочитай @src/auth/session.js и @src/middleware/auth.ts.
проанализируй, можем ли мы вынести логику в отдельный модуль.
```

В Plan Mode это особенно ценно: ты экономишь tool calls, оставляешь больше токенов на собственно планирование.

### 5.6 Re-plan при drift

Реальность планирования: **большинство многофайловых рефакторингов проходят 2-3 итерации плана**, а не одну. Это нормально. Если в процессе имплементации обнаруживаешь, что план не учитывал что-то важное:

```
Shift+Tab → возвращаешься в Plan Mode
"перечитай текущее состояние файлов и пересмотри план для оставшихся шагов"
```

Это не аварийная мера — это нормальный итеративный процесс. Изменить план посреди работы намного дешевле, чем продавливать неправильное решение до конца.

### 5.7 Cross-model review плана

Продвинутая практика: **отдай план второй модели на ревью**. Например, в плане сидит Claude Opus, а на ревью ты подтягиваешь стороннюю модель (Codex/GPT/etc.) с промптом «ты staff engineer, найди слабые места в этом плане». Этот приём ловит blind spots, которые одна модель не увидит сама.

## 6. Best practices, которые экономят время

### 6.1 Планируй маленькими кусками — 30 минут максимум

Большие планы, покрывающие фичу на 20 файлов, **теряют когерентность**. Контекст Claude забивается, и к десятому шагу он уже забывает, что было в третьем. Правило: **один план = одна логически замкнутая часть, имплементируемая за 30 минут**.

Большую фичу разбивай на последовательные планы:
- Plan 1: вынести auth в отдельный модуль
- Plan 2: переключить роуты на новый модуль
- Plan 3: написать миграционные тесты

Каждый план в своей сессии. Используй `/rename` для имён сессий: `claude --resume blog-api-auth-extraction`.

### 6.2 Делай план фазированным с гейтами и тестами

Плохой план: «изменить файлы A, B, C; написать тест X».
Хороший план:
```
Phase 1: Extract auth logic
  - Steps: ...
  - Gate: unit tests for new module pass
Phase 2: Migrate routes
  - Steps: ...
  - Gate: integration tests for /api/auth/* pass
Phase 3: Cleanup old code
  - Steps: ...
  - Gate: full e2e suite passes
```

Это ловит проблемы рано, и каждая фаза — естественная точка коммита.

### 6.3 Заполняй CLAUDE.md ограничениями, которые Claude не выведет из кода

В Plan Mode Claude задаст уточняющие вопросы. Но он **не спросит про то, о чём не догадался спросить**. Чтобы он не догадался — у него должен быть контекст. Положи в `CLAUDE.md`:

- Архитектурные принципы команды («у нас clean architecture, business logic не зависит от фреймворка»)
- Жёсткие нефункциональные требования («latency не должен расти больше 10% в P99»)
- Запреты («не используй любые библиотеки с GPL-лицензией»)
- Соглашения, которые не написаны в коде явно

Эти вещи невидимы из кода — но именно они роняют план.

### 6.4 Используй `/clear` между разными задачами

Контекст накапливается и начинает мешать. Для новой задачи — **новая сессия**. Те 5 секунд на `/clear` спасают от 30 минут затухающего качества.

### 6.5 Не используй Plan Mode там, где он не нужен

Если задачу можно описать одним предложением — **просто делай её**. Plan Mode даёт overhead в несколько минут на планирование. Для:
- Переименования переменной
- Простой починки опечатки
- Уже понятного шаблонного рефакторинга

— это лишнее. Правило простое: **если diff можно описать одним предложением, plan mode избыточен**.

### 6.6 Используй для незнакомого кода и многофайловых изменений

А вот когда Plan Mode оправдан почти всегда:
- Многофайловые изменения (3+ файла)
- Незнакомая часть кодовой базы
- Архитектурные решения
- Миграции
- Любая задача, где «потерять» что-то критичное больно (платежи, auth, данные)

## 7. Примеры на «человеческом» языке

**Сценарий 1: добавить новую фичу в незнакомый модуль**

```
[Shift+Tab дважды → Plan Mode]

прочитай @src/notifications/ — мне нужно понять, как у нас сейчас
устроена доставка уведомлений. не предлагай решение, просто опиши.
```

После того как Claude отчитался:

```
теперь план: я хочу добавить поддержку push-уведомлений через FCM.
существующая логика email/SMS должна работать без изменений.
ultrathink, составь поэтапный план с тестами на каждом этапе.
```

`Ctrl+G` → правишь план в редакторе → `Shift+Tab` обратно → имплементация.

**Сценарий 2: рефакторинг на много файлов**

```
[claude --permission-mode plan]

наша auth-логика размазана по 12 файлам. цель — вынести её в один
модуль /src/auth/ с чистым API. ограничения: zero downtime,
все существующие тесты должны проходить, обратная совместимость
публичного API.

разбей на фазы. в каждой фазе: атомарный шаг + тесты + коммит-поинт.
```

**Сценарий 3: расследование бага через план, не код**

```
у нас ломается оплата у ~2% пользователей в Европе. логи я приложил.

в Plan Mode: проанализируй код /src/payments/ и /src/billing/,
составь список всех мест, где это могло бы сломаться,
и план диагностики — что нужно проверить и в каком порядке.
без кода — мне нужен следственный план.
```

## 8. Примеры на техническом языке

**Сделать Plan Mode дефолтом для production-проекта:**

```jsonc
// .claude/settings.json
{
  "permissions": {
    "defaultMode": "plan"
  },
  "plansDirectory": "./plans",
  "alwaysThinkingEnabled": true
}
```

**Headless с автоматическим сохранением плана:**

```bash
claude --permission-mode plan \
  -p "ultrathink: проанализируй diff PR #1234 и составь план тестов,
      покрывающих новые риски" \
  --output-format json > pr-test-plan.json
```

**Pipeline с разделением ролей по моделям:**

```
/model
> Use Opus in plan mode, Sonnet 4.6 otherwise

[Shift+Tab → Plan Mode]
ultrathink: спланируй миграцию ORM с TypeORM на Prisma.
```

Opus с 1M контекстом строит план через всю кодовую базу. Затем `Shift+Tab` обратно — автоматически переключаемся на Sonnet для имплементации.

**Версионирование плана как living spec в репо:**

```bash
# планы лежат в ./plans/
git add plans/2026-04-29-oauth-migration.md
git commit -m "RFC: OAuth migration plan"
git push origin rfc/oauth-migration

# открываем PR на сам план — команда ревьюит план, а не код
gh pr create --title "RFC: OAuth migration plan" --body "Review the plan before implementation"

# после approve — стартуем имплементацию по плану
claude --resume oauth-migration
```

**Переплан при drift посреди имплементации:**

```
[В процессе имплементации обнаружилась неучтённая зависимость]

[Shift+Tab → Plan Mode]
обновить план: я только что узнал, что /src/legacy/auth-shim.js
тоже использует старую сессию. перечитай этот файл и пересмотри
оставшиеся шаги.

[Claude обновляет план с учётом нового контекста]
[Ctrl+G → ревью → Shift+Tab → продолжение]
```

## 9. Полная связка: ultrathink + Plan Mode + Opus Plan Mode + правильный workflow

Это конфигурация максимального качества для нетривиальной задачи. Используй её для всего, что страшно сломать:

```
1. Запуск:
   claude --permission-mode plan
   /model → Use Opus in plan mode, Sonnet 4.6 otherwise

2. EXPLORE (Plan Mode уже включён):
   "прочитай @src/feature/, опиши текущее состояние. без решений."

3. PLAN с глубоким мышлением:
   "ultrathink: на основе того, что ты увидел, составь поэтапный план
    [задача]. учти ограничения: [...]. разбей на фазы с тестами."

4. Ревью плана:
   - Ctrl+G → правка в редакторе
   - Если нужно — попроси Claude скритиковать собственный план:
     "найди слабые места в этом плане как staff engineer"

5. CODE:
   Shift+Tab → Normal Mode (модель автоматически переключится на Sonnet)
   "имплементируй план пошагово, после каждой фазы запускай тесты"

6. При drift → обратно в Plan Mode → re-plan → продолжение

7. COMMIT:
   "сделай коммит с осмысленным сообщением и открой PR"
```

Эта схема покрывает 80% всех сложных задач. Остальные 20% — либо BMAD/спеки для совсем больших проектов, либо наоборот — мелочёвка, где Plan Mode избыточен.

## 10. Главное

1. **Plan Mode — это read-only гарантия.** Claude физически не может изменить файлы, пока ты не одобрил план.
2. **`Shift+Tab` дважды** — самый частый способ войти. `claude --permission-mode plan` — для старта сессии в этом режиме.
3. **`Ctrl+G` — главный шорткат.** Открывает план в редакторе для прямой правки. Используй его.
4. **Workflow: Explore → Plan → Code → Commit.** Не смешивай фазы.
5. **Маленькие планы лучше больших.** 30 минут максимум на план. Большие фичи — это последовательность планов, а не один.
6. **Связка с `ultrathink` и Opus Plan Mode** — конфигурация максимального качества для серьёзных задач.
7. **Re-plan при drift — это норма, а не провал.** 2-3 итерации плана для многофайлового рефакторинга — стандарт.
8. **Не используй для тривиальщины.** Если diff описывается одним предложением — просто сделай его в Normal Mode.
9. **Версионируй планы через `plansDirectory` в проекте** для долгих рефакторингов — план становится living spec.
10. **Plan Mode + ясный CLAUDE.md** — Claude задаст уточняющие вопросы, но не догадается спросить о том, чего не знает. Заранее положи туда архитектурные ограничения.
