## Decision 2 — Tension #1 (Solo vs Team) — LOCKED

### Formula

**Solo-now-with-team-ready-scaffolding.**

НЕ phased "сначала solo потом команда". А: **действуем соло прямо сейчас**, НО **архитектура Jetix с Day 1 готова к тому что появится 2-й или 3-й пилот**, партнёр или team member.

### Operating mechanism

- По мере появления новых людей — **право влияния на стратегические документы** (в целом или на конкретное направление) распределяется
- Это структурная готовность, не временной переход: system design must accommodate multi-pilot с самого начала
- Ruslan один сейчас — но код / документы / процессы build'ятся так, что onboarding 2-го пилота не требует refactor

### Ruslan approval quote

> «значит, изначально соло, но ещё раз, чтобы система, да, и я в целом уже готовы на командную работу. Соответственно, чтобы система была готова, если там какой-то второй или третий пилот появится, адекватно под него подстроиться.
>
> Но по сути это будет так, что влияние на стратегический документ просто получит ещё там какой-то партнёр или кто-то из команды. Вот, или на стратегический документ какого-то определённого направления. В итоге вот сейчас соло, дальше по мере развития команда.»

### Implications

- **D1 (Vision)** — "operating model" section: solo-with-team-ready, не "solo-founder forever"
- **D2 (Philosophy)** — принцип: "system should absorb 2nd+ pilot без refactor"
- **D3 (Plan)** — Phase 1 operates solo; Phase 2+ onboarding partners on strategic docs. Но Phase 1 deliverables build'ятся так что Phase 2 onboarding = quick paste, не rewrite
- **D4 (Architecture-brief)** — критично для архитекторов: document ownership / governance structure должна support multi-pilot inheritance

## Decision 3 — Tension #2 (Open-source vs Patents) — LOCKED

### Formula

**Closed outside / Open inside.**

Для внешнего мира — **closed**. Для команды / community (inside members) — **open**, как open source, но **только для своих**.

### Clarification

Ruslan уточнил: **не open-source в общепринятом смысле** (публично в GitHub). А скорее "open для прошедших membrane": закрытая система с privileged access для partners.

### Red rhymes (coherence check)

- Syncс с Decision 1 (Gentleman inside / Predator outside): inside = open + civil = team/community; outside = closed + defensive = external world
- Syncс с Secure Network concept (addendum §3.3): hidden layer / open layer / quality-gate

### Ruslan approval quote

> «потом дальше третий вопрос: это мы значит делаем так, что это закрытая для внешнего мира система, но для команды, для вот комьюнити, она будет открытая. И как, ну вот, открыто, как open source, но только для команды. То есть вот так вот это будет в итоге. Но если только это для команды, то тогда получается это closed. Да, тогда мы говорим, что это closed, но для команды open.»

### Implications

- **D1 (Vision)** — "openness policy" section
- **D2 (Philosophy)** — принцип: "radical transparency inside the gate, opacity outside"
- **D3 (Plan)** — repository structure: private repos + selective access patterns; patents можно (не конфликт)
- **D4 (Architecture-brief)** — membership-gated access model должна быть dimension D (architecture)

### Post-€200K considerations

Ранее locked: "Post-€200K open-source timing". Это decision **уточняет**: post-€200K **можно** рассмотреть selective open-sourcing для дополнительных слоёв, но core всегда остаётся closed. Public open-source full platform — **НЕ на столе** (Ruslan clarified "closed outside" как постоянный режим, не temporary).

## Decision 5 — Phase priority (Consulting-first, Secure Network later) — LOCKED

### Formula

**Phase 1 = Consulting-first. Community = попутно в чате. Secure Network = natural evolution later.**

### Path

- **Phase 1 (сейчас → €50K):** Core revenue = consulting (€150/час, сессии, 10 шаблонов, услуги). Community = "собираю попутно" — сначала просто чат или basic группа, без platform.

- **Phase 2+ (post €200K validation):** Эта разросшаяся community эволюционирует в Secure Network — платформа как primary product.

### Нет противоречия между Note 1 и Note 2

Оба видения Ruslan'а — валидны, но **в разное время**:
- Note 1 ("сообщество попутно") = Phase 1 mode
- Note 2 ("Jetix = Secure Network") = Phase 2+ evolved state

### Ruslan approval quote

> «по пятому вопросу: да, у нас консалтинг first, и потом у нас будет просто комьюнити, да, там в каком-то чате или ещё что-то. Потом оно будет разрастаться на secure комьюнити. Вот так вот, никаких противоречий нет, отлично.»

### Implications

- **D1 (Vision)** — "core offer" section: Phase 1 описывается консультингом, Secure Network упоминается как Phase 2+ natural evolution
- **D3 (Plan)** — Phase 1 actions focus на consulting revenue; community-infrastructure work = low priority Phase 1, повышается Phase 2
- **D4 (Architecture-brief)** — архитекторы должны build для consulting-first, но **extensibility** в community/platform без rewrite
